# Project Goal: Rust-Based Trino Worker (Project Crucible/Forge)

## Executive Summary
The objective of this project is to architect and implement a drop-in replacement for the Trino Worker node, written entirely in Rust. This new worker must seamlessly integrate with an existing, unmodified Java-based Trino Coordinator. The overarching goal is to combine the distributed scheduling brilliance of Trino, the native asynchronous execution model of Apache DataFusion, and the highly optimized memory and vectorized compute concepts of Meta Velox.

## 1. Project Scope
* **In-Scope:** * Full implementation of a Trino-compatible Worker node in Rust.
  * Implementation of Trino's Control Plane (REST/JSON task management, status polling).
  * Implementation of Trino's Data Plane (token-based HTTP pull model for intra-node shuffle).
  * Implementation of Trino's Storage Plane (parsing split definitions and reading from object storage).
  * Execution of physical execution plans generated by the Trino Coordinator.
* **Out-of-Scope:** * Any modifications to the Trino Coordinator.
  * Logical SQL parsing, semantic analysis, or logical query optimization (handled by the Coordinator).

## 2. Core Compatibility Requirements
To function as a true drop-in replacement, the Rust worker must strictly adhere to Trino's existing wire protocols:
* **Task Lifecycle:** It must implement the `TaskResource` REST endpoints to accept `TaskUpdateRequest` JSON payloads, parse the physical plan fragments, and manage the `PLANNED -> RUNNING -> FINISHED` state machine.
* **Heartbeats & Status:** It must support Trino's version-based long-polling mechanism for task status updates and propagate dynamic filters back to the coordinator.
* **Data Exchange:** It must implement Trino's 12-byte header binary wire format for page serialization and support the idempotent, token-based `/v1/task/{taskId}/results/` HTTP pull protocol for shuffling data between workers.

## 3. Architectural Philosophy: The "Best of Breed" Approach
The worker will not be a 1:1 line-by-line port of the Java worker. Instead, it will synthesize the best architectural patterns from the industry's leading query engines:

* **From Trino (The Protocol & Pipeline Semantics):**
  * Strict adherence to Trino's non-blocking, push-pull semantics at the network boundary.
  * Trino's robust approach to memory revoking/spilling interfaces and OOM crash prevention via deterministic manual memory accounting.
* **From Apache DataFusion (The Engine & Concurrency):**
  * **Async Volcano Model:** Replacing Trino's custom `Driver` yield loop with Rust's native `tokio` asynchronous runtime and the `Stream` trait (`poll_next()`).
  * **Memory Safety:** Leveraging Rust's RAII (Resource Acquisition Is Initialization) pattern for memory reservations to eliminate the possibility of memory accounting leaks.
  * **Compute:** Delegating vectorized expression evaluation to Apache Arrow compute kernels.
* **From Meta Velox (The Memory & Optimization Patterns):**
  * **Cross-Language/Protocol Memory Bridges:** Adopting Velox's strict `MemoryPool` and `MemoryArbitrator` designs to safely manage memory when interacting with external constraints (like the Java Coordinator's expectations).
  * **Data Structures:** Adopting customized, highly optimized physical data layouts where standard Arrow buffers fall short (e.g., utilizing Velox-style `StringView` prefixes for faster sorting and hashing).

## 4. Technical Decision-Making Framework
Architecture is ultimately about trade-offs rather than finding a single "perfect" solution. Therefore, **every technical, architectural, and design decision made during this project MUST be accompanied by a thorough explanation, alternative approaches, and their respective trade-offs.** When deciding how to implement a specific component, the decision must explicitly reference prior research and provide the following:
1.  **How does Trino do it?** (Identify the baseline Java implementation).
2.  **How does DataFusion do it?** (Identify the Rust/Arrow implementation).
3.  **How does Velox do it?** (Identify the C++ implementation).
4.  **Alternative Approaches:** Outline the viable implementation paths for our Rust worker based on the engines above (or completely novel approaches).
5.  **Trade-off Analysis:** Detail the pros and cons of each alternative (e.g., memory overhead vs. CPU cache efficiency, implementation complexity vs. runtime performance, strict Trino compatibility vs. native Rust idioms).
6.  **The Verdict:** Provide a well-reasoned argument for the chosen path, proving why its specific trade-offs are the most acceptable for our goal of building a compatible, safe, and highly performant Rust worker.

### 4.1 Iterative Discovery & Knowledge Gaps (The "No Guessing" Rule)
If, during the planning or decision-making process, the internal behavior of Trino, DataFusion, or Velox is found to be uncertain, ambiguous, or insufficiently covered by the existing tracing manifests, **you must not guess or hallucinate an assumption.** Instead, you must immediately:
1. Formulate a new, highly specific, atomic tracing task.
2. Append this new task to the appropriate manifest (`TRINO_TRACING_GUIDE`, `DATAFUSION_TRACING_GUIDE`, or `VELOX_TRACING_GUIDE`).
3. Pause the design process and kick off the execution of this new research task to uncover the ground truth from the source code.
4. Resume the architectural decision-making only after the factual evidence has been established.

## 5. Next Steps
Using the comprehensive tracing guides, we will begin mapping the Trino Physical Plan JSON definitions to equivalent DataFusion-style asynchronous execution streams, invoking the Iterative Discovery rule whenever we hit an architectural blind spot.
