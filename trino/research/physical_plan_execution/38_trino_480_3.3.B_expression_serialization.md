# Module Teardown: Expression Serialization Format (Task 3.3.B) [KG-7]

## 0. Research Focus
* **Task ID:** 3.3.B
* **Focus:** Determine the exact wire format used when the coordinator sends expressions (filter predicates, projections, function calls) to workers as part of the plan fragment JSON. Understand whether expressions are transmitted as AST nodes, pre-compiled bytecode, or something else. Document the complete expression class hierarchy, function reference format, constant serialization, and the worker-side compilation path from deserialized AST to executable code.

## 1. High-Level Overview

**The wire format is a JSON-serialized expression AST** -- a tree of typed `Expression` nodes under `io.trino.sql.ir`. No bytecode or compiled code is ever sent over the wire. The coordinator performs all query analysis, type resolution, and function resolution, then serializes the resulting fully-typed IR tree as JSON within the plan fragment. Workers deserialize this AST and compile it locally into JVM bytecode using the `ExpressionCompiler` / `PageFunctionCompiler` pipeline.

The expression IR is a sealed algebraic data type with exactly 18 node types, using Jackson polymorphic serialization (`@JsonTypeInfo` / `@JsonSubTypes`) for wire encoding. Every expression carries full type information, and every function call carries a fully resolved `ResolvedFunction` that includes the function ID, bound signature, catalog handle, nullability, and determinism metadata. This means workers never need to perform function resolution -- the coordinator has already done it.

**Key architectural insight:** The coordinator sends a "ready to compile" AST, not raw SQL text or unresolved syntax trees. The `io.trino.sql.ir.Expression` tree is a post-analysis, post-optimization intermediate representation where all names are resolved, all types are concrete, and all function references point to specific implementations.

## 2. Detailed Analysis

### 2.1 Expression Class Hierarchy

The `Expression` interface is a sealed interface in `io.trino.sql.ir` with 18 permitted subtypes. Jackson uses `@JsonTypeInfo(use = JsonTypeInfo.Id.NAME)` with `@JsonSubTypes` to serialize each node with a `@type` discriminator field.

**Source:** `core/trino-main/src/main/java/io/trino/sql/ir/Expression.java`
```java
@Immutable
@JsonTypeInfo(use = JsonTypeInfo.Id.NAME)
@JsonSubTypes({
        @JsonSubTypes.Type(value = Array.class, name = "array"),
        @JsonSubTypes.Type(value = Between.class, name = "between"),
        @JsonSubTypes.Type(value = Bind.class, name = "bind"),
        @JsonSubTypes.Type(value = Call.class, name = "call"),
        @JsonSubTypes.Type(value = Case.class, name = "case"),
        @JsonSubTypes.Type(value = Cast.class, name = "cast"),
        @JsonSubTypes.Type(value = Coalesce.class, name = "coalesce"),
        @JsonSubTypes.Type(value = Comparison.class, name = "comparison"),
        @JsonSubTypes.Type(value = Constant.class, name = "constant"),
        @JsonSubTypes.Type(value = FieldReference.class, name = "field"),
        @JsonSubTypes.Type(value = In.class, name = "in"),
        @JsonSubTypes.Type(value = IsNull.class, name = "isnull"),
        @JsonSubTypes.Type(value = Lambda.class, name = "lambda"),
        @JsonSubTypes.Type(value = Logical.class, name = "logical"),
        @JsonSubTypes.Type(value = NullIf.class, name = "nullif"),
        @JsonSubTypes.Type(value = Reference.class, name = "reference"),
        @JsonSubTypes.Type(value = Row.class, name = "row"),
        @JsonSubTypes.Type(value = Switch.class, name = "switch"),
})
public sealed interface Expression
        permits Array, Between, Bind, Call, Case, Cast, Coalesce,
        Comparison, Constant, FieldReference, In, IsNull, Lambda, Logical,
        NullIf, Reference, Row, Switch
{
    Type type();

    @JsonIgnore
    List<? extends Expression> children();
}
```

#### 2.2.2 `Constant` (type name: `"constant"`)
A literal constant value. The value is **not** serialized as a native JSON value. Instead, the `value` field is `@JsonIgnore`'d and a `valueAsBlock` field is used, which serializes the value as a Trino `Block` via base64-encoded binary.

**Source:** `core/trino-main/src/main/java/io/trino/sql/ir/Constant.java`
```java
public record Constant(Type type, @JsonIgnore Object value) implements Expression
{
    @JsonCreator
    public static Constant fromJson(
            @JsonProperty Type type,
            @JsonProperty Block valueAsBlock)
    {
        return new Constant(type, readNativeValue(type, valueAsBlock, 0));
    }

    @JsonProperty
    public Block getValueAsBlock()
    {
        BlockBuilder blockBuilder = type.createBlockBuilder(null, 1);
        writeNativeValue(type, blockBuilder, value);
        return blockBuilder.build();
    }
}
```

**JSON fields:**
| Field | Type | Description |
|-------|------|-------------|
| `@type` | `"constant"` | Discriminator |
| `type` | string | Type ID (e.g., `"bigint"`) |
| `valueAsBlock` | string | Base64-encoded binary block (via `BlockJsonSerde`) |

**Block serialization detail:** `BlockJsonSerde.Serializer` uses `BlockEncodingSerde` to write the block to a `SliceOutput`, then base64-encodes the bytes (`Base64Variants.MIME_NO_LINEFEEDS`). This means constants are opaque binary blobs in the JSON -- not human-readable JSON values.

### 2.3 Function References: `ResolvedFunction`

Function calls carry a `ResolvedFunction` record with complete resolution metadata. This is the most important structure for a Rust port since it tells the worker exactly which function implementation to invoke.

**Source:** `core/trino-main/src/main/java/io/trino/metadata/ResolvedFunction.java`
```java

        BoundSignature signature,
        CatalogHandle catalogHandle,
        FunctionId functionId,
```

**JSON fields:**
| Field | Type | Description |
|-------|------|-------------|
| `signature` | BoundSignature | Fully bound function signature |
| `catalogHandle` | string | Catalog identifier (e.g., `"system:normal:1"`) |

| `functionKind` | string | `"SCALAR"`, `"AGGREGATE"`, `"WINDOW"`, or `"TABLE"` |
| `deterministic` | boolean | Whether the function is deterministic |

| `functionNullability` | object | `{returnNullable: bool, argumentNullable: [bool...]}` |

```java
    "schemaName": "builtin",       // always "builtin" for builtins
```

**CatalogHandle** serializes as a single string via `@JsonValue`: `"<catalogName>:<type>:<version>"` (e.g., `"system:normal:1"`). Deserialization uses `@JsonCreator fromId(String)`.

**FunctionId** serializes as a single string via `@JsonValue`: a lowercase identifier like `"$operator$add(bigint,bigint):bigint"` or `"system.builtin.upper(varchar):varchar"`.

### 2.4 Type Serialization

Types are serialized as simple strings via `@JsonValue` on `Type.getTypeId()`, which returns the type signature string. Examples:
- `"bigint"` -- BigintType
- `"integer"` -- IntegerType
- `"boolean"` -- BooleanType
- `"varchar"` -- VarcharType (unbounded)
- `"varchar(100)"` -- VarcharType(100)
- `"array(bigint)"` -- ArrayType(BigintType)
- `"row(x bigint, y varchar)"` -- named RowType

Deserialization uses `TypeDeserializer` which extends `FromStringDeserializer<Type>` and calls `typeManager.getType(TypeId.of(value))`.

### 2.5 Constant Serialization Detail

Constant values are **not** JSON primitives. They are serialized as base64-encoded Trino `Block` binary data. This is a critical detail for porting.

The path:
1. `Constant.getValueAsBlock()` writes the native Java value into a single-element `Block` via `writeNativeValue(type, blockBuilder, value)`
2. `BlockJsonSerde.Serializer` writes the block using `BlockEncodingSerde` to a binary `SliceOutput`, then base64-encodes it

This means:
- A BIGINT `10` is serialized as a single-element `LongArrayBlock` -> binary encoded -> base64 string
- A VARCHAR `'hello'` is serialized as a single-element `VariableWidthBlock` -> binary encoded -> base64 string
- A NULL value produces a block with a null flag set
- An ARRAY literal produces an `ArrayBlock` containing the element values

The block encoding format is Trino's internal binary format (`BlockEncoding` implementations like `LongArrayBlockEncoding`, `VariableWidthBlockEncoding`, etc.), each writing a name prefix string followed by binary data specific to the block type.

### 2.6 Expression Compilation Path (Worker Side)

When a worker deserializes the plan fragment, expressions exist as `io.trino.sql.ir.Expression` AST trees. The path to executable code follows three stages:

**Stage 1: IR Expression -> RowExpression** (`SqlToRowExpressionTranslator`)

The `LocalExecutionPlanner` calls `toRowExpression()` which invokes `SqlToRowExpressionTranslator.translate()`:

```java
// LocalExecutionPlanner.java
private RowExpression toRowExpression(Expression expression, Map<Symbol, Integer> layout) {
    return SqlToRowExpressionTranslator.translate(expression, layout, metadata, plannerContext.getTypeManager());
}
```

This translation:
- Converts `Reference` nodes into `InputReferenceExpression` (positional channel references) using the `layout` map
- Converts `Call` nodes into `CallExpression` (preserving the `ResolvedFunction`)
- Converts structural nodes (`Comparison`, `Logical`, `IsNull`, `Between`, `In`, `Case`, `Switch`, `Coalesce`, `Cast`, `NullIf`, `Bind`, `Array`, `Row`, `FieldReference`) into `SpecialForm` or `CallExpression` nodes
- Importantly, `Comparison` is lowered to function calls: `GREATER_THAN(a, b)` becomes `LESS_THAN(b, a)`, and `NOT_EQUAL(a, b)` becomes `$not(EQUAL(a, b))`
- `Case` (searched CASE) is lowered to nested `IF` special forms
- `Cast` is resolved to a coercion function call via `metadata.getCoercion()`

**Stage 2: RowExpression -> JVM Bytecode** (`PageFunctionCompiler` + `RowExpressionCompiler`)

`ExpressionCompiler.compilePageProcessor()` invokes `PageFunctionCompiler` which:
1. Short-circuits: `InputReferenceExpression` -> `InputPageProjection` (no codegen)
2. Short-circuits: `ConstantExpression` -> `ConstantPageProjection` (no codegen)
3. For complex expressions: generates a `ClassDefinition` using airlift bytecode library, creates JVM bytecode dynamically via `RowExpressionCompiler`

`RowExpressionCompiler` visits the `RowExpression` tree:
- `CallExpression` -> generates bytecode to invoke the function's `MethodHandle` via `BytecodeGeneratorContext.generateFullCall()`
- `SpecialForm` -> dispatches to specialized code generators: `IfCodeGenerator`, `SwitchCodeGenerator`, `BetweenCodeGenerator`, `IsNullCodeGenerator`, `CoalesceCodeGenerator`, `InCodeGenerator`, `AndCodeGenerator`, `OrCodeGenerator`, `DereferenceCodeGenerator`, `RowConstructorCodeGenerator`, `ArrayConstructorCodeGenerator`, `BindCodeGenerator`
- `ConstantExpression` -> emits LDC instructions for primitives (boolean/long/double/String) or binds objects via `CallSiteBinder` for complex types
- `InputReferenceExpression` -> emits code to read from the page's channel array

**Stage 3: Bytecode Class -> PageProcessor**

The generated class implements `PageProjectionWork` or `PageFilter`. `PageFunctionCompiler` uses `defineClass()` to load the generated bytecode into a new JVM class, then wraps it in a `GeneratedPageProjection`. The `PageProcessor` combines a `FilterEvaluator` (filter) with a list of `PageProjection`s (projections) for the final execution pipeline.

**ProjectNode** uses `Assignments` which is a `Map<Symbol, Expression>`:
```java
public record Assignments(@JsonProperty("assignments") Map<Symbol, Expression> assignments)
```

The map keys use a custom serialization format from `SymbolKeySerializer`: `"<type_id_length>|<type_id>|<symbol_name>"`. For example, a symbol `col1` of type `bigint` serializes as key `"6|bigint|col1"`.

## 3. Key Design Insights

1. **Fully resolved at the coordinator**: The coordinator performs all name resolution, type inference, function lookup, and operator resolution before serialization. The `ResolvedFunction` in each `Call` node contains everything the worker needs -- the exact function ID, its catalog, its signature, and its nullability contract. Workers never need access to a function catalog for resolution.

2. **AST over the wire, bytecode compiled locally**: The design choice to send AST (not bytecode) means workers can apply JVM-specific optimizations during compilation. The airlift bytecode library generates JVM bytecode tailored to the worker's JVM. This also avoids the complexity of serializing JVM bytecode (class files with linked dependencies) across the wire.

3. **Constants are binary, not JSON-native**: The choice to serialize constants as base64-encoded `Block` data rather than JSON primitives is unusual but consistent -- it reuses the same `Block` serialization infrastructure used for data exchange between operators and for page serialization in shuffle. This means the constant encoding for a BIGINT is the same binary format whether it appears in an expression or in a data page.

4. **Two-level expression IR**: There are two expression representations: `io.trino.sql.ir.Expression` (the wire/plan format) and `io.trino.sql.relational.RowExpression` (the compilation-ready format). The translation between them (`SqlToRowExpressionTranslator`) performs important lowerings: comparisons become function calls, searched CASE becomes nested IF, Cast resolves to coercion functions, column references become positional channel references.

5. **Sealed type hierarchy with 18 variants**: The expression IR is a closed set. The `sealed interface` + `permits` clause in Java guarantees exhaustive matching. For a Rust port, this maps directly to a Rust `enum` with 18 variants.

6. **Comparison lowering is lossy**: `GREATER_THAN(a, b)` is lowered to `LESS_THAN(b, a)` and `NOT_EQUAL(a, b)` to `$not(EQUAL(a, b))` during RowExpression translation. The IR preserves all 7 comparison operators, but the compilation layer only needs `EQUAL`, `LESS_THAN`, `LESS_THAN_OR_EQUAL`, and `IDENTICAL`.

7. **FunctionId as canonical key**: The `FunctionId` is derived from `(canonicalName + signature).toLowerCase()`. This means function implementations are addressed by their fully-qualified name plus concrete type signature, making them content-addressable. On the worker, `FunctionManager` uses this ID to look up the function's `MethodHandle`.

## 4. Porting Considerations (Java -> Rust)

### 4.1 Expression AST as Rust Enum

The 18-variant sealed Expression hierarchy maps directly to a Rust enum:

```rust
enum Expression {
    Array { element_type: TypeId, elements: Vec<Expression> },
    Between { value: Box<Expression>, min: Box<Expression>, max: Box<Expression> },
    Bind { values: Vec<Expression>, function: Box<Lambda> },
    Call { function: ResolvedFunction, arguments: Vec<Expression> },
    Case { when_clauses: Vec<WhenClause>, default_value: Box<Expression> },
    Cast { expression: Box<Expression>, target_type: TypeId },
    Coalesce { operands: Vec<Expression> },
    Comparison { operator: ComparisonOperator, left: Box<Expression>, right: Box<Expression> },
    Constant { type_id: TypeId, value: ConstantValue },  // NOT a Block
    FieldReference { base: Box<Expression>, field: u32 },
    In { value: Box<Expression>, value_list: Vec<Expression> },
    IsNull { value: Box<Expression> },
    Lambda { arguments: Vec<Symbol>, body: Box<Expression> },
    Logical { operator: LogicalOperator, terms: Vec<Expression> },
    NullIf { first: Box<Expression>, second: Box<Expression> },
    Reference { type_id: TypeId, name: String },
    Row { items: Vec<Expression>, row_type: TypeId },
    Switch { operand: Box<Expression>, when_clauses: Vec<WhenClause>, default_value: Box<Expression> },
}
```

### 4.2 Constant Deserialization

The most complex porting task is `Constant.valueAsBlock`. The Rust worker must:
1. Base64-decode the string
2. Parse the Trino `Block` binary encoding (block encoding name + type-specific binary layout)
3. Extract the single value from position 0

Alternatively, the Rust worker could implement a simpler constant representation and only parse the Block binary format during deserialization, extracting the native value immediately. The binary encoding for a single-element `LongArrayBlock` is: `"LONG_ARRAY"` (encoding name), `int positionCount=1`, `boolean hasNull=false`, `long[1] values`.

### 4.3 Function Resolution Strategy

The `ResolvedFunction` on the wire tells the Rust worker:
- `functionId`: The unique function key (e.g., `"$operator$add(bigint,bigint):bigint"`)
- `functionKind`: Whether it is `SCALAR`, `AGGREGATE`, `WINDOW`, or `TABLE`
- `signature.name`: The three-part catalog.schema.function name
- `deterministic` / `neverFails`: Optimization hints

For a Rust worker, the approach should be:
1. Build a static registry of builtin function implementations keyed by `FunctionId`
2. During expression evaluation, look up the function by its `functionId` string
3. For connector-provided functions, the `catalogHandle` identifies which catalog plugin to call

Since the Rust worker will not use JVM bytecode generation, expression evaluation can be:
- **Interpreted**: Walk the AST tree directly (simplest, slowest)
- **Vectorized**: Process expressions column-at-a-time over Arrow arrays (recommended for Rust)
- **JIT compiled**: Use Cranelift or LLVM to generate native code (highest performance, highest complexity)

### 4.4 The Two-Level Expression Problem

The Java worker has two expression IRs (`ir.Expression` and `relational.RowExpression`) because the compilation layer needs positional channel references rather than named symbol references. In Rust, this two-level approach may not be necessary if the expression evaluator can resolve symbol names to column positions during a single planning pass. Consider merging the lowerings from `SqlToRowExpressionTranslator` into the deserialization step.

### 4.5 Type Serialization is Simple

Types serialize as strings (their `TypeId`). The Rust worker needs a type registry that maps strings like `"bigint"`, `"varchar(100)"`, `"array(integer)"` to Rust type representations. This is straightforward string parsing of Trino's type signature grammar.

### 4.6 Key JSON Parsing Requirements

To deserialize the plan fragment, the Rust worker needs:
- Polymorphic JSON deserialization for `Expression` (discriminated by `@type` field)
- Custom deserialization for `Symbol` map keys (the `"<len>|<type>|<name>"` format)
- Base64 decoding + Block binary parsing for `Constant.valueAsBlock`
- Custom string deserialization for `CatalogHandle` (`"<name>:<type>:<version>"` format)
- Custom string deserialization for `FunctionId` (pass-through string)
- Custom string deserialization for `Type` (type signature string)

Libraries like `serde` with `#[serde(tag = "@type")]` can handle the polymorphic dispatch. The custom deserializers are all relatively simple string parsers.
