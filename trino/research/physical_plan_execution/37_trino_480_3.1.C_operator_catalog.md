## 0. Research Focus
* **Task ID:** 3.1.C
* **Focus:** Enumerate every concrete `Operator` implementation in Trino 480, classifying each along five dimensions: category, pipeline role, state model, spill support, and function.

## 1. High-Level Overview

Trino 480 contains **43 concrete operator implementations** across the `io.trino.operator` package and its sub-packages. These operators form the executable backbone of every query plan -- each pipeline in a Driver is a chain of operators that produce, transform, and consume pages of columnar data.

**Breakdown by category:**
- Source operators: 9
- Exchange operators: 5
- Transform operators (stateless): 7
- Join operators: 8
- Aggregation operators: 3
- Window & ranking operators: 4
- DML / output operators: 9
- Metadata & control operators: 6

(Some operators fall on boundaries -- the categorization here reflects primary purpose.)

**Two operator patterns coexist:**
1. **Classic `Operator` interface** -- the original push/pull model with `addInput()`/`getOutput()`/`finish()`. Used by ~28 operators.
2. **`WorkProcessorOperator` adapter** -- a pull-based model using `WorkProcessor<Page>` streams, wrapped via `WorkProcessorOperatorAdapter` to conform to the classic interface. Used by ~10 operators.
3. **`SourceOperator` sub-interface** -- extends `Operator` with split-based data ingestion (`addSplit()`/`noMoreSplits()`). Used by ~6 source operators.

The `WorkProcessorSourceOperator` variant combines the source and WorkProcessor patterns for scan+filter+project fusion.

## 2. Structural Architecture

### 2.1 The Operator Interface Contract

```java
// core/trino-main/.../operator/Operator.java
public interface Operator extends AutoCloseable {
    OperatorContext getOperatorContext();
    ListenableFuture<Void> isBlocked();      // async backpressure
    boolean needsInput();                      // can accept a page?
    void addInput(Page page);                  // push a page in
    Page getOutput();                          // pull a page out
    ListenableFuture<Void> startMemoryRevoke(); // spill trigger
    void finishMemoryRevoke();                 // spill cleanup
    void finish();                             // no more input
    boolean isFinished();                      // fully done?
}
```

The Driver loop calls these methods in a strict sequence: check `isBlocked()`, then `needsInput()`/`addInput()` or `getOutput()`, repeating until `isFinished()`. The `startMemoryRevoke()`/`finishMemoryRevoke()` pair is invoked by the memory management system when the operator needs to spill.

### 2.2 The SourceOperator Sub-Interface

```java
// core/trino-main/.../operator/SourceOperator.java
public interface SourceOperator extends Operator {
    PlanNodeId getSourceId();
    void addSplit(Split split);
    void noMoreSplits();
}
```

Source operators are always the first operator in a pipeline. They receive splits (units of work) from the scheduler rather than pages from a predecessor. `needsInput()` returns `false` and `addInput()` throws.

### 2.3 The WorkProcessorOperator Adapter Pattern

```java
// core/trino-main/.../operator/WorkProcessorOperator.java
public interface WorkProcessorOperator extends AutoCloseable {
    WorkProcessor<Page> getOutputPages();
}
```

`WorkProcessorOperator` exposes a lazy pull-based `WorkProcessor<Page>` stream. The `WorkProcessorOperatorAdapter` wraps this into the classic `Operator` interface by driving the WorkProcessor in `getOutput()`. This pattern enables cleaner composition of transformations (filter, project, merge, sort) via `WorkProcessor.transform()`, `flatMap()`, etc.

`WorkProcessorSourceOperator` extends this for source operators, adding physical I/O metrics.

### 2.4 OperatorFactory

Every operator has a corresponding `OperatorFactory` (or `SourceOperatorFactory`) that creates operator instances per Driver. Factories hold configuration immutably and are closed via `noMoreOperators()` once all drivers are created. Some factories implement `duplicate()` for plan cloning.

## 3. The Complete Operator Catalog

### 3.1 Source Operators

These are the first operator in a pipeline. They produce pages from external data or internal state, and never accept input via `addInput()`.

## 4. Key Design Insights

* **Two-pattern coexistence is deliberate.** The classic `Operator` interface is used where fine-grained control over blocking, spill, and memory is needed (e.g., `HashAggregationOperator`, `WindowOperator`, `OrderByOperator`). The `WorkProcessorOperator` pattern is used where the operator can be expressed cleanly as a lazy transformation pipeline (e.g., `TopNOperator`, `FilterAndProjectOperator`, `StreamingAggregationOperator`). The adapter layer (`WorkProcessorOperatorAdapter`) bridges the two.

* **Spill support is concentrated in four operators.** Only `HashAggregationOperator`, `OrderByOperator`, `WindowOperator`, and `spilling.HashBuilderOperator` implement `startMemoryRevoke()`/`finishMemoryRevoke()`. These are the operators that can accumulate unbounded state. All other operators either are stateless, have bounded state, or rely on the spilling infrastructure of their build-side partner.

* **Build/probe separation is fundamental to join execution.** Every join type (hash, nested-loop, semi, spatial) uses separate build and probe operators connected via a shared "bridge" (`JoinBridgeManager`, `NestedLoopJoinBridge`, `SetSupplier`, `PagesSpatialIndexFactory`). The build side runs in its own pipeline and publishes a lookup structure; the probe side consumes it. This clean separation enables the build side to spill independently.

* **Source operators form two sub-families.** The `SourceOperator` interface (with `addSplit()`) is used for operators that receive work units from the scheduler: `TableScanOperator`, `ExchangeOperator`, `MergeOperator`, `IndexSourceOperator`, `LeafTableFunctionOperator`. Non-split source operators like `LocalExchangeSourceOperator`, `ValuesOperator`, and `LookupOuterOperator` implement `Operator` directly with `needsInput() = false`.

* **The scan-filter-project fusion pattern.** `ScanFilterAndProjectOperator` combines three logical operations (table scan, predicate filtering, column projection) into a single physical operator using the `WorkProcessorSourceOperator` pattern. This avoids materializing intermediate pages between scan and filter, significantly improving I/O efficiency.

* **Sink operators have a `getOutput() = null` contract.** Operators at the end of a pipeline (`LocalExchangeSinkOperator`, `PartitionedOutputOperator`, `TaskOutputOperator`, `DevNullOperator`, `PageBufferOperator`, etc.) accept input via `addInput()` but always return `null` from `getOutput()`. They push data to external destinations (output buffers, exchange sinks, page sinks) rather than passing it to a downstream operator.

## 5. Porting Considerations (Java -> Rust Architecture)

* **The `Operator` interface maps naturally to a Rust trait** with `async fn` for `is_blocked()` and `start_memory_revoke()`. The `WorkProcessorOperator` pattern could be replaced with Rust's `Stream` trait for cleaner pull-based composition.

* **Spill support could be implemented via a generic `Spillable` trait** rather than optional method overrides. Only four operators need it, making the trait surface small and testable.

* **The build/probe bridge pattern** (shared state between build and probe pipelines) requires careful lifetime management in Rust. `Arc<Mutex<...>>` or channel-based designs could replace Java's `SettableFuture`/`ListenableFuture` coordination.

* **The `OperatorFactory` pattern** could be replaced with closures or builder patterns in Rust, avoiding the need for explicit factory classes while maintaining the ability to create per-driver instances.

* **The 54 operators** represent a large surface area. A phased approach prioritizing the ~15 most common operators (TableScan, ScanFilterProject, FilterProject, HashAgg, OrderBy, Window, HashBuilder, LookupJoin, Exchange, LocalExchangeSource/Sink, PartitionedOutput, TaskOutput, Limit, TopN) would cover the vast majority of queries.
