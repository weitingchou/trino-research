# Module Teardown: Exchange Protocol & Partition Hash Tests (Task 6.5.A) [KG-13]

## 0. Research Focus

This document catalogs the tests that cover three critical areas for a Rust drop-in replacement worker:

1. **Token-based pull protocol** -- the HTTP exchange where a downstream consumer fetches serialized pages from an upstream buffer using sequence tokens, ACKs, and buffer-destroy lifecycle.
2. **Custom HTTP headers** -- the 8 `X-Trino-*` headers that form the exchange wire protocol.
3. **Partition hash functions** -- the hashing algorithms that determine which partition a row is routed to, where even a single-bit discrepancy would silently corrupt query results.
4. **OutputBuffer back-pressure** -- the mechanism that blocks writers when buffers are full and unblocks them on consumer acknowledgment.
5. **Extractable test artifacts** -- concrete values, binary formats, and protocol constants that can be used for Rust-side conformance tests.

Source version: Trino 480.

## 2. Detailed Analysis

### 2.1 Token-Based Pull Protocol Tests

#### 2.1.1 `TestHttpPageBufferClient` (`.../operator/TestHttpPageBufferClient.java`)

This is the primary test for the single-buffer HTTP fetch lifecycle. It uses `MockExchangeRequestProcessor` as a fake HTTP server.

**`testHappyPath()`** -- The definitive lifecycle test:
1. Creates client pointed at `http://localhost:8080`
2. Adds a page to the mock processor, schedules a request
3. Verifies: 1 page received, 1 completed request, 0 finished buffers
4. Asserts status: `(location, "queued", pagesReceived=1, requests=1, completed=1, failed=0)`
5. Schedules request with no data -- gets empty response, no failure
6. Adds 2 more pages, fetches them in one request
7. Sets `complete`, schedules request -- gets buffer-complete signal
8. Schedules another request -- triggers DELETE (buffer destroy), verifies `finishedBuffers=1`
9. Final status: `"closed"`, 3 pages total, 5 requests, 0 failures

This test exercises the full state machine: `queued -> running -> queued -> ... -> closed`.

**`testLifecycle()`** -- Tests the `client.close()` path that sends a DELETE request.

**`testInvalidResponses()`** -- Tests error handling for:
- HTTP 404 (NOT_FOUND) -- expects `PageTransportErrorException`
- Wrong Content-Type (INVALID_TYPE) -- expects specific error message
- Unexpected Content-Type (text/plain) -- expects specific error message

**`testCloseDuringPendingRequest()`** -- Tests closing while a request is in flight.

**`testExceptionFromResponseHandler()`** -- Tests the error timeout mechanism:
- First error starts the error stopwatch
- 30s later: still not failed (within tolerance)
- 31s later: client fails with `PageTransportTimeoutException`

**`testMemoryExceededInAddPages()`** -- Tests memory limit handling during page receipt.

**`testAverageSizeOfRequest()`** -- Tests rolling average calculation:
```
requestSucceeded(0)    -> avg = 0
requestSucceeded(1000) -> avg = 600 (weighted)
requestSucceeded(800)  -> avg = 600
```

#### 2.1.3 `MockExchangeRequestProcessor` (`.../operator/MockExchangeRequestProcessor.java`)

This is the mock HTTP server that produces correct exchange responses. It is the **authoritative test reference for the wire protocol**. Key behaviors:

**Request handling:**
```java
// Reads and validates X-Trino-Max-Size header from request
assertThat(request.getHeaders().get(TRINO_MAX_SIZE_HEADER)).isNotEmpty();
DataSize maxSize = DataSize.valueOf(request.getHeader(TRINO_MAX_SIZE_HEADER));

// Parses sequence ID from URL: http://host:port/location/{sequenceId}
long sequenceId = Long.parseLong(string.substring(index + 1));

// DELETE requests return 204 NO_CONTENT immediately
if (request.getMethod().equalsIgnoreCase("DELETE")) {
    return new TestingResponse(HttpStatus.NO_CONTENT, ...);
}
```

**Response headers (all 6 custom headers present):**
```java
ImmutableListMultimap.<HeaderName, String>builder()
    .put(CONTENT_TYPE, TRINO_PAGES)                                    // "application/x-trino-pages"
    .put(TRINO_TASK_INSTANCE_ID_HEADER, String.valueOf(result.taskInstanceId()))  // "X-Trino-Task-Instance-Id"
    .put(TRINO_PAGE_TOKEN_HEADER, String.valueOf(result.token()))                 // "X-Trino-Page-Sequence-Id"
    .put(TRINO_PAGE_NEXT_TOKEN_HEADER, String.valueOf(result.nextToken()))        // "X-Trino-Page-End-Sequence-Id"
    .put(TRINO_BUFFER_COMPLETE_HEADER, String.valueOf(result.bufferComplete()))   // "X-Trino-Buffer-Complete"
    .put(TRINO_TASK_FAILED_HEADER, "false")                                      // "X-Trino-Task-Failed"
    .build()
```

**Binary response body format:**
```java
sliceOutput.writeInt(SERIALIZED_PAGES_MAGIC);           // 4 bytes: 0xfea4f001
sliceOutput.writeLong(calculateChecksum(serializedPages)); // 8 bytes: XxHash64 of all page slices
sliceOutput.writeInt(result.serializedPages().size());   // 4 bytes: page count
for (Slice page : result.serializedPages()) {
    sliceOutput.writeBytes(page);                        // N bytes: concatenated serialized pages
}
```

**Token management in `MockBuffer`:**
```java
long nextToken = token.get() + responsePages.size();
BufferResult bufferResult = new BufferResult(TASK_INSTANCE_ID, token.get(), nextToken, false, responsePages);
token.set(nextToken);
```

**Key constant:** `TASK_INSTANCE_ID = 0x1337`

### 2.3 Partition Hash Function Tests

#### 2.3.1 Hash Algorithm Summary

Trino uses a two-layer hash scheme for partitioning:

**Per-field hash (type-specific):**
- `HASH_CODE` operator: Used for hash-joins and grouping. For BIGINT: `rotateLeft(value * 0xC2B2AE3D27D4EB4FL, 31) * 0x9E3779B185EBCA87L` (xxHash64 mix function)
- `XX_HASH_64` operator: Used for checksums. For BIGINT: `XxHash64.hash(value)`. For VARCHAR: `XxHash64.hash(slice)`.

**Row hash combination (cross-field):**
```java
// CombineHashFunction.getHash()
return (31 * previousHashValue + value);
```

This is a simple polynomial hash combiner, starting from `previousHash=0` for the first field. It is **not** the same as Java's `Object.hashCode()`.

**Partition assignment:**
```java
// In PartitionFunction implementations:
int partition = (int) (hash % partitionCount);
```

#### 2.3.2 `TestInterpretedHashGenerator` (`.../operator/TestInterpretedHashGenerator.java`)

Tests correctness of the batched hash generator against a manually computed reference.

**`testHashPosition()`:**
- Creates a large type set (17 base types: BIGINT, BOOLEAN, CHAR(5), DECIMAL(18), DECIMAL(38), DOUBLE, INTEGER, IPADDRESS, REAL, TIMESTAMP_SECONDS/MILLIS/MICROS/NANOS/PICOS, UUID, VARBINARY, VARCHAR), plus ROW, ARRAY, MAP combinations.
- For sizes 1, 500, 501, 999, 1000, 1001, 2000, 2001 columns
- Generates random data (10 rows, 0% nulls)
- For positions 0, 5, 9: asserts `hashGenerator.hashPosition(pos, page)` equals `manualHash(types, blocks, pos)`
- Repeats with RLE blocks and Dictionary blocks

**`manualHash()` reference implementation:**
```java
long manualRowHash = 0;
for (int i = 0; i < types.size(); i++) {
    long fieldHash = 0;
    if (!block.isNull(position)) {
        fieldHash = typeOperators.getHashCodeOperator(type, ...).invoke(block, position);
    }
    manualRowHash = CombineHashFunction.getHash(manualRowHash, fieldHash);
    // i.e., manualRowHash = 31 * manualRowHash + fieldHash;
}
```

**`testBatchedRawHashesMatchSinglePositionHashes()`:**
- Tests 1024 positions with 25% null rate
- Verifies batched hash output matches per-position hash output
- Tests with regular, RLE, and dictionary blocks

**CRITICAL NOTE:** There are **no hardcoded expected hash values** in this test. All assertions compare `hashGenerator` output against the `manualHash()` reference. This means cross-language validation requires re-implementing the exact same hash chain.

#### 2.3.3 `TestFlatHashStrategy` (`.../operator/TestFlatHashStrategy.java`)

Tests the flat (compiled) hash strategy against the interpreted hash generator. Same type set and methodology as `TestInterpretedHashGenerator`. Also verifies flat serialization round-trip.

The key assertion: `flatHashStrategy.hash(blocks, position) == manualHash(types, blocks, position)`.

#### 2.3.4 `AbstractTestType` (`.../type/AbstractTestType.java`)

Base class for all type tests. For each type's test block, it:
- Computes `hashCodeOperator.hashCodeNullSafe(block, position)` and stores the expected hash
- Asserts that hash is stable across `getSingleValueBlock()`, `getRegion()`, and `copyPositions()`
- Asserts `flatHashCodeOperator` matches `hashCodeOperator`
- Asserts `flatXxHash64Operator` matches `xxHash64Operator`

This provides consistency guarantees but **no cross-language golden values**.

#### 2.3.5 Type-Specific Hash Formulas (for Rust Porting)

From the SPI source code:

| Type | HASH_CODE formula | XX_HASH_64 formula |
|---|---|---|
| BIGINT (AbstractLongType) | `rotateLeft(value * 0xC2B2AE3D27D4EB4FL, 31) * 0x9E3779B185EBCA87L` | `XxHash64.hash(value)` |
| INTEGER (AbstractIntType) | Same as BIGINT (casts to long first) | `XxHash64.hash((long)value)` |
| VARCHAR (AbstractVariableWidthType) | `XxHash64.hash(slice)` | `XxHash64.hash(slice)` |
| BOOLEAN | `value ? 1231 : 1237` | `XxHash64.hash(value ? 1L : 0L)` |
| DOUBLE | `hash(doubleToLongBits(value))` via AbstractLongType mix | `XxHash64.hash(doubleToLongBits(value))` |
| REAL | `hash((long) floatToIntBits(value))` via AbstractLongType mix | `XxHash64.hash((long) floatToIntBits(value))` |

The `HASH_CODE` operator is the one used in partition hash computation. `XX_HASH_64` is used only for checksums.

## 3. Key Design Insights

### 3.1 The Token Protocol is Simpler Than It Appears

The exchange protocol boils down to:
1. **GET** `/{location}/{sequenceId}` with `X-Trino-Max-Size` header
2. Response includes `token` (confirms what was requested), `nextToken` (sequenceId for next request), `bufferComplete` flag
3. Requesting with `sequenceId = nextToken` from previous response ACKs the received pages
4. When `bufferComplete=true`, client sends **DELETE** to destroy the buffer
5. DELETE returns 204 NO_CONTENT

The mock processors show this is a simple cursor protocol, not a complex state machine.

### 3.2 No Golden Test Vectors Exist

The most significant finding for the Rust port is that **there are no recorded binary fixtures or hardcoded expected hash values** anywhere in the test suite. All hash tests compare two implementations against each other:
- `InterpretedHashGenerator` vs `manualHash()` (both in Java)
- `FlatHashStrategy` vs `InterpretedHashGenerator` (both in Java)
- `hashCodeOperator` across different block representations (all Java)

This means the Rust port must create its own golden test vectors by running the Java implementation once to record expected outputs, then assert the Rust implementation matches.

### 3.3 The Partition Hash Is NOT XxHash64

A critical distinction: the `HASH_CODE` operator used for partitioning uses a custom xxHash64 **mix function** (multiply + rotate), NOT the full xxHash64 algorithm. For BIGINT:
```
hash = rotateLeft(value * 0xC2B2AE3D27D4EB4FL, 31) * 0x9E3779B185EBCA87L
```

The `XX_HASH_64` operator (full algorithm) is only used for data integrity checksums.

For VARCHAR, both operators use the full `XxHash64.hash(slice)`.

The row combiner is: `combined = 31 * previousHash + fieldHash`, starting from 0.

### 3.4 Back-Pressure Is ListenableFuture-Based

All back-pressure tests follow the pattern:
1. `buffer.isFull()` returns a `ListenableFuture<Void>`
2. When buffer exceeds threshold, the future is NOT done (blocks caller via `await`)
3. Consumer ACK (advancing sequenceId) frees space and completes the future
4. `setNoMorePages()` and `destroy()` also complete all pending futures

This maps cleanly to Rust's async/await pattern.

### 3.5 TASK_INSTANCE_ID is a Crash Detector

The `X-Trino-Task-Instance-Id` header (value `0x1337` in tests) serves as a crash/restart detector. If the coordinator restarts, it gets a new instance ID, and in-flight exchange requests can detect the mismatch.

## 4. Porting Considerations (Java -> Rust)

### 4.1 Priority 1: Generate Cross-Language Hash Test Vectors

Since no golden values exist, the Rust project must:

1. **Write a Java harness** that computes and prints hash values for known inputs:
   ```
   BIGINT(0)    -> hashCode = ?
   BIGINT(1)    -> hashCode = ?
   BIGINT(-1)   -> hashCode = ?
   BIGINT(1111) -> hashCode = ?
   VARCHAR("hello") -> hashCode = ?
   ROW(BIGINT(1), VARCHAR("a")) -> combined hash = ?
   ```
2. **Hard-code those values** in Rust tests.
3. The hash formula for BIGINT can be directly computed:
   ```rust
   fn bigint_hash(value: i64) -> i64 {
       let v = (value as u64).wrapping_mul(0xC2B2AE3D27D4EB4F);
       (v.rotate_left(31).wrapping_mul(0x9E3779B185EBCA87)) as i64
   }
   fn combine_hash(previous: i64, field: i64) -> i64 {
       31i64.wrapping_mul(previous).wrapping_add(field)
   }
   ```

### 4.2 Priority 2: Implement Wire Protocol Test from MockExchangeRequestProcessor

The `MockExchangeRequestProcessor` is effectively a specification:

- **Request:** GET `{base_uri}/{seq_id}`, Header: `X-Trino-Max-Size: {n}MB`
- **Response body (when pages present):**
  - `[4 bytes: 0xfea4f001]` (magic)
  - `[8 bytes: XxHash64 checksum]`
  - `[4 bytes: page_count]`
  - `[variable: serialized pages concatenated]`
- **Response body (no pages):** Empty, status 204
- **Response headers:** All 6 `X-Trino-*` headers always present
- **DELETE:** Returns 204, no body

### 4.3 Priority 3: Back-Pressure Conformance

The Rust buffer must:
- Return a future/waker from `enqueue()` when the buffer exceeds `maxBufferedBytes`
- Complete that future when any consumer advances its sequenceId past acknowledged pages
- Complete all pending futures on `set_no_more_pages()` or `destroy()`

Test pattern from Java:
```
let future = buffer.enqueue(page, partition);
assert!(!future.is_ready());  // blocked
buffer.get(consumer_id, seq_id + N, max_size); // ACK
assert!(future.is_ready());   // unblocked
```

| Constant | Value | Usage |
