# Module Teardown: Control Plane & Task Lifecycle Tests (Task 6.3.A) [KG-13]

## 0. Research Focus

This document analyzes Trino 480's test infrastructure for the worker control plane: task lifecycle state machines, JSON serialization of wire-protocol types, REST endpoint testing, plan fragment parsing, and expression evaluation. The goal is to identify which test patterns can be directly ported to validate a Rust-based drop-in replacement worker.

**Key questions:**
- Do task lifecycle tests create real tasks with plan fragments, or are they mocked?
- Are there standalone JSON roundtrip tests for `TaskUpdateRequest`, `TaskStatus`, `PlanFragment`?
- Are `TaskResource` REST endpoints tested in isolation? What harness is used?
- Are specific plan node types deserialized from JSON and validated?
- How are expression compilation and evaluation tested?

## 1. High-Level Overview

Trino's control plane test infrastructure is organized into **five layers**, each testing a different slice of the coordinator-to-worker protocol:

| Layer | What It Tests | Key Test Classes |
|-------|--------------|-----------------|
| **Task Lifecycle** | SqlTask/SqlTaskManager state machine (RUNNING -> FLUSHING -> FINISHED, cancel, abort, fail) | `TestSqlTask`, `BaseTestSqlTaskManager`, `TestSqlTaskExecution` |
| **JSON Serialization** | Roundtrip encode/decode of wire types | `TestTaskStats`, `TestSymbolSerializer`, `TestSymbolKeySerialization`, `TestPatternRecognitionNodeSerialization`, `TestRowPatternSerialization`, `TestStatisticAggregationsDescriptor`, `TestSliceSerialization`, `TestServerInfoSerialization` |
| **REST Endpoint (HTTP)** | End-to-end coordinator-to-worker protocol over HTTP | `TestHttpRemoteTask` (with embedded `TestingTaskResource` JAX-RS fake) |
| **Plan Fragment** | Fragment construction, partitioning, serialization of plan nodes | `TestPlanFragmentPartitionCount`, `TaskTestUtils.PLAN_FRAGMENT` |
| **Expression Evaluation** | IR-level optimization, constant folding, bytecode compilation | `TestExpressionInterpreter`, `TestExpressionCompiler`, `TestEvaluateIsNull` (and other IR optimizer tests) |

**No dedicated `TestTaskResource` exists.** The real `TaskResource` JAX-RS endpoint is never tested in isolation; instead, `TestHttpRemoteTask` stands up a `TestingTaskResource` (a fake JAX-RS resource) that mimics the wire protocol and validates the client side (`HttpRemoteTask`).

## 2. Detailed Analysis

### 2.1 Task Lifecycle Tests

#### 2.1.1 `TestSqlTask` — Unit-level task state machine
- **File:** `core/trino-main/src/test/java/io/trino/execution/TestSqlTask.java`
- **Creates real tasks** with actual plan fragments and a real `SqlTaskExecutionFactory`.
- Setup creates a `TimeSharingTaskExecutor` (8 runners, 16 concurrent), a `LocalExecutionPlanner` from `TaskTestUtils.createTestingPlanner()`, and wraps them in `SqlTaskExecutionFactory`.
- Each test calls `createInitialTask()` which builds a full `QueryContext` -> `TaskStateMachine` -> `SqlTask.createSqlTask()`.
- Tasks are driven by calling `sqlTask.updateTask()` with `PLAN_FRAGMENT` (a `TableScanNode`) and `SplitAssignment`.

**Test methods and what they validate:**

| Method | Validates |
|--------|-----------|
| `testEmptyQuery()` | Task starts RUNNING, transitions to FINISHED when given `noMoreSplits` with empty set |
| `testSimpleQuery()` | Task goes RUNNING -> FLUSHING (data in buffer) -> FINISHED after buffer drain |
| `testCancel()` | `sqlTask.cancel()` -> CANCELED (may race through CANCELLING) |
| `testAbort()` | Buffer destroyed -> FINISHED |
| `testBufferCloseOnFinish()` | Pending buffer future completes on task finish |
| `testBufferCloseOnCancel()` | Buffer future completes on cancel |
| `testBufferNotCloseOnFail()` | On `failed()`, buffer is NOT automatically closed (timeout expected) |
| `testDynamicFilters()` | Dynamic filter version increments unblock `getTaskStatus()` future |
| `testDynamicFilterFetchAfterTaskDone()` | Dynamic filters still retrievable after task finishes |

**Plan fragment used** (from `TaskTestUtils`):
```java
public static final PlanFragment PLAN_FRAGMENT = new PlanFragment(
    new PlanFragmentId("fragment"),
    new TableScanNode(TABLE_SCAN_NODE_ID, TEST_TABLE_HANDLE,
        ImmutableList.of(SYMBOL),
        ImmutableMap.of(SYMBOL, new TestingColumnHandle("column", 0, BIGINT)),
        TupleDomain.all(), ...),
    ImmutableSet.of(SYMBOL),
    SOURCE_DISTRIBUTION,
    OptionalInt.empty(),
    ImmutableList.of(TABLE_SCAN_NODE_ID),
    new PartitioningScheme(...).withBucketToPartition(Optional.of(new int[1])),
    ...);
```

This is a minimal but **structurally real** PlanFragment: it has a root `TableScanNode`, `SOURCE_DISTRIBUTION`, and a `PartitioningScheme` with `SINGLE_DISTRIBUTION`.

#### 2.1.2 `BaseTestSqlTaskManager` — Manager-level lifecycle
- **File:** `core/trino-main/src/test/java/io/trino/execution/BaseTestSqlTaskManager.java`
- Abstract base class with two concrete implementations:
  - `TestSqlTaskManagerTimeSharing` — uses `TimeSharingTaskExecutor`
  - `TestSqlTaskManagerThreadPerDriver` — uses `ThreadPerDriverTaskExecutor`
- Tests the full `SqlTaskManager` (the component that `TaskResource` delegates to).
- Creates `SqlTaskManager` with `LocalExecutionPlanner`, `LocalMemoryManager`, `TaskManagementExecutor`, etc.
- Tests: `testEmptyQuery`, `testSimpleQuery`, `testCancel`, `testAbort`, `testAbortResults`, `testRemoveOldTasks`, `testSessionPropertyMemoryLimitOverride`.
- Uses helper `createTask()` which calls `sqlTaskManager.updateTask()` with `PLAN_FRAGMENT`.

**Key mock classes defined here:**
- `MockDirectExchangeClientSupplier` — throws on use (exchange not needed for TableScan)
- `MockLocationFactory` — returns `http://fake.invalid/task/{taskId}` URIs
- `NoConnectorServicesProvider` — empty catalog provider

#### 2.1.3 `TestSqlTaskExecution` — Low-level execution with custom operators
- **File:** `core/trino-main/src/test/java/io/trino/execution/TestSqlTaskExecution.java`
- Bypasses `PlanFragment` entirely. Creates a `LocalExecutionPlan` directly with custom `DriverFactory` containing:
  - `TestingScanOperatorFactory` — a fake source operator that yields pages from `TestingSplit` data
  - `TaskOutputOperatorFactory` — writes to `PartitionedOutputBuffer`
- Drives execution by calling `sqlTaskExecution.addSplitAssignments()` directly.
- Tests split ordering (sequence IDs across plan node IDs), `Pauser` for controlling execution, buffer consumption.
- This tests the engine's execution loop without any plan parsing.

#### 2.1.4 `TestSqlTaskManagerRaceWithCatalogPrune`
- **File:** `core/trino-main/src/test/java/io/trino/execution/TestSqlTaskManagerRaceWithCatalogPrune.java`
- Stress test: races task updates against catalog pruning in a thread pool.
- Uses `WorkerDynamicCatalogManager` and `CatalogPruneTask`.
- Validates that concurrent catalog management does not corrupt task state.

### 2.2 JSON Serialization Tests

There is **no single `TestTaskUpdateRequest`** or `TestTaskStatus` that roundtrips those types in isolation. Instead, serialization coverage comes from:

#### 2.2.1 `TestTaskStats` — Stats roundtrip
- **File:** `core/trino-main/src/test/java/io/trino/operator/TestTaskStats.java`
- Creates a fully populated `TaskStats` object with every field set.
- `testJson()`: `codec.toJson(EXPECTED)` then `codec.fromJson(json)`, asserts every field.
- Pattern: `JsonCodec<TaskStats> codec = jsonCodec(TaskStats.class)` (airlift codec, no custom setup needed).

#### 2.2.2 `TestSymbolSerializer` / `TestSymbolKeySerialization` — Symbol encoding
- **Files:** `core/trino-main/src/test/java/io/trino/sql/planner/TestSymbolSerializer.java`, `TestSymbolKeySerialization.java`
- Validates `Symbol` (name + type) survives JSON roundtrip both as a value and as a map key.
- Symbols use a custom encoding format: `"7|integer|abc"` (version|type|name).
- Tests edge cases: special characters, complex types like `rowType`.

#### 2.2.3 `TestPatternRecognitionNodeSerialization` — Complex plan node roundtrip
- **File:** `core/trino-main/src/test/java/io/trino/sql/planner/plan/TestPatternRecognitionNodeSerialization.java`
- Sets up a full `JsonCodecFactory` with `TypeDeserializer`, `BlockJsonSerde`, `SymbolKeyDeserializer`.
- Tests roundtrip of: `ValuePointer`, `ExpressionAndValuePointers`, `Measure`, `PatternRecognitionNode`.
- Uses `assertJsonRoundTrip(codec, object)` pattern: `codec.fromJson(codec.toJson(obj)) == obj`.

#### 2.2.4 `TestRowPatternSerialization` — Row pattern IR roundtrip
- **File:** `core/trino-main/src/test/java/io/trino/sql/planner/rowpattern/TestRowPatternSerialization.java`
- Tests `IrQuantifier` and `IrRowPattern` serialization with nested patterns.

#### 2.2.5 `TestStatisticAggregationsDescriptor` — Statistics descriptor roundtrip
- **File:** `core/trino-main/src/test/java/io/trino/sql/planner/plan/TestStatisticAggregationsDescriptor.java`
- Creates descriptors with column statistics of every type, group columns, table statistics.

#### 2.2.6 `TestSliceSerialization` / `TestServerInfoSerialization`
- **Files:** `core/trino-main/src/test/java/io/trino/server/TestSliceSerialization.java`, `TestServerInfoSerialization.java`
- `Slice` serialization: binary data roundtripped through custom Jackson serializer/deserializer.
- `ServerInfo` serialization: tests that server-side `ServerInfo` deserializes correctly into client-side `ServerInfo`.

#### 2.2.7 Implicit `TaskUpdateRequest` serialization in `TestHttpRemoteTask`
- `TestHttpRemoteTask` registers `JsonCodec<TaskUpdateRequest>` via Guice/airlift bindings.
- The `HttpRemoteTaskFactory` serializes `TaskUpdateRequest` to JSON; the `TestingTaskResource` deserializes it.
- This exercises the full `TaskUpdateRequest` serialization path including `SessionRepresentation`, `PlanFragment`, `SplitAssignment`, `OutputBuffers`, `DynamicFilterDomains` -- but there is no standalone assertion on the JSON structure.

### 2.3 REST Endpoint Tests

#### 2.3.1 `TestHttpRemoteTask` — The primary REST protocol test
- **File:** `core/trino-main/src/test/java/io/trino/server/remotetask/TestHttpRemoteTask.java`
- Tests the **client side** (`HttpRemoteTask` / `HttpRemoteTaskFactory`) against a fake server.
- **Test harness:** Uses airlift's `JaxrsTestingHttpProcessor` + `TestingHttpClient` to stand up an in-process JAX-RS endpoint without a real HTTP server.
- The embedded `TestingTaskResource` (a `@Path("/task/{nodeId}")` class) implements the full task REST API:
  - `POST {taskId}` — `createOrUpdateTask(TaskUpdateRequest)` — deserializes and accumulates split assignments
  - `GET {taskId}` — `getTaskInfo()`
  - `GET {taskId}/status` — `getTaskStatus()` with long-polling
  - `GET {taskId}/dynamicfilters` — `acknowledgeAndGetNewDynamicFilterDomains()`
  - `DELETE {taskId}` — `deleteTask(abort)`

**Failure scenario tests:**
- `TASK_MISMATCH` — simulates worker restart (task instance ID changes at status fetch #10)
- `TASK_MISMATCH_WHEN_VERSION_IS_HIGH` — same but with artificially high version number
- `REJECTED_EXECUTION` — simulates HTTP client shutdown

**Dynamic filter tests** (5 separate test methods):
- Validates that dynamic filters are fetched lazily (not on every status poll).
- Tests failure/retry of dynamic filter fetch.
- Tests version mismatch detection.
- Tests outbound dynamic filter delivery from coordinator to worker.
- Tests adaptive request sizing for split batching.

**JSON codec wiring** (from `createHttpRemoteTaskFactory`):
```java
jsonCodecBinder(binder).bindJsonCodec(TaskStatus.class);
jsonCodecBinder(binder).bindJsonCodec(VersionedDynamicFilterDomains.class);
jsonCodecBinder(binder).bindJsonCodec(TaskInfo.class);
jsonCodecBinder(binder).bindJsonCodec(TaskUpdateRequest.class);
jsonCodecBinder(binder).bindJsonCodec(FailTaskRequest.class);
```

#### 2.3.2 `TaskResource` — The real endpoint (not tested in isolation)
- **File:** `core/trino-main/src/main/java/io/trino/server/TaskResource.java`
- `@Path("/v1/task")` with `@ResourceSecurity(INTERNAL_ONLY)`
- Endpoints:
  - `GET /` — `getAllTaskInfo()`
  - `POST /{taskId}` — `createOrUpdateTask(TaskUpdateRequest)` (async)
  - `GET /{taskId}` — `getTaskInfo()` with long-polling via `TRINO_CURRENT_VERSION` / `TRINO_MAX_WAIT` headers
  - `GET /{taskId}/status` — `getTaskStatus()` with long-polling
  - `GET /{taskId}/dynamicfilters` — dynamic filter domains
  - `DELETE /{taskId}` — cancel/abort
  - `POST /{taskId}/fail` — `failTask(FailTaskRequest)`
  - `GET /{taskId}/results/{bufferId}/{token}` — page data (binary `TRINO_PAGES` format)
  - `GET /{taskId}/results/{bufferId}/{token}/acknowledge` — acknowledge results
  - `DELETE /{taskId}/results/{bufferId}` — destroy results
  - `POST /pruneCatalogs` — catalog pruning

### 2.4 Plan Fragment Parsing Tests

There are **no standalone PlanFragment JSON deserialization tests** that read a JSON string and produce a `PlanFragment`. Instead:

#### 2.4.1 `TaskTestUtils` provides pre-built PlanFragments
- `PLAN_FRAGMENT` — `TableScanNode` with `SOURCE_DISTRIBUTION`
- `PLAN_FRAGMENT_WITH_DYNAMIC_FILTER_SOURCE` — `DynamicFilterSourceNode` wrapping a `TableScanNode`
- These are constructed programmatically (not parsed from JSON).

#### 2.4.2 `TestPlanFragmentPartitionCount`
- **File:** `core/trino-main/src/test/java/io/trino/sql/planner/TestPlanFragmentPartitionCount.java`
- Uses `PlanFragmenter` to split a logical plan into fragments and validates partition counts.
- Validates fragment construction at a higher level (planning, not serialization).

#### 2.4.3 Plan node roundtrip tests
- `TestPatternRecognitionNodeSerialization` roundtrips `PatternRecognitionNode` through JSON.
- `TestStatisticAggregationsDescriptor` roundtrips statistic descriptors.
- Individual plan node types (TableScan, Join, etc.) are tested via `TestingPlannerContext` but not through explicit JSON roundtrip tests in most cases.

### 2.5 Expression Evaluation Tests

#### 2.5.1 `TestExpressionInterpreter` — IR-level constant folding
- **File:** `core/trino-main/src/test/java/io/trino/sql/TestExpressionInterpreter.java`
- Operates on **IR expression nodes** (not SQL strings).
- `assertOptimizedEquals(input, expected)` — runs the expression interpreter (constant folder) and compares.
- Tests: `AND`, `OR`, `Comparison`, `IsNull`, `NullIf`, `Coalesce`, `Case`, `Switch`, `Between`, `In`, `Cast`, arithmetic operators.
- Uses `Reference("bound_value")` with `INPUTS` map for partial evaluation.

#### 2.5.2 `TestExpressionCompiler` — End-to-end SQL expression evaluation
- **File:** `core/trino-main/src/test/java/io/trino/sql/gen/TestExpressionCompiler.java`
- Uses `QueryAssertions` which boots a full `TestingTrinoServer`.
- Tests expressions as **SQL strings** with `.binding("a", value)`.
- Evaluates through the full compilation pipeline (IR -> bytecode -> execution).
- Tests: unary/binary operators, comparisons, arithmetic, string functions, timestamps, regex, JSON, LIKE, CASE, COALESCE, TRY, CAST across all type combinations.

#### 2.5.3 IR Optimizer Rule Tests (`sql/ir/optimizer/`)
- Files: `TestEvaluateIsNull`, `TestDesugarBetween`, `TestEvaluateFieldReference`, `TestRemoveRedundantInItems`, `TestSimplifyComplementaryLogicalTerms`, `TestSimplifyRedundantCase`, `TestSimplifyStackedNot`
- Each tests a single optimizer rule in isolation.
- Pattern: construct an IR `Expression`, call `rule.apply(expression, session, bindings)`, assert the `Optional<Expression>` result.

#### 2.5.4 `TestExpressionFormatter` — IR to string rendering
- **File:** `core/trino-main/src/test/java/io/trino/sql/ir/TestExpressionFormatter.java`
- Tests that IR expressions render to expected string representations.

## 3. Key Design Insights

### 3.1 Tests create real execution infrastructure, not mocks
`TestSqlTask` and `BaseTestSqlTaskManager` use a real `LocalExecutionPlanner` with real compilers (`ExpressionCompiler`, `JoinCompiler`, `PageFunctionCompiler`, etc.). The only things mocked are connectors (`TestingPageSourceProvider` returns a fixed 1-row page) and exchange clients. This means the test validates the full path from `PlanFragment` through `LocalExecutionPlanner` to `DriverFactory` to actual operator execution.

### 3.2 No JSON snapshot tests for wire types
There are no tests that say "given this exact JSON string, assert the deserialized object is X." All serialization tests use the roundtrip pattern (`fromJson(toJson(obj)) == obj`). The lack of snapshot tests means the JSON schema is implicitly defined by Jackson annotations, not explicitly documented.

### 3.3 REST endpoint tests validate the client, not the server
`TestHttpRemoteTask` validates that `HttpRemoteTask` correctly sends `TaskUpdateRequest`, polls status, fetches dynamic filters, handles failures, etc. It does this by standing up a fake server. There is no test that instantiates the real `TaskResource` and sends HTTP requests to it. The real `TaskResource` is only tested through integration tests (e.g., `DistributedQueryRunner`).

### 3.4 Two execution models are tested via abstract base class
`BaseTestSqlTaskManager` is extended by both `TestSqlTaskManagerTimeSharing` (cooperative multi-tasking with time slicing) and `TestSqlTaskManagerThreadPerDriver` (one OS thread per driver). The same lifecycle tests run on both, providing confidence that the state machine is executor-agnostic.

### 3.5 `TaskTestUtils.createTestingPlanner()` is the reusable setup
This single factory method creates a fully-wired `LocalExecutionPlanner` with:
- `PageSourceManager` -> `TestingPageSourceProvider`
- `ExpressionCompiler` (bytecode gen) and `ColumnarFilterCompiler`
- `JoinCompiler`, `JoinFilterFunctionCompiler`
- `OrderingCompiler`, `FlatHashStrategyCompiler`
- `GenericSpillerFactory` (throws on use -- spill not tested here)
- `DynamicFilterConfig`, `BlockTypeOperators`, `TypeOperators`
- `ExchangeManagerRegistry`

This is the single most important helper for task-level testing.

### 3.6 Expression IR nodes are the deserialization target for plan nodes
Plan nodes reference expressions via `io.trino.sql.ir.*` types (`Constant`, `Reference`, `Call`, `Comparison`, `Cast`, `Logical`, `IsNull`, `In`, `Case`, `Switch`, `Between`, `NullIf`, `Coalesce`, `Row`, `FieldReference`, `Lambda`). These are all Jackson-serializable and appear in `PlanFragment` JSON. The IR optimizer tests prove these types can be constructed and evaluated without a running server.

## 4. Porting Considerations (Java -> Rust)

### 4.1 Task lifecycle test porting
- **Port directly:** The `TestSqlTask` test structure maps cleanly to Rust. Create a `SqlTask`, call `update_task()` with a PlanFragment, assert state transitions. The state machine (RUNNING/FLUSHING/FINISHED/CANCELED/ABORTED/FAILED) is pure logic.
- **Simplification opportunity:** The Java tests need extensive boilerplate (QueryContext, MemoryPool, SpillSpaceTracker, GcMonitor, 4 executors). In Rust, tokio runtime handles scheduling natively, reducing setup complexity.
- **Buffer testing:** `BufferResult` with sequence IDs and completion tracking maps to a Rust channel/stream pattern. The "buffer not closed on fail" behavior is a specific design choice to preserve.

### 4.2 JSON serialization test porting — the critical path
- **Highest priority for a drop-in replacement.** The Rust worker must produce JSON that the Java coordinator can deserialize, and vice versa.
- **No JSON snapshots exist** in Trino's tests. For the Rust port, create **golden file tests**: capture actual JSON from a running Trino coordinator and assert the Rust implementation produces byte-identical output.
- **Key types to roundtrip test:** `TaskUpdateRequest`, `TaskStatus`, `TaskInfo`, `PlanFragment`, `SessionRepresentation`, `SplitAssignment`, `OutputBuffers`, `VersionedDynamicFilterDomains`, `FailTaskRequest`.
- **Jackson-specific behaviors to handle:**
  - `@JsonTypeInfo` / `@JsonSubTypes` for polymorphic types (plan nodes, output buffer types)
  - Symbol encoding format: `"7|integer|name"` (version byte, pipe-delimited type and name)
  - `Optional` fields: Jackson's `NON_ABSENT` inclusion
  - `Span` serialization via custom serializer/deserializer (OpenTelemetry trace context)
  - `Slice` serialization: base64-encoded byte arrays

### 4.3 REST endpoint test porting
- **Build a fake coordinator client** in Rust tests that sends `TaskUpdateRequest` JSON to the Rust worker's `/v1/task/{taskId}` endpoint and validates the `TaskInfo` response.
- **Test the specific protocol behaviors:**
  - Long-polling with `TRINO_CURRENT_VERSION` / `TRINO_MAX_WAIT` headers
  - Task instance ID (`TRINO_TASK_INSTANCE_ID`) for restart detection
  - Binary page format (`TRINO_PAGES` content type) for results endpoint
  - Dynamic filter version tracking and lazy fetching
  - `503 Service Unavailable` during startup

### 4.4 Expression evaluation strategy
- **Do not port the bytecode compiler.** Trino's `ExpressionCompiler` generates JVM bytecode. The Rust worker needs its own expression evaluation strategy.
- **Port the IR interpreter** (`ExpressionInterpreter`) or build a Cranelift/LLVM-based compiler.
- **Port IR optimizer rules** directly -- they are pure functional transforms on expression trees and test well in isolation.
- **Key IR types to implement:** `Constant`, `Reference`, `Call` (function invocation), `Comparison`, `Cast`, `Logical` (AND/OR), `IsNull`, `In`, `Case`, `Switch`, `Between`, `NullIf`, `Coalesce`, `Row`, `FieldReference`, `Lambda`.

### 4.5 Test data generation
- `TestingPageSourceProvider` returns a single page with one row of null BIGINT values per column. This is sufficient for lifecycle testing. Port this as a trivial connector.
- `TestingSplit` is a marker split with `isRemotelyAccessible=true`. Implement as a simple struct.
- `TestingColumnHandle` wraps a column name, ordinal, and type.

### 4.6 Recommended Rust test prioritization
1. **JSON roundtrip tests for all wire types** (highest value -- validates protocol compatibility)
2. **Task state machine unit tests** (pure logic, no I/O)
3. **REST endpoint integration tests** (actual HTTP, validate headers and content types)
4. **Plan fragment deserialization from captured JSON** (golden files from real coordinator)
5. **Expression evaluation correctness** (can be incremental, starting with constants and comparisons)
