# Module Teardown: Product Tests & Cross-Compatibility (Task 6.6.B) [KG-14]

## 0. Research Focus

This document analyzes Trino's product test infrastructure -- the Docker-based integration test system that exercises complete query execution across a cluster of containers -- with the specific goal of understanding:

1. **Architecture**: How product tests are structured (Testcontainers + Tempto framework, not docker-compose).
2. **Connector coverage**: Which connectors and query types are exercised by product tests.
3. **Docker infrastructure**: How coordinator and worker containers are created, configured, health-checked, and networked.
4. **Compatibility testing**: Whether mixed-version or mixed-implementation tests already exist.
5. **Rust worker injection**: Concrete strategies for replacing a Java worker container with a Rust worker process in the existing test harness.

Source version: Trino 480.

---

## 1. High-Level Overview

Trino's product tests form the highest-level integration test layer. Unlike the unit tests in Phase 6.1-6.5, these tests exercise real query execution across Docker containers with real connectors, real data, and real network communication.

| Component | Technology | Role |
|---|---|---|
| Container orchestration | **Testcontainers** (Java library) | Programmatically creates/manages Docker containers -- no docker-compose files |
| Test framework | **Tempto** (`github.com/trinodb/tempto`) | Test harness for running Java test classes and convention-based SQL tests |
| Container types | `presto-master`, `presto-worker`, `hadoop-master`, `tests`, service containers | Named containers with DNS aliases in a shared Docker network |
| Test organization | **Suites** > **Groups** > **Tests** | Suites select environments + test groups; groups tag individual test methods |
| CI execution | GitHub Actions matrix | `suite run --suite $SUITE --config config-$CONFIG` with impact analysis |

The test runner (the `tests` container) connects to the Trino coordinator via JDBC, never directly to workers. This means the Rust worker only needs to implement the internal Trino worker protocol, not the client-facing protocol.

### Key directories

```
testing/
  trino-product-tests/                      # Test code (Java + SQL convention tests)
    src/main/java/io/trino/tests/product/   # Java test classes
    src/main/resources/sql-tests/            # Convention-based SQL tests + expected results
    src/main/resources/tempto-configuration.yaml  # Database connection configs
  trino-product-tests-launcher/             # Docker/environment infrastructure
    src/main/java/.../launcher/env/         # Environment definitions
    src/main/java/.../launcher/suite/       # Suite definitions
    src/main/resources/docker/              # Shell scripts, configs, health checks
  trino-product-tests-groups/               # Test group constants
```

## 2. Detailed Analysis

### 2.1 Container Architecture (Testcontainers, not docker-compose)

Product tests use the Java Testcontainers library to create Docker containers programmatically. There are no docker-compose files anywhere in the codebase.

#### 2.1.1 Container Name Constants

**File:** `testing/trino-product-tests-launcher/src/main/java/io/trino/tests/product/launcher/env/EnvironmentContainers.java`

```java
public static final String TRINO = "presto";
public static final String COORDINATOR = TRINO + "-master";    // "presto-master"
public static final String WORKER = TRINO + "-worker";          // "presto-worker"
public static final String WORKER_NTH = WORKER + "-";           // "presto-worker-"
public static final String HADOOP = "hadoop-master";
public static final String TESTS = "tests";
```

The legacy `presto` naming is preserved for backward compatibility with Docker image names and DNS aliases.

#### 2.1.2 Container Factory: `Standard.createTrinoContainer()`

**File:** `testing/trino-product-tests-launcher/src/main/java/io/trino/tests/product/launcher/env/common/Standard.java`

This is the single factory method that creates ALL Trino containers (coordinator and worker):

```java
public static DockerContainer createTrinoContainer(
    DockerFiles dockerFiles, File serverPackage, JdkProvider jdkProvider,
    boolean debug, boolean tracing, boolean ipv6,
    String dockerImageName, String logicalName) {

    DockerContainer container = new DockerContainer(dockerImageName, logicalName)
        .withNetworkAliases(logicalName + ".docker.cluster")
        .withFileSystemBind(serverPackage.getPath(), "/docker/trino-server.tar.gz", READ_ONLY)
        .withCommand("/docker/trino-product-tests/run-trino.sh")
        .waitingForAll(
            forLogMessage(".*======== SERVER STARTED ========.*", 1),
            forHealthcheck())
        .withStartupTimeout(Duration.ofMinutes(5));
    // ... JDK, debug, tracing, IPv6 setup ...
}
```

Key observations:
- The server package (a tar.gz) is bind-mounted into the container
- The entry point is `run-trino.sh` (see section 2.2)
- Readiness is detected by both a log message pattern AND a health check
- All containers share a Docker network with DNS aliases like `presto-master.docker.cluster`

The `Standard.extendEnvironment()` method adds the coordinator container and configures default connectors: **blackhole**, **jmx**, **system**, and **tpch**.

#### 2.1.3 Worker Container Setup

**File:** `testing/trino-product-tests-launcher/src/main/java/io/trino/tests/product/launcher/env/common/StandardMultinode.java`

The worker is created by the same `createTrinoContainer()` factory, then configured with worker-specific properties:

```
coordinator=false
http-server.http.port=8081
discovery.uri=http://presto-master:8080
```

**File:** `testing/trino-product-tests-launcher/src/main/resources/docker/trino-product-tests/common/standard-multinode/multinode-worker-config.properties`

Connector catalog files are copied from the coordinator's config directory to the worker, ensuring both see the same set of connectors.

### 2.2 Container Startup: `run-trino.sh`

**File:** `testing/trino-product-tests-launcher/src/main/resources/docker/trino-product-tests/run-trino.sh`

This is the entrypoint script for every Trino container:

```bash
tar xf /docker/trino-server.tar.gz -C /docker
ln -s "$(echo /docker/trino-server-*/bin/launcher | sed 's@/bin/launcher$@@')" /docker/trino-server

if test -d /docker/presto-init.d; then
    for init_script in /docker/presto-init.d/*; do
        "${init_script}"
    done
fi

exec /docker/trino-server/bin/launcher \
  -Dnode.id="${HOSTNAME}" \
  --etc-dir="/docker/trino-product-tests/conf/trino/etc" \
  --data-dir=/var/trino \
  run
```

The init-script mechanism (`/docker/presto-init.d/`) runs arbitrary scripts before server start. Existing init scripts handle: debug agent setup, tracing (OpenTelemetry), and IPv6 configuration.

### 2.3 Health Checks

**File:** `testing/trino-product-tests-launcher/src/main/resources/docker/trino-product-tests/health-checks/trino-health-check.sh`

The health check is Java-specific:

```bash
jps | grep TrinoServer
```

This uses `jps` (Java Process Status) to check for a running `TrinoServer` process. A Rust worker would need a replacement health check.

### 2.4 Test Framework: Tempto

#### 2.4.1 Tempto Configuration

**File:** `testing/trino-product-tests/src/main/resources/tempto-configuration.yaml`

Defines database connections used by tests:
- **trino**: `jdbc:trino://localhost:8080/hive/default` (primary test target)
- **hive**: `jdbc:hive2://hadoop-master:10000` (Hive Metastore at port 9083)
- **mysql**: `jdbc:mysql://mysql:13306/test`
- **postgres**: `jdbc:postgresql://postgresql:15432/test`
- **cassandra**: Host `cassandra:9042`
- **sqlserver**: `jdbc:sqlserver://sqlserver`
- **spark**: `jdbc:hive2://spark:10213`
- **kafka**: Broker at `kafka:9092`, ZooKeeper at `zookeeper:2181`
- **delta/hudi**: Via Spark at `jdbc:hive2://spark:10213`
- **exasol**: `jdbc:exa://exasol:8563`
- **ldap**: `ldap://ldapserver`

User-based connections (`alice@trino`, `bob@trino`, `charlie@trino`, `hdfs@trino`, `CaseSensitiveUserName@trino`) are defined for authorization testing.

#### 2.4.2 Tempto Plugin

**File:** `testing/trino-product-tests/src/main/java/io/trino/tests/product/TrinoTemptoPlugin.java`

Registers:
- **Fulfillers**: `LdapObjectFulfiller` (creates LDAP test data)
- **Suite modules**: `LdapObjectModuleProvider`, `HiveVersionProvider`
- **Table managers**: `KafkaTableManager`
- **Table definitions**: Hive tables in various formats (TEXTFILE, RCFILE, AVRO, ORC, PARQUET) and bucketed/partitioned nation tables

#### 2.4.3 Test Entry Point

**File:** `testing/trino-product-tests/src/main/java/io/trino/tests/product/TemptoProductTestRunner.java`

The `tests` container runs: `java -jar /docker/test.jar --config <tempto-configs>`, scanning `io.trino.tests.product.*` for test classes, excluding the `quarantine` group.

### 2.5 Test Organization: Groups and Suites

#### 2.5.1 Test Groups

**File:** `testing/trino-product-tests-groups/src/main/java/io/trino/tests/product/TestGroups.java`

Approximately 50+ test groups. Key groups relevant to a Rust worker:

| Group | Description |
|---|---|
| `TPCH` | TPC-H benchmark queries |
| `TPCDS` | TPC-DS benchmark queries |
| `HIVE_*` (many variants) | Hive connector: partitioning, bucketing, transactional tables, views, coercion |
| `ICEBERG` | Iceberg connector tests |
| `DELTA_LAKE_*` | Delta Lake connector tests |
| `STORAGE_FORMATS` | Cross-format read/write tests |
| `FAULT_TOLERANT` | Fault-tolerant execution mode |
| `AUTHORIZATION` | Access control tests |
| `CASSANDRA`, `MYSQL`, `SQLSERVER`, `KAFKA`, `POSTGRESQL` | External connector tests |

#### 2.5.2 Suite Structure

Suites combine test groups with environment definitions. The CI runs suites via a matrix:

```yaml
# .github/workflows/ci.yml
suite run --suite ${{ matrix.suite }} --config config-${{ matrix.config }}
```

Each suite selects:
1. An environment (which Docker containers to start)
2. A set of test groups (which tests to run)

Impact analysis in CI can skip irrelevant suites by analyzing which source files changed and which connectors/features they affect.

### 2.6 Convention-Based SQL Tests

Tests can be defined as pure SQL files with expected results, without writing Java code.

#### 2.6.1 SQL Test Structure

**Location:** `testing/trino-product-tests/src/main/resources/sql-tests/testcases/`

Subdirectories: `hive_tpch/` (q01-q22), `tpcds/`, `aggregate/`, `functions/`, `connectors/`, etc.

**Example:** `sql-tests/testcases/hive_tpch/q01.sql`

```sql
-- database: trino; groups: tpch; tables: lineitem
SELECT
  l_returnflag,
  l_linestatus,
  sum(l_quantity)                                       AS sum_qty,
  ...
FROM lineitem
WHERE l_shipdate <= DATE '1998-12-01' - INTERVAL '90' DAY
GROUP BY l_returnflag, l_linestatus
ORDER BY l_returnflag, l_linestatus
```

The comment header (`-- database: trino; groups: tpch; tables: lineitem`) is parsed by Tempto to:
- Route the query to the correct JDBC connection
- Tag the test into groups
- Declare table dependencies (Tempto ensures these are created)

**Expected results** are in corresponding `.result` files with the same name.

#### 2.6.2 Dataset Definitions

**Location:** `testing/trino-product-tests/src/main/resources/sql-tests/datasets/`

Contains `.data` (CSV-like data) and `.ddl` (CREATE TABLE) files for test tables.

#### 2.7.1 TPC-H/TPC-DS Test Utilities

**File:** `testing/trino-product-tests/src/main/java/io/trino/tests/product/TpcTestUtils.java`

Creates TPC datasets by executing `SELECT * FROM tpch.sf1.<table>` into Parquet tables. The `assertResults()` utility compares query output against hardcoded expected values from text files. These utilities are connector-independent and would work with a Rust worker unmodified.

#### 2.7.2 Fault-Tolerant Execution Tests

**File:** `testing/trino-product-tests/src/main/java/io/trino/tests/product/faulttolerant/TestTaskRetriesFilesystemSmoke.java`

```java
@Test(groups = FAULT_TOLERANT)
public void testSimpleQuery() {
    assertThat(onTrino().executeQuery(
        "select count(*) from tpch.sf1.lineitem where orderkey < 100")
        .getOnlyValue())
        .isEqualTo(105L);
}
```

This test validates fault-tolerant execution mode, which is critical for a Rust worker since task retries must work correctly across worker implementations.

### 2.8 Compatibility Testing

#### 2.8.1 `SuiteCompatibility`

**File:** `testing/trino-product-tests-launcher/src/main/java/io/trino/tests/product/launcher/suite/suites/SuiteCompatibility.java`

Tests against the last 5 Trino releases (every 3rd version) plus PrestoSQL 350:

```java
private static final int NUMBER_OF_TESTED_VERSIONS = 5;
private static final int TESTED_VERSIONS_GRANULARITY = 3;
```

This suite uses `EnvSinglenodeCompatibility` to run two Trino instances side-by-side.

#### 2.8.2 `EnvSinglenodeCompatibility`

**File:** `testing/trino-product-tests-launcher/src/main/java/io/trino/tests/product/launcher/env/environment/EnvSinglenodeCompatibility.java`

This is the closest existing pattern to mixed-implementation testing. It:
1. Starts the current Trino version from the source build
2. Starts a second Trino instance from a Docker image (`trinodb/trino:<version>`)
3. Maps configuration directories based on version boundaries (pre-369, pre-359, post-477)
4. Tests run against the current version while the old version provides compatibility data

This environment provides a direct template for mixed-implementation testing with a Rust worker.

### 2.9 CLI and Test Execution Flow

**File:** `testing/trino-product-tests-launcher/src/main/java/io/trino/tests/product/launcher/cli/TestRun.java`

The `test run` CLI command orchestrates the full flow:
1. Creates the environment (Docker containers) from the specified environment provider
2. Configures the `tests` container with the test JAR and Tempto config files
3. Starts all containers
4. Waits for the `tests` container to complete
5. Collects exit code and logs

The test container runs Tempto via: `java -jar /docker/test.jar --config <tempto-configs>`

## 3. Key Design Insights

### 3.1 Tests Never Talk to Workers Directly

All product tests connect to the coordinator via JDBC. The coordinator distributes work to workers through the internal protocol. This means:
- A Rust worker does not need to implement the client-facing Trino protocol
- A Rust worker only needs to implement the coordinator-worker protocol (task creation, heartbeats, data exchange)
- Existing product tests can validate a Rust worker without modification to the test code itself

### 3.2 Testcontainers Provides Programmatic Control

Because environments are defined in Java (not docker-compose), injecting a Rust worker requires modifying Java environment classes. However, the `createTrinoContainer()` factory method is a clean injection point -- a new environment class could create a Rust worker container instead of calling this factory.

### 3.3 The Container Startup Script is the Key Injection Point

`run-trino.sh` is a simple shell script that extracts a tar.gz and runs the Trino launcher. Replacing this with a script that starts a Rust binary is straightforward. The init-script mechanism (`/docker/presto-init.d/`) provides an additional hook point without modifying the main script.

### 3.4 Health Check is Java-Specific

The `jps | grep TrinoServer` health check assumes a JVM process. A Rust worker needs a different health check (e.g., HTTP health endpoint check, or simply checking if the process is running).

### 3.5 Compatibility Environment is a Direct Template

`EnvSinglenodeCompatibility` already runs two different Trino versions side-by-side. This pattern can be adapted to run a Java coordinator + Rust worker. The version-based config mapping logic shows how to handle configuration differences.

### 3.6 Convention Tests are Immediately Reusable

The SQL convention tests (q01-q22 TPC-H, TPC-DS, etc.) with their `.result` files are pure SQL + expected output. They test end-to-end query correctness and require zero modification to validate a Rust worker, since they go through the coordinator's JDBC endpoint.

## 4. Porting Considerations

### 4.1 Strategy: Inject Rust Worker into Existing Docker Infrastructure

The recommended approach for validating a Rust worker is to create a new environment class (e.g., `EnvMultinodeRustWorker`) that:

1. **Reuses the existing coordinator container** from `Standard.createTrinoContainer()` unchanged
2. **Creates a custom worker container** that runs the Rust binary instead of the Java server
3. **Reuses the existing test container** and all Tempto infrastructure unchanged

#### 4.1.1 New Startup Script

Replace `run-trino.sh` with a Rust-specific script:

```bash
#!/bin/bash
# run-rust-worker.sh
# Copy config files that the Rust worker needs
cp /docker/trino-product-tests/conf/trino/etc/catalog/* /rust-worker/catalog/

# Start the Rust worker, pointing at the coordinator for discovery
exec /rust-worker/trino-rust-worker \
  --node-id="${HOSTNAME}" \
  --discovery-uri=http://presto-master:8080 \
  --http-port=8081 \
  --data-dir=/var/trino
```

#### 4.1.2 New Health Check

Replace the JVM-based health check with an HTTP-based check:

```bash
#!/bin/bash
# rust-worker-health-check.sh
curl -sf http://localhost:8081/v1/info > /dev/null
```

Or check the process directly:

```bash
pgrep trino-rust-worker > /dev/null
```

#### 4.1.3 New Environment Class

```java
// EnvMultinodeRustWorker.java (conceptual)
public class EnvMultinodeRustWorker extends EnvironmentProvider {
    @Override
    public void extendEnvironment(Environment.Builder builder) {
        // Coordinator: standard Java Trino, unchanged
        builder.addContainers(standard.createTrinoCoordinator());

        // Worker: Rust binary in a lightweight container
        DockerContainer rustWorker = new DockerContainer("rust-worker-image", WORKER)
            .withNetworkAliases("presto-worker.docker.cluster")
            .withFileSystemBind(rustBinary, "/rust-worker/trino-rust-worker", READ_ONLY)
            .withCommand("/docker/run-rust-worker.sh")
            .waitingFor(forHttp("/v1/info").forPort(8081))
            .withStartupTimeout(Duration.ofMinutes(2));
        builder.addContainer(rustWorker);
    }
}
```

### 4.2 Phased Test Adoption

#### Phase 1: Smoke Tests (Immediate)
- `TPCH` group: TPC-H queries (q01-q22) exercise scan, filter, join, aggregation, sort
- `TestTaskRetriesFilesystemSmoke`: Simple count query validates basic task execution
- These require: task creation, split processing, page serialization, exchange protocol

#### Phase 2: Storage Format Tests
- `STORAGE_FORMATS` group: Validates reading Parquet, ORC, Avro, etc.
- Exercises the connector/page-source path on the worker

#### Phase 3: Connector Tests
- Hive, Iceberg, Delta Lake tests
- Requires the Rust worker to handle connector-specific split types

#### Phase 4: Advanced Features
- `FAULT_TOLERANT`: Task retries, checkpoint/restore
- `AUTHORIZATION`: Access control propagation to workers
- Bucketed/partitioned table tests

### 4.3 Reusable Test Artifacts

The following artifacts can be reused without modification:

| Artifact | Location | Usage |
|---|---|---|
| SQL convention tests | `src/main/resources/sql-tests/testcases/` | End-to-end query validation |
| Expected result files | `src/main/resources/sql-tests/testcases/*.result` | Golden-file comparison |
| Dataset definitions | `src/main/resources/sql-tests/datasets/` | Test data DDL/DML |
| Tempto configuration | `src/main/resources/tempto-configuration.yaml` | JDBC connection setup |
| Connector properties | `src/main/resources/docker/.../catalog/` | Catalog configuration files |
| TPC test utilities | `TpcTestUtils.java` | Dataset creation and result assertion |
| Test group constants | `TestGroups.java` | Group tagging for selective execution |

### 4.4 What Cannot Be Reused Directly

| Component | Reason | Replacement Needed |
|---|---|---|
| `run-trino.sh` | Extracts Java server tar.gz, runs JVM launcher | New script that starts Rust binary |
| `trino-health-check.sh` | Uses `jps` (Java-specific) | HTTP or process-based health check |
| `Standard.createTrinoContainer()` | Hardcoded for Java server package | New factory for Rust worker container |
| JDK provider setup | JVM-specific | Not needed for Rust |
| Debug agent configuration | Java debug agent (`-agentlib:jdwp`) | Rust debugging tools (gdb/lldb) |

### 4.5 Configuration the Rust Worker Must Support

From the multinode worker config:
- `coordinator=false` -- The worker role flag
- `http-server.http.port=8081` -- HTTP port for internal communication
- `discovery.uri=http://presto-master:8080` -- How the worker finds the coordinator
- `node.id=${HOSTNAME}` -- Unique node identifier
- `catalog/*` properties files -- Connector configuration (same format as coordinator)

Additionally, the worker must:
- Register with the coordinator's discovery service
- Respond to `/v1/task` API for task creation/update
- Serve pages via `/v1/task/{taskId}/results/{bufferId}/{token}` for exchange
- Report task status via heartbeats
- Handle the `X-Trino-*` HTTP headers in exchange responses

### 4.6 No Existing External Worker Concept

A search for "external worker", "out of process", "mixed implementation", and "rust worker" across the entire codebase returned zero results. There is no existing concept of a non-Java worker in Trino. The compatibility testing infrastructure (Section 2.8) is the closest analog, but it runs different versions of the same Java implementation, not different implementations.

This means the Rust worker effort is breaking new ground. However, the clean separation between the test harness (which talks to the coordinator via JDBC) and the internal cluster protocol (which the worker implements) makes it architecturally feasible to inject a Rust worker without modifying any test logic.
