# 6.6.A DistributedQueryRunner & Query Assertions

## Research Focus

How Trino's integration test infrastructure works end-to-end: `DistributedQueryRunner` spins up in-process coordinator + workers, test base classes provide SQL assertion primitives, and `QueryAssertions` validates results against an H2 oracle. Critical question: can the Rust worker be plugged into this infrastructure, or does it require a separate test harness?

## High-Level Overview

Trino's test infrastructure is a layered stack:

1. **DistributedQueryRunner** -- creates a full Trino cluster in a single JVM. A coordinator `TestingTrinoServer` plus N worker `TestingTrinoServer` instances (default 2), connected via HTTP discovery. Each server is a real, Guice-bootstrapped Trino node.

2. **Test base class hierarchy** -- `AbstractTestQueryFramework` -> `AbstractTestQueries` -> `BaseConnectorTest` -> connector-specific tests. Each layer adds assertions, required TPC-H tables, and test methods.

3. **QueryAssertions** -- static utility that runs the query on Trino, runs the expected SQL on an in-memory H2 database pre-loaded with TPC-H data, then compares `MaterializedResult` objects.

4. **Worker discovery protocol** -- Workers announce themselves to the coordinator via `POST /v1/announce` (plain text body = worker's URI). Coordinator polls each worker's `GET /v1/info` to get `ServerInfo` JSON (node ID, state, version, environment, coordinator flag). This is a simple, HTTP-based protocol with no Java-specific coupling.

## Detailed Analysis

### DistributedQueryRunner Architecture

**Source**: `trino/testing/trino-testing/src/main/java/io/trino/testing/DistributedQueryRunner.java`

`DistributedQueryRunner` implements the `QueryRunner` interface and is the primary test harness for distributed Trino tests. It creates a multi-node Trino cluster entirely within a single JVM process.

**Construction flow**:
1. Creates a coordinator `TestingTrinoServer` with `node-scheduler.include-coordinator=true`
2. Creates N worker `TestingTrinoServer` instances (default 2), each configured with `discovery.uri` pointing to the coordinator
3. Calls `refreshNodes(true)` to wait until all workers are registered and active
4. All nodes share the same JVM but use separate class loaders, thread pools, and HTTP server ports

**Builder pattern**:
```java
TpchQueryRunner.builder()
    .setWorkerCount(3)
    .setExtraProperties(Map.of("task.concurrency", "8"))
    .setCoordinatorProperties(Map.of("http-server.http.port", "8080"))
    .withProtocolSpooling("json+zstd")
    .build();
```

**Dynamic worker management**:
- `addServers(int count)` -- adds workers at runtime
- `removeWorker()` -- stops and removes a worker
- `restartWorker(TestingTrinoServer)` -- simulates pod restart: stops old server, creates new one on same port with fresh instance ID. Used by `TestWorkerRestart` to verify cluster resilience.

**TestingTrinoServer** (`trino/core/trino-main/src/main/java/io/trino/server/testing/TestingTrinoServer.java`):
Each node is a full Guice-bootstrapped server using airlift's `Bootstrap`. Modules include `ServerMainModule`, `NodeManagerModule`, `TestingHttpServerModule`, etc. Key defaults for tests: `task.concurrency=4`, `task.max-worker-threads=4`. After construction, `Announcer.start()` is called to begin service discovery announcements.

### Worker Discovery Protocol

This is the most architecturally important finding for the Rust worker project. The protocol is entirely HTTP-based with no Java coupling:

**Step 1 -- Worker announces itself**:
- Worker sends `POST /v1/announce` to coordinator(s)
- Request body: plain text containing the worker's URI (e.g., `http://localhost:12345`)
- Header: `Content-Type: text/plain`
- Security: `INTERNAL_ONLY` access type (requires shared secret via `internal-communication.shared-secret`)
- Repeated every 5 seconds via `AnnounceNodeAnnouncer`

**Step 2 -- Coordinator maintains inventory**:
- `AnnounceNodeInventory` stores URIs in a cache with 30-second TTL
- If a worker stops announcing, it expires from the inventory after 30 seconds
- `CoordinatorNodeManager.refreshNodes()` polls every 5 seconds

**Step 3 -- Coordinator polls worker state**:
- For each URI in inventory, coordinator sends `GET /v1/info` to the worker
- Worker must respond with `ServerInfo` JSON:
  ```json
  {
    "nodeId": "unique-uuid",
    "state": "ACTIVE",
    "nodeVersion": {"version": "480"},
    "environment": "testenv",
    "coordinator": false,
    "coordinatorId": null,
    "starting": false,
    "uptime": "10.00s"
  }
  ```
- Coordinator validates: environment must match, nodeVersion must match
- States: ACTIVE, INACTIVE, DRAINING, DRAINED, SHUTTING_DOWN, INVALID, GONE

**Source files**:
- `AnnounceNodeAnnouncer.java` -- worker side, sends POST
- `AnnounceNodeResource.java` -- coordinator side, receives POST at `/v1/announce`
- `AnnounceNodeInventory.java` -- stores announced URIs with 30s TTL
- `RemoteNodeState.java` -- coordinator side, polls `GET /v1/info` on each worker
- `CoordinatorNodeManager.java` -- aggregates all node states, notifies listeners
- `ServerInfo.java` -- record type for the `/v1/info` response

### Test Base Class Hierarchy

**AbstractTestQueryFramework** (`trino/testing/trino-testing/src/main/java/io/trino/testing/AbstractTestQueryFramework.java`):
- Uses `@TestInstance(PER_CLASS)` and `@Execution(CONCURRENT)` -- single runner instance, tests run in parallel
- `@BeforeAll init()`: calls abstract `createQueryRunner()`, creates `H2QueryRunner`, creates `QueryAssertions`
- `@AfterAll close()`: runs post-test health checks -- `checkQueryMemoryReleased()`, `checkQueryInfosFinal()`, `checkTasksDone()`. These verify no memory leaks and all tasks completed cleanly.
- Provides convenience methods: `assertQuery()`, `assertQueryOrdered()`, `assertUpdate()`, `assertQueryFails()`, `assertExplain()`, `computeActual()`, `computeScalar()`

**AbstractTestQueries** (`trino/testing/trino-testing/src/main/java/io/trino/testing/AbstractTestQueries.java`):
- Extends `AbstractTestQueryFramework`
- Defines `REQUIRED_TPCH_TABLES = [NATION, ORDERS, REGION]`
- Contains hundreds of SQL correctness tests: aggregations, distinct, limit, joins, IN clauses, comparisons, subqueries, CASE expressions, etc.

**BaseConnectorTest** (`trino/testing/trino-testing/src/main/java/io/trino/testing/BaseConnectorTest.java`):
- Extends `AbstractTestQueries`, adds CUSTOMER to required tables
- Introduces `hasBehavior(TestingConnectorBehavior)` pattern for conditional test execution
- Connector tests override `hasBehavior()` to disable tests for unsupported features (e.g., TPCH connector disables SUPPORTS_INSERT, SUPPORTS_DELETE, etc.)
- Installs `MockConnectorPlugin` in `@BeforeAll initMockCatalog()`
- Extended by: Iceberg, Hive, Memory, TPCH, Delta Lake, MongoDB, Elasticsearch, etc.

### QueryAssertions and Result Comparison

**Source**: `trino/testing/trino-testing/src/main/java/io/trino/testing/QueryAssertions.java`

The core assertion flow:
1. Execute the "actual" query on DistributedQueryRunner -> get `MaterializedResult`
2. Execute the "expected" query on H2QueryRunner -> get `MaterializedResult`
3. Compare the two results

**Two execution paths**:
- If runner is `DistributedQueryRunner`: uses `assertDistributedQuery()` which also captures `QueryId` and optionally validates the `Plan`
- Otherwise: uses generic path via `runner.execute()`

**Comparison modes**:
- **Ordered** (`assertQueryOrdered`): compares row lists directly via `MaterializedResult.equals()`
- **Unordered** (`assertQuery`): converts each result to `ImmutableMultiset<MaterializedRow>`, then computes symmetric difference for mismatch reporting
- **Contains** (`assertContains`): checks every expected row exists in actual results
- **Update** (`assertUpdate`): validates update count matches expected value

**MaterializedResult** (`trino/core/trino-main/src/main/java/io/trino/testing/MaterializedResult.java`):
- Holds: `List<MaterializedRow> rows`, `List<Type> types`, `List<String> columnNames`, `Optional<String> updateType`, `OptionalLong updateCount`
- `toTestTypes()` normalizes internal Trino types to standard Java types: `SqlDate -> LocalDate`, `SqlTimestamp -> LocalDateTime`, `SqlDecimal -> BigDecimal`, etc.
- `resultBuilder()` factory for constructing results programmatically

**H2QueryRunner** (`trino/testing/trino-testing/src/main/java/io/trino/testing/H2QueryRunner.java`):
- In-memory H2 database pre-loaded with TPC-H tables (orders, lineitem, nation, region, part, customer, supplier, partsupp) from `TpchRecordSet`
- Serves as the "oracle" -- correct reference implementation for SQL semantics

**TestingTrinoClient** (`trino/testing/trino-testing/src/main/java/io/trino/testing/TestingTrinoClient.java`):
- Extends `AbstractTestingTrinoClient<MaterializedResult>`
- `MaterializedResultSession` converts wire protocol results (JSON from HTTP) into `MaterializedResult` with proper type conversions
- Handles all Trino types: primitives, decimals, dates/times with zones, intervals, arrays, maps, rows, UUID, IP addresses, variants, etc.

### TPC-H / TPC-DS Query Usage

- `AbstractTestQueries` requires NATION, ORDERS, REGION tables from TPC-H
- `BaseConnectorTest` adds CUSTOMER
- `QueryAssertions.copyTpchTables()` copies TPC-H data into test catalogs
- TPC-H "tiny" scale factor used by default (e.g., orders has 15,000 rows, lineitem has 60,175 rows)
- `TpchQueryRunner` installs `TpchPlugin` and sets schema to "tiny"
- No TPC-DS usage found in the core test infrastructure; TPC-DS benchmarks exist separately

### Concrete Test Examples

**TestTpchConnectorTest** (`trino/testing/trino-tests/src/test/java/io/trino/tests/tpch/TestTpchConnectorTest.java`):
- Extends `BaseConnectorTest`
- `createQueryRunner()` returns `TpchQueryRunner.builder().build()` -- 1 coordinator + 2 workers with TPC-H plugin
- Disables write behaviors: SUPPORTS_INSERT, SUPPORTS_DELETE, SUPPORTS_CREATE_TABLE, etc.
- Custom tests: IO explain validation, analyze, pre-sorted input optimization, predicate pushdown

**TestWorkerRestart** (`trino/testing/trino-tests/src/test/java/io/trino/tests/TestWorkerRestart.java`):
- Tests three scenarios: restart before query, restart during query, worker start during query
- Uses `DistributedQueryRunner.restartWorker()` to simulate failures
- Validates that the cluster recovers and subsequent queries succeed

**TestDistributedEngineOnlyQueries** (`trino/testing/trino-tests/src/test/java/io/trino/tests/TestDistributedEngineOnlyQueries.java`):
- Uses `MemoryQueryRunner` with TPC-H data copied in
- Installs `MockConnectorPlugin` for catalog property testing
- Extends `AbstractDistributedEngineOnlyQueries` for tests that only make sense in distributed mode

## Key Design Insights

1. **Discovery protocol is HTTP-only and language-agnostic**. Worker registration is `POST /v1/announce` with plain text URI body. State reporting is `GET /v1/info` returning `ServerInfo` JSON. No Java serialization, no RPC framework, no Guice coupling. A Rust process can implement these two endpoints and be recognized as a Trino worker.

2. **Environment and version validation is strict**. The coordinator checks `ServerInfo.environment` and `ServerInfo.nodeVersion` match its own values. The Rust worker must report the exact same environment string and version (`NodeVersion`) as the coordinator, or it will be marked INVALID and never receive work.

3. **30-second TTL with 5-second heartbeat**. Workers must re-announce every 5 seconds. If the coordinator does not see an announcement for 30 seconds, the worker is evicted. The coordinator also polls `/v1/info` every 5 seconds. If the HTTP connection fails with `ConnectException`, the worker is marked GONE (not just INACTIVE).

4. **Internal-only security on announce endpoint**. The `POST /v1/announce` endpoint uses `INTERNAL_ONLY` access type, which requires the `internal-communication.shared-secret` to match. The Rust worker must include this shared secret in its HTTP requests.

5. **DistributedQueryRunner is fundamentally in-process**. All workers are `TestingTrinoServer` instances created via `new TestingTrinoServer(...)` within the same JVM. There is no `addExternalWorker(URI)` method. The `restartWorker()` method creates a new Java server instance, not an arbitrary external process.

6. **Post-test health checks are Java-internal**. `AbstractTestQueryFramework.close()` calls `checkQueryMemoryReleased()`, `checkQueryInfosFinal()`, and `checkTasksDone()` which inspect the coordinator's `QueryManager` and `TaskManager` directly via Java object references. These checks verify internal state that an external worker cannot satisfy in the same way.

7. **The H2 oracle pattern is reusable regardless of worker implementation**. The assertion framework compares Trino query results (via HTTP client) against H2 expected results. The Trino client protocol between test and coordinator is standard HTTP/JSON. If the Rust worker correctly processes tasks, the coordinator will return correct results through the normal client protocol, and H2 comparisons will pass.

8. **MaterializedResult comparison is type-aware but normalized**. `toTestTypes()` converts all internal Trino types to standard Java types before comparison. The comparison itself uses `ImmutableMultiset` for unordered checks, providing clear diff output on failure.

## Porting Considerations

### Can the Rust Worker Use Trino's Existing Test Infrastructure?

**Partially yes, but not directly within DistributedQueryRunner.**

The most viable approach is a **hybrid external test harness**:

1. **Start a real Trino coordinator** (Java) using `TpchQueryRunner.main()` or similar standalone launcher. This is already supported -- `TpchQueryRunner` has a `main()` method that starts a coordinator on port 8080.

2. **Start the Rust worker as a separate process** that:
   - Periodically POSTs its URI to `http://coordinator:8080/v1/announce` (plain text, with shared secret header)
   - Serves `GET /v1/info` returning `ServerInfo` JSON with matching environment and version
   - Implements the task execution endpoints (`/v1/task/...`) that the coordinator will call

3. **Run SQL queries against the coordinator** via standard Trino client protocol and compare results against expected values (either H2 or pre-computed).

### What the Rust Worker Must Implement for Discovery

Minimum viable HTTP endpoints:

| Endpoint | Method | Purpose | Request/Response |
|----------|--------|---------|-----------------|
| `POST /v1/announce` | Client-side | Register with coordinator | Body: worker URI as plain text |
| `GET /v1/info` | Server-side | Report node state | Response: `ServerInfo` JSON |

`ServerInfo` fields the coordinator checks:
- `nodeId` -- unique identifier (UUID string)
- `state` -- must be `"ACTIVE"` for the coordinator to schedule work
- `nodeVersion.version` -- must match coordinator's version exactly
- `environment` -- must match coordinator's environment string exactly
- `coordinator` -- must be `false` for a worker

### Reusing vs. Rebuilding the Test Harness

**Reusable from Java side**:
- `TpchQueryRunner` as standalone coordinator launcher
- H2 oracle for expected results (could be a separate Java process or replaced with pre-computed expected results)
- The Trino JDBC driver or REST client for submitting queries

**Must be rebuilt for Rust**:
- Worker lifecycle management (start/stop Rust binary)
- Health check polling (wait for Rust worker to become ACTIVE)
- Assertion logic (can be simple row comparison in any language)

**Not reusable**:
- `DistributedQueryRunner` -- hardcoded to create Java `TestingTrinoServer` instances
- `AbstractTestQueryFramework` post-test checks -- inspect Java-internal state
- `BaseConnectorTest` conditional behavior system -- assumes Java connector plugins

### Recommended Test Strategy for Phase 1

1. Write a Rust test binary that starts the Rust worker, starts a Trino coordinator (via Java subprocess), waits for registration, runs TPC-H queries via the Trino REST API, and compares results against known-good values.

2. Start with `SELECT * FROM tpch.tiny.nation` (21 rows, simple scan) and progressively add more complex queries from `AbstractTestQueries`.

3. Use `DistributedQueryRunner` with `setWorkerCount(0)` and `node-scheduler.include-coordinator=false` to create a coordinator that will not execute queries itself -- forcing all work to the Rust worker. This confirms the Rust worker is actually processing tasks.

4. For regression testing, capture expected results from a Java-only Trino run and use them as golden files, eliminating the H2 dependency.
