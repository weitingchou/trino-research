# Module Teardown: Test Hierarchy & Organization (Task 6.1.A) [KG-13]

## 0. Research Focus

This document catalogs Trino 480's entire test infrastructure: the test categories (unit, integration, product, benchmark), the frameworks used, how tests are selectively run through Maven profiles and CI matrix partitioning, the directory conventions, and the GitHub Actions CI pipeline that orchestrates everything. The goal is to inform the testing strategy for the Rust-based worker replacement.

**Source version:** Trino 481-SNAPSHOT (tag 479, building toward 480)
**Key files examined:**
- Root `pom.xml` -- surefire configuration, Maven profiles (`gib`, `errorprone-compiler`)
- `.github/workflows/ci.yml` -- the single CI workflow (1134 lines)
- `testing/` directory -- 16 testing-related modules
- `core/trino-main/pom.xml` -- JMH benchmark dependencies
- `docs/src/main/sphinx/develop/tests.md` -- official test writing guidelines
- Connector `pom.xml` files (delta-lake, hive, iceberg, faulttolerant-tests) -- profile definitions
- `testing/trino-testing/` -- the `AbstractTestQueryFramework` class hierarchy

## 1. High-Level Overview

Trino's test infrastructure is organized into **four distinct tiers**, each with its own execution model, framework, and CI pipeline:

| Tier | Where | Framework | Runner | Approximate Scale |
|------|-------|-----------|--------|-------------------|
| **Unit tests** | `src/test/java/` in each module | JUnit 5 + AssertJ | `maven-surefire-plugin` | Thousands of tests across ~80 modules |
| **Integration tests (connector)** | `src/test/java/` in plugin modules | JUnit 5 + Testcontainers + `DistributedQueryRunner` | `maven-surefire-plugin` with Maven profiles | ~50 connector modules, each with Docker-based external systems |
| **Product tests** | `testing/trino-product-tests/` | Tempto (TestNG-based) + Docker Compose | Custom `trino-product-tests-launcher` | ~38 test suites, each with Docker environments |
| **Benchmarks** | `src/test/java/` (Benchmark*.java) | JMH (OpenJDK) | Included by surefire but intended for manual/benchto runs | ~20+ benchmark classes in trino-main alone |

**Key architectural insight:** Despite having four tiers, Trino heavily favors unit tests. The official docs explicitly state "Tests must be written as unit tests when possible" and "Avoid product tests." The project is actively migrating away from product tests toward unit tests backed by Testcontainers.

## 2. Detailed Analysis

### 2.1 Test Frameworks and Dependencies

#### 2.1.1 JUnit 5 (Primary Framework)

All new tests **must** use JUnit 5 (`org.junit.jupiter`). This is mandated in the official test guidelines at `docs/src/main/sphinx/develop/tests.md`.

Key JUnit 5 features used:

- **`@TestInstance(PER_CLASS)`** -- The standard lifecycle mode for framework tests. `AbstractTestQueryFramework` (the base class for connector tests) uses this to share a `QueryRunner` across all test methods in a class.

- **`@Execution(CONCURRENT)`** -- Test methods within a class run concurrently. This is the default set globally via surefire system properties.

- **Parallel execution** -- Configured globally in the root `pom.xml`:
  ```xml
  <!-- From pom.xml, maven-surefire-plugin configuration -->
  <junit.jupiter.execution.parallel.enabled>true</junit.jupiter.execution.parallel.enabled>
  <junit.jupiter.execution.parallel.mode.default>concurrent</junit.jupiter.execution.parallel.mode.default>
  <junit.jupiter.execution.parallel.mode.classes.default>concurrent</junit.jupiter.execution.parallel.mode.classes.default>
  ```
  This means both test methods AND test classes run concurrently by default. Some modules (faulttolerant-tests, redshift) override to `fixed` parallelism strategy with `parallelism=4`.

- **`@Timeout`** -- Used extensively for tests involving distributed queries or memory management (e.g., `@Timeout(240)` on memory tests, `@Timeout(60)` on system connector tests).

- **`@RepeatedTest`** -- Used selectively for reliability verification.

- **`@Disabled`** -- Officially discouraged: "Prefer to remove a test instead of disabling it."

- **Auto-detection of extensions** -- Enabled globally: `junit.jupiter.extensions.autodetection.enabled=true`

- **Timeout thread mode** -- `junit.jupiter.execution.timeout.thread.mode.default=SEPARATE_THREAD` -- timeouts run on a separate thread to avoid blocking the test executor.

#### 2.1.2 AssertJ (Primary Assertion Library)

All tests use statically imported AssertJ assertions (`org.assertj.core.api.Assertions`). The test guidelines mandate this explicitly.

#### 2.1.3 Tempto (Product Test Framework)

Product tests use `io.trino.tempto` -- Trino's own test framework built on TestNG. It provides:
- `ProductTest` base class
- `@Requires` annotations for environment prerequisites
- `@Groups` annotations for test categorization (maps to `TestGroups` constants)
- `TemptoRunner` for test execution
- Built-in HDFS, Kafka, LDAP fulfillment

The `TemptoProductTestRunner` excludes the "quarantine" group by default.

#### 2.1.4 JMH (Benchmark Framework)

JMH (`org.openjdk.jmh`) is used for microbenchmarks. Benchmark files live alongside regular tests in `src/test/java/` with the `Benchmark` prefix. The surefire plugin includes `**/Benchmark*.java` but excludes JMH-generated code (`**/*jmhTest*.java`, `**/*jmhType*.java`).

JMH benchmarks use standard annotations: `@Benchmark`, `@BenchmarkMode`, `@Fork`, `@Measurement`, `@Warmup`, `@Param`, `@State(Scope.Thread)`.

Each benchmark class also contains a JUnit `@Test` method that invokes the JMH runner, so benchmarks are runnable as regular tests (though they run a single abbreviated iteration, not a proper benchmark).

#### 2.1.5 Testcontainers

`org.testcontainers:testcontainers` is a key dependency for connector integration tests. The `trino-testing-containers` module wraps Testcontainers for Trino-specific containers. The CI configures `TESTCONTAINERS_PULL_PAUSE_TIMEOUT=600` to handle slow image pulls.

#### 2.1.6 Custom Testing Services (`trino-testing-services`)

The `trino-testing-services` module provides cross-cutting test infrastructure:

- **`LogTestDurationListener`** -- A JUnit `TestExecutionListener` (auto-discovered via SPI) that logs test durations and detects hung tests. Enabled automatically in CI (`CONTINUOUS_INTEGRATION=true`) and detects tests idle for >8 minutes.

- **`@Flaky` annotation** (legacy, TestNG only) -- Marks tests that may fail intermittently. Requires a GitHub issue reference and a regex match pattern. **Deprecated** -- new tests cannot use it since new tests must use JUnit 5.

- **`FlakyTestRetryAnalyzer`** -- TestNG retry logic for `@Flaky` tests.

#### 2.1.7 Airlift Testing Utilities

`io.airlift:testing` and `io.airlift:junit-extensions` provide additional test utilities used throughout.

### 2.2 Test Class Hierarchy (The `AbstractTestQueryFramework` Tree)

The connector test framework is built on a deep inheritance hierarchy rooted in `AbstractTestQueryFramework`:

```
AbstractTestQueryFramework                    (testing/trino-testing/)
  |-- BaseConnectorSmokeTest                   (minimal connector validation)
  |-- BaseConnectorTest                        (comprehensive connector test suite)
  |     |-- extends BaseConnectorSmokeTest behavior via composition
  |-- BaseFailureRecoveryTest                  (fault-tolerant execution tests)
  |     |-- ExtendedFailureRecoveryTest
  |-- AbstractTestQueries                      (query correctness against H2)
  |     |-- AbstractTestEngineOnlyQueries
  |     |     |-- AbstractDistributedEngineOnlyQueries
  |-- AbstractTestJoinQueries
  |-- AbstractTestOrderByQueries
  |-- AbstractTestAggregations
  |-- AbstractTestWindowQueries
  |-- BaseDynamicPartitionPruningTest
  |-- AbstractTestDynamicRowFiltering
  |-- AbstractTestFaultTolerantExecution{Aggregations,JoinQueries,OrderByQueries,WindowQueries}
```

**How it works:**
1. `AbstractTestQueryFramework.init()` (`@BeforeAll`) creates a `QueryRunner` + `H2QueryRunner`
2. Subclasses implement `createQueryRunner()` to provide a connector-specific runner
3. `close()` (`@AfterAll`) verifies no leaked memory, all queries final, all tasks done
4. The base class provides `assertQuery()`, `assertUpdate()`, `computeActual()` utilities

**`DistributedQueryRunner`** is the in-process Trino cluster used by most tests:
- Starts a coordinator + configurable number of workers
- Uses `TestingTrinoServer` (real Trino server with in-process discovery)
- Supports OpenTelemetry span collection for test observability
- Supports failure injection, exchange spooling, and custom Guice modules

**`TestingConnectorBehavior`** is an enum-based capability matrix:
- Connectors declare supported behaviors (e.g., `SUPPORTS_INSERT`, `SUPPORTS_PREDICATE_PUSHDOWN`)
- `BaseConnectorTest`/`BaseConnectorSmokeTest` use `hasBehavior()` to skip inapplicable tests
- This is Trino's approach to parameterizing connector tests without data providers

### 2.3 Maven Surefire Configuration

**Root `pom.xml` surefire settings:**

```xml
<includes>
    <include>**/Test*.java</include>     <!-- Primary: TestFoo.java -->
    <include>**/*Test.java</include>     <!-- Also: FooTest.java -->
    <include>**/Benchmark*.java</include> <!-- JMH benchmarks -->
</includes>
<excludes>
    <exclude>**/*jmhTest*.java</exclude>  <!-- JMH generated code -->
    <exclude>**/*jmhType*.java</exclude>
</excludes>
```

**Convention:** Test class names **must** start with `Test` (e.g., `TestExample`). The `**/*Test.java` pattern is a fallback for incorrectly named tests.

**JVM settings for tests:**
- `-Xmx3g` heap
- `-XX:G1HeapRegionSize=32M` (to handle large byte[] allocations from connector tests)
- `--sun-misc-unsafe-memory-access=allow`
- Timezone forced to `America/Bahia_Banderas` (has DST transitions for datetime testing)

### 2.4 Maven Profiles for Test Selection

Trino uses **Maven profiles** to partition tests within a module. There are no `@Tag` or `@Category` annotations -- all selective execution is via surefire include/exclude patterns in profiles.

#### Common Profile Patterns

| Profile | Used By | Purpose |
|---------|---------|---------|
| `default` | trino-faulttolerant-tests | Runs all tests except connector-specific FTE tests |
| `cloud-tests` | delta-lake, iceberg, filesystem-{s3,azure,gcs}, bigquery, redshift, snowflake | Tests requiring real cloud credentials |
| `cloud-tests-1` / `cloud-tests-2` | bigquery | Split cloud tests into two CI jobs |
| `fte-tests` | delta-lake, hive, iceberg, redshift | Fault-tolerant execution recovery tests |
| `test-parquet` | hive | Parquet-specific reader tests |
| `minio-and-avro` | iceberg | MinIO + Avro format tests |
| `test-fault-tolerant-{hive,delta,iceberg}` | trino-faulttolerant-tests | Connector-specific FTE tests |
| `test-isolated-jvm-suites` | trino-hdfs | Tests requiring JVM isolation (`reuseForks=false`) |
| `aws-tests` | hive | Tests requiring AWS credentials |
| `errorprone-compiler` | root | Runs Error Prone static analysis (not tests) |
| `gib` | root | Git-flow Incremental Builder for change detection |
| `ci` | root | CI-specific checks (checkstyle, etc.) |

**Example from `trino-faulttolerant-tests/pom.xml`:**
```xml
<profile>
    <id>test-fault-tolerant-hive</id>
    <build><plugins><plugin>
        <artifactId>maven-surefire-plugin</artifactId>
        <configuration>
            <includes>
                <include>**/io/trino/faulttolerant/hive/Test*FaultTolerant*.java</include>
            </includes>
        </configuration>
    </plugin></plugins></build>
</profile>
```

### 2.5 Product Test Architecture

Product tests are a multi-module system:

```
testing/trino-product-tests-groups/     -- TestGroups constants (80+ group names)
testing/trino-product-tests/            -- Tempto-based test code
testing/trino-product-tests-launcher/   -- Java launcher with Suite classes
```

#### Test Groups (`TestGroups.java`)

Over 80 string constants define test groups: `TPCH`, `TPCDS`, `FUNCTIONS`, `CLI`, `JDBC`, `HIVE_TRANSACTIONAL`, `ICEBERG`, `DELTA_LAKE_DATABRICKS`, `KAFKA`, `CASSANDRA`, `LDAP`, `OAUTH2`, etc.

Tests are tagged with these groups via Tempto's `@Groups` annotation.

#### Suite System

38 concrete `Suite` classes in `trino-product-tests-launcher` define which groups run on which Docker environments:

```
Suite1, Suite2, Suite3, Suite5, Suite6NonGeneric, Suite7NonGeneric
SuiteAllConnectorsSmoke, SuiteAzure, SuiteCassandra, SuiteClickhouse
SuiteClients, SuiteCompatibility, SuiteDeltaLakeDatabricks{133,143,154,164,173}
SuiteDeltaLakeOss, SuiteExasol, SuiteFunctions, SuiteGcs
SuiteHive4, SuiteHiveTransactional, SuiteHmsOnly, SuiteHudi
SuiteIceberg, SuiteIgnite, SuiteKafka, SuiteLdap, SuiteLoki
SuiteMysql, SuiteOauth2, SuiteParquet, SuiteRanger, SuiteSnowflake
SuiteStorageFormatsDetailed, SuiteTpcds, SuiteTpch
```

Each suite specifies an environment class (e.g., `EnvMultinode`) and optionally includes/excludes specific test groups.

#### Launcher Execution

The product test launcher is a self-contained executable JAR:
```bash
testing/trino-product-tests-launcher/target/trino-product-tests-launcher-*-executable.jar \
    suite run --suite suite-1 --config config-default --timeout 2h
```

### 2.6 Benchmark Infrastructure

Two separate benchmark systems exist:

1. **JMH microbenchmarks** -- `Benchmark*.java` files in `src/test/java/`, using `org.openjdk.jmh`. Included in surefire but run as quick validation (1 iteration). Full benchmark runs are manual.

2. **Benchto system** -- `testing/trino-benchto-benchmarks/` uses the Benchto driver for macro-level TPC-H/TPC-DS benchmarks against a running Trino cluster. `testing/trino-benchmark-queries/` contains the SQL queries.

### 2.7 CI Infrastructure (`.github/workflows/ci.yml`)

The CI is a single GitHub Actions workflow with **12 jobs** (plus a meta `build-success` gate):

#### Job Breakdown

| Job | Trigger | Timeout | What it does |
|-----|---------|---------|--------------|
| `path-filters` | Always | - | Determines if changes are docs-only vs. code |
| `maven-checks` | Always | 45m | Checkstyle, SPI compatibility; matrix: Java 25, 26-ea, 27-ea |
| `artifact-checks` | Non-docs | 45m | JDBC shading test, Docker image build (multi-arch) |
| `check-commits-dispatcher` | PR only | - | Prepares per-commit compilation matrix |
| `check-commit` | PR only | 20m | Verifies each PR commit compiles independently |
| `error-prone-checks` | Non-docs | 45m | Error Prone static analysis on impacted modules |
| `test-jdbc-compatibility` | Non-docs | 30m | Old JDBC driver vs. new server, and vice versa |
| `hive-tests` | Non-docs | 60m | Hive AWS tests (S3); matrix: config-hdp3 |
| `test-other-modules` | Non-docs | 60m | Tests for ~30 smaller modules not in the matrix |
| `build-test-matrix` + `test` | Non-docs | 60m | **Main test matrix** -- ~50 entries with profiles |
| `build-pt` + `pt` | Non-docs | 130m | **Product test matrix** -- ~38 suites with configs |
| `build-success` | Always | - | Meta-job: gates merge on all jobs passing |

#### Test Matrix Construction

The test matrix is **dynamically built** using Git-flow Incremental Builder (GIB):

1. `build-test-matrix` runs `$MAVEN validate` with GIB to detect impacted modules
2. A Python script (`.github/bin/build-matrix-from-impacted.py`) cross-references impacted modules against a hardcoded YAML matrix definition
3. Only impacted modules' test entries are included in the matrix

The full matrix YAML (embedded in `ci.yml`) defines ~50 entries like:
```yaml
- { modules: core/trino-main }
- { modules: plugin/trino-hive, profile: fte-tests }
- { modules: plugin/trino-iceberg, profile: cloud-tests }
- { modules: testing/trino-faulttolerant-tests, profile: test-fault-tolerant-hive }
```

#### GIB (Git-flow Incremental Builder)

The `gib` Maven profile activates `io.github.gitflow-incremental-builder:gitflow-incremental-builder:4.6.0` which:
- Compares current branch to the merge base of the target branch
- Determines which Maven modules are impacted by the diff
- Skips building/testing unaffected modules
- On `master`, GIB is disabled (all modules run)

#### Product Test Matrix Construction

Similar to the test matrix but uses a different Python script (`.github/bin/build-pt-matrix-from-impacted-connectors.py`) and the `trino-plugin-reader` tool to map impacted modules to product test features.

Product tests that require cloud credentials (Azure, GCS, Databricks, Snowflake) are conditionally excluded when secrets are unavailable.

#### CI Labels for Test Control

PR labels control test scope:
- `tests:all` -- Run all unit tests and all product tests
- `tests:all-product` / `product-tests:all` -- Run all product tests

#### Test Result Processing

The `.github/actions/process-test-results` composite action:
1. Uploads surefire XML reports as artifacts
2. Uploads TestNG results from product tests
3. Uses `trinodb/github-actions/action-surefire-report` for CI annotations
4. Optionally uploads heap dumps (`.hprof`) on failure

#### CI Self-Testing

`TestCiWorkflow.java` in `testing/trino-tests/` parses the `ci.yml` YAML and verifies:
- All test steps have proper IDs
- The `build-success` job lists all required jobs
- Upload test results conditions reference all test step outcomes

### 2.8 Directory Structure Convention

Standard Maven layout with no deviation:
```
module/
  src/
    main/java/io/trino/{module}/    -- Production code
    test/java/io/trino/{module}/    -- Test code (unit + integration)
    test/resources/                  -- Test resources
```

The `testing/` top-level directory contains 16 modules:

| Module | Purpose |
|--------|---------|
| `trino-testing` | Core test framework (`AbstractTestQueryFramework`, `DistributedQueryRunner`, `BaseConnectorTest`, etc.) -- shipped as a normal dependency, not test-scoped |

| `trino-testing-containers` | Testcontainers wrappers for Trino-specific Docker images |
| `trino-testing-kafka` | Kafka test utilities |
| `trino-testing-resources` | Shared test resource files |
| `trino-tests` | Integration tests for Trino core (distributed queries, memory, scheduling) |
| `trino-faulttolerant-tests` | Fault-tolerant execution tests across connectors |
| `trino-product-tests` | Tempto-based product test code |
| `trino-product-tests-groups` | `TestGroups` and `SuiteGroups` constants |
| `trino-product-tests-launcher` | Docker-based product test launcher with Suite classes |
| `trino-benchmark-queries` | SQL files for benchmarks |
| `trino-benchto-benchmarks` | Benchto macro-benchmark driver |

This is because `BaseConnectorTest`, `DistributedQueryRunner`, etc. must be available at compile time for connector test classes.

## 3. Key Design Insights

### 3.1 No Annotation-Based Test Categorization

Unlike many Java projects, Trino does **not** use JUnit 5 `@Tag` annotations or TestNG `@Category` for test selection. Instead, test partitioning is done entirely through:
- **Maven profile + surefire include/exclude patterns** (based on file path/name)
- **Tempto `@Groups`** for product tests only

This means the filename and package path *are* the categorization mechanism.

### 3.2 Parallel-by-Default Test Execution

Trino globally enables JUnit 5 parallel execution at both the method and class levels. This is aggressive -- most projects use sequential-by-default. It means:
- Tests must be stateless or use `@TestInstance(PER_CLASS)` with proper synchronization
- The `DistributedQueryRunner` is expensive to create, so `PER_CLASS` lifecycle + `@Execution(CONCURRENT)` is the standard pattern: one cluster per test class, methods run concurrently
- Some modules override to fixed parallelism (e.g., `parallelism=4`) to control resource usage

### 3.3 GIB-Based Change Detection Is Central

Git-flow Incremental Builder is not an afterthought -- it is deeply integrated into both the Maven build and the CI pipeline. It determines:
- Which modules to compile
- Which modules to test
- Which product test suites to run

On `master` pushes, GIB is disabled and everything runs. On PRs, only impacted modules are tested.

### 3.4 Product Tests Are Being Deprecated

The official test guidelines explicitly state: "Avoid product tests. If you can create a unit test for a feature, use a unit test." Product tests are reserved for:
- Plugin classloader verification
- Kerberos/LDAP/OAuth2 environments that need real infrastructure
- Cross-system integration that cannot be mocked

The `@Flaky` annotation is deprecated and only available for legacy TestNG tests. New JUnit 5 tests cannot be marked flaky -- they must be fixed or removed.

### 3.5 The `TestingConnectorBehavior` Pattern

Rather than using test inheritance to customize which tests run for each connector, Trino uses a capability enum. Connectors override `hasBehavior(TestingConnectorBehavior)` to declare their capabilities, and `BaseConnectorTest` uses `assumeTrue(hasBehavior(...))` to skip inapplicable tests. This is more maintainable than deep inheritance trees and allows the test framework to evolve independently of connector implementations.

### 3.6 CI Validates Its Own Structure

`TestCiWorkflow.java` parses the CI YAML at test time and verifies structural invariants (all jobs in build-success, all test steps have IDs, etc.). This is an unusually rigorous approach that prevents CI configuration drift.

## 4. Porting Considerations (Java -> Rust)

### 4.1 Test Framework Mapping

| Java | Rust Equivalent | Notes |
|------|-----------------|-------|
| JUnit 5 | `#[test]` + `cargo test` | Rust's built-in test framework is sufficient for unit tests |
| JUnit 5 `@TestInstance(PER_CLASS)` | `lazy_static!` / `OnceCell` per module | Share expensive fixtures across tests in a module |
| JUnit 5 parallel execution | `cargo test` (parallel by default) | Rust tests are parallel by default; use `#[serial]` crate for exceptions |
| AssertJ | `assert_eq!`, `assert!`, `pretty_assertions` crate | Consider `speculoos` or `assertables` for richer assertions |
| Testcontainers | `testcontainers-rs` crate | Direct Rust equivalent exists |
| JMH benchmarks | `criterion` crate | Standard Rust benchmarking |
| Tempto product tests | Custom Docker-compose based harness | No direct equivalent; build a launcher in Rust or shell scripts |

### 4.2 Test Categorization in Rust

Since Trino does not use annotation-based categorization, the Rust port does not need to replicate `@Tag`. Instead:

- **Module-level partitioning** -- Rust's module system (`mod tests`) naturally groups tests. Cargo's `--test` flag runs specific test files, and `#[cfg(test)]` gates test compilation.
- **Feature flags** -- Use Cargo features for cloud/integration tests: `#[cfg(feature = "cloud-tests")]`, `#[cfg(feature = "fte-tests")]`.
- **Test binary separation** -- Put integration tests in `tests/` (separate binary per file) and unit tests in `src/` (`#[cfg(test)]` modules). This mirrors the unit vs. integration split.

### 4.3 The `DistributedQueryRunner` Equivalent

The Rust worker will need an in-process test harness similar to `DistributedQueryRunner` that:
1. Starts a Trino coordinator (Java) + one or more Rust workers
2. Provides query submission and result collection APIs
3. Supports OpenTelemetry span collection for debugging
4. Tears down cleanly after tests

This is likely the most complex piece of test infrastructure to build. Consider:
- Using `testcontainers-rs` to run the Java coordinator as a Docker container
- Implementing a lightweight Rust process launcher for the worker
- Building the query assertion layer in Rust using the Trino JDBC or REST protocol

### 4.4 Connector Behavior Pattern

The `TestingConnectorBehavior` enum pattern should be replicated in Rust. A trait with default implementations and per-connector overrides would work well:

```rust
trait ConnectorBehavior {
    fn supports_insert(&self) -> bool { true }
    fn supports_predicate_pushdown(&self) -> bool { true }
    // ...
}
```

Test functions can then check capabilities and skip via `#[ignore]` or runtime checks.

### 4.5 CI Strategy

The Trino CI structure suggests the Rust project should:
1. Use GitHub Actions with a matrix strategy for connector-specific tests
2. Implement change detection (GIB equivalent) -- perhaps via `cargo-diff` or path-based filters
3. Separate fast (unit) and slow (integration/product) test jobs
4. Upload test results (use `cargo-nextest` for JUnit-compatible XML output)
5. Consider self-testing the CI configuration (following Trino's `TestCiWorkflow` pattern)

### 4.6 Protocol Compatibility Tests (Highest Priority)

For a drop-in worker replacement, the most critical tests are **protocol compatibility tests** that verify:
- The Rust worker correctly handles all Thrift/REST protocol messages from the coordinator
- Task execution produces bit-identical results to Java workers
- Memory accounting, spilling behavior, and error reporting match

These do not have a direct equivalent in Trino's test suite (since Trino tests assume Java workers). This is new test infrastructure the Rust project must build.

### 4.7 Benchmark Infrastructure

Use `criterion` for microbenchmarks with a structure mirroring Trino's:
- Benchmarks in `benches/` directory (Cargo convention)
- Macro benchmarks as separate integration test binaries that run queries against a cluster
- Consider reusing Trino's benchmark queries from `testing/trino-benchmark-queries/`
