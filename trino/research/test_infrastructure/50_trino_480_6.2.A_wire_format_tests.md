# Module Teardown: Block & Page Serialization Tests (Task 6.2.A) [KG-13]

## 0. Research Focus

This document maps the test infrastructure for Trino 480's block encoding and page serialization -- the wire format used when workers exchange data. The goal is to identify which byte-level behaviors are pinned down by tests so that a Rust replacement worker can use these tests as its conformance suite.

Key questions:
1. Round-trip coverage for each of the 13 block encoding types
2. Null bitmap (MSB-first) test coverage
3. HTTP envelope (magic number, xxHash64 checksum, chunk compression) test coverage
4. Cross-version compatibility tests
5. Test data generators for Block and Page objects

---

## 1. High-Level Overview

Trino's serialization tests are split across two tiers:

| Tier | Location | Purpose |
|------|----------|---------|
| **SPI-level block encoding** | `core/trino-spi/src/test/java/io/trino/spi/block/` | Tests individual block encoding round-trips (serialize -> deserialize) for each block type. Uses `TestingBlockEncodingSerde`. |
| **trino-main block behavior** | `core/trino-main/src/test/java/io/trino/block/` | Tests block semantics (values, nulls, regions, compactness, retained size) with incidental serde via `AbstractTestBlock.copyBlockViaBlockSerde()`. |
| **Page-level serde** | `core/trino-main/src/test/java/io/trino/execution/buffer/` | Tests full Page serialization with compression (NONE/LZ4/ZSTD), encryption, and the 12-byte page header. |
| **HTTP exchange envelope** | `core/trino-main/src/test/java/io/trino/operator/` | Tests the 16-byte HTTP envelope (magic + checksum + page count) and data integrity verification (ABORT/RETRY). |

#### BaseBlockEncodingTest: The Core Round-Trip Harness

File: `core/trino-spi/src/test/java/io/trino/spi/block/BaseBlockEncodingTest.java`

This abstract class provides a structured round-trip test framework. Key characteristics:

- **Test sizes**: `{2, 4, 8, 9, 16, 17, 32, 33, 64, 65, 1000, 1000000}` -- covers power-of-2 boundaries and off-by-one
- **Fill patterns**: `ONLY_NULLS`, `ONLY_VALUES`, `MIXED` (50% random nulls)
- **Structured test**: `testBlocksOf8()` chains 8 groups of 8 elements with varying fill patterns
- **Subclass contract**: Each subclass implements `getType()`, `write()`, and `randomValue()`
- **Fixed seed**: `Random(32167)` ensures deterministic test data

The `roundTrip()` method:
```java
BlockBuilder expectedBlockBuilder = createBlockBuilder(values.length);
// ... write values ...
Block expectedBlock = expectedBlockBuilder.build();
DynamicSliceOutput sliceOutput = new DynamicSliceOutput(1024);
blockEncodingSerde.writeBlock(sliceOutput, expectedBlock);
Block actualBlock = blockEncodingSerde.readBlock(sliceOutput.slice().getInput());
assertBlockEquals(getType(), actualBlock, expectedBlock);
```

#### AbstractTestBlock: The Behavioral Serde Harness

File: `core/trino-main/src/test/java/io/trino/block/AbstractTestBlock.java`

This base class performs block serde as a side effect of its `assertBlock()` method:

```java
assertBlockPositions(block, expectedValues);
assertBlockPositions(copyBlockViaBlockSerde(block), expectedValues);  // <-- round-trip here
```

Additionally, `assertBlockPosition()` tests serde on every region slice:
```java
assertPositionValue(copyBlockViaBlockSerde(block.getRegion(position, 1)), 0, expectedValue);
assertPositionValue(copyBlockViaBlockSerde(block.getRegion(0, position + 1)), position, expectedValue);
assertPositionValue(copyBlockViaBlockSerde(block.getRegion(position, block.getPositionCount() - position)), 0, expectedValue);
```

This means every block test class that extends AbstractTestBlock implicitly tests serde round-trips on subregions.

### 2.2 Null Bitmap Tests (MSB-First Convention)

The null bitmap uses an MSB-first bit-packing convention, tested in `TestEncoderUtil`.

File: `core/trino-spi/src/test/java/io/trino/spi/block/TestEncoderUtil.java`

**Encoding format (from `EncoderUtil.java`)**:
1. Write a boolean flag: `true` if nulls are present, `false` if not
2. If nulls present, pack boolean[] into bytes, MSB-first: position 0 goes to bit 7, position 1 to bit 6, ..., position 7 to bit 0
3. Last byte is padded with zeros for positions beyond the block length

**Bit layout per byte**:
```
bit:     7    6    5    4    3    2    1    0
pos: [pos+0][pos+1][pos+2][pos+3][pos+4][pos+5][pos+6][pos+7]
```

**Test coverage**:
- `testEncodeNullBitsScalarEqualsVectorized()`: Verifies scalar and vectorized encoders produce identical output
- **Test lengths**: `{0, 3, 255, 257, 512, 530, 1024, 2048, 8192}` -- covers boundaries near 256 (byte alignment), non-aligned sizes
- **Test offsets**: `{0, 2, 256}` -- tests encoding from non-zero offset
- **Null patterns**: 4 cases per (length, offset) combination:
  - All false (no nulls)
  - All true (all nulls)
  - Alternating true/false
  - Random (30% null rate, seed 42)
- Round-trip verified: encode -> decode -> compare to input

**The fixed-width block encoding tests also test null compaction**:
Each of `TestByteArrayBlockEncoding`, `TestShortArrayBlockEncoding`, `TestIntegerArrayBlockEncoding`, and `TestLongArrayBlockEncoding` has a `testCompressAndExpand*ScalarEqualsVector()` method that:
1. Generates random values with the same null patterns as TestEncoderUtil
2. Compresses values (removing null positions) using both scalar and vectorized paths
3. Verifies the compressed bytes are identical
4. Expands (re-inserts gaps at null positions) using both scalar and vectorized paths
5. Verifies the resulting blocks are identical

### 2.3 Page Serialization Envelope Tests

#### 2.3.A The 12-Byte Page Header

File: `core/trino-main/src/main/java/io/trino/execution/buffer/PagesSerdeUtil.java`

The serialized page header is **12 bytes** (not 16):

| Offset | Size | Field |
|--------|------|-------|
| 0 | 4 bytes | `positionCount` (int32) |
| 4 | 4 bytes | `uncompressedSize` (int32) |
| 8 | 4 bytes | `compressedSize` (int32, high bit = compression flag) |

The header is followed by the compressed/encrypted block stream.

**Test coverage** (`TestPagesSerde.java`):
- `testBigintSerializedSize()`: Asserts exact byte sizes -- empty BIGINT page = 35 bytes, each added value = 8 bytes
- `testVarcharSerializedSize()`: Asserts exact byte sizes -- empty VARCHAR page = 43 bytes, "alice" adds 9 bytes (4-byte offset + 5 chars)
- `testRoundTrip()`: Tests pages from tiny (0-1 rows) through jumbo (120,000 rows) with all compression codecs (NONE, LZ4, ZSTD) and with/without AES encryption
- `testDeserializationWithRollover()`: Tests deserialization when block data spans chunk boundaries (variable block sizes from 100-500, 500-1000 entries)

**Compression chunk format** (from `CompressingEncryptingPageSerializer`):

Default block size: **64KB** (`SERIALIZED_PAGE_DEFAULT_BLOCK_SIZE_IN_BYTES = 64 * 1024`)

Each chunk in the compressed block stream has:
```
[4 bytes: blockMarker = size | (compressed ? 0x80000000 : 0)]
[N bytes: compressed or uncompressed data]
```

The high bit of the 4-byte block marker indicates whether the chunk is compressed. Compression is skipped when the ratio is worse than 80% (`MINIMUM_COMPRESSION_RATIO = 0.8`).

The test exercises block sizes of `{107, 1009, 64*1024, 128*1024}` to cover corner cases.

#### 2.3.B The 16-Byte HTTP Envelope

File: `core/trino-main/src/main/java/io/trino/server/PagesInputStreamFactory.java`

The HTTP response body wrapping serialized pages is:

| Offset | Size | Field |
|--------|------|-------|
| 0 | 4 bytes | Magic number: `0xfea4f001` (`SERIALIZED_PAGES_MAGIC`) |
| 4 | 8 bytes | xxHash64 checksum (or `NO_CHECKSUM = 0x0123456789abcdef`) |
| 12 | 4 bytes | Page count (int32) |
| 16+ | variable | Concatenated serialized pages |

**Checksum details** (from `PagesSerdeUtil.calculateChecksum()`):
- Uses airlift `XxHash64` over all serialized page slices
- Special case: if the computed hash equals `NO_CHECKSUM` (`0x0123456789abcdef`), it returns `hash + 1` to avoid ambiguity

**Test coverage**:

1. `MockExchangeRequestProcessor` (used by `TestDirectExchangeClient`, `TestExchangeOperator`):
   - Constructs the full 16-byte envelope: magic + checksum + page count + serialized pages
   - Used as the HTTP response body in exchange client tests

2. `TestingExchangeHttpClientHandler`:
   - Same envelope construction for the testing exchange protocol
   - Handles both data pages and empty (buffer complete) responses

3. `TestDirectExchangeClient`:
   - `testStreamingAbortOnDataCorruption()`: Corrupts byte 42 of the response, verifies `TrinoException` with checksum mismatch message showing both read and calculated checksums
   - `testStreamingRetryDataCorruption()`: Corrupts first response, verifies retry delivers correct pages
   - The corruption test explicitly increments `bytes[42]` after serialization to simulate wire corruption

### 2.4 Cross-Version Compatibility Tests

**Finding: There are no cross-version compatibility tests.**

Trino does not maintain serialized byte fixtures from older versions. There are no golden files, no version-tagged test data, and no tests that deserialize bytes produced by a different Trino version. The block encoding format is treated as an internal protocol that changes freely between versions.

The encoding names are string-based (e.g., `"LONG_ARRAY"`, `"VARIABLE_WIDTH"`) and written as length-prefixed UTF-8 in the serialized stream, but there is no versioning mechanism in the wire format itself.

## 3. Key Design Insights

### 3.1 Two-Layer Serde Architecture

Block encoding (SPI layer) and page serialization (main layer) are cleanly separated:
- **Block encoding** is a `BlockEncodingSerde` that reads/writes individual blocks with a string-based encoding name prefix
- **Page serialization** wraps the block-level serde, adding the 12-byte page header, chunk-based compression, and optional encryption
- **HTTP envelope** adds the 16-byte magic+checksum+count wrapper on top of serialized pages

This means a Rust implementation needs three layers: block codec, page framing, and HTTP envelope.

### 3.2 TestingBlockEncodingSerde Registers All 13 Encodings

The `TestingBlockEncodingSerde` constructor (`core/trino-spi/src/test/java/io/trino/spi/block/TestingBlockEncodingSerde.java`) explicitly registers all 13 block encodings with their vectorization flags:

```java
addBlockEncoding(new VariableWidthBlockEncoding(true));
addBlockEncoding(new ByteArrayBlockEncoding(true, true, true));
addBlockEncoding(new ShortArrayBlockEncoding(true, true, true));
addBlockEncoding(new IntArrayBlockEncoding(true, true, true));
addBlockEncoding(new LongArrayBlockEncoding(true, true, true));
addBlockEncoding(new Fixed12BlockEncoding(true));
addBlockEncoding(new Int128ArrayBlockEncoding(true));
addBlockEncoding(new VariantBlockEncoding(true));
addBlockEncoding(new DictionaryBlockEncoding());
addBlockEncoding(new ArrayBlockEncoding(true));
addBlockEncoding(new MapBlockEncoding(true));
addBlockEncoding(new RowBlockEncoding(true));
addBlockEncoding(new RunLengthBlockEncoding());
```

### 3.3 Null Compaction in Fixed-Width Encodings

For the fixed-width types (BYTE_ARRAY, SHORT_ARRAY, INT_ARRAY, LONG_ARRAY), the encoding does NOT write values at null positions -- it compacts non-null values into a contiguous array. The null bitmap is written separately. On read, the values are expanded back with gaps at null positions.

This is heavily tested: each of these four encoding tests has a `testCompressAndExpand*ScalarEqualsVector()` method comparing scalar and vectorized implementations with identical test parameters.

### 3.4 Dictionary Encoding is Optimization-Aware

`TestDictionaryBlockEncoding` tests show that the serde is not a simple round-trip for dictionaries:
- **Sequential dictionary unnest**: When ids are `{0,1,2,3}`, the round-trip produces a flat `VariableWidthBlock` (not a DictionaryBlock)
- **Non-sequential with gaps**: When ids skip entries (e.g., `{3,2,0}`), the round-trip also produces a flat block
- **Non-sequential without gaps**: When ids are `{3,2,1,0}`, the round-trip preserves dictionary encoding

This means the encoder has a `replacementBlockForWrite()` hook that can unwrap dictionaries.

### 3.5 No Deterministic Byte-Level Golden Tests

The tests verify semantic equality (same values at same positions) rather than byte-level identity. There are no golden byte fixtures. The exact byte layout is only pinned indirectly through:
- `testBigintSerializedSize()` asserting exact total sizes (35 bytes for empty BIGINT page)
- `testVarcharSerializedSize()` asserting exact total sizes (43 bytes for empty VARCHAR page)
- Scalar-vs-vectorized byte identity checks for null bitmaps and value compaction

### 3.6 Compression is Tested Exhaustively via Parameterization

`TestPagesSerde.testRoundTrip()` tests every combination of:
- Compression: `{NONE, LZ4, ZSTD}` (all `CompressionCodec` values)
- Encryption: `{enabled, disabled}`
- Block sizes: `{107, 1009, 65536, 131072}` bytes
- Page sizes: from 0 rows to 120,000 rows
- Column types: various combinations of `{BIGINT, VARCHAR, DOUBLE}`

## 4. Porting Considerations (Java -> Rust)

### 4.1 Conformance Test Strategy

The Rust implementation should replicate the following test layers:

1. **Block encoding round-trips**: Port `BaseBlockEncodingTest` pattern -- for each encoding type, build blocks from random data (using the same seeds), serialize, deserialize, compare. The fixed seed `Random(32167)` means the exact test data can be reproduced in Rust.

2. **Null bitmap**: Port `TestEncoderUtil` with the exact MSB-first packing convention. The bit layout is `isNull[pos+0]` in bit 7, `isNull[pos+7]` in bit 0. This is the opposite of Arrow's LSB-first convention.

3. **Page header**: Implement the 12-byte header (positionCount, uncompressedSize, compressedSize with high-bit flag). The `testBigintSerializedSize` and `testVarcharSerializedSize` tests provide exact expected byte counts.

4. **HTTP envelope**: Implement the 16-byte wrapper (magic `0xfea4f001`, xxHash64 checksum, page count). The checksum avoids the sentinel value `0x0123456789abcdef`.

### 4.2 Critical Byte-Level Details

- **Null bitmap**: MSB-first. Position 0 is bit 7 of byte 0. Rust's `bitvec` crate uses LSB-first by default -- needs custom packing or `Msb0` order.
- **Block encoding name**: Written as `int32 length` + UTF-8 bytes. The name is the first thing in each block's serialized form.
- **Fixed-width null compaction**: Non-null values are written contiguously (no gaps for nulls). The count of non-null values is inferred from the null bitmap. Null positions get arbitrary (typically zero) values on expansion.
- **Dictionary unnesting**: The encoder may flatten a DictionaryBlock into its underlying value block if the ids form a permutation. The Rust encoder should match this behavior for compatibility.
- **Compression block marker**: Each compressed chunk starts with a 4-byte int where bit 31 (`0x80000000`) indicates the data is compressed. The remaining 31 bits give the chunk size.
- **Compression threshold**: 80% ratio -- if compressed size >= 80% of uncompressed, the uncompressed data is written instead.

### 4.3 Missing Test Coverage (Risks for Porting)

- **No byte-level golden fixtures**: The test suite does not check exact serialized bytes. A Rust implementation could produce different-but-valid serializations (e.g., different dictionary compaction decisions) that pass Trino's tests but fail in cross-implementation interop. Consider generating golden byte fixtures from the Java implementation.
- **No cross-version tests**: The wire format is implicitly versioned by the Trino release. A Rust worker must match the exact format of the coordinator's Trino version.
- **ARRAY, MAP, RUN_LENGTH lack SPI-level encoding tests**: These three types only get serde coverage through `AbstractTestBlock.copyBlockViaBlockSerde()`, which is less structured than the `BaseBlockEncodingTest` harness.
- **Variant encoding test is minimal**: `TestVariantBlockEncoding` only uses `Variant.ofLong()`. Other variant forms (string, binary, null, nested) are not tested in the encoding round-trip.

### 4.4 Recommended Approach for Rust Test Suite

1. **Snapshot-based golden tests**: Serialize reference blocks in Java (using the same random seeds), save the byte arrays to files, and load them in Rust tests for exact byte comparison. This catches endianness, padding, and alignment differences.

2. **Integration test**: Run Trino Java coordinator + Rust worker, execute TPC-H queries that exercise all block types, verify results match.

3. **Null bitmap unit tests**: Port the exact test parameters from `TestEncoderUtil`:
   - Lengths: `{0, 3, 255, 257, 512, 530, 1024, 2048, 8192}`
   - Offsets: `{0, 2, 256}`
   - Patterns: all-false, all-true, alternating, random(seed=42, rate=0.3)

4. **Page serde property tests**: Use Rust property-testing libraries (e.g., `proptest`) to generate random pages and verify round-trip correctness. Supplement with the exact TPC-H data source used in `TestPagesSerde`.
