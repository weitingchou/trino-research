# Module Teardown: Operator Correctness Tests (Task 6.4.A) [KG-13]

## 0. Research Focus

This document examines how Trino 480 tests individual operators in isolation. The focus areas are:

1. The `OperatorAssertion` / `TestingOperatorContext` test harness that drives operators outside of a real query execution
2. How synthetic input pages are constructed and fed into operators
3. How stateful operators (joins, aggregations, windows) are tested, including spill code paths
4. Edge case coverage: null handling, empty pages, single-row pages, large pages, dirty nulls
5. Parameterized testing across data types and spill/non-spill configurations

Primary source directories:

## 1. High-Level Overview

Trino's operator tests are **unit-level tests that exercise operators in isolation**, without spinning up a coordinator, catalog, or connector. They follow a consistent pattern:

```
Build synthetic input pages  -->  Create OperatorFactory  -->  Drive the operator  -->  Assert output
```

### Key Infrastructure Classes

| Class | File | Role |
|-------|------|------|
| `OperatorAssertion` | `operator/OperatorAssertion.java` | Static utilities: `toPages()`, `assertOperatorEquals()`, `assertOperatorEqualsIgnoreOrder()`, `finishOperator()`. The core test driver loop. |
| `TestingOperatorContext` | `operator/TestingOperatorContext.java` | Factory for a lightweight `OperatorContext` + `DriverContext` + `PipelineContext` + `TaskContext` stack with in-memory memory tracking. |
| `TestingTaskContext` | `testing/TestingTaskContext.java` | Factory for `TaskContext` with configurable memory limits, spill budgets, and `MemoryPool` sizing. Builder pattern. |
| `RowPagesBuilder` | `RowPagesBuilder.java` | Fluent builder: `.row(...)`, `.addSequencePage(n, initVals...)`, `.pageBreak()`, `.build()`. Produces `List<Page>`. |
| `PageAssertions` | `operator/PageAssertions.java` | `assertPageEquals()` -- channel-by-channel, position-by-position block comparison. |
| `DummySpillerFactory` | `operator/DummySpillerFactory.java` | In-memory spiller: stores spilled pages in `List<Iterable<Page>>`, counts spills. Used to verify spill occurred. |
| `GroupByHashYieldAssertion` | `operator/GroupByHashYieldAssertion.java` | Specialized harness for testing memory-pressure yield behavior on GroupByHash-based operators. |
| `AggregationTestUtils` | `operator/aggregation/AggregationTestUtils.java` | `assertAggregation()` for individual aggregation functions (SINGLE, PARTIAL+FINAL decomposition). |
| `AbstractTestAggregationFunction` | `operator/aggregation/AbstractTestAggregationFunction.java` | Abstract base providing standard test cases (no positions, single position, all nulls, mixed nulls, sliding window) for every aggregation function. |
| `TestingSourcePage` | `operator/TestingSourcePage.java` | `SourcePage` implementation that tracks which columns were lazily loaded (for testing column pruning). |

### The Standard Test Pattern

Almost every operator test class follows this structure:

```java
// 1. Set up executors (JUnit @BeforeEach / @BeforeAll)
executor = newCachedThreadPool(...);
scheduledExecutor = newScheduledThreadPool(2, ...);

// 2. Create a DriverContext (via TestingTaskContext)
DriverContext driverContext = createTaskContext(executor, scheduledExecutor, TEST_SESSION)
        .addPipelineContext(0, true, true, false)
        .addDriverContext();

// 3. Build synthetic input pages
List<Page> input = rowPagesBuilder(BIGINT, DOUBLE)
        .row(1L, 0.1)
        .row(2L, 0.2)
        .pageBreak()
        .row(-1L, -0.1)
        .build();

// 4. Create the OperatorFactory
OperatorFactory factory = new TopNOperatorFactory(...);

// 5. Build expected output
MaterializedResult expected = resultBuilder(session, BIGINT, DOUBLE)
        .row(2L, 0.2)
        .row(1L, 0.1)
        .build();

// 6. Assert
assertOperatorEquals(factory, driverContext, input, expected);
```

## 2. Detailed Analysis

### 2.1 The OperatorAssertion Driver Loop

`OperatorAssertion` is the heart of operator testing. It replaces the `Driver` loop with a simplified test-friendly version. The key method is `toPagesPartial()`:

```java
// Simplified from OperatorAssertion.toPagesPartial()
for (int loopsSinceLastPage = 0; loopsSinceLastPage < 1_000; loopsSinceLastPage++) {
    if (handledBlocked(operator)) continue;    // Wait for blocked operators
    if (revokeMemory) handleMemoryRevoking(operator);  // Trigger spill if revocable > 0
    if (input.hasNext() && operator.needsInput()) {
        operator.addInput(input.next());
        loopsSinceLastPage = 0;
    }
    Page outputPage = operator.getOutput();
    if (outputPage != null && outputPage.getPositionCount() != 0) {
        outputPages.add(outputPage);
        loopsSinceLastPage = 0;
    }
}
```

Key design decisions:
- **Loop limit of 1,000 iterations without output** prevents infinite loops in broken operators.
- **`revokeMemory` parameter** (default `true`): when enabled, every iteration checks `getReservedRevocableBytes() > 0` and forces `startMemoryRevoke()` / `finishMemoryRevoke()`. This means **spill tests automatically work** -- just set `spillEnabled=true` on the factory and pass `revokeMemoryWhenAddingPages=true`.
- **`handledBlocked()`** polls `isBlocked()` with a 1ms timeout, returning true if operator is still blocked.
- **`finishOperator()`** calls `operator.finish()` in a loop, draining remaining output, then asserts the operator is truly finished (`isFinished()==true`, `needsInput()==false`, `isBlocked().isDone()==true`).

### 2.2 Assertion Variants

| Method | Behavior |
|--------|----------|
| `assertOperatorEquals(factory, driverContext, input, expected)` | Ordered comparison via `containsExactlyElementsOf` |
| `assertOperatorEqualsIgnoreOrder(factory, driverContext, input, expected)` | Unordered comparison via `ImmutableMultiset` equality |
| `assertOperatorEquals(factory, types, driverContext, input, expectedPages)` | Page-level comparison (position counts + block-by-block) |
| `assertOperatorIsBlocked(operator)` / `assertOperatorIsUnblocked(operator)` | Timed assertions on `isBlocked()` future |

The `toMaterializedResult()` helper converts `List<Page>` to `MaterializedResult` for row-level comparison with tolerance for floating-point.

### 2.3 Synthetic Input Construction

**`RowPagesBuilder`** (`core/trino-main/src/test/java/io/trino/RowPagesBuilder.java`):

- `rowPagesBuilder(Type... types)` -- no hash channels
- `rowPagesBuilder(List<Integer> hashChannels, Type... types)` -- with hash channels (appends a hash column)
- `.row(Object... values)` -- add a single row (null allowed for nullable columns)
- `.rows(Object[]... rows)` -- add multiple rows
- `.addSequencePage(int length, int... initialValues)` -- generate `length` rows with sequential values starting at `initialValues` per column
- `.addBlocksPage(Block... blocks)` -- use pre-built blocks
- `.pageBreak()` -- start a new Page
- `.build()` -- returns `List<Page>`

Tests also build pages directly for edge cases:
```java
// RunLengthEncoded probe input (join tests)
new Page(RunLengthEncodedBlock.create(VARCHAR, Slices.utf8Slice("20"), 2))

// Dirty null bytes in masks (aggregation tests)
new ByteArrayBlock(4, Optional.of(new boolean[]{true, true, false, false}),
    new byte[]{0, 27 /* dirty null */, 0, 75 /* non-zero */})
```

### 2.4 Context Hierarchy for Tests

Tests must construct the full context stack: `TaskContext -> PipelineContext -> DriverContext -> OperatorContext`.

**`TestingTaskContext`** (in `core/trino-main/src/main/java/io/trino/testing/TestingTaskContext.java`) provides:
- `createTaskContext(executor, scheduledExecutor, session)` -- default 256MB query limit, 1GB pool
- `createTaskContext(executor, scheduledExecutor, session, DataSize maxMemory)` -- custom memory limit
- `Builder` pattern with `setQueryMaxMemory()`, `setMemoryPoolSize()`, `setMaxSpillSize()`

**`TestingOperatorContext`** (in `operator/TestingOperatorContext.java`) creates a minimal `OperatorContext` with:
- `directExecutor()` for synchronous execution
- `newSimpleAggregatedMemoryContext()` for memory tracking
- A PlanNodeId of "test"

### 2.5 Testing Stateful Operators

#### 2.5.A Hash Aggregation (TestHashAggregationOperator)

File: `core/trino-main/src/test/java/io/trino/operator/TestHashAggregationOperator.java`

This is one of the most thorough test classes. It tests with multiple configuration axes:

```java
testHashAggregation(true, true, 8, Integer.MAX_VALUE);   // spill ON, revoke ON, merge limits
testHashAggregation(true, false, 8, Integer.MAX_VALUE);  // spill ON, revoke OFF
testHashAggregation(false, false, 0, 0);                 // spill OFF
testHashAggregation(true, true, 0, 0);                   // etc.
```

Parameters: `(spillEnabled, revokeMemoryWhenAddingPages, memoryLimitForMerge, memoryLimitForMergeWithMemory)`

Test coverage includes:
- **Basic correctness**: 40,000 rows grouped by VARCHAR, with COUNT, SUM, AVG, MAX, COUNT(varchar), COUNT(boolean)
- **Empty input with global aggregation groups** (null output rows for group IDs 42 and 49)
- **Memory reservation tracking**: verifies `getOperatorStats().getUserMemoryReservation()` returns to 0 after completion
- **Memory limit violations**: `ExceededMemoryLimitException` with 10-byte limit
- **Hash builder resize**: forces resize with 200KB VARCHAR values
- **Memory reservation yield** (`GroupByHashYieldAssertion`): tests both VARCHAR and BIGINT key types
- **Multi-slice output**: verifies operator produces >1 output page for large result sets
- **Partial flushes**: verifies `needsInput()` returns true after partial flush
- **Merge-with-memory spill**: `testMergeWithMemorySpill()` with threshold-based spilling
- **Spill metrics**: verifies `SPILL_COUNT_METRIC_NAME` and `SPILL_DATA_SIZE` histograms after forced spill
- **Spiller failure**: `FailingSpillerFactory` that throws `IOException("Failed to spill")`
- **Adaptive partial aggregation**: tests that PA is disabled when unique-row ratio > 80%, then re-enabled after enough good flushes
- **Async spill blocking**: `SlowSpiller` that verifies the operator blocks and unblocks correctly

#### 2.5.B Hash Join (TestHashJoinOperator -- spilling variant)

File: `core/trino-main/src/test/java/io/trino/operator/join/TestHashJoinOperator.java`

The join tests are structurally different because joins have two sides (build + probe) and require the `JoinBridgeManager` / `LookupSourceFactory` infrastructure:

```java
// Build side: construct pages, set up build side via JoinTestUtils
BuildSideSetup buildSideSetup = setupBuildSide(partitionFunctionProvider, parallelBuild,
    taskContext, buildPages, filterFunction, spillEnabled, spillerFactory);

// Build drivers run to completion to populate the lookup source
instantiateBuildDrivers(buildSideSetup, taskContext);
buildLookupSource(executor, buildSideSetup);

// Probe side: create operator factory and assert
OperatorFactory joinOperatorFactory = innerJoinOperatorFactory(lookupSourceFactory, probePages, spillerFactory);
assertOperatorEquals(joinOperatorFactory, driverContext, probeInput, expected);
```

Spill test coverage uses a `WhenSpill` enum: `{DURING_BUILD, AFTER_BUILD, DURING_USAGE, NEVER}`. Tests enumerate all combinations:

```java
innerJoinWithSpill(nCopies(PARTITION_COUNT, WhenSpill.DURING_BUILD), ...);    // spill all during build
innerJoinWithSpill(nCopies(PARTITION_COUNT, WhenSpill.AFTER_BUILD), ...);     // spill all after build
innerJoinWithSpill(nCopies(PARTITION_COUNT, WhenSpill.DURING_USAGE), ...);    // spill all during probe
innerJoinWithSpill(concat(singletonList(WhenSpill.DURING_BUILD),
    nCopies(PARTITION_COUNT - 1, WhenSpill.NEVER)), ...);                     // spill one partition
```

Failing-spill tests verify error propagation for both `failSpill()` and `failUnspill()` variants.

Null handling tests:
- `testInnerJoinWithNullProbe` -- nulls on probe side filtered out correctly
- `testInnerJoinWithNullBuild` -- nulls on build side never match
- `testInnerJoinWithNullOnBothSides` -- both sides have nulls, no null-null matches

Join type tests: `innerJoin`, `probeOuterJoin` (LEFT OUTER), `lookupOuterJoin` (RIGHT OUTER), `fullOuterJoin`.

Special input encodings: `testInnerJoinWithRunLengthEncodedProbe()` tests RLE blocks as probe input.

#### 2.5.C Hash Join (unspilled variant)

File: `core/trino-main/src/test/java/io/trino/operator/join/unspilled/TestHashJoinOperator.java`

Parallel test class for the non-spilling code path. Same test patterns but uses `OperatorFactories.join()` instead of `OperatorFactories.spillingJoin()`. Tests `parallelBuild=true/false` for all join types.

#### 2.5.D Window Operator (TestWindowOperator)

File: `core/trino-main/src/test/java/io/trino/operator/TestWindowOperator.java`

Uses the same spill pattern: `testRowNumber(spillEnabled, revokeMemoryWhenAddingPages, memoryLimit)`.

Window function definitions are pre-built constants: `ROW_NUMBER`, `RANK`, `FIRST_VALUE`, `LAST_VALUE`, `NTH_VALUE`, `LAG`, `LEAD`.

Tests: multiple output pages (80,000 rows), row_number, rank, first/last/nth value, lag/lead, partitioned windows, memory limit exceeded.

#### 2.5.E Order By (TestOrderByOperator)

Same pattern: `testSingleFieldKey(spillEnabled, revokeMemoryWhenAddingPages, memoryLimit)`. Tests ASC/DESC, multi-field keys, memory limit, spill verification via `DummySpillerFactory.getSpillsCount()`.

#### 2.5.F Streaming Aggregation (TestStreamingAggregationOperator)

Tests pre-sorted grouped aggregation. Notable: tests `Double.NaN` and `null` group keys, verifying NaN groups correctly, null groups separately.

### 2.6 Edge Case Testing Patterns

#### Null Handling

Null testing is pervasive throughout operator tests:

1. **Join null handling**: Dedicated tests (`testInnerJoinWithNullProbe`, `testInnerJoinWithNullBuild`, `testInnerJoinWithNullOnBothSides`) verify null keys never match in inner/outer joins. Nulls are inserted via `row((String) null)`.

2. **Aggregation null handling**: `AbstractTestAggregationFunction` provides standard tests:
   - `testAllPositionsNull()` -- all-null input via `RunLengthEncodedBlock.create(type, null, 10)`
   - `testMixedNullAndNonNullPositions()` -- alternating null/non-null blocks
   - `getExpectedValueIncludingNulls()` override for functions with special null semantics

3. **Dirty nulls**: `TestAggregationOperator.testMaskWithDirtyNulls()` tests `ByteArrayBlock` with null positions containing non-zero bytes (`27`, `75`), verifying the engine reads the null bitmap, not the value.

4. **Null masks in distinct aggregation**: `testDistinctMaskWithNulls()` tests with `RunLengthEncodedBlock` masks where all positions are null-but-true.

5. **NaN handling**: `TestStreamingAggregationOperator` explicitly tests `Double.NaN` group keys.

#### Empty / Single-Row / Large Pages

- **Empty input**: Many tests call `toPages(operatorFactory, driverContext)` (no input pages) or feed `emptyIterator()`.
- **Single-row pages**: `TestMarkDistinctOperator` uses `addSequencePage(1, ...)` for single-position pages and verifies they don't get RLE-encoded.
- **Large pages**: `TestHashAggregationOperator.testMultipleOutputPages()` uses 40,000 rows; window tests use 80,000 rows to force multi-page output. `testHashBuilderResize()` creates a 200KB VARCHAR value to trigger hash table resize.
- **Page boundaries**: `TestLimitOperator` tests limit exactly at page boundary (`testLimitWithPageAlignment`) and mid-page truncation (`testLimitWithBlockView`).

### 2.7 Parameterized Type Testing

Trino does not use JUnit `@ParameterizedTest` extensively for operator tests. Instead, it uses two patterns:

**Pattern 1: Manual iteration** (in test methods)
```java
// TestHashAggregationOperator
public void testMemoryReservationYield() {
    testMemoryReservationYield(VARCHAR);
    testMemoryReservationYield(BIGINT);
}

// TestRowNumberOperator
public void testMemoryReservationYield() {
    for (Type type : Arrays.asList(VARCHAR, BIGINT)) { ... }
}
```

**Pattern 2: Abstract test class inheritance** (for aggregation functions)

`AbstractTestAggregationFunction` defines abstract methods:
```java
protected abstract Block[] getSequenceBlocks(int start, int length);
protected abstract String getFunctionName();
protected abstract List<Type> getFunctionParameterTypes();
protected abstract Object getExpectedValue(int start, int length);
```

Each concrete subclass (e.g., `TestBigintAverageAggregation`, `TestApproximateCountDistinctVarchar`, etc.) implements these for a specific type/function combination. The base class provides 7 standard test methods: `testNoPositions`, `testSinglePosition`, `testMultiplePositions`, `testAllPositionsNull`, `testMixedNullAndNonNullPositions`, `testNegativeOnlyValues`, `testPositiveOnlyValues`, `testSlidingWindow`.

There are ~133 files in the aggregation test directory covering approximate count distinct (per-type), approximate set (per-type), min/max (per-type), sum/avg (per-type), etc.

### 2.8 Memory Pressure and Yield Testing

`GroupByHashYieldAssertion` is a dedicated harness for testing memory-pressure yield behavior:

1. Creates a `MemoryPool` of 1GB
2. Before each page, reserves almost all remaining pool memory (leaving only `additionalMemoryInBytes`)
3. After `addInput()`, checks if the operator yielded (needs more memory than available)
4. Verifies the operator correctly blocked on `isWaitingForMemory()`
5. Verifies hash capacity did not increase until memory was freed
6. After freeing memory, verifies the operator unblocked and the hash capacity doubled

This is used by `TestHashAggregationOperator`, `TestDistinctLimitOperator`, `TestMarkDistinctOperator`, and `TestRowNumberOperator`.

### 2.9 Semi-Join Testing (TestHashSemiJoinOperator)

Uses a **two-stage driver pattern**: first runs a `ValuesOperator -> SetBuilderOperator` driver to build the set, then tests `HashSemiJoinOperator` with probe input. Tests cover:
- BIGINT set membership
- VARCHAR set membership
- Memory limit exceeded

### 2.10 DummySpillerFactory Mechanics

```java
public class DummySpillerFactory implements SpillerFactory {
    private long spillsCount;

    public Spiller create(List<Type> types, SpillContext ctx, AggregatedMemoryContext mem) {
        return new Spiller() {
            private final List<Iterable<Page>> spills = new ArrayList<>();

            public ListenableFuture<DataSize> spill(Iterator<Page> pages) {
                spillsCount++;
                spills.add(ImmutableList.copyOf(pages));     // in-memory copy
                return immediateFuture(DataSize.ofBytes(0));  // instant completion
            }

            public List<Iterator<Page>> getSpills() {
                return spills.stream().map(Iterable::iterator).collect(toImmutableList());
            }
        };
    }
}
```

For join spill tests, there is also a `DummySpillerFactory` variant in `JoinTestUtils` that supports `failSpill()` and `failUnspill()` modes, returning `immediateFailedFuture(new RuntimeException("Spill failed"))`.

## 3. Key Design Insights

### Insight 1: OperatorAssertion replaces the real Driver loop
Rather than constructing a full `Driver` pipeline, tests call `OperatorAssertion.toPages()` which implements the same `needsInput() -> addInput() -> getOutput()` loop that the real Driver uses, but with test-friendly additions like automatic memory revocation and a loop-iteration safety limit. This means operator tests exercise the **exact same protocol contract** as production code.

### Insight 2: Spill testing is a first-class concern, not an afterthought
Every stateful operator test (aggregation, join, window, order-by) systematically tests with `spillEnabled=true/false` and `revokeMemoryWhenAddingPages=true/false`. The `DummySpillerFactory` makes this zero-cost: no temp files, no I/O, just in-memory page lists. The test asserts `spillerFactory.getSpillsCount() > 0` to verify spill actually occurred.

### Insight 3: Manual "parameterized" testing over (spill, revoke, memoryLimit) axes
Rather than using JUnit 5 `@ParameterizedTest`, Trino tests manually enumerate configurations:
```java
testHashAggregation(true, true, 8, Integer.MAX_VALUE);
testHashAggregation(true, false, 8, Integer.MAX_VALUE);
testHashAggregation(false, false, 0, 0);
// ... 7 combinations total
```
This gives fine control over which parameter combinations are worth testing.

### Insight 4: AbstractTestAggregationFunction standardizes aggregation coverage
By providing 7-8 template tests in an abstract base class (no positions, single, multi, all-null, mixed-null, negative, positive, sliding window), Trino guarantees every aggregation function has baseline edge-case coverage simply by subclassing.

### Insight 5: RowPagesBuilder enables both structured and adversarial input
The `addSequencePage(40_000, ...)` method generates large deterministic datasets for correctness and multi-page output testing, while `.row((String) null)` and direct `ByteArrayBlock` construction allow precise edge-case inputs (dirty nulls, RLE blocks, oversized values).

### Insight 6: The context stack is real but lightweight
Tests construct the full `TaskContext -> PipelineContext -> DriverContext -> OperatorContext` hierarchy because operators depend on it for memory tracking, session access, and spill context. But `TestingTaskContext` and `TestingOperatorContext` keep this cheap by using `directExecutor()` and in-memory memory pools.

### Insight 7: Join tests exercise the build-probe lifecycle
Join tests construct the full two-stage pipeline: build-side drivers populate the `LookupSourceFactory`, then probe-side operators are tested through `OperatorAssertion`. This is necessary because the build/probe handshake (blocking on lookup source readiness) is part of correctness.

## 4. Porting Considerations (Java -> Rust)

### 4.1 OperatorAssertion -> Rust test harness

The `OperatorAssertion.toPages()` pattern maps cleanly to Rust:
```rust
fn drive_operator(op: &mut dyn Operator, input: Vec<Page>) -> Vec<Page> {
    let mut output = Vec::new();
    let mut input_iter = input.into_iter();
    loop {
        if op.needs_input() { if let Some(page) = input_iter.next() { op.add_input(page); } }
        if let Some(page) = op.get_output() { output.push(page); }
        if op.is_finished() { break; }
    }
    output
}
```
The memory revocation in the loop should be modeled as a trait method on the operator or a test wrapper.

### 4.2 DummySpillerFactory -> In-memory spill trait

```rust
trait Spiller {
    fn spill(&mut self, pages: Vec<Page>) -> Result<()>;
    fn get_spills(&self) -> Vec<Vec<Page>>;
}
struct InMemorySpiller { spills: Vec<Vec<Page>> }
```
This avoids filesystem dependencies in unit tests while exercising the same spill code paths.

### 4.3 RowPagesBuilder -> Rust page builder

A `PageBuilder` with `.add_row(&[Value])`, `.add_sequence_page(len, &[initial_vals])`, `.page_break()`, `.build() -> Vec<Page>` would replicate the Java pattern. Rust's type system can enforce column types at compile time.

### 4.4 TestingTaskContext -> Lightweight context stack

The full context hierarchy (`TaskContext -> PipelineContext -> DriverContext -> OperatorContext`) exists primarily for memory accounting. In Rust, this could be simplified to a single `TestContext` struct with an `Arc<MemoryTracker>` if the memory tracking interface is uniform across operators.

### 4.5 Spill test matrix

Port the `(spill_enabled, revoke_during_add, memory_limit)` parameter matrix. Rust's `#[test_case]` or `rstest` crates handle parameterized tests more elegantly than Java's manual enumeration:
```rust
#[rstest]
#[case(true, true, 8)]
#[case(true, false, 8)]
#[case(false, false, 0)]
fn test_hash_aggregation(#[case] spill: bool, #[case] revoke: bool, #[case] limit: u64) { ... }
```

### 4.6 AbstractTestAggregationFunction -> Macro-based test generation

The abstract base class pattern for aggregation tests maps to Rust macros or a `test_aggregation!` proc macro that generates the standard 7 test cases for each function implementation:
```rust
test_aggregation!(sum_bigint, SumAccumulator<i64>, {
    sequence_blocks: |start, len| vec![create_bigint_block(start, len)],
    expected_value: |start, len| (start..start+len).sum::<i64>(),
});
```

### 4.7 Key test categories to replicate

1. **Stateless operators** (Filter, Project, Limit): feed pages, assert output. Simplest category.
2. **Accumulating operators** (TopN, OrderBy, Aggregation): feed all input, call finish, assert output. Must test memory tracking.
3. **Two-input operators** (HashJoin, SemiJoin): need build-side setup before probe-side testing. Must test null handling on both sides.
4. **Windowed operators**: need partition/ordering specification plus frame bounds.
5. **Spill paths**: every accumulating/stateful operator needs `with_spill` and `without_spill` variants.
6. **Edge cases**: empty input, single row, null keys, dirty nulls, RLE-encoded blocks, oversized values, page boundaries, NaN group keys.
7. **Memory pressure**: yield-on-rehash (GroupByHash pattern), memory limit exceeded (ExceededMemoryLimitException pattern).
