# Module Teardown: Block-Level Binary Encoding (Task 4.2.D)

## 0. Research Focus
* **Task ID:** 4.2.D [KG-1]
* **Focus:** Trace the exact binary layout written by each `BlockEncoding.writeBlock()` method. Document how each block type serializes its position count, null bitmap, and value data. Analyze the `BlockEncodingSerde` envelope that prefixes block type names. Cover primitive blocks, variable-width blocks, dictionary/RLE wrappers, and complex nested types (Row, Array, Map).

## 1. High-Level Overview

* **Core Responsibility:** The block encoding subsystem converts in-memory columnar `Block` objects into a self-describing binary format for transfer between Trino workers. Each block type has a dedicated `BlockEncoding` implementation that knows exactly how to serialize and deserialize that block's internal structure. The format is designed for efficiency: nulls are bit-packed (1 bit per position), null values are compacted out of the value stream, and nested types use recursive encoding.

* **Key Design Principles:**
  1. **Self-describing envelope:** Every serialized block is prefixed with a length-prefixed UTF-8 encoding name string, allowing the reader to dispatch to the correct `BlockEncoding` without external schema.
  2. **Bit-packed null bitmaps:** Nullability is encoded as 1 bit per position (MSB-first within each byte), not 1 byte per position. This is an 8x compression over the in-memory `boolean[]` representation.
  3. **Null compaction:** When nulls are present, only non-null values are written to the wire. The reader expands them back into a dense array using the null bitmap. This saves `nullCount * valueWidth` bytes.
  4. **Little-endian byte order:** All multi-byte integers and longs are written in little-endian order (the native order of the airlift `Slice` library, which stores all primitives in `ByteOrder.LITTLE_ENDIAN`).
  5. **Recursive nesting:** Complex types (Row, Array, Map, Dictionary, RLE) serialize their child blocks by calling back into `BlockEncodingSerde.writeBlock()`, producing a recursively self-describing tree.
  6. **SIMD-accelerated paths:** Trino 480 includes optional vectorized code paths (using `jdk.incubator.vector`) for null bit-packing, null compaction (compress), and null expansion (expand). The scalar and vectorized paths produce identical wire formats.

* **Key Triggers:** Block encoding is invoked during page serialization -- specifically by `PagesSerdeUtil.writeRawPage()` which calls `BlockSerdeUtil.writeBlock()` (which delegates to `BlockEncodingSerde.writeBlock()`) for each block/column in a page. This happens when a producing task writes output pages to its `OutputBuffer` for downstream consumption.

## 2. Detailed Analysis

### 2.1 The Serde Envelope: `InternalBlockEncodingSerde` and `BlockEncodingManager`

**Source:** `io.trino.metadata.InternalBlockEncodingSerde`

Every block serialization begins and ends at `InternalBlockEncodingSerde`. This class wraps each block's type-specific encoding with a name-based envelope.

**Write path:**

```java
// InternalBlockEncodingSerde.writeBlock()
public void writeBlock(SliceOutput output, Block block)
{
    while (true) {
        BlockEncoding blockEncoding = blockToEncoding.apply(block.getClass());
        Optional<Block> replacementBlock = blockEncoding.replacementBlockForWrite(block);
        if (replacementBlock.isPresent()) {
            block = replacementBlock.get();
            continue;
        }
        writeLengthPrefixedString(output, blockEncoding.getName());
        blockEncoding.writeBlock(this, output, block);
        break;
    }
}
```

The `while(true)` loop handles block replacement -- for example, a `DictionaryBlock` with identity-mapping ids gets replaced by its underlying dictionary block. The loop runs until no more replacements are needed.

**Wire format of the name prefix:**

```java
private static void writeLengthPrefixedString(SliceOutput output, String value)
{
    byte[] bytes = value.getBytes(UTF_8);
    output.writeInt(bytes.length);     // 4 bytes, little-endian
    output.writeBytes(bytes);          // N bytes, UTF-8
}
```

**Read path:**

```java
// InternalBlockEncodingSerde.readBlock()
public Block readBlock(SliceInput input)
{
    String encodingName = readLengthPrefixedString(input);
    BlockEncoding blockEncoding = nameToEncoding.apply(encodingName);
    return blockEncoding.readBlock(this, input);
}
```

**Envelope wire layout (every block):**

```
+----------------------------+
| name_length : int32 LE     |  4 bytes
+----------------------------+
| name_bytes : UTF-8         |  name_length bytes (e.g. "LONG_ARRAY" = 10 bytes)
+----------------------------+
| <block-specific payload>   |  variable
+----------------------------+
```

### 2.2 The Null Bitmap: `EncoderUtil`

**Source:** `io.trino.spi.block.EncoderUtil`

All block encodings share a common null bitmap format implemented in `EncoderUtil`. This is the most critical shared primitive.

**Encoding (`encodeNullsAsBitsScalar`):**

```java
public static void encodeNullsAsBitsScalar(SliceOutput sliceOutput,
        @Nullable boolean[] isNull, int offset, int length)
{
    sliceOutput.writeBoolean(isNull != null);  // 1 byte: 0x00 = no nulls, 0x01 = has nulls
    if (isNull == null) {
        return;  // No null bitmap follows
    }
    byte[] packedIsNull = new byte[(length + 7) / 8];
    int currentByte = 0;
    for (int position = 0; position < (length & ~0b111); position += 8) {
        int value = ((isNull[position + offset] ? 1 : 0) << 7) |       // bit 7 (MSB)
                ((isNull[position + offset + 1] ? 1 : 0) << 6) |       // bit 6
                ((isNull[position + offset + 2] ? 1 : 0) << 5) |       // ...
                ((isNull[position + offset + 3] ? 1 : 0) << 4) |
                ((isNull[position + offset + 4] ? 1 : 0) << 3) |
                ((isNull[position + offset + 5] ? 1 : 0) << 2) |
                ((isNull[position + offset + 6] ? 1 : 0) << 1) |
                (isNull[position + offset + 7] ? 1 : 0);               // bit 0 (LSB)
        packedIsNull[currentByte++] = (byte) (value & 0xFF);
    }
    // Handle remaining positions (< 8) in the last byte, MSB-first with zero-padding
    if ((length & 0b111) > 0) {
        int value = 0;
        int mask = 0b1000_0000;
        for (int position = length & ~0b111; position < length; position++) {
            value |= isNull[position + offset] ? mask : 0;
            mask >>>= 1;
        }
        packedIsNull[currentByte++] = (byte) (value & 0xFF);
    }
    sliceOutput.writeBytes(packedIsNull, 0, currentByte);
}
```

**Decoding (`retrieveNullBits` + `decodeNullBitsScalar`):**

```java
@Nullable
public static byte[] retrieveNullBits(SliceInput sliceInput, int positionCount)
{
    if (!sliceInput.readBoolean()) {
        return null;  // hasNulls flag was false
    }
    return sliceInput.readNBytes((positionCount + 7) / 8);
}
```

**Null bitmap wire layout:**

```
+-------------------------------+
| hasNulls : byte               |  1 byte: 0x00 = all non-null, 0x01 = has nulls
+-------------------------------+
| (if hasNulls = 0x01):         |
| packed_bits : byte[]          |  ceil(positionCount / 8) bytes
+-------------------------------+
```

**Bit ordering within each byte:** MSB-first. Position `i` within a group of 8 maps to bit `(7 - (i % 8))`. The first position in a byte occupies bit 7, the last occupies bit 0. For a trailing byte with fewer than 8 positions, unused low bits are zero.

**Example:** For 10 positions where positions 2 and 7 are null:
- Byte 0: positions 0-7 -> `0b00100001` = `0x21` (bit 5 for pos 2, bit 0 for pos 7)
- Byte 1: positions 8-9 -> `0b00000000` = `0x00` (both non-null, 6 zero-padding bits)

### 2.3 `LongArrayBlockEncoding` (BIGINT, TIMESTAMP, etc.)

**Source:** `io.trino.spi.block.LongArrayBlockEncoding` -- NAME = `"LONG_ARRAY"`

```java
public void writeBlock(BlockEncodingSerde blockEncodingSerde,
        SliceOutput sliceOutput, Block block)
{
    LongArrayBlock longArrayBlock = (LongArrayBlock) block;
    int positionCount = longArrayBlock.getPositionCount();
    sliceOutput.appendInt(positionCount);                       // [1] positionCount

    int rawOffset = longArrayBlock.getRawValuesOffset();
    boolean[] isNull = longArrayBlock.getRawValueIsNull();
    long[] rawValues = longArrayBlock.getRawValues();

    encodeNullsAsBits...(sliceOutput, isNull, rawOffset, positionCount);  // [2] null bitmap

    if (isNull == null) {
        sliceOutput.writeLongs(rawValues, rawOffset, positionCount);      // [3a] all values
    }
    else {
        compactLongsWithNullsScalar(sliceOutput, rawValues, isNull, rawOffset, positionCount);  // [3b] compacted
    }
}
```

**Null compaction (`compactLongsWithNullsScalar`):**

```java
static void compactLongsWithNullsScalar(SliceOutput sliceOutput,
        long[] values, boolean[] isNull, int offset, int length)
{
    long[] compacted = new long[length];
    int compactedIndex = 0;
    for (int i = 0; i < length; i++) {
        compacted[compactedIndex] = values[i + offset];
        compactedIndex += isNull[i + offset] ? 0 : 1;    // skip null positions
    }
    sliceOutput.writeInt(compactedIndex);                  // non-null count
    sliceOutput.writeLongs(compacted, 0, compactedIndex);  // only non-null values
}
```

**Wire layout -- no nulls:**

```
+-----------------------------------+
| positionCount : int32 LE          |  4 bytes
+-----------------------------------+
| hasNulls : 0x00                   |  1 byte
+-----------------------------------+
| values[0..N-1] : int64 LE each   |  positionCount * 8 bytes
+-----------------------------------+
```

**Wire layout -- with nulls:**

```
+-----------------------------------+
| positionCount : int32 LE          |  4 bytes
+-----------------------------------+
| hasNulls : 0x01                   |  1 byte
+-----------------------------------+
| nullBitmap : byte[]               |  ceil(positionCount / 8) bytes
+-----------------------------------+
| nonNullCount : int32 LE           |  4 bytes
+-----------------------------------+
| values[0..M-1] : int64 LE each   |  nonNullCount * 8 bytes
+-----------------------------------+
```

Where `M = nonNullCount <= positionCount`.

### 2.7 `Int128ArrayBlockEncoding` (DECIMAL(p>18), UUID, IPv6)

**Source:** `io.trino.spi.block.Int128ArrayBlockEncoding` -- NAME = `"INT128_ARRAY"`

Each value is stored as 2 consecutive `long` values (16 bytes total). The null compaction writes the count of non-null *positions* (not longs), then writes `nonNullCount * 2` longs.

```java
public void writeBlock(...)
{
    int positionCount = int128ArrayBlock.getPositionCount();
    sliceOutput.appendInt(positionCount);
    encodeNullsAsBits...(sliceOutput, isNull, rawArrayOffset, positionCount);

    if (isNull == null) {
        sliceOutput.writeLongs(rawValues, rawArrayOffset * 2, positionCount * 2);
    } else {
        long[] valuesWithoutNull = new long[positionCount * 2];
        int nonNullPositionCount = 0;
        for (int i = 0; i < positionCount; i++) {
            int rawValuesIndex = (i + rawArrayOffset) * 2;
            valuesWithoutNull[nonNullPositionCount] = rawValues[rawValuesIndex];
            valuesWithoutNull[nonNullPositionCount + 1] = rawValues[rawValuesIndex + 1];
            nonNullPositionCount += isNull[i + rawArrayOffset] ? 0 : 2;
        }
        sliceOutput.writeInt(nonNullPositionCount / 2);  // position count, not long count
        sliceOutput.writeLongs(valuesWithoutNull, 0, nonNullPositionCount);
    }
}
```

**Wire layout -- with nulls:**

```
+-------------------------------------------+
| positionCount : int32 LE                  |  4 bytes
+-------------------------------------------+
| hasNulls : 0x01                           |  1 byte
+-------------------------------------------+
| nullBitmap : byte[]                       |  ceil(positionCount / 8) bytes
+-------------------------------------------+
| nonNullPositionCount : int32 LE           |  4 bytes (positions, not longs)
+-------------------------------------------+
| values[0..2*M-1] : int64 LE each         |  nonNullPositionCount * 16 bytes
| (pairs of [high, low] per position)       |
+-------------------------------------------+
```

**Important:** Unlike `LongArrayBlockEncoding` which has SIMD-accelerated compress/expand, `Int128ArrayBlockEncoding` uses a simple scalar loop with a stride of 2 longs. No `jdk.incubator.vector` usage here because the 128-bit pair semantics don't map cleanly to SIMD lanes.

### 2.9 `VariableWidthBlockEncoding` (VARCHAR, VARBINARY)

**Source:** `io.trino.spi.block.VariableWidthBlockEncoding` -- NAME = `"VARIABLE_WIDTH"`

This is the most complex primitive encoding because it must handle variable-length byte data with an offsets array.

```java
public void writeBlock(...)
{
    VariableWidthBlock variableWidthBlock = (VariableWidthBlock) block;
    int positionCount = variableWidthBlock.getPositionCount();
    sliceOutput.appendInt(positionCount);                                      // [1]

    int arrayBaseOffset = variableWidthBlock.getRawArrayBase();
    boolean[] isNull = variableWidthBlock.getRawValueIsNull();
    encodeNullsAsBits...(sliceOutput, isNull, arrayBaseOffset, positionCount); // [2]

    int[] rawOffsets = variableWidthBlock.getRawOffsets();
    writeOffsetsWithNullsCompacted(sliceOutput, rawOffsets, isNull,
            arrayBaseOffset, positionCount);                                   // [3]

    int startingOffset = rawOffsets[arrayBaseOffset];
    int totalLength = rawOffsets[positionCount + arrayBaseOffset] - startingOffset;
    sliceOutput.writeBytes(variableWidthBlock.getRawSlice(),
            startingOffset, totalLength);                                      // [4]
}
```

**Offset compaction (`writeOffsetsWithNullsCompacted`):**

The offsets array stores ending offsets (not starting offsets). The first value is implicit (0). When nulls are present, only the offsets for non-null positions are written.

```java
private static void writeOffsetsWithNullsCompacted(SliceOutput sliceOutput,
        int[] rawOffsets, @Nullable boolean[] valueIsNull, int baseOffset, int positionCount)
{
    int startingOffset = rawOffsets[baseOffset];
    if (valueIsNull == null && startingOffset == 0) {
        // Fast path: write ending offsets directly
        sliceOutput.appendInt(positionCount)
                   .writeInts(rawOffsets, baseOffset + 1, positionCount);
    }
    else {
        int[] nonNullOffsets;
        int nonNullOffsetsCount;
        if (valueIsNull == null) {
            // Translate offsets to start from zero
            nonNullOffsets = new int[positionCount];
            for (int i = 0; i < nonNullOffsets.length; i++) {
                nonNullOffsets[i] = rawOffsets[i + baseOffset + 1] - startingOffset;
            }
            nonNullOffsetsCount = nonNullOffsets.length;
        }
        else {
            // Translate and suppress null offsets
            nonNullOffsets = new int[positionCount];
            nonNullOffsetsCount = 0;
            for (int i = 0; i < positionCount; i++) {
                nonNullOffsets[nonNullOffsetsCount] = rawOffsets[i + baseOffset + 1] - startingOffset;
                nonNullOffsetsCount += valueIsNull[i + baseOffset] ? 0 : 1;
            }
        }
        sliceOutput.appendInt(nonNullOffsetsCount)
                   .writeInts(nonNullOffsets, 0, nonNullOffsetsCount);
    }
}
```

**Critical detail:** The offsets written are *ending offsets* (each offset is the cumulative byte position after the entry's data). The starting offset of entry `i` is `offsets[i-1]` (or 0 for the first entry). This is different from Arrow-style offsets which store `positionCount + 1` start/end offsets.

**Wire layout -- no nulls:**

```
+-------------------------------------------+
| positionCount : int32 LE                  |  4 bytes
+-------------------------------------------+
| hasNulls : 0x00                           |  1 byte
+-------------------------------------------+
| offsetCount : int32 LE                    |  4 bytes (= positionCount)
+-------------------------------------------+
| endOffsets[0..N-1] : int32 LE each        |  positionCount * 4 bytes
+-------------------------------------------+
| byteData : raw bytes                      |  endOffsets[N-1] bytes total
+-------------------------------------------+
```

**Wire layout -- with nulls:**

```
+-------------------------------------------+
| positionCount : int32 LE                  |  4 bytes
+-------------------------------------------+
| hasNulls : 0x01                           |  1 byte
+-------------------------------------------+
| nullBitmap : byte[]                       |  ceil(positionCount / 8) bytes
+-------------------------------------------+
| nonNullOffsetCount : int32 LE             |  4 bytes
+-------------------------------------------+
| endOffsets[0..M-1] : int32 LE each        |  nonNullOffsetCount * 4 bytes
+-------------------------------------------+
| byteData : raw bytes                      |  endOffsets[M-1] bytes total
+-------------------------------------------+
```

Note that the byte data only contains the data for non-null entries. Null entries contribute zero bytes to the data slice (their offsets are suppressed). The reader uses the null bitmap and the compacted offsets to reconstruct the full offsets array with repeated offsets for null positions.

### 2.10 `DictionaryBlockEncoding`

**Source:** `io.trino.spi.block.DictionaryBlockEncoding` -- NAME = `"DICTIONARY"`

Dictionary encoding wraps another block (the dictionary of unique values) with an `int[]` array of ids mapping logical positions to dictionary entries.

**Write:**

```java
public void writeBlock(BlockEncodingSerde blockEncodingSerde,
        SliceOutput sliceOutput, Block block)
{
    DictionaryBlock dictionaryBlock = (DictionaryBlock) block;
    if (!dictionaryBlock.isCompact()) {
        throw new IllegalArgumentException("Dictionary should be compact");
    }
    int positionCount = dictionaryBlock.getPositionCount();
    sliceOutput.appendInt(positionCount);

    // Recursively serialize the dictionary block
    Block dictionary = dictionaryBlock.getDictionary();
    blockEncodingSerde.writeBlock(sliceOutput, dictionary);

    // Write the raw id mapping
    sliceOutput.writeInts(dictionaryBlock.getRawIds(),
            dictionaryBlock.getRawIdsOffset(), dictionaryBlock.getPositionCount());
}
```

**Read (with flattening):**

```java
public Block readBlock(BlockEncodingSerde blockEncodingSerde, SliceInput sliceInput)
{
    int positionCount = sliceInput.readInt();
    Block dictionaryBlock = blockEncodingSerde.readBlock(sliceInput);  // recursive
    int[] ids = new int[positionCount];
    sliceInput.readInts(ids);
    return dictionaryBlock.copyPositions(ids, 0, ids.length);  // FLATTEN on read!
}
```

**Critical observation:** On read, `DictionaryBlockEncoding` **does not reconstruct a `DictionaryBlock`**. Instead, it calls `copyPositions()` which flattens the dictionary back into a plain block. This means dictionary encoding is only a wire-format optimization (reducing size for repeated values), not a runtime optimization that survives deserialization.

**Write-time replacement (`replacementBlockForWrite`):**

```java
public Optional<Block> replacementBlockForWrite(Block block)
{
    DictionaryBlock dictionaryBlock = (DictionaryBlock) block;
    if (!dictionaryBlock.isCompact()) {
        return Optional.of(dictionaryBlock.compact());
    }
    if (dictionaryBlock.getUniqueIds() == dictionaryBlock.getPositionCount()
            && dictionaryBlock.isSequentialIds()) {
        return Optional.of(dictionaryBlock.getDictionary());  // identity -> unwrap
    }
    return Optional.empty();
}
```

If the dictionary has one unique id per position with sequential mapping, it is replaced with the underlying block (no dictionary encoding written). If the dictionary is not compact (has unused entries), it is compacted first.

**Wire layout:**

```
+-------------------------------------------+
| positionCount : int32 LE                  |  4 bytes
+-------------------------------------------+
| <recursive: dictionary block>             |  self-describing (with its own envelope)
|   nameLength + name + block data          |
+-------------------------------------------+
| ids[0..N-1] : int32 LE each              |  positionCount * 4 bytes
+-------------------------------------------+
```

### 2.12 `RowBlockEncoding` (ROW types, struct-like)

**Source:** `io.trino.spi.block.RowBlockEncoding` -- NAME = `"ROW"`

Row blocks contain N field blocks (one per struct field), each with the same positionCount.

```java
public void writeBlock(BlockEncodingSerde blockEncodingSerde,
        SliceOutput sliceOutput, Block block)
{
    RowBlock rowBlock = (RowBlock) block;
    sliceOutput.appendInt(rowBlock.getPositionCount());          // [1]

    List<Block> fieldBlocks = rowBlock.getFieldBlocks();
    sliceOutput.appendInt(fieldBlocks.size());                   // [2] numFields
    for (Block fieldBlock : fieldBlocks) {
        blockEncodingSerde.writeBlock(sliceOutput, fieldBlock);  // [3] recursive per field
    }

    encodeNullsAsBits...(sliceOutput, rowBlock.getRawRowIsNull(),
            rowBlock.getOffsetBase(), rowBlock.getPositionCount());  // [4] row-level nulls
}
```

**Wire layout:**

```
+-------------------------------------------+
| positionCount : int32 LE                  |  4 bytes
+-------------------------------------------+
| numFields : int32 LE                      |  4 bytes
+-------------------------------------------+
| <recursive: fieldBlock[0]>                |  self-describing
| <recursive: fieldBlock[1]>                |  self-describing
| ...                                       |
| <recursive: fieldBlock[numFields-1]>      |  self-describing
+-------------------------------------------+
| hasNulls : byte                           |  1 byte
+-------------------------------------------+
| (if hasNulls) nullBitmap : byte[]         |  ceil(positionCount / 8) bytes
+-------------------------------------------+
```

**Note:** The null bitmap for the row block is written *after* the field blocks, which is the opposite order from primitive blocks (where nulls come before values). This is because the ROW-level null means "the entire row is null" -- even if the child field blocks have values at those positions, they should be treated as absent.

### 2.13 `ArrayBlockEncoding` (ARRAY types)

**Source:** `io.trino.spi.block.ArrayBlockEncoding` -- NAME = `"ARRAY"`

```java
public void writeBlock(BlockEncodingSerde blockEncodingSerde,
        SliceOutput sliceOutput, Block block)
{
    ArrayBlock arrayBlock = (ArrayBlock) block;
    int positionCount = arrayBlock.getPositionCount();
    int offsetBase = arrayBlock.getOffsetBase();
    int[] offsets = arrayBlock.getOffsets();

    int valuesStartOffset = offsets[offsetBase];
    int valuesEndOffset = offsets[offsetBase + positionCount];
    Block values = arrayBlock.getRawElementBlock()
            .getRegion(valuesStartOffset, valuesEndOffset - valuesStartOffset);

    blockEncodingSerde.writeBlock(sliceOutput, values);              // [1] element block

    sliceOutput.appendInt(positionCount);                            // [2]
    for (int position = 0; position < positionCount + 1; position++) {
        sliceOutput.writeInt(offsets[offsetBase + position] - valuesStartOffset);  // [3]
    }

    encodeNullsAsBits...(sliceOutput, arrayBlock.getRawValueIsNull(),
            offsetBase, positionCount);                              // [4]
}
```

**Wire layout:**

```
+-------------------------------------------+
| <recursive: element block>                |  self-describing (all array elements)
+-------------------------------------------+
| positionCount : int32 LE                  |  4 bytes
+-------------------------------------------+
| offsets[0..N] : int32 LE each             |  (positionCount + 1) * 4 bytes
+-------------------------------------------+
| hasNulls : byte                           |  1 byte
+-------------------------------------------+
| (if hasNulls) nullBitmap : byte[]         |  ceil(positionCount / 8) bytes
+-------------------------------------------+
```

**Offsets here are start/end offsets** (the standard `positionCount + 1` convention). The element block at position `i` spans elements `offsets[i]` to `offsets[i+1]`. This is different from `VariableWidthBlockEncoding` which only writes ending offsets.

### 2.14 `MapBlockEncoding` (MAP types)

**Source:** `io.trino.spi.block.MapBlockEncoding` -- NAME = `"MAP"`

The most complex encoding. A map block contains: a type descriptor (MapType), separate key and value blocks, an optional hash table, offsets, and a null bitmap.

```java
public void writeBlock(BlockEncodingSerde blockEncodingSerde,
        SliceOutput sliceOutput, Block block)
{
    MapBlock mapBlock = (MapBlock) block;
    int positionCount = mapBlock.getPositionCount();
    int offsetBase = mapBlock.getOffsetBase();
    int[] offsets = mapBlock.getOffsets();
    Optional<int[]> hashTable = mapBlock.getHashTables().tryGet();

    int entriesStartOffset = offsets[offsetBase];
    int entriesEndOffset = offsets[offsetBase + positionCount];

    blockEncodingSerde.writeType(sliceOutput, mapBlock.getMapType());  // [1] type

    blockEncodingSerde.writeBlock(sliceOutput,
            mapBlock.getRawKeyBlock().getRegion(
                entriesStartOffset, entriesEndOffset - entriesStartOffset));   // [2] keys
    blockEncodingSerde.writeBlock(sliceOutput,
            mapBlock.getRawValueBlock().getRegion(
                entriesStartOffset, entriesEndOffset - entriesStartOffset));   // [3] values

    if (hashTable.isPresent()) {
        int hashTableLength = (entriesEndOffset - entriesStartOffset) * HASH_MULTIPLIER;
        sliceOutput.appendInt(hashTableLength);
        sliceOutput.writeInts(hashTable.get(),
                entriesStartOffset * HASH_MULTIPLIER, hashTableLength);  // [4a] hash table
    } else {
        sliceOutput.appendInt(-1);                                       // [4b] no hash table
    }

    sliceOutput.appendInt(positionCount);                                // [5]
    for (int position = 0; position < positionCount + 1; position++) {
        sliceOutput.writeInt(offsets[offsetBase + position] - entriesStartOffset);  // [6]
    }

    encodeNullsAsBits...(sliceOutput, mapBlock.getRawMapIsNull(),
            offsetBase, positionCount);                                  // [7]
}
```

**Wire layout:**

```
+-------------------------------------------+
| <type envelope>                           |  length-prefixed UTF-8 MapType descriptor
|   typeNameLen : int32 LE                  |
|   typeName : UTF-8 bytes                  |  e.g. "map(varchar,bigint)"
+-------------------------------------------+
| <recursive: key block>                    |  self-describing
+-------------------------------------------+
| <recursive: value block>                  |  self-describing
+-------------------------------------------+
| hashTableLength : int32 LE               |  4 bytes (-1 = absent)
+-------------------------------------------+
| (if >= 0) hashTable : int32 LE each      |  hashTableLength * 4 bytes
+-------------------------------------------+
| positionCount : int32 LE                  |  4 bytes
+-------------------------------------------+
| offsets[0..N] : int32 LE each             |  (positionCount + 1) * 4 bytes
+-------------------------------------------+
| hasNulls : byte                           |  1 byte
+-------------------------------------------+
| (if hasNulls) nullBitmap : byte[]         |  ceil(positionCount / 8) bytes
+-------------------------------------------+
```

**Hash table detail:** `HASH_MULTIPLIER = 2`, so the hash table has `2 * totalEntries` slots. Each slot is an `int32`. The value `-1` for `hashTableLength` signals that no hash table was serialized (it will be recomputed lazily on read).

### 2.15 `VariantBlockEncoding` (Semi-structured JSON-like data)

**Source:** `io.trino.spi.block.VariantBlockEncoding` -- NAME = `"VARIANT"`

```java
public void writeBlock(...)
{
    VariantBlock variantBlock = (VariantBlock) block;
    sliceOutput.appendInt(variantBlock.getPositionCount());
    blockEncodingSerde.writeBlock(sliceOutput, variantBlock.getMetadata());  // metadata block
    blockEncodingSerde.writeBlock(sliceOutput, variantBlock.getValues());    // values block
    encodeNullsAsBits...(sliceOutput, variantBlock.getRawIsNull(), ...);
}
```

**Wire layout:**

```
+-------------------------------------------+
| positionCount : int32 LE                  |  4 bytes
+-------------------------------------------+
| <recursive: metadata block>               |  self-describing
+-------------------------------------------+
| <recursive: values block>                 |  self-describing
+-------------------------------------------+
| hasNulls : byte                           |  1 byte
+-------------------------------------------+
| (if hasNulls) nullBitmap : byte[]         |  ceil(positionCount / 8) bytes
+-------------------------------------------+
```

### 2.16 Unified Summary: Primitive Block Encoding Pattern

All fixed-width primitive blocks follow the same pattern:

```
[positionCount: int32] [nullBitmap: 1+N bytes] [values: compacted if nulls present]
```

The parameterization by value width:

| Encoding | Value Width | Write Method | Read Method |
|----------|------------|--------------|-------------|
| `BYTE_ARRAY` | 1 byte | `writeBytes` | `readBytes` |
| `SHORT_ARRAY` | 2 bytes | `writeShorts` | `readShorts` |
| `INT_ARRAY` | 4 bytes | `writeInts` | `readInts` |
| `LONG_ARRAY` | 8 bytes | `writeLongs` | `readLongs` |
| `INT128_ARRAY` | 16 bytes (2 longs) | `writeLongs` | `readLongs` |
| `FIXED12` | 12 bytes (3 ints) | `writeInts` | `readInts` |

The `BYTE_ARRAY`, `SHORT_ARRAY`, `INT_ARRAY`, and `LONG_ARRAY` encodings all have SIMD-accelerated compress/expand paths using `jdk.incubator.vector`. The `INT128_ARRAY` and `FIXED12` encodings use scalar loops because their multi-element-per-position structure does not map cleanly to SIMD lanes.

### 2.17 Concrete Byte-Level Example

**Example: LongArrayBlock with 4 positions, positions 1 and 3 are null**

In-memory state:
- `positionCount = 4`
- `values = [100, 0, 200, 0]` (values at null positions are don't-care)
- `isNull = [false, true, false, true]`

**Serialized bytes (hex, little-endian):**

```
Envelope:
  0A 00 00 00              name_length = 10
  4C 4F 4E 47 5F 41 52 52  "LONG_ARR"
  41 59                     "AY"

Block payload:
  04 00 00 00              positionCount = 4

  01                       hasNulls = true

  50                       nullBitmap = 0b01010000
                           (pos0=0, pos1=1, pos2=0, pos3=1, pad=0000)
                           MSB-first: bit7=pos0(0), bit6=pos1(1),
                                      bit5=pos2(0), bit4=pos3(1)

  02 00 00 00              nonNullCount = 2

  64 00 00 00 00 00 00 00  values[0] = 100 (0x64) LE
  C8 00 00 00 00 00 00 00  values[1] = 200 (0xC8) LE
```

Total: 14 (envelope: 4 len + 10 name) + 4 (posCount) + 1 (hasNulls) + 1 (bitmap) + 4 (nonNullCount) + 16 (values) = 40 bytes.

Without null compaction (hypothetical), the values would be 32 bytes (4 * 8). With compaction: 16 bytes (2 * 8) + 4 bytes (count) + 1 byte (bitmap) = 21 bytes for the values+nulls section. The savings scale with null density.

**Example: VariableWidthBlock with 3 positions ["ab", null, "cdef"]**

In-memory state:
- `positionCount = 3`
- `rawSlice = "abcdef"` (6 bytes)
- `rawOffsets = [0, 2, 2, 6]` (null entry has same start/end offset)
- `isNull = [false, true, false]`

**Serialized bytes:**

```
Envelope:
  0E 00 00 00              name_length = 14
  56 41 52 49 41 42 4C 45  "VARIABLE"
  5F 57 49 44 54 48        "_WIDTH"

Block payload:
  03 00 00 00              positionCount = 3

  01                       hasNulls = true
  40                       nullBitmap = 0b01000000 = 0x40
                           MSB-first: bit7=pos0(0), bit6=pos1(1),
                                      bit5=pos2(0), pad=00000

  02 00 00 00              nonNullOffsetCount = 2 (offsets for "ab" and "cdef")
  02 00 00 00              endOffset[0] = 2 (end of "ab")
  06 00 00 00              endOffset[1] = 6 (end of "cdef")

  61 62 63 64 65 66        byteData = "abcdef" (6 bytes)
```

**Example: RLE block with value 42 repeated 1000 times**

```
Envelope (RLE):
  03 00 00 00              name_length = 3
  52 4C 45                 "RLE"

RLE payload:
  E8 03 00 00              positionCount = 1000

  Nested single-value block (e.g., LongArrayBlock with 1 position):
    0A 00 00 00            name_length = 10
    4C 4F 4E 47 5F 41 52 52 41 59  "LONG_ARRAY"
    01 00 00 00            positionCount = 1
    00                     hasNulls = false
    2A 00 00 00 00 00 00 00  value = 42 LE
```

Total: 7 (RLE envelope) + 4 (count) + 14 (nested envelope) + 4 + 1 + 8 (nested block) = 38 bytes to represent 1000 values. Without RLE: ~8000+ bytes.

## 3. Key Design Insights

1. **Null compaction is the core wire-format optimization.** Every primitive block encoding writes only non-null values, preceded by a count. Combined with the bit-packed null bitmap, this can dramatically reduce wire size for nullable columns. A column that is 50% null saves nearly 50% of value bytes.

2. **The null bitmap bit ordering is MSB-first, not LSB-first.** This is a departure from Apache Arrow's convention (which uses LSB-first within each byte). The encoding packs position 0 into bit 7 of byte 0, position 7 into bit 0 of byte 0, position 8 into bit 7 of byte 1, etc. A Rust implementation must match this exact ordering.

3. **VariableWidthBlock stores ending offsets, not start/end pairs.** The offset array has `nonNullOffsetCount` elements (one per non-null position), where each element is the cumulative byte position after that entry's data. This is different from Arrow's `offsets[positionCount + 1]` convention and from ArrayBlock/MapBlock's `positionCount + 1` convention used within Trino itself.

4. **DictionaryBlock is flattened on read.** The reader does NOT reconstruct a `DictionaryBlock`. It calls `copyPositions()` to produce a plain block. This means the dictionary format is purely a serialization-time compression -- the receiver never operates on dictionary-encoded data from the wire. This is important for Rust porting: the deserializer need only produce the flattened block type.

5. **Recursive self-describing format.** Complex types (Row, Array, Map, Dictionary, RLE) call back into `BlockEncodingSerde.writeBlock()` for their children, which re-emits the full envelope (name prefix + block data) for each child. This means the format is a tree of self-describing blocks with no external schema needed. A parser can decode the stream knowing only the block encoding names.

6. **MapBlock includes a serialized hash table.** Unlike other complex types, MapBlock carries an optional precomputed hash table (with `HASH_MULTIPLIER = 2`, so 2 slots per key). This avoids recomputing hashes on the receiver side. If the hash table is absent (`-1` sentinel), it is lazily rebuilt.

7. **MapBlock includes a type descriptor.** MapBlock is the only block encoding that serializes type information (the `MapType`). This is because `MapHashTables` needs the `MapType` for correct hash computation during lazy rebuild.

8. **Ordering of metadata varies by block type.** There is no single field order that applies to all block types:
   - Primitives: `positionCount -> nullBitmap -> values`
   - VariableWidth: `positionCount -> nullBitmap -> offsets -> data`
   - Row: `positionCount -> numFields -> fieldBlocks -> nullBitmap` (nulls last)
   - Array: `elementBlock -> positionCount -> offsets -> nullBitmap` (element first)
   - Map: `type -> keyBlock -> valueBlock -> hashTable -> positionCount -> offsets -> nullBitmap`
   - Dictionary: `positionCount -> dictionaryBlock -> ids`
   - RLE: `positionCount -> valueBlock`

9. **SIMD acceleration for null bit-packing.** The vectorized path uses `ByteVector.SPECIES_64` (8-byte SIMD operations) to pack/unpack 8 boolean values at a time. For compress/expand of value arrays, full-width SIMD species (`SPECIES_PREFERRED`) are used with `compress` and `expand` vector operations. These produce bit-identical wire output to the scalar paths.

10. **Block replacement loop in the serde.** `InternalBlockEncodingSerde.writeBlock()` uses a `while(true)` loop that checks `replacementBlockForWrite()` before serializing. This allows `DictionaryBlockEncoding` to compact non-compact dictionaries or unwrap identity-mapped dictionaries without the caller knowing. This is an extensibility pattern worth preserving.

## 4. Porting Considerations (Java -> Rust)

### 4.1 Byte Order
All multi-byte values are little-endian. Rust's `byteorder` crate or direct `to_le_bytes()` / `from_le_bytes()` methods should be used consistently. On little-endian platforms (x86_64, ARM), this is a no-op for bulk writes. Use `std::io::Write` with explicit LE encoding, or write through a buffer with `LittleEndian` byte order.

### 4.2 Null Bitmap Implementation
The MSB-first bit ordering requires explicit shifting: position `i` within byte `i/8` uses mask `0x80 >> (i % 8)`. This maps naturally to:

```rust
fn encode_null_bitmap(is_null: &[bool]) -> Vec<u8> {
    let mut packed = vec![0u8; (is_null.len() + 7) / 8];
    for (i, &null) in is_null.iter().enumerate() {
        if null {
            packed[i / 8] |= 0x80 >> (i % 8);
        }
    }
    packed
}
```

Consider using SIMD intrinsics (e.g., `std::simd` or `packed_simd2`) for the vectorized path, processing 8 booleans at a time.

### 4.3 Null Compaction Pattern
The compress/expand pattern for nullable data is idiomatic in Rust using iterators:

```rust
// Compress (write only non-null values)
fn compress_with_nulls<T: Copy>(values: &[T], is_null: &[bool]) -> Vec<T> {
    values.iter().zip(is_null).filter(|(_, &n)| !n).map(|(&v, _)| v).collect()
}
```

However, the SIMD-accelerated in-place expand on read is more complex. The Java code reads compacted values into a portion of the target array and then uses SIMD `expand` operations to spread them back. A Rust equivalent would use `std::simd` `simd_swizzle` or scatter operations if available, or fall back to the scalar reverse-scan approach.

### 4.4 String/Length-Prefixed Encoding Name
The encoding name is a `int32 length + UTF-8 bytes` prefix. In Rust, this maps to reading 4 LE bytes as `u32`, then reading that many bytes as `&str`. Encoding names are ASCII-only in practice (`"LONG_ARRAY"`, `"VARIABLE_WIDTH"`, etc.), so `str::from_utf8()` should always succeed.

### 4.5 Recursive Dispatch
The recursive `blockEncodingSerde.writeBlock()/readBlock()` pattern maps to Rust's trait dispatch. Define a `BlockEncoding` trait:

```rust
trait BlockEncoding {
    fn name(&self) -> &str;
    fn write_block(&self, serde: &dyn BlockEncodingSerde, out: &mut dyn Write, block: &Block);
    fn read_block(&self, serde: &dyn BlockEncodingSerde, input: &mut dyn Read) -> Block;
}
```

Use a `HashMap<String, Box<dyn BlockEncoding>>` for name-to-encoding dispatch on read, and either `TypeId`-based dispatch or enum matching for write.

### 4.6 DictionaryBlock Flattening on Read
Since the Java reader flattens dictionary blocks via `copyPositions()`, the Rust reader can simply expand dictionary-encoded data inline. However, for write-side optimization, consider keeping a dictionary representation to avoid re-encoding overhead. The key insight: the wire format is the contract, not the in-memory representation.

### 4.7 VariableWidthBlock Offset Convention
The VariableWidth encoding writes *ending* offsets only (not `positionCount + 1` start/end offsets). The Rust reader must reconstruct the full offsets array:
- Position `i` data spans `[offsets[i-1], offsets[i])` where `offsets[-1] = 0`
- When nulls are present, decompress by scanning the null bitmap and repeating the previous offset for null positions

### 4.8 MapBlock Hash Table
The hash table is optional and can be omitted (written as `-1`). For a Rust implementation, lazy hash table construction on first lookup is the cleanest approach. The hash table uses open addressing with `HASH_MULTIPLIER = 2` (2 slots per entry).

### 4.9 No Alignment Guarantees
The wire format has no alignment padding between fields. Values are packed immediately after the preceding field. This means a Rust reader cannot simply cast byte slices to `&[i64]` without handling alignment. Use `read_i64::<LittleEndian>()` or copy into aligned buffers.

### 4.10 Wire Format Compatibility Matrix
For interoperability with Java Trino workers, the Rust implementation must produce byte-identical output for the same logical data. Key pitfalls:
- The `hasNulls` flag byte must be `0x01` when `isNull` is non-null (even if all values happen to be non-null). The Java code checks `isNull != null`, not "any null exists."
- Dictionary blocks must be compacted before serialization. The `replacementBlockForWrite` logic must be replicated.
- MapBlock must serialize the `MapType` as a length-prefixed UTF-8 type ID string.

### 4.11 Page-Level Wire Format Context
Each serialized block sits within a page wire format:

```
Page header (12 bytes):
  positionCount : int32 LE
  uncompressedSize : int32 LE
  compressedSize : int32 LE (high bit = compression flag)

Page body (may be LZ4/zstd compressed):
  channelCount : int32 LE
  block[0] : self-describing (envelope + payload)
  block[1] : self-describing
  ...
  block[channelCount-1] : self-describing
```

The compression wraps the entire page body, not individual blocks. The Rust implementation should serialize all blocks into a buffer first, then optionally compress the entire buffer.
