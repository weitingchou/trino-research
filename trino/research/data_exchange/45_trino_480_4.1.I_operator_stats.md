## Section 1: High-Level Overview

Trino uses a four-level stats hierarchy to report execution progress from worker to coordinator:

```
TaskStats                          (one per task, top-level)
  +-- List<PipelineStats>          (one per pipeline within the task)
        +-- List<DriverStats>      (one per driver within the pipeline)
        |     +-- List<OperatorStats>  (one per operator within the driver)
        +-- List<OperatorStats>    (operatorSummaries -- merged across drivers)
```

All four classes are immutable value objects serialized as JSON via Jackson `@JsonProperty` annotations. The worker builds these objects from live counters and returns them inside `TaskInfo` responses. The coordinator then aggregates `TaskStats` from all worker tasks to build `StageStats` and ultimately `QueryStats`.

### Data Flow: Worker -> Coordinator

1. Worker periodically snapshots live counters into `TaskStats` (a Java record).
2. `TaskStats` is embedded inside `TaskInfo`, which is returned via HTTP to the coordinator.
3. The coordinator's `StageStateMachine.getStageInfo()` iterates over all `TaskInfo` objects and sums their `TaskStats` fields into `StageStats`.
4. `StageStateMachine.getBasicStageStats()` performs a lighter-weight aggregation for fast status polling.
5. `QueryStateMachine` further aggregates `StageStats` into `QueryStats`.

### Summarization

Each stats level has a `summarize()` method that strips detailed sub-objects to reduce JSON payload size:
- `TaskStats.summarize()` -- drops the `pipelines` list entirely (empty list).
- `TaskStats.summarizeFinal()` -- keeps pipelines but calls `PipelineStats.summarize()` on each.
- `PipelineStats.summarize()` -- drops the `drivers` list and calls `OperatorStats.summarize()` on each operator summary.
- `OperatorStats.summarize()` -- strips non-final `OperatorInfo` (sets `info` to null).

### 2.5 JSON Representation of Duration and DataSize

Airlift's `Duration` and `DataSize` use `@JsonValue` on their `toString()` methods and `@JsonCreator` on their static `valueOf(String)` methods. They serialize as human-readable strings:

**Duration format:** `"<value><unit>"` where unit is one of `ns`, `us`, `ms`, `s`, `m`, `h`, `d`.
- Examples: `"3.00ns"`, `"15.50ms"`, `"2.30s"`, `"1.25m"`
- The `convertToMostSuccinctTimeUnit()` method picks the most readable unit.

**DataSize format:** `"<value><unit>"` where unit is one of `B`, `kB`, `MB`, `GB`, `TB`, `PB`.
- Examples: `"0B"`, `"51B"`, `"1024kB"`, `"3.50MB"`
- The `succinctBytes()` and `succinctDataSize()` helpers pick the most readable unit.

In the `OperatorStats.add()` method, the pattern for aggregation is:
```java
// Convert to raw units for arithmetic
long addInputWall = this.addInputWall.roundTo(NANOSECONDS);
long physicalInputDataSize = this.physicalInputDataSize.toBytes();

// Sum across operators...
addInputWall += operator.getAddInputWall().roundTo(NANOSECONDS);

// Convert back to succinct form
new Duration(addInputWall, NANOSECONDS).convertToMostSuccinctTimeUnit()
DataSize.ofBytes(physicalInputDataSize)
```

**Instant format:** ISO-8601 string, e.g., `"2024-01-15T10:30:00.000Z"`. Used by Jackson's built-in `JavaTimeModule`.

### 2.6 The Metrics Type

**Source:** `core/trino-spi/src/main/java/io/trino/spi/metrics/Metrics.java`

`Metrics` wraps a `Map<String, Metric<?>>` and uses `@JsonValue`/`@JsonCreator` so it serializes as a plain JSON object whose keys are metric names and whose values are polymorphic `Metric` objects.

```java
public class Metrics implements Mergeable<Metrics> {
    public static final Metrics EMPTY = new Metrics(Map.of());
    private final Map<String, Metric<?>> metrics;

    @JsonCreator
    public Metrics(Map<String, Metric<?>> metrics) { ... }

    @JsonValue
    public Map<String, Metric<?>> getMetrics() { return metrics; }
}
```

The `Metric<T>` interface uses `@JsonTypeInfo(use = JsonTypeInfo.Id.CLASS)`, meaning the full Java class name appears as `@class` in JSON. This is different from the `@type` name-based dispatch used elsewhere in Trino.

**Metric implementations in the SPI:**

| Interface | Purpose |
|-----------|---------|
| `Count<T>` | Has `getTotal(): long` |
| `Distribution<T>` | Has `getTotal()`, `getMin()`, `getMax()`, `getPercentiles()` |
| `Timing<T>` | Has `getDuration(): java.time.Duration` |

**Concrete implementations used by Trino internally (in `trino-plugin-toolkit`):**

| Class | Extends | JSON Fields |
|-------|---------|-------------|
| `LongCount` | `Count<LongCount>` | `{ "@class": "io.trino.plugin.base.metrics.LongCount", "total": 42 }` |
| `DurationTiming` | `Timing<DurationTiming>` | timing with duration |
| `TDigestHistogram` | `Distribution<TDigestHistogram>` | T-digest percentile histogram |

The `Metrics.Accumulator` class handles merging: it groups metrics by key name and calls `mergeWith()` on each `Metric` implementation (e.g., `LongCount` sums, distributions merge their digests).

### 2.9 Which Fields the Coordinator Requires

**Source:** `core/trino-main/src/main/java/io/trino/execution/StageStateMachine.java`

The coordinator reads `TaskStats` fields in two methods:

**`getBasicStageStats()` -- lightweight polling (lines 260-406):** Reads these `TaskStats` fields:
- `totalDrivers`, `queuedDrivers`, `runningDrivers`, `completedDrivers`, `blockedDrivers`
- `cumulativeUserMemory`
- `userMemoryReservation().toBytes()`, `revocableMemoryReservation().toBytes()`
- `totalScheduledTime().roundTo(NANOSECONDS)`, `totalCpuTime().roundTo(NANOSECONDS)`
- `fullyBlocked`, `blockedReasons`
- `physicalInputDataSize().toBytes()`, `physicalInputPositions`
- `physicalInputReadTime().roundTo(NANOSECONDS)`
- `physicalWrittenDataSize().toBytes()`
- `internalNetworkInputDataSize().toBytes()`, `internalNetworkInputPositions`
- `processedInputPositions`
- `spilledDataSize().toBytes()`

**`getStageInfo()` -- full aggregation (lines 408-688):** Reads ALL of the above PLUS:
- `totalBlockedTime().roundTo(NANOSECONDS)`
- `processedInputDataSize().toBytes()`, `processedInputPositions`
- `inputBlockedTime().roundTo(NANOSECONDS)`
- `outputDataSize().toBytes()`, `outputPositions`
- `outputBlockedTime().roundTo(NANOSECONDS)`
- `physicalWrittenDataSize().toBytes()`
- `fullGcCount`, `fullGcTime().roundTo(SECONDS)`
- `pipelines()` -- iterates over `PipelineStats` to merge `OperatorStats`

**Fields the coordinator accesses from nested `PipelineStats`:**
```java
for (PipelineStats pipeline : taskStats.pipelines()) {
    taskOperatorSummaries += pipeline.getOperatorSummaries().size();
}
```

**`combineTaskOperatorSummaries()` -- operator merging (lines 706-734):**
```java
for (PipelineStats pipeline : taskInfo.stats().pipelines()) {
    long pipelineKeyMask = Integer.toUnsignedLong(pipeline.getPipelineId()) << 32;
    for (OperatorStats operator : pipeline.getOperatorSummaries()) {
        long combinedKey = pipelineKeyMask | Integer.toUnsignedLong(operator.getOperatorId());
        pipelineAndOperatorToStats.computeIfAbsent(combinedKey, statsListCreator).add(operator);
    }
}
```

This merges operators with the same `(pipelineId, operatorId)` across tasks using `OperatorStats.add()`.

**Required vs. best-effort:**

ALL `TaskStats` fields are structurally required -- the coordinator will NPE or fail if any non-nullable field is missing. The Java code uses `requireNonNull` for all object-typed fields and `checkArgument(>= 0)` for position/driver counts. However, the coordinator only _reads_ a subset of fields for its aggregation logic. Fields like `elapsedTime`, `queuedTime`, `terminatingStartTime`, `writerInputDataSize`, `maxWriterCount` are part of the JSON contract but are primarily used for the REST API response rather than for scheduling decisions.

## Section 3: Key Design Insights

### 3.1 Everything Is a String in the JSON Wire Format

Despite being numeric underneath, `Duration` and `DataSize` are serialized as human-readable strings (`"15.50ms"`, `"3.50MB"`). The coordinator immediately converts them back to raw numeric units (`toBytes()`, `roundTo(NANOSECONDS)`) for aggregation. This is a deliberate design trade-off: human-readable REST API responses at the cost of parse/format overhead on every stats update.

### 3.2 Hierarchical Summarization Controls Payload Size

The `summarize()` / `summarizeFinal()` pattern is critical for network efficiency. A task with 100 drivers and 10 operators each would produce 1,000 `OperatorStats` objects. The summarization pipeline:
- In-flight (non-done) tasks: `TaskStats.summarize()` drops pipelines entirely, sending only the 44 top-level fields.
- Completed tasks: `TaskStats.summarizeFinal()` preserves pipeline + operator summaries (merged across drivers) but drops per-driver stats and non-final `OperatorInfo`.

### 3.3 Operator Merging Uses a Clever Combined Key

In `combineTaskOperatorSummaries()`, the coordinator packs `pipelineId` and `operatorId` into a single `long` key (`pipelineId << 32 | operatorId`) for efficient grouping in a `Long2ObjectOpenHashMap`. This avoids `Pair<Integer, Integer>` boxing overhead.

### 3.4 Metrics Use Class-Based Polymorphism (Not Name-Based)

The `Metric<T>` interface uses `@JsonTypeInfo(use = JsonTypeInfo.Id.CLASS)`, meaning JSON contains the full Java class name (e.g., `"io.trino.plugin.base.metrics.LongCount"`). This is distinct from the name-based `@type` dispatch used for `OperatorInfo` and `PlanNode`. The class-based approach allows connectors to define arbitrary metric types without central registration.

### 3.5 TaskStats Is a Record, Others Are Classes

`TaskStats` is defined as a Java `record`, making it structurally different from `OperatorStats`, `DriverStats`, and `PipelineStats` which are traditional classes. Records get automatic `@JsonCreator` support in newer Jackson versions, but the JSON wire format is identical -- just `@JsonProperty` annotated fields.

### 3.6 PipelineStats Contains Both Summaries and Raw Drivers

`PipelineStats` holds two lists:
- `operatorSummaries: List<OperatorStats>` -- operators merged across all drivers in the pipeline
- `drivers: List<DriverStats>` -- individual per-driver stats

The coordinator only reads `operatorSummaries` for its aggregation. The `drivers` list is used for the detailed query info REST API but is stripped during summarization.

### 3.7 DistributionSnapshot at Pipeline Level

`PipelineStats` uses `DistributionSnapshot` (from `io.airlift.stats`) for `queuedTime` and `elapsedTime`, providing statistical distributions (count, min, max, mean, p50, p75, p90, p95, p99). At the `DriverStats` level, these are simple `Duration` values. The aggregation from driver-level to pipeline-level creates the distribution.

## Section 4: Porting Considerations (Java -> Rust)

### 4.1 Duration and DataSize Parsing/Formatting

The Rust worker must serialize `Duration` and `DataSize` using the exact same string format as airlift:
- Duration: `{value}{unit}` where unit is `ns`, `us`, `ms`, `s`, `m`, `h`, `d`. The value uses up to 2 decimal places.
- DataSize: `{value}{unit}` where unit is `B`, `kB`, `MB`, `GB`, `TB`, `PB`.

Recommended approach: Create Rust newtypes `TrinoDataSize(u64)` and `TrinoDuration(u64)` that store raw bytes/nanoseconds internally but implement custom `serde::Serialize`/`serde::Deserialize` that produce the human-readable string format. Use `convertToMostSuccinctTimeUnit()` logic when serializing.

```rust
// Sketch of the serialization approach
struct TrinoDuration(u64); // nanoseconds

impl Serialize for TrinoDuration {
    fn serialize<S: Serializer>(&self, s: S) -> Result<S::Ok, S::Error> {
        s.serialize_str(&self.to_succinct_string())
    }
}
```

### 4.2 Struct Definitions for All Four Levels

Each level needs a `#[derive(Serialize, Deserialize)]` struct with `#[serde(rename_all = "camelCase")]`. The field names in JSON use camelCase matching the Java `@JsonProperty` names.

Key type mappings:
| Java Type | Rust Type |
|-----------|-----------|
| `int` | `i32` |
| `long` | `i64` |
| `double` | `f64` |
| `boolean` | `bool` |
| `String` | `String` |
| `Duration` | `TrinoDuration` (custom serde) |
| `DataSize` | `TrinoDataSize` (custom serde) |
| `Instant` | `chrono::DateTime<Utc>` or `String` (ISO-8601) |
| `Optional<T>` / `@Nullable` | `Option<T>` |
| `OptionalInt` | `Option<i32>` |
| `Set<BlockedReason>` | `HashSet<BlockedReason>` or `Vec<BlockedReason>` |
| `List<T>` | `Vec<T>` |
| `PlanNodeId` | `String` (serialized as plain string via `@JsonValue`) |
| `Metrics` | `HashMap<String, MetricValue>` (see below) |
| `DistributionSnapshot` | Custom struct with percentile fields |

### 4.3 Metrics Polymorphism Strategy

The `Metric` interface uses `@JsonTypeInfo(use = JsonTypeInfo.Id.CLASS)`, putting the full Java class name in `@class`. For the Rust worker, there are two approaches:

**Option A: Implement known types, pass-through others.**
```rust
#[derive(Serialize, Deserialize)]
#[serde(tag = "@class")]
enum MetricValue {
    #[serde(rename = "io.trino.plugin.base.metrics.LongCount")]
    LongCount { total: i64 },
    #[serde(rename = "io.trino.plugin.base.metrics.DurationTiming")]
    DurationTiming { duration: String },
    #[serde(other)]
    Unknown(serde_json::Value),
}
```

**Option B: Store as opaque JSON.** Since the Rust worker only produces metrics (it does not interpret coordinator-side metrics), store `Metrics` as `serde_json::Value` for pass-through, and populate known types when generating worker stats.

### 4.4 OperatorInfo Polymorphism

The Rust worker needs to produce `OperatorInfo` subtypes for operators it implements. Use a serde-tagged enum:
```rust
#[derive(Serialize, Deserialize)]
#[serde(tag = "@type")]
enum OperatorInfo {
    #[serde(rename = "partitionedOutput")]
    PartitionedOutput { rows_added: i64 },
    #[serde(rename = "directExchangeClientStatus")]
    DirectExchangeClientStatus { /* ... */ },
    // ... other types as needed
}
```

The `isFinal()` semantics should be tracked as a method on the Rust enum so `summarize()` knows which infos to strip.

### 4.5 Summarization Must Be Implemented

The coordinator calls `taskInfo.summarize()` on received `TaskInfo`, but the worker also summarizes before sending to reduce payload size. The Rust worker should implement:
- `TaskStats::summarize()` -- return self with empty `pipelines` vec
- `TaskStats::summarize_final()` -- return self with `pipelines` having their `drivers` cleared and `OperatorStats` summarized
- `OperatorStats::summarize()` -- set `info` to `None` unless `info.is_final()`

### 4.6 Field Count Is Large But Predictable

The total field counts (40 + 27 + 39 + 44 = 150 fields across the four levels) are large but fully determined at compile time. No dynamic schema. The Rust implementation benefits from `#[derive(Default)]` for zero-initialization and builder patterns for construction.

### 4.7 Aggregation Semantics

The `OperatorStats.add()` method reveals the aggregation contract:
- Counters (calls, positions, data sizes): **summed**
- Peak memory values: **max** across operators
- Blocked reason: **last non-empty value wins**
- Metrics: **merged** via `Metrics.Accumulator` (which delegates to each `Metric.mergeWith()`)
- OperatorInfo: **merged** via `Mergeable.mergeWith()` if the info type implements `Mergeable`; otherwise dropped

The Rust worker needs to implement this aggregation for its own internal summarization (pipeline-level summaries from driver-level stats).

### 4.8 BlockedReason Is Currently Single-Variant

The `BlockedReason` enum only has `WAITING_FOR_MEMORY`. However, since it appears as a `Set<BlockedReason>` in `DriverStats`/`PipelineStats`/`TaskStats` and as `Optional<BlockedReason>` in `OperatorStats`, the Rust code should treat it as an extensible enum (use `#[serde(other)]` for forward compatibility).

### 4.9 Priority of Implementation

For a minimal viable Rust worker, the critical path is:
1. `TaskStats` -- this is what the coordinator reads via the `TaskInfo` HTTP response.
2. `PipelineStats` -- needed inside `TaskStats.pipelines`.
3. `OperatorStats` -- needed inside `PipelineStats.operatorSummaries`.
4. `DriverStats` -- lowest priority; only needed for detailed query info, stripped during summarization.
