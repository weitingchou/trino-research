## 0. Research Focus
* **Task ID:** 4.2.F
* **Focus:** Understand the complete hash-based partitioning pipeline for shuffle: how per-type hash codes are computed, how multi-column hashes are combined, how 64-bit hashes are reduced to partition numbers, and how this drives the PartitionedOutputOperator.

## 1. High-Level Overview

Trino's shuffle partitioning uses a four-layer hash pipeline to deterministically route each row to a target partition:

1. **Per-Type Hash** -- Each type implements a `HASH_CODE` operator (or falls back to `XX_HASH_64`). Most fixed-width types use a custom xxHash64-derived "mix" function: `rotateLeft(value * 0xC2B2AE3D27D4EB4FL, 31) * 0x9E3779B185EBCA87L`. Variable-width types (VARCHAR, VARBINARY) use `XxHash64.hash()` from the airlift library directly.

2. **Multi-Column Combining** -- For multi-column partition keys, column hashes are combined left-to-right using `CombineHashFunction.getHash()`: `hash = 31 * previousHash + columnHash`. The initial seed is `0`.

3. **Hash-to-Partition Reduction** -- The 64-bit combined hash is reduced to `[0, partitionCount)` via a multiply-and-shift technique that avoids the modulo bias problem: `(Integer.toUnsignedLong(Long.hashCode(rawHash)) * partitionCount) >>> 32`.

4. **Bucket-to-Partition Indirection** -- A `bucketToPartition` array maps the computed bucket to the final partition index, allowing flexible partition assignment (e.g., multiple buckets mapping to the same partition).

The key design insight is that Trino does NOT use raw modulo for partition assignment. The multiply-shift reduction provides uniform distribution regardless of whether `partitionCount` is a power of two.

## 2. Detailed Analysis

### 2.1 The Partitioning Class Hierarchy

The partitioning system is organized into several distinct layers:

**Source:** `io.trino.operator.PartitionFunction` (interface)

```java
public interface PartitionFunction
{
    int partitionCount();
    int getPartition(Page page, int position);

    default void getPartitions(Page page, int positionOffset, int length, int[] partitions)
    {
        for (int i = 0; i < length; i++) {
            partitions[i] = getPartition(page, positionOffset + i);
        }
    }
}
```

The call chain for FIXED_HASH_DISTRIBUTION is:

```
PartitionedOutputOperator.addInput(page)
  -> PagePartitioner.partitionPage(page)
    -> PagePartitioner.partitionPositions(page)
      -> partitionFunction.getPartitions(args, offset, length, partitions)
        -> BucketPartitionFunction.getPartitions(...)
          -> HashBucketFunction.getBuckets(...)
            -> HashGenerator.getPartitions(bucketCount, offset, page, length, partitions)
              -> InterpretedHashGenerator.hash(page, offset, length, hashes)
                -> NullSafeHash.hash(block, position) per column
                -> CombineHashFunction.getHash(prevHash, colHash) to combine
              -> processRawHash(hashes[i], partitionCount)  // hash -> bucket
          -> bucketToPartition[bucket]  // bucket -> partition
```

### 2.3 The BucketPartitionFunction Indirection

**Source:** `io.trino.operator.BucketPartitionFunction`

This wraps a `BucketFunction` with a `bucketToPartition` lookup array, providing a level of indirection between hash buckets and output partitions:

```java
public class BucketPartitionFunction implements PartitionFunction
{
    private final BucketFunction bucketFunction;
    private final int[] bucketToPartition;

    @Override
    public int getPartition(Page functionArguments, int position) {
        int bucket = bucketFunction.getBucket(functionArguments, position);
        return bucketToPartition[bucket];
    }

    @Override
    public void getPartitions(Page functionArguments, int positionOffset, int length, int[] partitions) {
        bucketFunction.getBuckets(functionArguments, positionOffset, length, partitions);
        for (int i = 0; i < length; i++) {
            int bucket = partitions[i];
            partitions[i] = bucketToPartition[bucket];
        }
    }
}
```

This array is typically an identity mapping (bucket `i` maps to partition `i`), but it allows connector-specific bucketing schemes where the number of buckets differs from the number of partitions.

### 2.4 Per-Type Hash Code Computation

Types provide hash code operators through the `TypeOperatorDeclaration` system. The `NullSafeHashCompiler` retrieves the hash method at compile time via `typeOperators.getHashCodeOperator(type, ...)`.

**Critical fallback logic** in `TypeOperators.getOperatorMethodHandles()`:

```java
case HASH_CODE -> {
    Collection<OperatorMethodHandle> hashCodeOperators =
        typeOperatorDeclaration.getHashCodeOperators();
    if (hashCodeOperators.isEmpty()) {
        yield typeOperatorDeclaration.getXxHash64Operators();  // fallback!
    }
    yield hashCodeOperators;
}
```

This means some types use `HASH_CODE` (which is a custom fast-path function), while others fall back to `XX_HASH_64` (the full xxHash64 algorithm).

#### BIGINT / TIMESTAMP (short) / SHORT DECIMAL -- `AbstractLongType.hash()`

**Source:** `io.trino.spi.type.AbstractLongType`

```java
public static long hash(long value) {
    // xxHash64 mix
    return rotateLeft(value * 0xC2B2AE3D27D4EB4FL, 31) * 0x9E3779B185EBCA87L;
}

@ScalarOperator(HASH_CODE)
private static long hashCodeOperator(long value) {
    return hash(value);
}
```

This is a stripped-down single-round xxHash64 finalization mix. The constants `0xC2B2AE3D27D4EB4F` (PRIME64_2) and `0x9E3779B185EBCA87` (PRIME64_1) come directly from the xxHash64 algorithm. BigintType, ShortTimestampType, and ShortDecimalType all use this same function since they store their values as `long`.

#### INTEGER / SMALLINT / TINYINT -- `AbstractIntType` / `TinyintType` / `SmallintType`

These cast down to the appropriate width before hashing:

```java
// AbstractIntType (INTEGER):
@ScalarOperator(HASH_CODE)
private static long hashCodeOperator(long value) {
    return AbstractLongType.hash((int) value);
}

// SmallintType:
private static long hashCodeOperator(long value) {
    return AbstractLongType.hash((short) value);
}

// TinyintType:
private static long hashCodeOperator(long value) {
    return AbstractLongType.hash((byte) value);
}
```

The narrowing cast is significant: `AbstractLongType.hash(42L)` and `AbstractLongType.hash((int) 42)` produce different results because the cast changes the input from `0x000000000000002A` to `0x000000000000002A` (same in this case, but `hash(-1L)` vs `hash((int)-1)` would differ since -1L = `0xFFFFFFFFFFFFFFFF` while `(int)-1` sign-extends to `0x00000000FFFFFFFF` as a long... actually the cast to `(int)` is from long to int then implicitly widened back, so it sign-extends: `(int)-1 = -1` as a long = `0xFFFFFFFFFFFFFFFF`). The key point is that the narrowing truncates first, then `hash()` operates on the sign-extended long value.

#### DOUBLE

**Source:** `io.trino.spi.type.DoubleType`

```java
@ScalarOperator(HASH_CODE)
private static long hashCodeOperator(double value) {
    if (value == 0) {
        value = 0;  // Normalize -0.0 to +0.0
    }
    return AbstractLongType.hash(doubleToLongBits(value));
}
```

The `if (value == 0) value = 0` trick normalizes negative zero (`-0.0`) to positive zero (`+0.0`) so they hash identically, matching the equality semantics where `-0.0 == +0.0`.

#### REAL (float)

**Source:** `io.trino.spi.type.RealType`

```java
@ScalarOperator(HASH_CODE)
private static long hashCodeOperator(long value) {
    float realValue = intBitsToFloat((int) value);
    if (realValue == 0) {
        realValue = 0;  // Normalize -0.0
    }
    return AbstractLongType.hash(floatToIntBits(realValue));
}
```

Same negative-zero normalization as DOUBLE. Note that REAL is stored as a `long` in the block (only the low 32 bits are meaningful), but `floatToIntBits()` produces a 32-bit int that is then passed to `hash()`.

#### BOOLEAN -- Fallback to XX_HASH_64

**Source:** `io.trino.spi.type.BooleanType`

```java
private static final long TRUE_XX_HASH = XxHash64.hash(1);
private static final long FALSE_XX_HASH = XxHash64.hash(0);

@ScalarOperator(XX_HASH_64)
private static long xxHash64Operator(boolean value) {
    return value ? TRUE_XX_HASH : FALSE_XX_HASH;
}
```

BooleanType has NO `HASH_CODE` operator. Due to the fallback logic in `TypeOperators`, it uses `XX_HASH_64`. The hash values are precomputed constants since there are only two possible inputs. This means BOOLEAN's partition hash uses a DIFFERENT algorithm (full xxHash64) than BIGINT's custom mix function.

#### VARCHAR / VARBINARY -- Fallback to XX_HASH_64

**Source:** `io.trino.spi.type.AbstractVariableWidthType.DefaultComparableOperators`

```java
@ScalarOperator(XX_HASH_64)
private static long xxHash64Operator(Slice value) {
    return XxHash64.hash(value);
}

@ScalarOperator(XX_HASH_64)
private static long xxHash64Operator(@BlockPosition VariableWidthBlock block,
                                      @BlockIndex int position) {
    return XxHash64.hash(block.getRawSlice(),
                         block.getRawSliceOffset(position),
                         block.getSliceLength(position));
}
```

VARCHAR and VARBINARY have no `HASH_CODE` operator and fall back to `XX_HASH_64`. The hash is computed over the raw UTF-8 bytes of the slice using the full xxHash64 algorithm from the airlift library.

#### LONG DECIMAL (Int128)

**Source:** `io.trino.spi.type.LongDecimalType`

```java
@ScalarOperator(XX_HASH_64)
private static long xxHash64Operator(Int128 value) {
    return xxHash64(value.getHigh(), value.getLow());
}

private static long xxHash64(long high, long low) {
    return XxHash64.hash(high) ^ XxHash64.hash(low);
}
```

Long decimal has no `HASH_CODE` operator. It falls back to `XX_HASH_64`, which XORs the full xxHash64 of the high and low 64-bit halves.

#### LONG TIMESTAMP (precision > 6)

**Source:** `io.trino.spi.type.LongTimestampType`

```java
@ScalarOperator(XX_HASH_64)
private static long xxHash64Operator(LongTimestamp value) {
    return xxHash64(value.getEpochMicros(), value.getPicosOfMicro());
}

private static long xxHash64(long epochMicros, int fraction) {
    return XxHash64.hash(epochMicros) ^ XxHash64.hash(fraction);
}
```

Same XOR pattern as long decimal. Falls back to `XX_HASH_64`.

### 2.5 Multi-Column Hash Combining

**Source:** `io.trino.operator.scalar.CombineHashFunction`

```java
public static long getHash(long previousHashValue, long value) {
    return (31 * previousHashValue + value);
}
```

The initial hash value is `0` (from `HashGenerator.INITIAL_HASH_VALUE = 0`).

**Source:** `io.trino.operator.InterpretedHashGenerator.hashPosition()`

```java
public long hashPosition(int position, Page page) {
    long result = INITIAL_HASH_VALUE;  // 0
    for (int i = 0; i < hashCodeOperators.length; i++) {
        Block block = page.getBlock(hashChannels == null ? i : hashChannels[i]);
        result = CombineHashFunction.getHash(result,
            hashCodeOperators[i].hash(block.getUnderlyingValueBlock(),
                                       block.getUnderlyingValuePosition(position)));
    }
    return result;
}
```

For a 3-column key `(a, b, c)`:
```
hash = 31 * (31 * (31 * 0 + hash(a)) + hash(b)) + hash(c)
     = 31^2 * hash(a) + 31 * hash(b) + hash(c)
```

This is the same polynomial hash combining strategy used by Java's `Arrays.hashCode()` and `String.hashCode()`.

**Null handling:** The `NullSafeHashCompiler` generates code that checks `block.isNull(position)` and substitutes `NULL_HASH_CODE = 0` for null values:

```java
// Generated bytecode equivalent:
long hash;
if (block.isNull(position)) {
    hash = 0;  // NULL_HASH_CODE
} else {
    hash = typeHashCode(block, position);  // actual hash operator
}
```

### 2.6 Hash-to-Partition Reduction (processRawHash)

**Source:** `io.trino.operator.HashGenerator`

```java
private static int processRawHash(long rawHash, int partitionCount) {
    // This function reduces the 64 bit rawHash to [0, partitionCount) uniformly.
    // It first reduces the rawHash to 32 bit integer x then normalizes it to
    // x / 2^32 * partitionCount to reduce the range from [0, 2^32) to [0, partitionCount)
    return (int) ((Integer.toUnsignedLong(Long.hashCode(rawHash)) * partitionCount) >>> 32);
}
```

This is a three-step process:

1. **64-to-32 bit reduction:** `Long.hashCode(rawHash)` computes `(int)(rawHash ^ (rawHash >>> 32))`, mixing the high and low 32 bits via XOR. This produces a signed 32-bit integer.

2. **Unsigned interpretation:** `Integer.toUnsignedLong(x)` interprets the 32-bit result as an unsigned value in `[0, 2^32)`.

3. **Multiply-shift:** `(unsignedX * partitionCount) >>> 32` maps the unsigned 32-bit range `[0, 2^32)` uniformly to `[0, partitionCount)` without modulo bias. This is the "fastrange" technique from Daniel Lemire.

This avoids the classic `hash % N` modulo bias problem where values are not uniformly distributed when N is not a power of two.

### 2.7 Batched Hash Computation in InterpretedHashGenerator

The `InterpretedHashGenerator` optimizes batched hashing with special handling for different block types:

**Source:** `io.trino.operator.InterpretedHashGenerator`

```java
private static void hashFirstBlock(Block rawBlock, int positionOffset, int length,
                                    NullSafeHash hashOperator, long[] hashes) {
    switch (rawBlock) {
        case RunLengthEncodedBlock rleBlock -> {
            long hash = hashOperator.hash(rleBlock.getUnderlyingValueBlock(), 0);
            Arrays.fill(hashes, 0, length, hash);
        }
        case DictionaryBlock dictionaryBlock -> {
            if (isDictionaryProcessingFaster(dictionaryBlock, length)) {
                ValueBlock dictionary = dictionaryBlock.getDictionary();
                long[] dictionaryHashes = new long[dictionary.getPositionCount()];
                hashOperator.hashBatched(dictionary, dictionaryHashes, 0, dictionary.getPositionCount());
                for (int i = 0; i < length; i++) {
                    hashes[i] = dictionaryHashes[dictionaryBlock.getId(i + positionOffset)];
                }
            } else {
                hashOperator.hashBatchedDictionary(dictionaryBlock, hashes, positionOffset, length);
            }
        }
        case ValueBlock valueBlock ->
            hashOperator.hashBatched(valueBlock, hashes, positionOffset, length);
    }
}
```

Three optimizations:
- **RLE blocks:** Compute hash once, fill the entire array with the same value.
- **Dictionary blocks (large dictionary ratio):** If the block has 20% more positions than dictionary entries, hash the dictionary once and look up positions.
- **Plain value blocks:** Hash each position directly.

For subsequent columns (non-first), the same optimizations apply but each hash value is combined with the running hash via `CombineHashFunction.getHash()`.

### 2.8 PagePartitioner: End-to-End Usage

**Source:** `io.trino.operator.output.PagePartitioner`

The `PagePartitioner.partitionPage()` method is the entry point. It decides between two strategies:

```java
public void partitionPage(Page page, OperatorContext operatorContext) {
    // ...
    if (page.getPositionCount() < partitionFunction.partitionCount() * 2) {
        partitionPageByRow(page);    // row-at-a-time (small pages or many partitions)
    } else {
        partitionPageByColumn(page); // column-at-a-time (batched, optimized path)
    }
}
```

The column-wise path:
1. Extract partition function arguments (a sub-page of just the partitioning columns, possibly with constant columns substituted).
2. Call `partitionFunction.getPartitions()` to get the partition index for every row in batch.
3. Build per-partition position lists.
4. For each non-empty partition, append rows at those positions to that partition's output buffer.

**Null row handling:** When a null channel is present (outer joins), null rows are replicated to ALL partitions. The PagePartitioner separates null rows from non-null rows before partitioning:

```java
private void partitionNullablePositions(Page partitionFunctionArgs, int startPosition,
                                         Block nullsBlock, IntArrayList[] partitionPositions) {
    // Separate nulls from non-nulls
    for (int i = startPosition; i < partitionFunctionArgs.getPositionCount(); i++) {
        // ... branchless null/non-null separation
    }
    // Nulls go to ALL partitions
    for (IntArrayList positions : partitionPositions) {
        positions.addElements(startPosition, nullPositions, 0, nullCount);
    }
    // Non-nulls are hash-partitioned normally
    for (int i = 0; i < nonNullCount; i++) {
        int partition = partitionFunction.getPartition(partitionFunctionArgs, nonNullPositions[i]);
        partitionPositions[partition].add(nonNullPositions[i]);
    }
}
```

### 2.9 Skewed Partition Rebalancing

**Source:** `io.trino.operator.output.SkewedPartitionFunction`

For scaled writer hash partitioning, `SkewedPartitionFunction` wraps the normal partition function and adds dynamic rebalancing:

```java
public int getPartition(Page page, int position) {
    int partition = partitionFunction.getPartition(page, position);
    return skewedPartitionRebalancer.getTaskId(partition, partitionRowCount[partition]++);
}
```

The `SkewedPartitionRebalancer` monitors per-partition row counts and can redirect rows from overloaded partitions to different tasks, increasing downstream parallelism when data is skewed.

## 3. Key Design Insights

1. **Two hash algorithms, not one.** Trino uses two distinct hash functions depending on the type: a lightweight xxHash64 mix for fixed-width numeric types (BIGINT, INTEGER, DOUBLE, REAL, TIMESTAMP short, DECIMAL short) and the full xxHash64 algorithm for variable-width types (VARCHAR, VARBINARY) and types without a `HASH_CODE` operator (BOOLEAN, LONG DECIMAL, LONG TIMESTAMP). The `HASH_CODE -> XX_HASH_64` fallback in `TypeOperators` is the key mechanism.

2. **Combine function is `31 * prev + col`.** The multi-column combining step uses a simple polynomial hash (the same as Java's `String.hashCode()`). The initial seed is `0`. This means single-column hash = `31 * 0 + hash(col0) = hash(col0)`, so the combine step is a no-op for single-column keys.

3. **Modulo-free partition assignment.** The hash-to-partition reduction uses the multiply-shift ("fastrange") technique: `(unsigned32(hash) * N) >>> 32`. This avoids modulo bias and avoids the division instruction entirely, which matters at shuffle-level throughput.

4. **Bucket-to-partition indirection.** The `bucketToPartition` array decouples the number of hash buckets from the number of output partitions. This enables connector-specific bucketing (e.g., Hive bucket count) to coexist with Trino's partition count.

5. **NULL_HASH_CODE is 0.** Null values always hash to `0`. Since the combine function is `31 * prev + 0 = 31 * prev`, a trailing null column does not change the hash from the perspective of ordering (it just multiplies by 31).

6. **Bytecode generation for hash operators.** The `NullSafeHashCompiler` generates specialized bytecode classes at runtime for each type, inlining the null check and hash computation. This avoids virtual dispatch on the hot path.

7. **Block-type-aware batching.** The `InterpretedHashGenerator` has specialized paths for RLE blocks (compute once, fill array), dictionary blocks (hash dictionary, then lookup), and plain value blocks. This avoids redundant hash computation for compressed columnar data.

8. **Column-wise vs row-wise partitioning.** When the page has at least `2 * partitionCount` positions, the `PagePartitioner` uses the column-wise batched path. Below that threshold, it falls back to row-at-a-time to avoid the overhead of allocating per-partition position lists.

## 4. Porting Considerations (Java -> Rust)

### 4.1 Hash Algorithm Reproduction

The per-type `HASH_CODE` function (`rotateLeft(value * PRIME64_2, 31) * PRIME64_1`) must be reproduced **exactly** -- the constants are from xxHash64 but the function is NOT a standard xxHash64 call. It is a single round of the xxHash64 accumulation step, applied as a standalone mix function. A Rust port needs:

```rust
fn trino_long_hash(value: i64) -> i64 {
    let v = (value as u64).wrapping_mul(0xC2B2AE3D27D4EB4F_u64);
    let v = v.rotate_left(31);
    let v = v.wrapping_mul(0x9E3779B185EBCA87_u64);
    v as i64
}
```

For VARCHAR/VARBINARY, the standard xxHash64 algorithm must be used (the `xxhash-rust` crate implements this).

### 4.2 Combine Function

The combine function `31 * prev + col_hash` uses wrapping (overflow) arithmetic on signed 64-bit integers. In Rust, use `i64::wrapping_mul` and `i64::wrapping_add`:

```rust
fn combine_hash(previous: i64, value: i64) -> i64 {
    previous.wrapping_mul(31).wrapping_add(value)
}
```

### 4.3 processRawHash Reduction

The multiply-shift reduction is the most subtle part to port. In Java:

```java
(int) ((Integer.toUnsignedLong(Long.hashCode(rawHash)) * partitionCount) >>> 32)
```

This requires:
1. `Long.hashCode(x)` = `(int)(x ^ (x >>> 32))` -- XOR the high and low halves.
2. `Integer.toUnsignedLong(y)` = interpret the 32-bit result as unsigned (zero-extend).
3. Multiply by `partitionCount` (as `u64`), then shift right 32 bits.

In Rust:

```rust
fn process_raw_hash(raw_hash: i64, partition_count: u32) -> u32 {
    let xored = (raw_hash ^ (raw_hash >> 32)) as i32;
    let unsigned = xored as u32;
    ((unsigned as u64 * partition_count as u64) >> 32) as u32
}
```

### 4.4 Null Hash Code

`NULL_HASH_CODE = 0` is defined in `TypeUtils`. The Rust port must ensure null values produce hash `0`.

### 4.5 Type-Specific Considerations

- **DOUBLE/REAL:** The `-0.0` normalization is critical. In Rust, `f64::to_bits()` on `-0.0` produces `0x8000000000000000`, which is different from `+0.0` producing `0x0`. The port must normalize before converting to bits.
- **BOOLEAN:** Uses full xxHash64 (via the fallback), not the custom mix. The precomputed constants `XxHash64.hash(0)` and `XxHash64.hash(1)` can be hard-coded.
- **LONG DECIMAL / LONG TIMESTAMP:** Use `XxHash64.hash(high) ^ XxHash64.hash(low)` -- the XOR is significant (not the combine function).

### 4.6 Stability Guarantees

The hash function is NOT explicitly documented as stable across Trino versions. The `HASH_CODE` operator is an internal operator (not user-facing) that could theoretically change. However, changing it would break rolling upgrades where nodes run different versions, since both sides of a shuffle must agree on the hash function. In practice, these hash functions have been stable for many versions. A Rust worker must match the exact hash computation of the Java Trino version it will interoperate with.

### 4.7 Bytecode Generation Alternative

Java's `NullSafeHashCompiler` generates specialized bytecode at runtime. In Rust, this can be replaced by:
- **Monomorphization:** Generic functions parameterized by type, which the compiler specializes at compile time.
- **Enum dispatch:** A match on the type enum, with each arm calling the type-specific hash function. LLVM optimizes this well.
- No runtime code generation is needed since Rust's type system and generics provide compile-time specialization.

### 4.8 Batched Computation

The column-wise batched hashing pattern (hash all positions for column 0, then combine all positions for column 1, etc.) is cache-friendly and SIMD-amenable. In Rust, this maps naturally to operating on `&[i64]` slices. The RLE and dictionary optimizations should be preserved.
