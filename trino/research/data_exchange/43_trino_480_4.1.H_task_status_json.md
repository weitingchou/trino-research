**Source Version:** Trino 480

## Section 1: High-Level Overview

When a Trino coordinator communicates with workers, the primary feedback channel is a nested hierarchy of immutable stats objects that flow back as JSON. A Rust worker must be able to **produce** these exact JSON shapes so that the coordinator can deserialize them without error.

### Object Hierarchy

```
TaskInfo                          (top-level response from worker)
  +-- taskStatus: TaskStatus      (lightweight heartbeat payload)
  |     +-- taskId: TaskId                     ("queryId.stageId.partitionId.attemptId")
  |     +-- state: TaskState                   (enum: PLANNED, RUNNING, FLUSHING, ...)
  |     +-- outputBufferStatus: OutputBufferStatus
  |     +-- failures: List<ExecutionFailureInfo>
  |     +-- DataSize fields (5)                (memory, output, written)
  |     +-- Duration field (1)                 (fullGcTime)
  |     +-- int/long counters (7)
  +-- lastHeartbeat: Instant
  +-- outputBuffers: OutputBufferInfo          (buffer state + per-partition stats)
  |     +-- pipelinedBufferStates: Optional<List<PipelinedBufferInfo>>
  |     +-- utilization: Optional<TDigestHistogram>
  |     +-- spoolingOutputStats: Optional<SpoolingOutputStats.Snapshot>
  +-- noMoreSplits: Set<PlanNodeId>
  +-- stats: TaskStats                         (aggregate metrics)
  |     +-- Duration fields (11)
  |     +-- DataSize fields (11)
  |     +-- Instant fields (6)
  |     +-- int/long counters (12)
  |     +-- pipelines: List<PipelineStats>     (per-pipeline metrics)
  |           +-- DistributionSnapshot fields (2)
  |           +-- operatorSummaries: List<OperatorStats>
  |           |     +-- info: OperatorInfo     (polymorphic, 8 @JsonSubTypes)
  |           |     +-- metrics: Metrics       (Map<String, Metric<?>>)
  |           +-- drivers: List<DriverStats>
  |                 +-- operatorStats: List<OperatorStats>
  +-- estimatedMemory: Optional<DataSize>
  +-- needsPlan: boolean
```

### Two Distinct Polling Loops

The coordinator uses two separate HTTP polling loops with different cadences:

1. **`ContinuousTaskStatusFetcher`** -- polls `GET /v1/task/{taskId}/status` for the lightweight `TaskStatus`. This is the high-frequency heartbeat loop used for scheduling decisions (memory pressure, split counts, dynamic filter versions).

2. **`TaskInfoFetcher`** -- polls `GET /v1/task/{taskId}` for the full `TaskInfo` (which contains `TaskStatus` as a nested field plus `TaskStats`, `OutputBufferInfo`, etc.). This is lower frequency and used for UI dashboards and final task result collection.

### Serialization Strategy

Trino uses two different serialization approaches depending on the class shape:

- **Java records** (`TaskStatus`, `TaskInfo`, `TaskStats`, `OutputBufferInfo`, `OutputBufferStatus`, `PipelinedBufferInfo`, `ExecutionFailureInfo`): Jackson automatically derives JSON property names from the record component names. No explicit `@JsonProperty` or `@JsonCreator` annotations are needed on records.

- **Traditional classes** (`PipelineStats`, `OperatorStats`, `DriverStats`): Use explicit `@JsonCreator` on the constructor with `@JsonProperty("fieldName")` on each parameter, and `@JsonProperty` on getter methods.

### Value Type Serialization

| Java Type | JSON Representation | Example |
|---|---|---|
| `io.airlift.units.Duration` | String: `"<value> <unit>"` | `"4.00ns"`, `"15.00ns"`, `"2.00s"` |
| `io.airlift.units.DataSize` | String: `"<bytes>B"` (raw bytes by default) | `"12B"`, `"191B"` |
| `java.time.Instant` | String: ISO-8601 | `"1970-01-01T00:00:00.001Z"` |
| `TaskId` | String (`@JsonValue`): dotted | `"20210101_000000_00000_xxxxx.0.0.0"` |
| `PlanNodeId` | String (`@JsonValue`) | `"test"` |
| `TaskState` | String (enum name) | `"RUNNING"` |
| `BufferState` | String (enum name) | `"OPEN"` |
| `BlockedReason` | String (enum name) | `"WAITING_FOR_MEMORY"` |
| `OptionalInt` | JSON number or absent | `2` or omitted |
| `Optional<T>` | Value or `null` | `null` or nested object |
| `OutputBufferId` | JSON integer (`@JsonValue` on `id()`) | `0` |

## Section 2: Detailed Analysis

### 2.1 TaskStatus (Lightweight Heartbeat Object)

**Source:** `core/trino-main/src/main/java/io/trino/execution/TaskStatus.java`

`TaskStatus` is a Java record with 22 components. Being a record, Jackson uses the canonical constructor and component accessor methods automatically.

```java
public record TaskStatus(
        TaskId taskId,                              // string: "queryId.stageId.partitionId.attemptId"
        long taskInstanceId,                        // long
        long version,                               // long, >= STARTING_VERSION (0)
        TaskState state,                            // enum string
        URI self,                                   // string URI
        String nodeId,                              // string
        boolean speculative,                        // boolean
        List<ExecutionFailureInfo> failures,         // array of objects
        int queuedPartitionedDrivers,               // int >= 0
        int runningPartitionedDrivers,              // int >= 0
        OutputBufferStatus outputBufferStatus,      // nested object
        DataSize outputDataSize,                    // string: "NB"
        DataSize writerInputDataSize,               // string: "NB"
        DataSize physicalWrittenDataSize,            // string: "NB"
        OptionalInt maxWriterCount,                 // integer or absent
        DataSize memoryReservation,                 // string: "NB"
        DataSize peakMemoryReservation,             // string: "NB"
        DataSize revocableMemoryReservation,         // string: "NB"
        long fullGcCount,                           // long >= 0
        Duration fullGcTime,                        // string: "0.00ns"
        long dynamicFiltersVersion,                 // long >= INITIAL (0)
        long queuedPartitionedSplitsWeight,          // long >= 0
        long runningPartitionedSplitsWeight)         // long >= 0
```

Key constants:
```java
public static final long STARTING_VERSION = 0;
private static final long MAX_VERSION = Long.MAX_VALUE;
```

**Versioning protocol:** The coordinator long-polls with `currentVersion`, and the worker blocks until `TaskStatus.version > currentVersion`. On terminal failure, the version is set to `MAX_VERSION` (Long.MAX_VALUE) via `failWith()`, which forces immediate notification.

### 2.7 Value Type Serialization Details

#### Duration Serialization

The airlift `Duration` class has `@JsonCreator` static factory (`valueOf(String)`) and `@JsonValue` (`toString()`). The `toString()` format is: `"<value> <unit>"` where value is a formatted double and unit is the time unit abbreviation.

Examples from test data:
- `new Duration(4, NANOSECONDS)` serializes as `"4.00ns"`
- `succinctDuration(0, MILLISECONDS)` serializes as `"0.00ns"` (converted to most succinct unit)

The constructor `convertToMostSuccinctTimeUnit()` is called during aggregation to keep values human-readable (e.g., nanoseconds for small values, seconds for large ones).

#### DataSize Serialization

**Source:** `core/trino-main/src/main/java/io/trino/server/DataSizeSerializer.java`

```java
public class DataSizeSerializer extends JsonSerializer<DataSize> {
    public static final String SUCCINCT_DATA_SIZE_ENABLED = "dataSize.succinct.enabled";

    @Override
    public void serialize(DataSize dataSize, JsonGenerator jsonGenerator,
                          SerializerProvider serializerProvider) throws IOException {
        if (Boolean.TRUE.equals(serializerProvider.getAttribute(SUCCINCT_DATA_SIZE_ENABLED))) {
            jsonGenerator.writeString(dataSize.succinct().toString());
            return;
        }
        jsonGenerator.writeString(dataSize.toBytesValueString());
    }
}
```

The custom `DataSizeSerializer` is registered globally via `ServerMainModule`:
```java
jsonBinder(binder).addSerializerBinding(DataSize.class).to(DataSizeSerializer.class);
```

**Default format:** `dataSize.toBytesValueString()` which produces `"<N>B"` (raw byte count as a string). For example, `DataSize.ofBytes(12)` serializes as `"12B"`.

**Succinct mode:** When the `SUCCINCT_DATA_SIZE_ENABLED` serializer attribute is set to true, it uses `dataSize.succinct().toString()` which produces human-readable strings like `"1.5MB"`, `"4.2GB"`. This mode is NOT used for worker-to-coordinator communication -- it is only enabled for specific UI contexts.

**Deserialization:** Uses the airlift `DataSize.valueOf(String)` factory which parses strings like `"12B"`, `"1.5MB"`, etc.

#### TaskId Serialization

**Source:** `core/trino-main/src/main/java/io/trino/execution/TaskId.java`

```java
public record TaskId(StageId stageId, int partitionId, int attemptId) {
    @JsonCreator
    public static TaskId valueOf(String fullId) {
        List<String> parts = parseDottedId(fullId, 4, "taskId");
        return new TaskId(
            new StageId(new QueryId(parts.get(0)), parseInt(parts.get(1))),
            parseInt(parts.get(2)),
            parseInt(parts.get(3)));
    }

    @Override @JsonValue
    public String toString() {
        return stageId.queryId().id() + '.' + stageId.id() + '.' + partitionId + '.' + attemptId;
    }
}
```

The `@JsonValue` / `@JsonCreator` pair means TaskId is a simple dotted string in JSON: `"20210101_000000_00000_xxxxx.0.0.0"`.

### 2.8 Summarization Protocol

The `TaskResource` endpoint supports summarization via query parameter:

```java
@GET @Produces(MediaType.APPLICATION_JSON)
public List<TaskInfo> getAllTaskInfo(@Context UriInfo uriInfo) {
    List<TaskInfo> allTaskInfo = taskManager.getAllTaskInfo();
    if (shouldSummarize(uriInfo)) {
        allTaskInfo = ImmutableList.copyOf(transform(allTaskInfo, TaskInfo::summarize));
    }
    return allTaskInfo;
}
```

The summarization chain operates as follows:

| Level | `summarize()` behavior | `summarizeFinal()` behavior |
|---|---|---|
| `TaskInfo` | Calls `outputBuffers.summarize()` + `stats.summarize()` | Calls `outputBuffers.summarizeFinal()` + `stats.summarizeFinal()` |
| `TaskStats` | Sets `pipelines = []` (empty) | Retains pipelines, calls `pipeline.summarize()` on each |
| `PipelineStats` | Strips `drivers`, calls `op.summarize()` on each operatorSummary | N/A |
| `OperatorStats` | Strips non-final `info` to null | N/A |
| `OutputBufferInfo` | Strips `pipelinedBufferStates`, `utilization`, `spoolingOutputStats`, `metrics` | Strips `pipelinedBufferStates` only |

## Section 3: Key Design Insights

### 3.1 Two-Tier Status Architecture

Trino separates "lightweight status" (`TaskStatus`, 22 fields) from "full stats" (`TaskInfo`, containing `TaskStats` with 39+ fields plus nested `PipelineStats`/`OperatorStats`/`DriverStats`). The lightweight `TaskStatus` is polled at high frequency for scheduling decisions and back-pressure signaling, while the full `TaskInfo` is fetched less frequently for monitoring and final result collection.

**Insight for Rust:** The Rust worker must produce both JSON shapes on separate endpoints. The `TaskStatus` endpoint must be extremely low-latency since it is on the critical scheduling path.

### 3.2 Version-Based Long Polling

The `TaskStatus.version` field is the backbone of the notification protocol. The coordinator polls with `currentVersion`, and the worker blocks until `version > currentVersion`. On failure, `version` is set to `Long.MAX_VALUE` to force immediate notification. This is a lock-free optimistic concurrency mechanism.

**Insight for Rust:** Implement this as an `AtomicU64` version counter with a `tokio::sync::watch` channel. Increment on every state change. Set to `u64::MAX` on terminal states.

### 3.3 Aggressive Summarization for Bandwidth Control

The multi-level summarization protocol (`summarize()` / `summarizeFinal()`) is critical for controlling JSON payload size. A task with 100 pipelines, each with 50 drivers and 10 operators, could produce megabytes of JSON. The summarization strips `drivers` lists and non-final `OperatorInfo` payloads during periodic polling, only retaining full stats in the final report.

**Insight for Rust:** Build the stats structs with `Option` wrappers for the fields that get stripped during summarization, and implement `summarize()` as `clone_with(drivers: None, ...)`.

### 3.4 DataSize Uses Raw Bytes by Default (Not Human-Readable)

The custom `DataSizeSerializer` writes `"12B"` (raw bytes) for worker-to-coordinator communication, not `"12 bytes"` or `"1.5MB"`. The succinct human-readable format is only used for UI display when the `SUCCINCT_DATA_SIZE_ENABLED` serializer attribute is set. This is a deliberate choice to avoid lossy conversions.

### 3.5 Duration Uses airlift's Custom Format (Not ISO-8601)

Airlift `Duration` serializes as `"4.00ns"` or `"2.00s"` -- a decimal value followed by a compact unit suffix. This is NOT ISO-8601 period format (e.g., `"PT0.000000004S"`). The parser must handle unit suffixes: `ns`, `us`, `ms`, `s`, `m`, `h`, `d`.

### 3.6 OperatorInfo is Polymorphic with 8 Known Subtypes

The `OperatorInfo` interface uses `@JsonTypeInfo(use = Id.NAME, property = "@type")` with 8 registered subtypes. For minimum viable compatibility, the Rust worker must be able to at least serialize whichever subtypes its operators produce (likely `PartitionedOutputInfo`, `DirectExchangeClientStatus`). Deserialization of all 8 types is optional for the worker (only needed if the worker receives OperatorInfo from somewhere, which it does not).

### 3.7 Records vs Classes: Two Serialization Styles

The newer types (`TaskStatus`, `TaskInfo`, `TaskStats`, `OutputBufferInfo`, `OutputBufferStatus`, `PipelinedBufferInfo`) are Java records, which means their JSON property names match the record component names exactly (camelCase). The older types (`PipelineStats`, `OperatorStats`, `DriverStats`) are traditional classes with explicit `@JsonProperty` annotations on getters, where the JSON property name is derived by stripping the `get`/`is` prefix from the getter name. In both cases the result is the same: camelCase JSON keys.

## Section 4: Porting Considerations (Java -> Rust)

### 4.1 Struct Layout Strategy

Define the hierarchy as nested Rust structs with serde derives:

```rust
#[derive(Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct TaskStatus {
    pub task_id: String,           // TaskId is a simple dotted string
    pub task_instance_id: i64,
    pub version: i64,
    pub state: TaskState,
    pub self_uri: String,          // "self" is a Rust keyword -- use #[serde(rename = "self")]
    pub node_id: String,
    pub speculative: bool,
    pub failures: Vec<ExecutionFailureInfo>,
    pub queued_partitioned_drivers: i32,
    pub running_partitioned_drivers: i32,
    pub output_buffer_status: OutputBufferStatus,
    pub output_data_size: DataSize,
    // ... etc
}
```

### 4.2 Custom Serialization for DataSize and Duration

Implement custom serde serializers/deserializers:

```rust
// DataSize: serialize as "12B", deserialize from "12B" or "1.5MB"
pub struct DataSize(u64); // internal bytes

impl Serialize for DataSize {
    fn serialize<S: Serializer>(&self, s: S) -> Result<S::Ok, S::Error> {
        s.serialize_str(&format!("{}B", self.0))
    }
}

// Duration: serialize as "4.00ns", deserialize from "<value><unit>"
pub struct AirliftDuration {
    value: f64,
    unit: TimeUnit,
}
```

The Duration parser must handle unit suffixes: `ns`, `us` (microseconds), `ms`, `s`, `m`, `h`, `d`. The serialized format always includes two decimal places (e.g., `"4.00ns"`, not `"4ns"`).

### 4.3 Nullable Fields and Optional Handling

Java `@Nullable` fields (e.g., `firstStartTime`, `endTime` in `TaskStats`) serialize as JSON `null` when absent. Java `Optional<T>` fields serialize as the value or `null`. In Rust:

- `@Nullable Instant` -> `Option<String>` with `#[serde(skip_serializing_if = "Option::is_none")]`
- `Optional<DataSize>` -> `Option<DataSize>`
- `OptionalInt` -> `Option<i32>`
- `OptionalLong` -> `Option<i64>`

### 4.4 Enum Serialization

All enums (`TaskState`, `BufferState`, `BlockedReason`) serialize as their name strings in UPPER_SNAKE_CASE:

```rust
#[derive(Serialize, Deserialize)]
#[serde(rename_all = "SCREAMING_SNAKE_CASE")]
pub enum TaskState {
    Planned,
    Running,
    Flushing,
    Finished,
    Canceling,
    Canceled,
    Aborting,
    Aborted,
    Failing,
    Failed,
}
```

### 4.5 OperatorInfo Polymorphism

Use serde's tagged enum representation:

```rust
#[derive(Serialize, Deserialize)]
#[serde(tag = "@type")]
pub enum OperatorInfo {
    #[serde(rename = "directExchangeClientStatus")]
    DirectExchangeClientStatus(DirectExchangeClientStatus),
    #[serde(rename = "partitionedOutput")]
    PartitionedOutput(PartitionedOutputInfo),
    #[serde(rename = "tableFinish")]
    TableFinish(TableFinishInfo),
    // ... 5 more variants
}
```

### 4.6 Field Count and Bandwidth

The full stats objects are large. Summary counts:

| Object | Field Count | Nested Arrays |
|---|---|---|
| `TaskStatus` | 22 | `failures` |
| `TaskInfo` | 7 | `noMoreSplits`, nested `outputBuffers`, `stats` |
| `TaskStats` | 39 | `pipelines` |
| `PipelineStats` | 35 | `operatorSummaries`, `drivers` |
| `OperatorStats` | 37 | none (but `info` is polymorphic) |
| `DriverStats` | 26 | `operatorStats` |

A fully unsummarized `TaskInfo` for a task with 3 pipelines, 10 drivers each, 5 operators each, would produce: 1 TaskInfo + 1 TaskStats + 3 PipelineStats + 30 DriverStats + 150 OperatorStats + 15 OperatorStats (summaries) = ~200 nested objects. This is why summarization is critical.

### 4.7 Metrics Map

The `Metrics` type serializes as a plain JSON object (map), but its values are polymorphic `Metric<?>` objects. For the Rust worker, the simplest approach is to use `serde_json::Value` for the metrics maps initially, and only implement concrete deserialization for the specific metric types the worker produces (e.g., `LongCount`).

### 4.8 Critical Path Fields

For minimum viable compatibility, these `TaskStatus` fields are consumed by the coordinator for scheduling decisions:
- `state` -- determines task lifecycle transitions
- `version` -- drives the long-poll notification protocol
- `queuedPartitionedDrivers` / `runningPartitionedDrivers` -- used by `SplitSchedulerStats` for split assignment
- `outputBufferStatus.overutilized` -- back-pressure signal
- `memoryReservation` / `peakMemoryReservation` -- memory accounting
- `dynamicFiltersVersion` -- triggers dynamic filter fetching
- `failures` -- error propagation
