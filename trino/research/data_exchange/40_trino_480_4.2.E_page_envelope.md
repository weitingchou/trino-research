# Module Teardown: Block Type Tags & Page Envelope (Task 4.2.E) [KG-2]

## 0. Research Focus
* **Task ID:** 4.2.E
* **Focus:** Understand the complete binary wire format for serialized pages: the 12-byte page header, how block types are tagged with length-prefixed strings, how compression and encryption are applied per-chunk, how pages are framed in HTTP responses, and the full byte layout end-to-end.

## 1. High-Level Overview

Trino's inter-worker data exchange transmits columnar `Page` objects serialized into a self-describing binary format. The format has four concentric layers:

1. **HTTP Response Envelope** -- A 16-byte response header (magic + checksum + page count) wraps one or more serialized pages in a single HTTP response body.
2. **Page Envelope** -- Each serialized page begins with a 12-byte header (`positionCount`, `uncompressedSize`, `compressedSize`), followed by the page payload.
3. **Page Payload** -- The payload starts with a 4-byte channel count, then each block (column) is serialized sequentially. The payload is chunked into 64 KB blocks for compression and encryption.
4. **Block Encoding** -- Each block is self-describing: a length-prefixed UTF-8 string tag (e.g., `"LONG_ARRAY"`, `"VARIABLE_WIDTH"`) followed by the block-specific binary data (position count, null bitmap, values).

Compression (LZ4 or ZSTD) and encryption (AES-256-CBC) are applied to chunks of the raw page payload, not to individual blocks. This is a critical design choice: the serializer writes raw block data into a 64 KB buffer, then compresses/encrypts each full buffer as a unit. The page header's `compressedSize` field covers the total compressed payload, allowing readers to read the exact number of bytes needed.

## 2. Detailed Analysis

### 2.1 HTTP Response Envelope (PagesInputStreamFactory)

When a downstream worker fetches data via `GET /v1/task/{taskId}/results/{bufferId}/{token}`, the `TaskResource` constructs a streaming HTTP response with content type `application/x-trino-pages`. The response body is written by `PagesInputStreamFactory.write()`:

**Source:** `io.trino.server.PagesInputStreamFactory`

```java
public static final int SERIALIZED_PAGES_MAGIC = 0xfea4f001;

public void write(OutputStream stream, List<Slice> serializedPages)
        throws IOException
{
    try (stream) {
        SliceOutput header = new OutputStreamSliceOutput(stream);
        header.writeInt(SERIALIZED_PAGES_MAGIC);
        header.writeLong(dataIntegrityVerificationEnabled
                ? calculateChecksum(serializedPages) : NO_CHECKSUM);
        header.writeInt(serializedPages.size());
        header.flush();

        for (Slice page : serializedPages) {
            page.getInput().transferTo(stream);
            stream.flush();
        }
    }
    // ... EOF exception handling omitted
}
```

The response body layout is:

```
+--------------------------------------------------+
| HTTP Response Body                               |
+==================================================+
| Offset 0  : magic        (int32, LE) 0xfea4f001  |
| Offset 4  : checksum     (int64, LE) XxHash64    |
|              or NO_CHECKSUM = 0x0123456789abcdef  |
| Offset 12 : pageCount    (int32, LE)             |
+--------------------------------------------------+
| Serialized Page 0: [12-byte header][payload...]  |
| Serialized Page 1: [12-byte header][payload...]  |
| ...                                              |
| Serialized Page N-1                              |
+--------------------------------------------------+
```

All integers are little-endian (Airlift `Slice` uses native byte order, which on x86/ARM is little-endian).

The checksum is computed using XxHash64 across all serialized page slices. The special sentinel `NO_CHECKSUM = 0x0123456789abcdef` indicates data integrity verification is disabled. If the computed checksum happens to equal `NO_CHECKSUM`, it is incremented by 1 to avoid ambiguity.

**Source:** `io.trino.execution.buffer.PagesSerdeUtil`

```java
public static final long NO_CHECKSUM = 0x0123456789abcdefL;

public static long calculateChecksum(List<Slice> pages)
{
    XxHash64 hash = new XxHash64();
    for (Slice page : pages) {
        hash.update(page);
    }
    long checksum = hash.hash();
    if (checksum == NO_CHECKSUM) {
        return checksum + 1;
    }
    return checksum;
}
```

On the client side, `HttpPageBufferClient.PagesResponseProcessor` reads the response:

```java
try (LittleEndianDataInputStream input =
        new LittleEndianDataInputStream(response.getInputStream())) {
    int magic = input.readInt();
    if (magic != SERIALIZED_PAGES_MAGIC) {
        throw new IllegalStateException(...);
    }
    long checksum = input.readLong();
    int pagesCount = input.readInt();
    List<Slice> pages = ImmutableList.copyOf(readSerializedPages(input));
    verifyChecksum(checksum, pages);
    checkState(pages.size() == pagesCount, ...);
    return createPagesResponse(taskInstanceId, token, nextToken,
            pages, complete, remoteTaskFailed);
}
```

HTTP headers used in the response:
- `X-Trino-Task-Instance-Id` -- identifies the task instance
- `X-Trino-Page-Sequence-Id` -- sequence token for acknowledgement
- `X-Trino-Page-End-Sequence-Id` -- next token for subsequent request
- `X-Trino-Buffer-Complete` -- whether all data has been sent
- `X-Trino-Task-Failed` -- whether the producing task has failed

### 2.2 Page Header (12 Bytes)

Each serialized page starts with a fixed 12-byte header:

**Source:** `io.trino.execution.buffer.PagesSerdeUtil`

```java
static final int SERIALIZED_PAGE_POSITION_COUNT_OFFSET = 0;
static final int SERIALIZED_PAGE_UNCOMPRESSED_SIZE_OFFSET =
        SERIALIZED_PAGE_POSITION_COUNT_OFFSET + Integer.BYTES;        // = 4
static final int SERIALIZED_PAGE_COMPRESSED_SIZE_OFFSET =
        SERIALIZED_PAGE_UNCOMPRESSED_SIZE_OFFSET + Integer.BYTES;     // = 8
static final int SERIALIZED_PAGE_HEADER_SIZE =
        SERIALIZED_PAGE_COMPRESSED_SIZE_OFFSET + Integer.BYTES;       // = 12
```

Layout:

```
Byte 0-3:   positionCount     (int32) -- number of rows in the page
Byte 4-7:   uncompressedSize  (int32) -- total uncompressed payload size
Byte 8-11:  compressedSize    (int32) -- total compressed payload size
```

The header is written by `CompressingEncryptingPageSerializer.SerializedPageOutput`:

```java
public void startPage(int positionCount, int sizeInBytes)
{
    WriteBuffer buffer = new WriteBuffer(
            round(sizeInBytes * 1.2F) + SERIALIZED_PAGE_HEADER_SIZE);
    buffer.writeInt(positionCount);
    // leave space for uncompressed and compressed sizes
    buffer.skip(Integer.BYTES * 2);
    buffers[buffers.length - 1] = buffer;
    uncompressedSize = 0;
}
```

After all blocks are written, `closePage()` fills in the deferred size fields:

```java
public SerializedPage closePage()
{
    compress();
    encrypt();

    WriteBuffer pageBuffer = buffers[buffers.length - 1];
    int serializedPageSize = pageBuffer.getPosition();
    int compressedSize = serializedPageSize - SERIALIZED_PAGE_HEADER_SIZE;
    Slice slice = pageBuffer.getSlice();
    slice.setInt(SERIALIZED_PAGE_UNCOMPRESSED_SIZE_OFFSET, uncompressedSize);
    slice.setInt(SERIALIZED_PAGE_COMPRESSED_SIZE_OFFSET, compressedSize);
    // ...
}
```

When reading, `PagesSerdeUtil.readSerializedPage()` first reads the 12-byte header, then uses `compressedSize` to know exactly how many bytes to read for the payload:

```java
public static Slice readSerializedPage(Slice headerSlice, InputStream inputStream)
        throws IOException
{
    int compressedSize = headerSlice.getIntUnchecked(
            SERIALIZED_PAGE_COMPRESSED_SIZE_OFFSET);
    byte[] outputBuffer = new byte[SERIALIZED_PAGE_HEADER_SIZE + compressedSize];
    headerSlice.getBytes(0, outputBuffer, 0, SERIALIZED_PAGE_HEADER_SIZE);
    int bytes = inputStream.readNBytes(
            outputBuffer, SERIALIZED_PAGE_HEADER_SIZE, compressedSize);
    verify(bytes == compressedSize, ...);
    return Slices.wrappedBuffer(outputBuffer);
}
```

### 2.3 Block Type Tags (Length-Prefixed UTF-8 Strings)

Blocks are **not** identified by numeric enums. Instead, each block is tagged with a **length-prefixed UTF-8 string** -- the encoding name. This is written by `InternalBlockEncodingSerde.writeBlock()`:

**Source:** `io.trino.metadata.InternalBlockEncodingSerde`

```java
@Override
public void writeBlock(SliceOutput output, Block block)
{
    while (true) {
        BlockEncoding blockEncoding = blockToEncoding.apply(block.getClass());

        // see if a replacement block should be written instead
        Optional<Block> replacementBlock = blockEncoding.replacementBlockForWrite(block);
        if (replacementBlock.isPresent()) {
            block = replacementBlock.get();
            continue;
        }

        // write the name to the output
        writeLengthPrefixedString(output, blockEncoding.getName());

        // write the block to the output
        blockEncoding.writeBlock(this, output, block);
        break;
    }
}
```

The string is framed as:

```java
private static void writeLengthPrefixedString(SliceOutput output, String value)
{
    byte[] bytes = value.getBytes(UTF_8);
    output.writeInt(bytes.length);     // 4-byte length prefix (int32)
    output.writeBytes(bytes);          // UTF-8 string bytes (no null terminator)
}
```

The deserialization side mirrors this:

```java
@Override
public Block readBlock(SliceInput input)
{
    String encodingName = readLengthPrefixedString(input);
    BlockEncoding blockEncoding = nameToEncoding.apply(encodingName);
    return blockEncoding.readBlock(this, input);
}

private static String readLengthPrefixedString(SliceInput input)
{
    int length = input.readInt();
    byte[] bytes = new byte[length];
    input.readBytes(bytes);
    return new String(bytes, UTF_8);
}
```

The `BlockEncodingManager` maintains a registry mapping both directions:
- **Name -> Encoding** (for deserialization): `Map<String, BlockEncoding>`
- **Block Class -> Encoding** (for serialization): `Map<Class<? extends Block>, BlockEncoding>`

The 13 registered block encoding names in Trino 480:

| Encoding Name | Block Class | Purpose |
|---|---|---|
| `VARIABLE_WIDTH` | `VariableWidthBlock` | VARCHAR, VARBINARY, and other variable-length types |
| `BYTE_ARRAY` | `ByteArrayBlock` | BOOLEAN, TINYINT |
| `SHORT_ARRAY` | `ShortArrayBlock` | SMALLINT |
| `INT_ARRAY` | `IntArrayBlock` | INTEGER, DATE |
| `LONG_ARRAY` | `LongArrayBlock` | BIGINT, TIMESTAMP, DOUBLE (as raw bits) |
| `FIXED12` | `Fixed12Block` | TIMESTAMP WITH TIME ZONE (12 bytes per value) |
| `INT128_ARRAY` | `Int128ArrayBlock` | DECIMAL (128-bit), UUID |
| `VARIANT` | `VariantBlock` | Variant type |
| `DICTIONARY` | `DictionaryBlock` | Dictionary-encoded wrapper around any other block |
| `RLE` | `RunLengthEncodedBlock` | Run-length-encoded wrapper (single value repeated N times) |
| `ARRAY` | `ArrayBlock` | ARRAY type (nested) |
| `MAP` | `MapBlock` | MAP type (nested keys + values) |
| `ROW` | `RowBlock` | ROW type (nested field blocks) |

The `replacementBlockForWrite()` mechanism allows optimizations at write time. For example, `DictionaryBlockEncoding` will compact the dictionary (removing unused entries) or replace it with the raw inner block if the dictionary is an identity mapping.

### 2.4 Block-Level Binary Layout

After the encoding name tag, each block type writes its own structure. The pattern is consistent but varies by type:

#### 2.4.1 Fixed-Width Blocks (LongArrayBlock Example)

```
[4 bytes: encoding name length] [N bytes: "LONG_ARRAY"]
[4 bytes: positionCount]
[1 byte:  hasNulls flag (boolean)]
  if hasNulls:
    [ceil(positionCount/8) bytes: null bitmap, MSB-first]
  if hasNulls:
    [4 bytes: nonNullCount]
    [nonNullCount * 8 bytes: compacted long values]
  else:
    [positionCount * 8 bytes: long values]
```

The null bitmap encoding is MSB-first (bit 7 of each byte corresponds to the first position in the group of 8). Non-null values are compacted -- nulls are elided from the values array, and the count of non-null values is written before the array. This is a space optimization: a block with 50% nulls transmits only half the value bytes.

**Source:** `io.trino.spi.block.LongArrayBlockEncoding.writeBlock()`

```java
sliceOutput.appendInt(positionCount);
encodeNullsAsBitsScalar(sliceOutput, isNull, rawOffset, positionCount);
if (isNull == null) {
    sliceOutput.writeLongs(rawValues, rawOffset, positionCount);
} else {
    compactLongsWithNullsScalar(sliceOutput, rawValues, isNull,
            rawOffset, positionCount);
}
```

The null encoding from `EncoderUtil`:

```java
public static void encodeNullsAsBitsScalar(SliceOutput sliceOutput,
        @Nullable boolean[] isNull, int offset, int length)
{
    sliceOutput.writeBoolean(isNull != null);  // 1-byte flag
    if (isNull == null) return;
    byte[] packedIsNull = new byte[(length + 7) / 8];
    // ... pack 8 booleans per byte, MSB first ...
    sliceOutput.writeBytes(packedIsNull, 0, currentByte);
}
```

#### 2.4.2 Variable-Width Blocks

```
[4 bytes: encoding name length] [N bytes: "VARIABLE_WIDTH"]
[4 bytes: positionCount]
[1 byte:  hasNulls flag]
  if hasNulls:
    [ceil(positionCount/8) bytes: null bitmap]
[4 bytes: nonNullOffsetCount]
[nonNullOffsetCount * 4 bytes: end-offsets (int32 each), null positions elided]
[totalDataLength bytes: raw byte data]
```

Offsets are end-offsets (not start-offsets). The starting offset for position `i` is `offsets[i-1]` (or 0 for the first position). Null positions are suppressed from the offsets array.

#### 2.4.3 Dictionary Blocks (Recursive)

```
[4 bytes: encoding name length] [N bytes: "DICTIONARY"]
[4 bytes: positionCount]
[... recursive: inner block (the dictionary) ...]
[positionCount * 4 bytes: id array (int32 indices into dictionary)]
```

The inner block itself has its own encoding tag, making the format recursively self-describing. On deserialization, `DictionaryBlockEncoding.readBlock()` flattens the dictionary by calling `dictionaryBlock.copyPositions(ids)`.

#### 2.4.4 Run-Length Encoded Blocks

```
[4 bytes: encoding name length] [N bytes: "RLE"]
[4 bytes: positionCount (the run length)]
[... recursive: single-value inner block ...]
```

#### 2.4.5 Nested Blocks (ROW, ARRAY, MAP)

**ROW:**
```
[4 bytes: "ROW" name length] [3 bytes: "ROW"]
[4 bytes: positionCount]
[4 bytes: numFields]
  for each field:
    [... recursive: field block with its own encoding tag ...]
[1 byte: hasNulls] [null bitmap if applicable]
```

**ARRAY:**
```
[4 bytes: "ARRAY" name length] [5 bytes: "ARRAY"]
[... recursive: values block ...]
[4 bytes: positionCount]
[(positionCount+1) * 4 bytes: offsets]
[1 byte: hasNulls] [null bitmap if applicable]
```

**MAP:**
```
[4 bytes: "MAP" name length] [3 bytes: "MAP"]
[... type serialization for MapType ...]
[... recursive: key block ...]
[... recursive: value block ...]
[4 bytes: hashTableLength (-1 if absent)]
  if hashTableLength >= 0:
    [hashTableLength * 4 bytes: hash table]
[4 bytes: positionCount]
[(positionCount+1) * 4 bytes: offsets]
[1 byte: hasNulls] [null bitmap if applicable]
```

MAP encoding uniquely includes a `Type` serialization (also length-prefixed string) via `blockEncodingSerde.writeType()`.

### 2.5 Compression: Chunk-Based, Not Per-Block

Compression is applied to the page payload as a whole, in 64 KB chunks. This is a critical distinction: blocks are not individually compressed. Instead, the raw bytes for all blocks flow through a `WriteBuffer[0]` (the compression input buffer), and whenever it fills to capacity, the chunk is compressed and appended to the next buffer in the chain.

**Source:** `io.trino.execution.buffer.PagesSerdeFactory`

```java
private static final int SERIALIZED_PAGE_DEFAULT_BLOCK_SIZE_IN_BYTES = 64 * 1024;
```

The buffer chain in `CompressingEncryptingPageSerializer.SerializedPageOutput`:
- **buffers[0]** -- compression input buffer (64 KB)
- **buffers[1]** (if encryption) -- encryption input buffer (holds compressed chunk)
- **buffers[N-1]** -- page output buffer (accumulates the final serialized page)

When the compression buffer fills, `compress()` is called:

```java
private void compress()
{
    if (this.compressor.isEmpty()) return;

    Compressor compressor = this.compressor.get();
    WriteBuffer sourceBuffer = buffers[0];
    WriteBuffer sinkBuffer = buffers[1];

    int maxCompressedLength = compressor.maxCompressedLength(
            sourceBuffer.getPosition());
    sinkBuffer.ensureCapacityFor(maxCompressedLength + Integer.BYTES);

    int uncompressedSize = sourceBuffer.getPosition();
    int compressedSize = compressor.compress(
            sourceBuffer.getSlice().byteArray(),
            sourceBuffer.getSlice().byteArrayOffset(),
            uncompressedSize,
            sinkBuffer.getSlice().byteArray(),
            sinkBuffer.getSlice().byteArrayOffset()
                    + sinkBuffer.getPosition() + Integer.BYTES,
            maxCompressedLength);

    boolean compressed =
            uncompressedSize * MINIMUM_COMPRESSION_RATIO > compressedSize;
    int blockSize;
    if (!compressed) {
        // Copy uncompressed -- compression did not help enough
        System.arraycopy(sourceBuffer..., sinkBuffer...);
        blockSize = uncompressedSize;
    } else {
        blockSize = compressedSize;
    }

    sinkBuffer.writeInt(createBlockMarker(compressed, blockSize));
    sinkBuffer.skip(blockSize);
    sourceBuffer.reset();
}
```

Each compressed chunk is preceded by a 4-byte block marker:

```java
static final int SERIALIZED_PAGE_COMPRESSED_BLOCK_MASK = 1 << (Integer.SIZE - 1);

private static int createBlockMarker(boolean compressed, int size)
{
    if (compressed) {
        return size | SERIALIZED_PAGE_COMPRESSED_BLOCK_MASK;  // MSB set = compressed
    }
    return size;  // MSB clear = uncompressed
}
```

So the marker is: `bit 31 = isCompressed`, `bits 0-30 = chunk byte size`. This allows individual chunks within a single page to be independently compressed or left uncompressed (if the compression ratio is worse than 0.8x).

The `MINIMUM_COMPRESSION_RATIO` threshold is 0.8 (80%): if the compressed output is not at least 20% smaller than the input, the chunk is stored uncompressed. This avoids wasting CPU on data that doesn't compress well.

Compression codec is determined by the session property `exchange_compression_codec`:

```java
public enum CompressionCodec
{
    NONE, LZ4, ZSTD;
}
```

There is no codec identifier in the wire format itself. Both sender and receiver must agree on the codec through configuration. The compressor/decompressor is selected at serializer/deserializer creation time and does not change per-page.

### 2.6 Encryption: Per-Chunk AES-256-CBC with Random IV

When encryption is enabled (a 256-bit `SecretKey` is provided), each compression chunk (or raw chunk if compression is disabled) is encrypted with AES/CBC/PKCS5Padding:

```java
static final String SERIALIZED_PAGE_CIPHER_NAME = "AES/CBC/PKCS5Padding";
```

The `encrypt()` method processes one chunk at a time:

```java
private void encrypt()
{
    if (encryptionKey.isEmpty()) return;

    Cipher cipher = initCipher(encryptionKey.get());
    byte[] iv = cipher.getIV();  // random IV from ENCRYPT_MODE init

    WriteBuffer sourceBuffer = buffers[buffers.length - 2];
    WriteBuffer sinkBuffer = buffers[buffers.length - 1];

    int maxEncryptedSize = cipher.getOutputSize(sourceBuffer.getPosition())
            + iv.length;
    sinkBuffer.ensureCapacityFor(maxEncryptedSize
            + Integer.BYTES   // encrypted block length
            + iv.length);     // initialization vector

    // reserve space for encrypted block length
    sinkBuffer.skip(Integer.BYTES);
    // write initialization vector
    sinkBuffer.writeBytes(iv, 0, iv.length);

    int encryptedSize;
    encryptedSize = cipher.update(source..., sink...);
    encryptedSize += cipher.doFinal(sink...);

    sinkBuffer.getSlice().setInt(
        sinkBuffer.getPosition() - Integer.BYTES - iv.length,
        encryptedSize);
    sinkBuffer.skip(encryptedSize);

    sourceBuffer.reset();
}
```

Each encrypted chunk in the payload has this layout:

```
[4 bytes: encryptedSize (int32)]
[16 bytes: IV (AES block size = 16 bytes for CBC)]
[encryptedSize bytes: ciphertext]
```

The IV is randomly generated by the JCE `Cipher.init(ENCRYPT_MODE, key)` call each time `encrypt()` is invoked. This means each 64 KB chunk gets a unique IV, which is correct for CBC mode security.

On the decryption side (`CompressingDecryptingPageDeserializer`):

```java
private void decrypt()
{
    ReadBuffer source = buffers[buffers.length - 1];
    ReadBuffer sink = buffers[buffers.length - 2];
    int bytesPreserved = sink.rollOver();

    int encryptedSize = source.readInt();
    int ivSize = cipher.orElseThrow().getBlockSize();  // 16 for AES
    IvParameterSpec iv = new IvParameterSpec(
            source.getSlice().byteArray(),
            source.getSlice().byteArrayOffset() + source.getPosition(),
            ivSize);
    source.setPosition(source.getPosition() + ivSize);

    Cipher cipher = initCipher(encryptionKey.get(), iv);
    decryptedSize = cipher.update(source..., sink...);
    decryptedSize += cipher.doFinal(sink...);
    // ...
}
```

### 2.7 Raw Page Serialization (writeRawPage / readRawPage)

The raw page content (before compression/encryption) is straightforward:

**Source:** `io.trino.execution.buffer.PagesSerdeUtil`

```java
static void writeRawPage(Page page, SliceOutput output, BlockEncodingSerde serde)
{
    output.writeInt(page.getChannelCount());
    for (int channel = 0; channel < page.getChannelCount(); channel++) {
        writeBlock(serde, output, page.getBlock(channel));
    }
}

static Page readRawPage(int positionCount, SliceInput input,
        BlockEncodingSerde blockEncodingSerde)
{
    int numberOfBlocks = input.readInt();
    Block[] blocks = new Block[numberOfBlocks];
    for (int i = 0; i < blocks.length; i++) {
        blocks[i] = readBlock(blockEncodingSerde, input);
    }
    return new Page(positionCount, blocks);
}
```

### 2.8 Multi-Page Framing (writePages / readPages)

Multiple pages are written sequentially with no per-page delimiter -- the 12-byte header's `compressedSize` field is the framing mechanism. The reader reads the header, extracts `compressedSize`, reads exactly that many bytes, and repeats until EOF.

```java
public static long writePages(PageSerializer serializer,
        SliceOutput sliceOutput, Iterator<Page> pages)
{
    long size = 0;
    while (pages.hasNext()) {
        Page page = pages.next();
        sliceOutput.writeBytes(serializer.serialize(page));
        size += page.getSizeInBytes();
    }
    return size;
}
```

The reader is a lazy iterator:

```java
private static class PageReader extends AbstractIterator<Page> {
    private final byte[] headerBuffer = new byte[SERIALIZED_PAGE_HEADER_SIZE];

    @Override
    protected Page computeNext() {
        int read = inputStream.readNBytes(headerBuffer, 0, headerBuffer.length);
        if (read <= 0) return endOfData();
        if (read != headerBuffer.length) throw new EOFException();
        return deserializer.deserialize(
                readSerializedPage(headerSlice, inputStream));
    }
}
```

### 2.9 Complete Byte Layout -- Full Example

A serialized page with 2 columns (a `BIGINT` and a `VARCHAR`), with LZ4 compression and no encryption, for N rows:

```
=== HTTP Response Body ===
[Offset 0]  0x01f0a4fe       -- SERIALIZED_PAGES_MAGIC (LE)
[Offset 4]  <8 bytes>        -- XxHash64 checksum (or NO_CHECKSUM)
[Offset 12] <4 bytes>        -- pageCount (e.g. 1)

=== Serialized Page 0 ===
--- Page Header (12 bytes) ---
[Offset 0]  <4 bytes>        -- positionCount = N
[Offset 4]  <4 bytes>        -- uncompressedSize
[Offset 8]  <4 bytes>        -- compressedSize

--- Compressed Page Payload (compressedSize bytes) ---
  One or more compressed chunks, each formatted as:
    [4 bytes: blockMarker]   -- bit31=isCompressed, bits0-30=chunkSize
    [chunkSize bytes: data]  -- compressed or uncompressed chunk

  When decompressed, the raw payload is:
    [4 bytes: channelCount = 2]

    --- Block 0: BIGINT column ---
    [4 bytes: nameLength = 10]
    [10 bytes: "LONG_ARRAY"]
    [4 bytes: positionCount = N]
    [1 byte: hasNulls]
      if hasNulls:
        [ceil(N/8) bytes: null bitmap, MSB-first]
        [4 bytes: nonNullCount]
        [nonNullCount * 8 bytes: compacted long values]
      else:
        [N * 8 bytes: long values]

    --- Block 1: VARCHAR column ---
    [4 bytes: nameLength = 14]
    [14 bytes: "VARIABLE_WIDTH"]
    [4 bytes: positionCount = N]
    [1 byte: hasNulls]
      if hasNulls:
        [ceil(N/8) bytes: null bitmap, MSB-first]
    [4 bytes: nonNullOffsetCount]
    [nonNullOffsetCount * 4 bytes: end-offsets]
    [totalDataLength bytes: raw string data]
```

When encryption is also enabled, each chunk gains an additional layer:

```
[4 bytes: encryptedSize]
[16 bytes: IV]
[encryptedSize bytes: ciphertext of the compressed-or-raw chunk]
```

## 3. Key Design Insights

1. **String-based block tags are self-describing but not compact.** Every block in the wire format carries a UTF-8 encoding name (e.g., `"LONG_ARRAY"` = 14 bytes with the length prefix, `"VARIABLE_WIDTH"` = 18 bytes). This is generous for debuggability and forward compatibility but adds overhead. For a page with 20 columns, that's 280+ bytes of encoding names alone. A numeric enum would use only 20-40 bytes.

2. **Chunk-based compression decouples block boundaries from compression boundaries.** The 64 KB chunk size means that block data from different columns may be compressed together in the same chunk. This improves compression ratio (adjacent data from different columns may have correlated patterns) but means the decompressor must decompress the entire chunk even if only one column is needed. For Trino's pull-based full-page consumption model, this is acceptable.

3. **Per-chunk compression decisions prevent incompressible data from wasting CPU.** The 0.8x ratio threshold allows each chunk to independently fall back to uncompressed storage. The `blockMarker` MSB flag makes this transparent to the reader.

4. **Random IV per chunk is correct for AES-CBC.** Each chunk gets a fresh IV from `Cipher.init(ENCRYPT_MODE, key)`, ensuring identical plaintext chunks encrypt to different ciphertexts. The IV is stored inline with each chunk, adding 16 bytes per ~64 KB.

5. **The `compressedSize` field in the page header is the only framing mechanism.** There are no sentinels or delimiters between pages. The reader calculates exactly how many bytes to read from `compressedSize`. This is efficient for streaming but means corruption in the size field leads to a completely unrecoverable stream.

6. **Null handling is space-optimized via bitmap + value compaction.** The null bitmap uses 1 bit per position (MSB-first packing). Values for null positions are elided from the data array and a separate `nonNullCount` field tells the reader how many values to expect. For high-null-rate columns, this can halve the data size.

7. **The replacement block mechanism is a serialization-time optimization.** `DictionaryBlockEncoding.replacementBlockForWrite()` can replace an identity-mapping dictionary with the raw inner block, avoiding dictionary overhead for blocks where every position maps to a unique dictionary entry.

8. **The buffer chain architecture (buffers[0..N-1]) creates a pipeline.** Data flows from the raw write buffer through compression, then encryption, then into the page output buffer. Each stage drains the previous buffer when it fills. This avoids materializing the entire uncompressed page in memory before compressing.

9. **No codec metadata in the wire format.** The compression codec (NONE/LZ4/ZSTD) is not encoded in the serialized page. Both endpoints must agree via session properties. This simplifies the format but means incompatible configuration silently produces garbage.

10. **XxHash64 provides integrity, not security.** The checksum detects accidental corruption (bit flips, truncation) but is not cryptographically secure. Encryption (when enabled) provides confidentiality, and the PKCS5 padding provides implicit integrity for encrypted chunks.

## 4. Porting Considerations (Java -> Rust)

### 4.1 Byte Order
Airlift `Slice` uses native byte order. On x86-64 and ARM (little-endian), all integers are written in little-endian format. A Rust implementation should use `u32::to_le_bytes()` / `u32::from_le_bytes()` explicitly, or use a `byteorder` crate with `LittleEndian`. Do not assume native endianness in Rust.

### 4.2 String-Based Block Tags
The encoding name registry can be implemented as a `HashMap<&str, Box<dyn BlockEncoding>>` in Rust. The string names must match exactly (case-sensitive, UTF-8). A Rust port could also define an enum with `TryFrom<&str>` for faster dispatch while maintaining wire compatibility.

### 4.3 Null Bitmap Encoding
The MSB-first packing order differs from Arrow's LSB-first convention. If the Rust port uses Arrow internally, the null bitmap must be transposed at the serialization boundary. The formula is: Trino's bit position `i` within a byte is at `(7 - (i % 8))`, while Arrow uses `(i % 8)`.

### 4.4 Null Value Compaction
The compacted-values optimization (only non-null values are transmitted) requires careful implementation. On write, iterate with a null-check and skip nulls. On read, expand the compacted array back to full length using the null bitmap. Trino 480 uses SIMD (`VectorSpecies<Long>` with `compress`/`expand` operations) for this -- Rust can use `std::simd` (nightly) or manual SIMD via `packed_simd2` / architecture-specific intrinsics.

### 4.5 Compression Chunk Pipeline
The 64 KB chunk-based compression pipeline is a natural fit for Rust's ownership model. Each buffer stage can own its allocation, and the drain-on-full logic maps to a `write()` method that checks capacity before each write. Use `lz4_flex` (pure Rust) or `lz4` (C binding) for LZ4, and `zstd` crate for ZSTD.

### 4.6 Encryption
Use the `aes` and `cbc` crates for AES-256-CBC with PKCS7 padding (PKCS7 is the same as PKCS5 for 16-byte blocks). Generate a random 16-byte IV per chunk using `rand::thread_rng()`. Store the IV inline before each encrypted chunk, exactly as Trino does.

### 4.7 Recursive Block Encoding
The `DICTIONARY`, `RLE`, `ARRAY`, `MAP`, and `ROW` encodings are recursive -- they call back into the serde to encode child blocks. In Rust, this naturally maps to a trait with recursive dispatch:
```rust
trait BlockEncoding {
    fn name(&self) -> &str;
    fn write(&self, serde: &dyn BlockEncodingSerde, out: &mut dyn Write, block: &Block);
    fn read(&self, serde: &dyn BlockEncodingSerde, input: &mut dyn Read) -> Block;
}
```

### 4.8 Wire Compatibility for Zero-Copy Reading
The serialized page `Slice` returned by `PageSerializer.serialize()` can be sent directly over the wire without additional framing. In Rust, the `bytes::Bytes` type provides a reference-counted immutable buffer that can be used similarly. The `SerializedPageReference` pattern (atomic refcount over a shared buffer) maps to `Arc<Bytes>`.

### 4.9 Missing Codec Identifier
Since the wire format does not identify the compression codec, a Rust worker must either (a) negotiate the codec out-of-band (matching the Java behavior) or (b) add a codec byte to the page header as an extension. For a mixed Java/Rust cluster, the codec must match the Java coordinator's session property. For a pure Rust cluster, adding explicit codec identification would be a worthwhile improvement.

### 4.10 MAP Type Serialization
MAP blocks include an optional hash table and a Trino `Type` serialization (for the `MapType`). The type is serialized as a length-prefixed string containing the type's ID (e.g., `"map(varchar, bigint)"`). A Rust port must parse and reconstruct the type from this string, or maintain a type registry keyed by these type IDs.
