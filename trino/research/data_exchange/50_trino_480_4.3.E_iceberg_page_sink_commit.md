## 0. Research Focus

This teardown traces the Iceberg write path in Trino 480, from the moment a `Page` arrives at `IcebergPageSink.appendPage()` through to the coordinator's atomic Iceberg commit in `IcebergMetadata.finishInsert()`. The five research questions:

1. How does `IcebergPageSink` write pages to Parquet files on object storage? How is the output file path determined?
2. How does the partition spec control file organization? How are rows routed to partition-specific writers?
3. What do the fragment `Slice`s returned by `finish()` contain?
4. How does `IcebergMetadata.finishInsert()` collect fragments from all workers and perform the atomic Iceberg commit?
5. How does the write transaction handle rollback on failure (`abort()`)?

**Key source files** (all under `plugin/trino-iceberg/src/main/java/io/trino/plugin/iceberg/`):

| File | Role |
|------|------|
| `IcebergPageSink.java` | Core write operator -- partitions pages, manages per-partition writers, produces commit fragments |
| `IcebergPageSinkProvider.java` | Factory that wires up the PageSink from handle metadata |
| `IcebergWritableTableHandle.java` | Serializable handle carrying schema, partition spec, sort order, output path |
| `IcebergFileWriter.java` | Interface extending `FileWriter` with Iceberg-specific `FileMetrics` |
| `IcebergFileWriterFactory.java` | Creates Parquet/ORC/Avro writers |
| `IcebergParquetFileWriter.java` | Parquet-specific writer wrapping Trino's `ParquetFileWriter` |
| `IcebergSortingFileWriter.java` | Decorator that sorts data before writing |
| `CommitTaskData.java` | Record serialized as the fragment Slice -- carries file path, metrics, partition data |
| `MetricsWrapper.java` | JSON-serializable wrapper around Iceberg `Metrics` |
| `PartitionData.java` | Iceberg `StructLike` impl with JSON ser/de for partition values |
| `PartitionTransforms.java` | Maps Iceberg partition transforms (identity, year, bucket, truncate...) to Trino block transforms |
| `IcebergMetadata.java` | Coordinator-side metadata operations including `finishInsert()` and `finishMerge()` |
| `IcebergUtil.java` | Utility: `getLocationProvider()`, `commit()` for SnapshotUpdate |

## 1. High-Level Overview

The Iceberg write protocol in Trino follows a two-phase architecture:

```
Phase 1 (Workers - distributed):
  Page -> IcebergPageSink.appendPage()
       -> partition routing (PagePartitioner)
       -> per-partition IcebergFileWriter.appendRows()
       -> on finish/close: serialize CommitTaskData as Slice fragments

Phase 2 (Coordinator - centralized):
  Collect all fragment Slices from all workers
       -> IcebergMetadata.finishInsert()
       -> deserialize CommitTaskData
       -> build Iceberg DataFile objects
       -> AppendFiles.appendFile() for each
       -> transaction.commitTransaction()  (atomic Iceberg commit)
```

Key architectural properties:
- **Workers never touch Iceberg metadata** -- they only write data files and produce JSON fragments
- **The coordinator is the sole committer** -- it collects fragments from all workers and performs a single atomic Iceberg snapshot append
- **Rollback is file-deletion** -- each writer registers a rollback action that deletes its output file from object storage
- **Partition routing is hash-based** -- a `PageIndexer` assigns each row to a writer index based on its transformed partition values

## 2. Detailed Analysis

### 2.1 IcebergPageSink: Writing Pages to Data Files

`IcebergPageSink` implements `ConnectorPageSink`, the SPI interface for writing:

```java
public interface ConnectorPageSink {
    CompletableFuture<?> appendPage(Page page);       // receive data
    CompletableFuture<Collection<Slice>> finish();     // finalize, return fragments
    void abort();                                       // cleanup on failure
    default void closeIdleWriters() {}                  // close idle partition writers
}
```

The core write flow in `IcebergPageSink`:

```java
// appendPage() delegates to writePage()
public CompletableFuture<?> appendPage(Page page) {
    writePage(page);
    return NOT_BLOCKED;  // always synchronous, never back-pressured
}
```

**`writePage(Page page)`** does three things:

1. **Compute writer indexes** via `getWriterIndexes(page)` -- maps each row position to a writer index (creating new writers as needed)
2. **Build position arrays** -- groups row positions by writer index
3. **Dispatch pages** -- for each writer, extracts the relevant row positions as a sub-page via `page.getPositions(positions, 0, len)` and calls `writer.appendRows(pageForWriter)`

The position-based page splitting is efficient: when all rows go to the same writer, the full page is passed without copying. Only when rows span multiple partitions does it create dictionary-based filtered sub-pages.

**Writer lifecycle per partition:**
- Created lazily when the first row for that partition arrives
- Closed and replaced when `writer.getWrittenBytes() > targetMaxFileSize` (session-configurable)
- Closed via `closeIdleWriters()` when marked inactive and above `idleWriterMinFileSize`
- Hard limit: `maxOpenWriters` (throws `ICEBERG_TOO_MANY_OPEN_PARTITIONS` if exceeded)

### 2.2 Output File Path Determination

File paths are constructed in `getWriterIndexes()` when creating a new writer:

```java
// File name: <queryId>-<UUID>.<format_extension>
String fileName = fileFormat.toIceberg().addExtension(session.getQueryId() + "-" + randomUUID());

// Full path depends on whether the table is partitioned
String outputPath = partitionData
    .map(partition -> locationProvider.newDataLocation(partitionSpec, partition, fileName))
    .orElseGet(() -> locationProvider.newDataLocation(fileName));
```

**The `queryId` prefix is deliberate**: it enables opportunistic cleanup of orphan files from failed queries with retry/recovery. If a query fails partway through, the coordinator can identify and delete all files written by that query based on the query ID prefix.

**LocationProvider** determines the directory structure:
- `IcebergUtil.getLocationProvider()` chooses between:
  - `DefaultLocationProvider` -- standard Iceberg hive-style layout: `<table_location>/data/<partition_path>/<filename>`
  - `ObjectStoreLocationProvider` -- hash-based layout that avoids hot-prefix problems on S3: distributes files across hash-prefixed directories
- Custom `WRITE_LOCATION_PROVIDER_IMPL` is explicitly **rejected** by Trino (throws NOT_SUPPORTED)

For partitioned tables, the partition path is generated by the Iceberg library itself using the `PartitionSpec` and `PartitionData` objects, producing paths like `data/year=2024/month=01/<queryId>-<uuid>.parquet`.

### 2.3 Partition Routing: How Rows Reach Partition-Specific Writers

Partition routing is handled by the inner `PagePartitioner` class.

**Setup** (in `IcebergPageSink` constructor):

```java
this.pagePartitioner = new PagePartitioner(
    pageIndexerFactory,
    toPartitionColumns(partitionColumns, partitionSpec, outputSchema));
```

`toPartitionColumns()` maps each `PartitionField` in the spec to a `PartitionColumn` record containing:
- `field` -- the Iceberg `PartitionField` (source column ID, transform)
- `sourceChannels` -- the path of column indexes from the page to the source column (supports nested structs)
- `sourceType` / `resultType` -- the Trino types before and after the partition transform
- `blockTransform` -- a `Function<Block, Block>` that applies the partition transform

**Per-page partitioning** (`PagePartitioner.partitionPage()`):

```java
public int[] partitionPage(Page page) {
    Block[] blocks = new Block[columns.size()];
    for (int i = 0; i < columns.size(); i++) {
        PartitionColumn column = columns.get(i);
        Block block = getPartitionBlock(column, page);         // extract source column
        blocks[i] = column.blockTransform().apply(block);      // apply transform (year, bucket, etc.)
    }
    Page transformed = new Page(page.getPositionCount(), blocks);
    return pageIndexer.indexPage(transformed);  // hash to writer indexes
}
```

The `PageIndexer` (provided by Trino SPI) assigns a stable integer index to each unique combination of transformed partition values. This is effectively a hash-group-by on the partition columns.

**Supported partition transforms** (from `PartitionTransforms`):
- `identity` -- value as-is
- `year`, `month`, `day`, `hour` -- temporal bucketing from date/timestamp types
- `bucket[N]` -- hash bucketing into N buckets (Murmur3)
- `truncate[W]` -- truncation of integers, strings, decimals
- `void` -- all values map to null (used for hidden partitioning)

**Nested struct support**: `getPartitionBlock()` handles partition fields sourced from nested struct columns by traversing `sourceChannels` indices through `RowBlock.getRowFieldsFromBlock()`.

### 2.4 Fragment Slices: What finish() Returns

When `finish()` is called (all pages have been written), it closes every open writer and collects the commit fragments:

```java
public CompletableFuture<Collection<Slice>> finish() {
    for (int writerIndex = 0; writerIndex < writers.size(); writerIndex++) {
        closeWriter(writerIndex);
    }
    writers.clear();
    return completedFuture(commitTasks);
}
```

**`closeWriter(int writerIndex)`** does the critical work:

1. **Commits the file writer** -- `writer.commit()` flushes and closes the Parquet/ORC file, returns a rollback `Closeable` (which, if invoked, deletes the file)
2. **Builds a `CommitTaskData`** record:

```java
CommitTaskData task = new CommitTaskData(
    writeContext.getPath(),                              // full output file path
    fileFormat,                                           // PARQUET, ORC, or AVRO
    writer.getWrittenBytes(),                             // file size in bytes
    new MetricsWrapper(writer.getFileMetrics().metrics()), // column-level stats
    PartitionSpecParser.toJson(partitionSpec),             // partition spec JSON
    writeContext.getPartitionData().map(PartitionData::toJson), // partition values JSON
    DATA,                                                 // FileContent.DATA
    Optional.empty(),                                     // referencedDataFile (only for deletes)
    writer.getFileMetrics().splitOffsets(),                // Parquet row group offsets
    sortOrderId,                                          // sort order ID
    Optional.empty());                                    // serializedDeletionVector (only for v3 deletes)
```

3. **Serializes to JSON bytes, wraps as Slice**:

```java
commitTasks.add(wrappedBuffer(jsonCodec.toJsonBytes(task)));
```

**The `CommitTaskData` record** is the fragment contract between workers and coordinator:

```java
public record CommitTaskData(
    String path,                          // Full path to written data file
    IcebergFileFormat fileFormat,          // PARQUET / ORC / AVRO
    long fileSizeInBytes,                 // Exact file size after close
    MetricsWrapper metrics,               // Iceberg Metrics (record count, column sizes,
                                          //   value counts, null counts, NaN counts,
                                          //   lower bounds, upper bounds)
    String partitionSpecJson,             // Partition spec as JSON
    Optional<String> partitionDataJson,   // Partition values as JSON (empty for unpartitioned)
    FileContent content,                  // DATA, POSITION_DELETES, or EQUALITY_DELETES
    Optional<String> referencedDataFile,  // For position deletes: the data file being deleted from
    Optional<List<Long>> fileSplitOffsets,// Parquet row group start offsets (for split planning)
    int sortOrderId,                      // Sort order ID (0 for unsorted)
    Optional<byte[]> serializedDeletionVector) // V3 deletion vectors
```

**MetricsWrapper** wraps the Iceberg `Metrics` object with Jackson JSON annotations, carrying per-column statistics:
- `recordCount` -- number of rows
- `columnSizes` -- bytes per column (by field ID)
- `valueCounts`, `nullValueCounts`, `nanValueCounts` -- per column
- `lowerBounds`, `upperBounds` -- column min/max as `ByteBuffer` (by field ID)

### 2.5 The Atomic Iceberg Commit: finishInsert()

The coordinator calls `IcebergMetadata.finishInsert()` after collecting fragments from all workers. This is the centralized commit point.

**Transaction lifecycle:**

```
beginInsert()  -->  [workers write data files]  -->  finishInsert()
     |                                                     |
     v                                                     v
 beginTransaction(icebergTable)                    deserialize fragments
 transaction = catalog.newTransaction(icebergTable)  build DataFile objects
                                                   appendFiles.appendFile(...)
                                                   commitUpdate(appendFiles)
                                                   commitTransaction(transaction)
                                                   transaction = null
```

**`finishInsert()` implementation** (simplified):

```java
public Optional<ConnectorOutputMetadata> finishInsert(
        ConnectorSession session, ConnectorInsertTableHandle insertHandle,
        List<ConnectorTableHandle> sourceTableHandles,
        Collection<Slice> fragments, Collection<ComputedStatistics> computedStatistics)
{
    // 1. Deserialize all fragment Slices into CommitTaskData
    List<CommitTaskData> commitTasks = fragments.stream()
        .map(Slice::getInput)
        .map(commitTaskCodec::fromJson)
        .collect(toImmutableList());

    if (commitTasks.isEmpty()) {
        transaction = null;
        return Optional.empty();
    }

    // 2. Choose append strategy
    Table icebergTable = transaction.table();
    AppendFiles appendFiles = isMergeManifestsOnWrite(session)
        ? transaction.newAppend()        // full manifest merge
        : transaction.newFastAppend();   // fast append (no manifest merge)

    // 3. Build DataFile objects from each fragment and append
    for (CommitTaskData task : commitTasks) {
        DataFiles.Builder builder = DataFiles.builder(icebergTable.spec())
            .withPath(task.path())
            .withFileSizeInBytes(task.fileSizeInBytes())
            .withFormat(table.fileFormat().toIceberg())
            .withMetrics(task.metrics().metrics())
            .withSortOrder(sortOrders.get(task.sortOrderId()));
        task.fileSplitOffsets().ifPresent(builder::withSplitOffsets);

        if (!icebergTable.spec().fields().isEmpty()) {
            builder.withPartition(PartitionData.fromJson(
                task.partitionDataJson().orElseThrow(), partitionColumnTypes));
        }
        appendFiles.appendFile(builder.build());
    }

    // 4. Commit the append (creates a new snapshot)
    appendFiles.scanManifestsWith(icebergScanExecutor);
    commitUpdate(appendFiles, session, "insert");

    // 5. Optionally write statistics
    if (!computedStatistics.isEmpty()) { ... }

    // 6. Commit the transaction
    commitTransaction(transaction, "insert");
    transaction = null;

    return Optional.of(new IcebergCommitMetadata(icebergTable.currentSnapshot().summary()));
}
```

**Key details:**

- **`newFastAppend()` vs `newAppend()`**: `FastAppend` creates new manifest files without rewriting existing ones -- faster but leaves more small manifests. `newAppend()` merges manifests. Controlled by `isMergeManifestsOnWrite` session property.

- **`commitUpdate()` helper**: Tags the snapshot with `TRINO_QUERY_ID` and `TRINO_USER` before committing:

```java
public static void commit(SnapshotUpdate<?> update, ConnectorSession session) {
    update.set(TRINO_QUERY_ID_NAME, session.getQueryId());
    update.set(TRINO_USER_NAME, session.getUser());
    update.commit();
}
```

- **Error handling**: Catches `CommitFailedException`, `CommitStateUnknownException`, `ValidationException`, `RESTException` and wraps them in `ICEBERG_COMMIT_ERROR`.

- **`finishCreateTable()` delegates to `finishInsert()`** when there are fragments (CTAS with data), or does a bare `newFastAppend()` commit for empty tables.

### 2.6 Abort and Rollback Handling

Rollback operates at two levels: individual file cleanup and metadata transaction abandonment.

**`IcebergPageSink.abort()`:**

```java
public void abort() {
    List<Closeable> rollbackActions = Streams.concat(
        writers.stream()                          // still-open writers
            .filter(Objects::nonNull)
            .map(writer -> writer::rollback),
        closedWriterRollbackActions.stream())      // already-closed writers
        .collect(toImmutableList());

    RuntimeException error = null;
    for (Closeable rollbackAction : rollbackActions) {
        try { rollbackAction.close(); }
        catch (Throwable t) { /* suppress, continue cleanup */ }
    }
    if (error != null) throw error;
}
```

**What rollback actions do:**

For a Parquet file, the rollback action is:
```java
Closeable rollbackAction = () -> fileSystem.deleteFile(outputPath);
```

This is set when the writer is created (`IcebergFileWriterFactory.createParquetWriter()`). The `ParquetFileWriter.commit()` method closes the Parquet writer (flushing footer) and returns this rollback action. If `commit()` itself fails, it invokes the rollback action immediately.

**Two sources of rollback actions:**
1. **`closedWriterRollbackActions`** -- from writers that were successfully closed via `closeWriter()` (the file exists on storage). The `commit()` method returns the rollback Closeable, which is stored.
2. **Live writers** -- still-open writers whose `rollback()` method both closes the writer AND deletes the file.

**Transaction-level rollback**: If `finishInsert()` is never called (e.g., query fails), the Iceberg `Transaction` object is simply abandoned -- since no `commitTransaction()` was called, no metadata changes are persisted. The written data files become orphans that can be cleaned up by Iceberg's `expire_snapshots` or `remove_orphan_files` procedures.

### 2.7 Sorted Writing Support

When the Iceberg table has a sort order and `sortedWritingEnabled` is true, `IcebergPageSink` wraps the base writer in an `IcebergSortingFileWriter`:

```java
if (!sortFields.isEmpty() && sortedWritingEnabled) {
    String tempName = "sorting-file-writer-%s-%s".formatted(session.getQueryId(), randomUUID());
    Location tempFilePrefix = tempDirectory.appendPath(tempName);
    WriteContext writerContext = createWriter(outputPath, partitionData);
    IcebergFileWriter sortedFileWriter = new IcebergSortingFileWriter(
        fileSystem, tempFilePrefix,
        writerContext.getWriter(),        // the real output writer
        sortingFileWriterBufferSize,
        sortingFileWriterMaxOpenFiles,
        columnTypes, sortColumnIndexes, sortOrders,
        pageSorter, typeManager.getTypeOperators());
    writer = new WriteContext(sortedFileWriter, outputPath, partitionData);
}
```

**How it works:**
- `IcebergSortingFileWriter` delegates to Hive's `SortingFileWriter`
- Incoming pages are buffered in memory up to `sortingFileWriterBufferSize`
- When the buffer is full, pages are sorted and spilled to temp files
- On `commit()`, all temp files are merge-sorted and written to the final output writer
- The temp directory is either a configured local path or `<table_location>/trino-tmp-files/`

**Sort field filtering**: Trino only supports identity transforms for sort fields. If the Iceberg table's sort order uses non-identity transforms, those fields are excluded and the `sortOrderId` is set to unsorted. Only when ALL sort fields are identity-transform and supported does Trino write the Iceberg sort order ID into the data file metadata.

### 2.8 File Writer Factory and Format Abstraction

`IcebergFileWriterFactory.createDataFileWriter()` dispatches by format:

```java
public IcebergFileWriter createDataFileWriter(...) {
    return switch (fileFormat) {
        case PARQUET -> createParquetWriter(metricsConfig, fileSystem, outputPath, icebergSchema, session, storageProperties);
        case ORC -> createOrcWriter(metricsConfig, fileSystem, outputPath, icebergSchema, session, storageProperties, ...);
        case AVRO -> createAvroWriter(fileSystem, outputPath, icebergSchema, storageProperties);
    };
}
```

**Parquet path** (`IcebergParquetFileWriter`):
- Wraps `io.trino.plugin.hive.parquet.ParquetFileWriter` which wraps `io.trino.parquet.writer.ParquetWriter`
- Configurable page size, page value count, block size, batch size, bloom filter columns
- Compression codec from table properties or session config
- Metrics collected post-write from Parquet footer via `footerMetrics()`
- Split offsets computed from row group starting positions

**ORC path** (`IcebergOrcFileWriter`):
- Uses Trino's ORC writer with configurable stripe size, row group size, dictionary memory
- Optional write validation (re-reads and validates the file after writing)
- Bloom filter columns configurable via table properties

**Avro path** (`IcebergAvroFileWriter`):
- Uses Iceberg's own `ForwardingOutputFile` adapter
- Simplest path, minimal configuration

**The `IcebergFileWriter` interface** adds Iceberg-specific metrics to the base `FileWriter`:

```java
public interface IcebergFileWriter extends FileWriter {
    FileMetrics getFileMetrics();

    record FileMetrics(Metrics metrics, Optional<List<Long>> splitOffsets) {}
}
```

Where `FileWriter` provides:
```java
public interface FileWriter {
    long getWrittenBytes();
    long getMemoryUsage();
    void appendRows(Page dataPage);
    Closeable commit();     // returns rollback action
    void rollback();
    long getValidationCpuNanos();
}
```

### 2.9 Merge (UPDATE/DELETE) Commit Path

For UPDATE/DELETE/MERGE operations, `IcebergMergeSink` wraps an `IcebergPageSink` for data writes and additionally writes position delete files or deletion vectors. The commit path goes through `finishMerge()` -> `finishWrite()`:

```java
public void finishMerge(...) {
    finishWrite(session, handle, fragments);
}
```

`finishWrite()` differs from `finishInsert()` in several ways:
- Uses `RowDelta` instead of `AppendFiles` (supports both inserts and deletes atomically)
- Separates `CommitTaskData` into data tasks (`FileContent.DATA`) and delete tasks (`FileContent.POSITION_DELETES`)
- Adds conflict detection: validates from a base snapshot, checks for conflicting data files and delete files
- For V3 format: handles deletion vectors (merging with existing delete files)
- For V2 format: creates position delete files via `FileMetadata.deleteFileBuilder()`

The conflict detection uses `IsolationLevel` from table properties:
- `SERIALIZABLE`: validates no conflicting data files
- Both levels validate deleted files and no conflicting delete files

## 3. Key Design Insights

### 3.1 Two-Phase Commit Without Distributed Transactions

Trino avoids distributed transactions by separating data writing (distributed, idempotent, retryable) from metadata commit (centralized, atomic). Workers are pure data writers that produce self-describing fragment metadata. The coordinator is the sole committer. This means:
- No two-phase commit protocol needed
- No distributed locks
- Failure recovery is simple: abandon the transaction, orphan files get cleaned up later

### 3.2 Fragment Serialization as the Worker-Coordinator Contract

The `CommitTaskData` JSON Slice is the complete interface between worker and coordinator. It carries everything needed to register the data file with Iceberg: path, size, full column metrics, partition values, sort order. The coordinator never reads the data files -- it trusts the fragment metadata.

### 3.3 Lazy Writer Management with Size-Based Rotation

Writers are created lazily on first row and rotated when they exceed `targetMaxFileSize`. This naturally produces well-sized files without requiring upfront knowledge of data distribution. The `closeIdleWriters()` mechanism prevents unbounded memory use for long-running writes across many partitions.

### 3.4 Rollback Actions as Closeable Chains

Every writer produces a rollback `Closeable` on commit. These are accumulated and invoked on abort. This creates a clean separation: the writer knows how to clean up its file, the PageSink knows the order of operations. Failed `commit()` calls self-clean immediately.

### 3.5 Fast Append vs Full Append

The `isMergeManifestsOnWrite` session property controls whether Trino uses `newFastAppend()` (no manifest merging, faster, more small manifests) or `newAppend()` (merges manifests, slower but better read performance). For streaming/frequent inserts, fast append is preferred; for batch loads, full append is better.

### 3.6 Query ID in File Names

Prepending the query ID to file names enables deterministic identification of all files written by a specific query. This is critical for fault tolerance: if a query partially fails and retries, the cleanup logic can identify and delete orphan files from the failed attempt.

## 4. Porting Considerations (Java -> Rust)

### 4.1 Core Interfaces to Define

```
trait ConnectorPageSink {
    fn append_page(&mut self, page: &Page) -> Result<()>;
    fn finish(&mut self) -> Result<Vec<Bytes>>;     // fragment slices
    fn abort(&mut self) -> Result<()>;
    fn close_idle_writers(&mut self);
}

trait IcebergFileWriter {
    fn append_rows(&mut self, page: &Page) -> Result<()>;
    fn commit(&mut self) -> Result<Box<dyn FnOnce() -> Result<()>>>;  // rollback action
    fn rollback(&mut self) -> Result<()>;
    fn written_bytes(&self) -> u64;
    fn memory_usage(&self) -> u64;
    fn file_metrics(&self) -> FileMetrics;
}
```

### 4.2 Fragment Serialization

`CommitTaskData` must be a well-defined serialization format (likely serde JSON or Protocol Buffers). The Rust engine needs to produce fragments that the coordinator can deserialize and use to build Iceberg `DataFile` objects. If using the `iceberg-rust` crate, the fragment should carry enough information to call `DataFileBuilder`.

### 4.3 Partition Routing

The `PagePartitioner` concept maps well to Rust. Key components:
- A partition transform function per partition field (identity, year, bucket, truncate)
- A hash-based grouping mechanism mapping transformed values to writer indices
- Consider using `HashMap<PartitionKey, WriterIndex>` instead of Trino's `PageIndexer` for clarity

### 4.4 Rollback Action Pattern

Rust's ownership model actually improves the rollback pattern. A writer can return a `Box<dyn FnOnce()>` cleanup closure on commit. The `abort()` implementation collects all closures and invokes them. Consider using `scopeguard` or explicit drop handlers.

### 4.5 LocationProvider

The path construction logic (query-id prefix, partition path from partition spec + values, format extension) can be ported directly. The Iceberg LocationProvider interface from `iceberg-rust` should be used for compatibility. Key concern: `ObjectStoreLocationProvider` hashing for S3.

### 4.6 Sorted Writing

The sorting file writer pattern (buffer in memory, spill to temp files, merge-sort on commit) is a significant piece of machinery. Consider:
- Using `arrow-rs` `RecordBatch` sorting for in-memory sorts
- External sort via temp Parquet files for spilling
- The `polars` crate also provides efficient external sorting

### 4.7 Coordinator Commit Logic

The coordinator's `finishInsert()` logic depends heavily on the Iceberg Java library (`Transaction`, `AppendFiles`, `DataFiles.builder()`). In Rust:
- Use `iceberg-rust` crate's `Transaction` and `AppendAction` equivalents
- The fragment deserialization -> `DataFile` building -> atomic commit pattern stays the same
- Error handling for `CommitFailedException` / `CommitStateUnknownException` must be robust

### 4.8 Metrics Collection

Iceberg metrics (record count, column sizes, min/max bounds, null counts) must be collected from the Parquet/ORC writer. The `parquet-rs` crate provides row group metadata from which these can be computed. This is what Trino's `IcebergParquetFileWriter.getFileMetrics()` -> `footerMetrics()` does.

### 4.9 Memory Management

Trino's `memoryUsage` tracking is manual (delta-based). In Rust:
- Use `jemalloc` or a custom allocator with stats
- Or track buffer sizes explicitly as Trino does
- The `targetMaxFileSize` and `idleWriterMinFileSize` thresholds should be configurable session properties

### 4.10 Concurrency Model

Trino's `IcebergPageSink` is single-threaded per task (one task per partition-split). In Rust, maintain this model: each task gets its own `PageSink` instance. The coordinator collects fragments from all tasks after they complete. No locking is needed within a single PageSink.
