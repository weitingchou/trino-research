# Module Teardown: Split JSON Formats -- Hive & Iceberg (Task 4.3.C)

## 0. Research Focus
* **Task ID:** 4.3.C
* **Knowledge Graph Tag:** KG-10
* **Focus:** Document the complete JSON serialization format for splits -- how `ConnectorSplit` polymorphism is handled, the full field-by-field JSON structure of `HiveSplit` and `IcebergSplit`, the wrapper hierarchy (`SplitAssignment` -> `ScheduledSplit` -> `Split` -> `CatalogHandle` + `ConnectorSplit`), and how the coordinator serializes splits within `TaskUpdateRequest`. Identify porting considerations for Rust.

## 1. High-Level Overview

* **Core Responsibility:** Splits are the fundamental unit of work in Trino's distributed execution. The coordinator discovers splits via connector-specific `ConnectorSplitManager` implementations, then sends them to workers over HTTP as JSON inside `TaskUpdateRequest`. Each split describes a specific byte range (or logical partition) of a data file that a single worker thread should read. The JSON format must handle polymorphic `ConnectorSplit` types transparently -- a Hive split and an Iceberg split have completely different fields, but they travel through the same generic plumbing.

* **Key Design Decisions:**
  - **Polymorphic dispatch via `@type` property:** Trino uses a custom Jackson module (`AbstractTypedJacksonModule`) that injects an `@type` field containing the fully-qualified class name (prefixed with a classloader ID) into every serialized `ConnectorSplit`. On deserialization, this `@type` value is resolved through `HandleResolver` to locate the concrete class. This avoids requiring a closed set of known split types at compile time.
  - **ClassLoader-scoped type IDs:** The `@type` value is not a simple class name -- it is `<classLoaderId>:<fullyQualifiedClassName>` (e.g., `hive:io.trino.plugin.hive.HiveSplit`). This is critical because connector classes are loaded by isolated `PluginClassLoader` instances. The resolver maps the classloader ID back to the correct classloader to find the class.
  - **Four-level nesting hierarchy:** Splits travel over the wire wrapped in four layers: `SplitAssignment` (groups splits by plan node) -> `ScheduledSplit` (adds a monotonic sequence ID) -> `Split` (pairs a `CatalogHandle` with a `ConnectorSplit`) -> the concrete `ConnectorSplit` (e.g., `HiveSplit`). Each level adds metadata needed by the worker.
  - **Addresses intentionally excluded from JSON:** Both `HiveSplit` and `IcebergSplit` annotate `getAddresses()` with `@JsonIgnore`. Host addresses are used by the coordinator for scheduling decisions but are not needed by the worker, so they are stripped from the serialized payload to save bandwidth.
  - **No inheritance, just interface implementation:** `ConnectorSplit` is a plain Java interface with no `@JsonTypeInfo` annotation. The polymorphism is handled entirely through Trino's custom Jackson module system, not through Jackson's built-in annotation-based polymorphism.
  - **Records serialize transparently:** Java `record` types (`HivePartitionKey`, `Schema`, `AcidInfo`, `DeleteFile`) serialize via Jackson's built-in record support -- component names become JSON field names with no explicit `@JsonProperty` annotations needed.

## 2. Detailed Analysis

### 2.1 The ConnectorSplit Interface (SPI)

The `ConnectorSplit` interface is minimal -- it defines behavioral contracts, not data:

```java
// core/trino-spi/.../connector/ConnectorSplit.java
public interface ConnectorSplit
{
    default boolean isRemotelyAccessible() { return true; }
    default List<HostAddress> getAddresses() { ... }
    default SplitWeight getSplitWeight() { return SplitWeight.standard(); }
    default long getRetainedSizeInBytes() { throw new UnsupportedOperationException(...); }
}
```

There are no Jackson annotations on this interface. All serialization is handled by the concrete implementations and the `AbstractTypedJacksonModule` infrastructure. The interface exists purely at the SPI boundary -- connectors implement it, and the engine treats it as an opaque marker type that Jackson must be taught to serialize/deserialize polymorphically.

### 2.2 Polymorphic Serialization: AbstractTypedJacksonModule

This is the most architecturally significant piece of the split serialization story.

**Registration (HandleJsonModule):**

```java
// core/trino-main/.../metadata/HandleJsonModule.java
@ProvidesIntoSet
public static com.fasterxml.jackson.databind.Module splitModule(HandleResolver resolver)
{
    return new AbstractTypedJacksonModule<>(
        ConnectorSplit.class,
        resolver::getId,        // Object -> String (type tag)
        resolver::getHandleClass // String -> Class<?> (reverse lookup)
    ) {};
}
```

This is registered as a Guice `@ProvidesIntoSet` Jackson module. The same pattern is used for `ConnectorTableHandle`, `ColumnHandle`, `ConnectorTransactionHandle`, and 10+ other polymorphic connector types. The `HandleJsonModule` installs the `HandleResolver` singleton and creates one `AbstractTypedJacksonModule` per polymorphic SPI interface.

**The Type ID Format (HandleResolver):**

```java
// core/trino-main/.../metadata/HandleResolver.java
public String getId(Object handle) {
    return classId(handle.getClass());
}

private static String classId(Class<?> handleClass) {
    return classLoaderId(handleClass) + ":" + handleClass.getName();
}

private static String classLoaderId(Class<?> handleClass) {
    ClassLoader classLoader = handleClass.getClassLoader();
    if (classLoader instanceof PluginClassLoader pluginClassLoader) {
        return pluginClassLoader.getId();  // e.g., "hive", "iceberg"
    }
    // system classloader
    return "system";
}
```

The plugin name comes from `PluginClassLoader.getId()`, which returns the `pluginName` string set at construction time. For Hive, this is typically `"hive"`; for Iceberg, `"iceberg"`.

The reverse direction:

```java
public Class<?> getHandleClass(String id) {
    int splitPoint = id.lastIndexOf(':');
    String classLoaderId = id.substring(0, splitPoint);   // e.g., "hive"
    String className = id.substring(splitPoint + 1);      // e.g., "io.trino.plugin.hive.HiveSplit"

    ClassLoader classLoader = classLoaders.get(classLoaderId);
    return classLoader.loadClass(className);
}
```

**The Serialization Mechanism:**

```java
// core/trino-main/.../metadata/AbstractTypedJacksonModule.java
private static final String TYPE_PROPERTY = "@type";

// Serializer wraps each object with a type tag:
public void serialize(T value, JsonGenerator generator, SerializerProvider provider)
        throws IOException
{
    JsonSerializer<T> serializer = serializerCache.get(type, () -> createSerializer(provider, type));
    serializer.serializeWithType(value, generator, provider, typeSerializer);
}
```

The `typeSerializer` is an `AsPropertyTypeSerializer` configured to write the `@type` property. The `typeDeserializer` is an `AsPropertyTypeDeserializer` that reads the `@type` property to resolve the concrete class. This is Jackson's `As.PROPERTY` inclusion strategy -- the type tag is a sibling property inside the JSON object, not a wrapper.

**Resulting wire format for ConnectorSplit:**

```json
{
  "@type": "hive:io.trino.plugin.hive.HiveSplit",
  "partitionName": "...",
  "path": "...",
  ...
}
```

### 2.5 The Split Assignment Hierarchy

Splits travel from coordinator to worker inside `TaskUpdateRequest` wrapped in a four-level hierarchy:

**Level 1: SplitAssignment** (groups splits by plan node)

```java
// core/trino-main/.../execution/SplitAssignment.java
public class SplitAssignment {
    @JsonProperty PlanNodeId planNodeId;     // serialized as plain string via @JsonValue
    @JsonProperty Set<ScheduledSplit> splits; // sorted by sequenceId
    @JsonProperty boolean noMoreSplits;       // terminal signal
}
```

**Level 2: ScheduledSplit** (adds monotonic sequence ID)

```java
// core/trino-main/.../execution/ScheduledSplit.java
public record ScheduledSplit(long sequenceId, PlanNodeId planNodeId, Split split)
```

Java record -- serialized by component names. The `sequenceId` provides idempotent, ordered split delivery. Equality is defined solely by `sequenceId` (not by the contained split), enabling the worker to detect duplicates.

**Level 3: Split** (pairs catalog identity with connector-specific split)

```java
// core/trino-main/.../metadata/Split.java
public final class Split {
    @JsonProperty CatalogHandle catalogHandle;    // serialized as string "name:type:version"
    @JsonProperty ConnectorSplit connectorSplit;   // polymorphic via @type
}
```

**Level 4: ConnectorSplit** (the concrete split -- HiveSplit, IcebergSplit, etc.)

This is where the `@type` property injection from `AbstractTypedJacksonModule` occurs.

**CatalogHandle** serialization:

```java
// core/trino-main/.../connector/CatalogHandle.java
@JsonValue
public String getId() {
    return catalogName + ":" + type.toString().toLowerCase(ROOT) + ":" + version;
}

@JsonCreator
public static CatalogHandle fromId(String id) { /* parses "name:type:version" */ }
```

Example: `"hive:normal:42"` means catalog named "hive", type NORMAL (not INFORMATION_SCHEMA or SYSTEM), version 42.

```java
// core/trino-main/.../planner/plan/PlanNodeId.java
```

### 2.6 Coordinator Serialization Path

The serialization path in `HttpRemoteTask.sendUpdateInternal()`:

```java
// core/trino-main/.../server/remotetask/HttpRemoteTask.java (line 771)
TaskUpdateRequest updateRequest = new TaskUpdateRequest(
        session.toSessionRepresentation(),
        session.getIdentity().getExtraCredentials(),
        stageSpan,
        fragment,
        tableCredentials,
        splitAssignments,    // <-- List<SplitAssignment>
        outputBuffers.get(),
        dynamicFilterDomains.getDynamicFilterDomains(),
        session.getExchangeEncryptionKey(),
        speculative.get());
byte[] taskUpdateRequestJson = taskUpdateRequestCodec.toJsonBytes(updateRequest);
```

The `taskUpdateRequestCodec` is a `JsonCodec<TaskUpdateRequest>` injected by Guice. It uses the Jackson `ObjectMapper` configured with:
- All `AbstractTypedJacksonModule` instances registered via `HandleJsonModule` (including the `ConnectorSplit` module)
- The standard airlift JSON module configuration

The serialized bytes are then sent as the HTTP POST body:

```java
Request request = preparePost()
        .setUri(uriBuilder.build())
        .setHeader(CONTENT_TYPE, MediaType.JSON_UTF_8.toString())
        .setBodyGenerator(createStaticBodyGenerator(taskUpdateRequestJson))
        .build();
```

The coordinator also implements **adaptive request sizing** -- if the serialized JSON exceeds `maxRequestSizeInBytes`, it reduces `splitBatchSize` and reschedules the update with fewer splits per batch.

## 3. Key Design Insights

1. **The `@type` field is the linchpin of extensibility.** Without it, the engine would need compile-time knowledge of all possible split types. The `classLoaderId:className` format ties into the plugin classloader isolation model -- each connector plugin has its own classloader, and the resolver knows which classloader can deserialize which split type. This is what makes the connector SPI truly pluggable at runtime.

2. **Addresses are coordinator-only metadata.** Both `HiveSplit` and `IcebergSplit` use `@JsonIgnore` on `getAddresses()`. The coordinator uses addresses to decide *which* worker gets a split (for data locality). Once assigned, the worker does not need to know where the data is "preferably" located -- it just reads from the path. This is a bandwidth optimization: split payloads can be large when multiplied by thousands of splits.

3. **Split serialization is a major bandwidth concern.** The `HttpRemoteTask` has adaptive request sizing (`adjustSplitBatchSize`) that dynamically reduces the number of splits per HTTP request if the serialized JSON payload exceeds the configured maximum. This implies that split JSON can be large -- particularly for Hive splits with many partition keys, complex schemas, and ACID info.

4. **Java records simplify serialization dramatically.** `Schema`, `HivePartitionKey`, `AcidInfo`, `DeleteFile`, `ScheduledSplit`, `TaskUpdateRequest` are all Java records. Jackson auto-discovers record components and uses them as JSON field names. This eliminates the need for `@JsonProperty` annotations on every getter. The Rust port should use `#[derive(Serialize, Deserialize)]` with serde to achieve the same effect.

5. **`OptionalInt` / `OptionalLong` serialization is non-obvious.** Java's `OptionalInt` and `OptionalLong` are serialized as `null` (when empty) or as the bare number (when present) by Jackson. This works because Jackson has built-in support for Java optional types via its `jdk8` module. The Rust port needs to handle `Option<i32>` / `Option<i64>` the same way.

6. **SplitWeight uses fixed-point representation.** The standard weight is 100 (representing 1.0). A weight of 0.5 is stored as 50. This avoids floating-point precision issues in split count accounting. The `@JsonValue`/`@JsonCreator` annotations mean it serializes as a bare long, not as an object.

7. **The `noMoreSplits` flag is part of split assignment, not individual splits.** This is significant -- it means the coordinator must send a final `SplitAssignment` with `noMoreSplits: true` for each plan node. The worker uses this to know when it has received all splits for a given source node and can proceed to the finishing phase.

8. **CatalogHandle includes a version number.** The version is part of the `CatalogHandle` identity (format: `name:type:version`). This enables catalog versioning for hot-reload scenarios. If a catalog is dropped and recreated, the version changes, ensuring stale splits from the old catalog version are rejected.

## 4. Porting Considerations (Java -> Rust)

### 4.1 Polymorphic Split Deserialization

The Java approach uses runtime classloading to resolve `@type` strings to concrete classes. In Rust, there is no equivalent runtime class resolution. Options:

- **serde `#[serde(tag = "@type")]` with enum dispatch:** Define an enum `ConnectorSplit { Hive(HiveSplit), Iceberg(IcebergSplit), ... }` with `#[serde(tag = "@type")]`. The `@type` values become the enum discriminant strings. This is the most natural serde approach but requires all split types to be known at compile time.
- **Two-pass deserialization:** First parse the raw JSON value, extract the `@type` field, then dispatch to the appropriate deserializer. This is more flexible but more complex.
- **Registry-based deserialization:** Maintain a `HashMap<String, Box<dyn Fn(&str) -> Box<dyn ConnectorSplit>>>` that maps type IDs to deserialization functions. Connectors register their deserializers at startup. This is closest to the Java model.

**Recommendation:** Use serde's `#[serde(tag = "@type")]` on an enum. The Rust worker only needs to support the connectors it is compiled with, so compile-time dispatch is acceptable and far simpler.

### 4.2 Type ID Format

The Java type ID format is `classLoaderId:fullyQualifiedClassName` (e.g., `"hive:io.trino.plugin.hive.HiveSplit"`). The Rust worker must recognize these exact strings since the coordinator (remaining in Java) will send them. Use `#[serde(rename = "hive:io.trino.plugin.hive.HiveSplit")]` on each enum variant.

### 4.6 Fields to Watch

- **`TupleDomain<IcebergColumnHandle>`** in `IcebergSplit`: This is a complex generic type involving predicate domains. Its JSON format needs separate analysis (it was covered in the predicate serialization task). The Rust port may initially deserialize this as `serde_json::Value` and refine later.
- **`DeleteFile` references Apache Iceberg enums** (`FileContent`, `FileFormat`): These serialize as their enum names. The Rust port needs matching enums with `#[serde(rename = "...")]` for any names that differ from Rust conventions.
- **`BucketingVersion` in `BucketConversion`**: An enum (`BUCKETING_V1`, `BUCKETING_V2`) from Hive.
- **`HiveColumnHandle` in `BucketConversion` and `BucketValidation`**: A complex type with many fields including `Type` (Trino's type system). This is a deep dependency tree that needs careful modeling.

### 4.7 Bandwidth and Performance

Split JSON payloads can be significant. For a query over a table with 10,000 partitions, each HiveSplit carries the full `Schema` (serde properties map) and partition keys. The Rust worker must efficiently parse these payloads. Consider:

- Use `serde_json` with borrowed string deserialization where possible
- Pre-allocate vectors for splits based on expected batch sizes
- Profile deserialization of large split assignment payloads early in development
- For Hive splits specifically, the `Schema.serdeProperties` map and `HivePartitionKey` lists are the most repetitive data -- consider whether these can be deduplicated across splits in the same assignment (the Java coordinator does not do this, but a Rust optimization could)

### 4.8 Test Strategy

Create JSON fixtures matching the exact format the coordinator sends. The `TestHiveSplit.java` test provides a reference for field values, but it tests only the inner split (without the `@type` wrapper). For the Rust worker, test both:
1. **Inner split deserialization:** Parse a `HiveSplit` JSON (without `@type`) into the Rust struct
2. **Full polymorphic deserialization:** Parse a `ConnectorSplit` JSON (with `@type`) into the enum
3. **Full hierarchy deserialization:** Parse a complete `SplitAssignment` JSON through all four levels
