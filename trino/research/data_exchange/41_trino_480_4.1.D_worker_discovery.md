# Module Teardown: Worker Discovery & Registration Protocol (Task 4.1.D)

## 0. Research Focus
* **Task ID:** 4.1.D
* **Knowledge Graph Tag:** KG-3
* **Focus:** How does a Trino worker register itself with the coordinator? What properties does it advertise? Is there a periodic heartbeat separate from task-level status? How does the coordinator maintain its list of active workers? What configuration is required? How does the coordinator detect worker failure?

## 1. High-Level Overview

* **Core Responsibility:** Trino 480 implements a two-layer node discovery system. The first layer is **node inventory** -- how the coordinator learns that a worker exists (its URI). The second layer is **node state verification** -- how the coordinator confirms that a known worker is healthy, compatible, and in the correct lifecycle state. The two layers are decoupled and pluggable.

* **Key Design Decisions:**
  - **Three pluggable inventory backends:** The `discovery.type` config property selects among `ANNOUNCE` (default, workers push their URI to the coordinator), `AIRLIFT_DISCOVERY` (legacy, uses Airlift's embedded discovery server with service selectors), or `DNS` (headless DNS resolution, for Kubernetes). Each backend implements the `NodeInventory` interface that returns `Set<URI>`.
  - **Coordinator-driven state polling:** Regardless of which inventory backend discovers the worker URI, the coordinator independently polls each worker's `/v1/info` endpoint to verify environment, version, and lifecycle state. This is **not** a task-level heartbeat -- it is a cluster-membership heartbeat.
  - **Pull-not-push health model:** Workers never push health status to the coordinator. The coordinator maintains a `RemoteNodeState` per known URI and asynchronously refreshes it every 5 seconds. This is the same architectural pattern as task-level status polling (Task 4.1.B), applied to cluster membership.
  - **Separate failure detection layer:** The `HeartbeatFailureDetector` (only in `AIRLIFT_DISCOVERY` mode) runs its own independent heartbeat loop, pinging `/v1/status` on each worker every 500ms. It uses exponential decay counters to compute a failure ratio. Workers that exceed the failure threshold are excluded from the `NodeInventory` before they even reach the node state poller.
  - **Workers are stateless regarding cluster membership:** Workers have no knowledge of the full cluster topology. `WorkerInternalNodeManager` throws `UnsupportedOperationException` for all node listing operations. Workers only know their own identity and the coordinator URI(s) to announce to.
  - **Environment and version validation as a firewall:** The coordinator rejects any worker that advertises a different `node.environment` or `NodeVersion`. Such workers are marked `INVALID` and are invisible to the scheduler.

## 2. Detailed Analysis

### 2.1 Discovery Mode Selection: NodeManagerModule

The `NodeManagerModule` is the entry point for all node management wiring. It selects the appropriate discovery backend based on `discovery.type`:

```java
// io.trino.node.NodeManagerModule
public class NodeManagerModule extends AbstractConfigurationAwareModule {
    @Override
    protected void setup(Binder binder) {
        ServerConfig serverConfig = buildConfigObject(ServerConfig.class);
        if (serverConfig.isCoordinator()) {
            binder.bind(CoordinatorNodeManager.class).in(Scopes.SINGLETON);
            binder.bind(InternalNodeManager.class).to(CoordinatorNodeManager.class).in(Scopes.SINGLETON);
            // HTTP client for polling worker /v1/info endpoints
            install(internalHttpClientModule("node-manager", ForNodeManager.class)
                    .withConfigDefaults(config -> {
                        config.setIdleTimeout(new Duration(30, SECONDS));
                        config.setRequestTimeout(new Duration(10, SECONDS));
                    }).build());
        } else {
            binder.bind(InternalNodeManager.class).to(WorkerInternalNodeManager.class).in(Scopes.SINGLETON);
        }

        switch (buildConfigObject(NodeInventoryConfig.class).getType()) {
            case AIRLIFT_DISCOVERY -> install(new AirliftNodeInventoryModule(nodeVersion));
            case ANNOUNCE -> install(new AnnounceNodeInventoryModule());
            case DNS -> install(new DnsNodeInventoryModule());
        }
    }
}
```

The `NodeInventoryConfig` maps the `discovery.type` property to an enum:

```java
// io.trino.node.NodeInventoryConfig
public class NodeInventoryConfig {
    public enum NodeInventoryType { ANNOUNCE, DNS, AIRLIFT_DISCOVERY }
    private NodeInventoryType type = NodeInventoryType.ANNOUNCE;  // default

    @Config("discovery.type")
    public NodeInventoryConfig setType(NodeInventoryType type) { ... }
}
```

**Key insight:** The default discovery type in Trino 480 is `ANNOUNCE`, not the legacy `AIRLIFT_DISCOVERY`. This is a significant change from older Trino versions.

### 2.2 Worker Registration -- The Three Inventory Backends

#### 2.2.1 ANNOUNCE Mode (Default)

In `ANNOUNCE` mode, each worker periodically POSTs its own URI to the coordinator's `/v1/announce` endpoint.

**Worker side -- `AnnounceNodeAnnouncer`:**

```java
// io.trino.node.AnnounceNodeAnnouncer
public class AnnounceNodeAnnouncer implements Announcer {
    @Inject
    public AnnounceNodeAnnouncer(InternalNode currentNode, AnnounceNodeAnnouncerConfig config,
                                  @ForAnnouncer HttpClient httpClient) {
        this(currentNode.getInternalUri(), config.getCoordinatorUris(), currentNode.isCoordinator(), httpClient);
    }

    public AnnounceNodeAnnouncer(URI internalUri, Collection<URI> coordinatorUris,
                                  boolean coordinator, HttpClient httpClient) {
        // Build list of /v1/announce URIs, filtering out self
        this.announceUris = coordinatorUris.stream()
                .filter(not(currentUri::equals))
                .map(uri -> uriBuilderFrom(uri).appendPath("/v1/announce").build())
                .distinct()
                .toList();
        this.currentHostAnnouncement = createStaticBodyGenerator(currentUri.toString(), UTF_8);
    }

    @Override
    public void start() {
        // Schedule periodic announcements every 5 seconds
        announceExecutor.scheduleWithFixedDelay(() -> {
            try { forceAnnounce(); }
            catch (RuntimeException e) { log.error(e, "Error announcing node to coordinators"); }
        }, 5, 5, SECONDS);
        forceAnnounce();  // also announce immediately
    }

    private ListenableFuture<?> announce(URI announceUri) {
        // POST the worker's own URI as plain text body to coordinator
        Request request = preparePost()
                .setUri(announceUri)
                .setBodyGenerator(currentHostAnnouncement)
                .setHeader(CONTENT_TYPE, TEXT_PLAIN)
                .build();
        return httpClient.executeAsync(request, createStatusResponseHandler());
    }
}
```

The configuration for announce mode maps `discovery.uri` to the list of coordinator URIs:

```java
// io.trino.node.AnnounceNodeAnnouncerConfig
public class AnnounceNodeAnnouncerConfig {
    private List<URI> coordinatorUris = List.of();

    @Config("discovery.uri")
    public AnnounceNodeAnnouncerConfig setCoordinatorUris(List<URI> coordinatorUris) { ... }
}
```

**Coordinator side -- `AnnounceNodeInventory` and `AnnounceNodeResource`:**

```java
// io.trino.node.AnnounceNodeInventory
public class AnnounceNodeInventory implements NodeInventory {
    // Time-expiring set: entries expire 30 seconds after last write
    private final Set<URI> nodes = newSetFromMap(SafeCaches.<URI, Boolean>buildNonEvictableCache(
            CacheBuilder.newBuilder().expireAfterWrite(30, SECONDS)).asMap());

    public void announce(URI node) {
        nodes.add(node);  // refreshes the 30-second TTL
    }

    @Override
    public Set<URI> getNodes() {
        return ImmutableSet.copyOf(nodes);
    }
}
```

```java
// io.trino.node.AnnounceNodeResource
@Path("/v1/announce")
public class AnnounceNodeResource {
    @ResourceSecurity(INTERNAL_ONLY)
    @POST
    @Consumes(TEXT_PLAIN)
    public void announce(String uri) {
        announceNodeInventory.announce(URI.create(uri));
    }
}
```

**Critical timing:** The worker announces every 5 seconds. The inventory entry expires after 30 seconds of no announcements. This means a worker can miss up to 5 consecutive announcements before being evicted from the inventory. Combined with the coordinator's 5-second polling interval for node state, a crashed worker can take up to 35 seconds to disappear from the active set.

#### 2.2.2 AIRLIFT_DISCOVERY Mode (Legacy)

In `AIRLIFT_DISCOVERY` mode, Trino uses Airlift's built-in discovery framework. Workers announce as a "trino" service through the Airlift `Announcer`, and the coordinator queries a `ServiceSelector` for all "trino" services.

```java
// io.trino.node.AirliftNodeInventoryModule -- both coordinator and worker
install(new DiscoveryModule());
binder.bind(Announcer.class).to(AirliftAnnouncer.class).in(Scopes.SINGLETON);
discoveryBinder(binder).bindHttpAnnouncement("trino")
        .addProperty("node_version", nodeVersion)
        .addProperty("coordinator", String.valueOf(coordinator));
```

On the coordinator side only:

```java
if (coordinator) {
    if (buildConfigObject(EmbeddedDiscoveryConfig.class).isEnabled()) {
        install(new EmbeddedDiscoveryModule());  // embedded discovery server
    }
    binder.bind(NodeInventory.class).to(AirliftNodeInventory.class).in(Scopes.SINGLETON);
    discoveryBinder(binder).bindSelector("trino");       // query for all "trino" services
    discoveryBinder(binder).bindHttpAnnouncement("trino-coordinator");  // also announce as coordinator
    install(new FailureDetectorModule());
}
```

The `AirliftNodeInventory` reads from the service selector and filters out failed nodes:

```java
// io.trino.node.AirliftNodeInventory
public class AirliftNodeInventory implements NodeInventory {
    @Override
    public Set<URI> getNodes() {
        Set<ServiceDescriptor> failed = failureDetector.getFailed();
        return serviceSelector.selectAllServices().stream()
                .filter(not(failed::contains))
                .map(this::getHttpUri)
                .filter(Objects::nonNull)
                .collect(toImmutableSet());
    }
}
```

**Properties advertised through Airlift discovery:**

| Property | Value | Source |
|----------|-------|--------|
| `http` or `https` | Worker HTTP(S) URI | Airlift HTTP announcement |
| `node_version` | e.g., "480" | Explicitly added via `addProperty` |
| `coordinator` | "true" or "false" | Explicitly added via `addProperty` |
| Service type | "trino" | Announcement binding |

#### 2.2.3 DNS Mode

DNS mode is designed for Kubernetes / headless service environments. The coordinator resolves DNS hostnames and constructs URIs from the results. Workers do not actively announce.

```java
// io.trino.node.DnsNodeInventory
public class DnsNodeInventory implements NodeInventory {
    @Override
    public Set<URI> getNodes() {
        return hosts.stream()
                .map(hostResolver::getAllByName)   // DNS resolve
                .flatMap(Set::stream)
                .map(address -> new URI(https ? "https" : "http", null,
                        InetAddresses.toUriString(address), port, null, null, null))
                .filter(Objects::nonNull)
                .collect(toImmutableSet());
    }
}
```

```java
// io.trino.node.DnsNodeInventoryConfig
@Config("cluster.hosts")
@ConfigDescription("List of host names to resolve for Trino nodes")
public DnsNodeInventoryConfig setHosts(Set<String> hosts) { ... }
```

In DNS mode, the `Announcer` is a no-op:

```java

```

### 2.3 Worker Identity: What the Worker Advertises

When the coordinator polls a worker's `/v1/info` endpoint, it receives a `ServerInfo` record:

```java
// io.trino.server.ServerInfo
public record ServerInfo(
        String nodeId,            // from node.id property (via Airlift NodeInfo)
        NodeState state,          // ACTIVE, DRAINING, DRAINED, SHUTTING_DOWN
        NodeVersion nodeVersion,  // e.g., "480"
        String environment,       // from node.environment property
        boolean coordinator,      // from coordinator= property
        Optional<String> coordinatorId,  // coordinator's ID, if this is a coordinator
        boolean starting,         // true until StartupStatus.startupComplete()
        Duration uptime)          // time since server start
{
}
```

The `/v1/info` endpoint is served by `ServerInfoResource`:

```java
// io.trino.server.ServerInfoResource
@Path("/v1/info")
public class ServerInfoResource {
    @ResourceSecurity(PUBLIC)
    @GET
    @Produces(APPLICATION_JSON)
    public ServerInfo getInfo() {
        boolean starting = !startupStatus.isStartupComplete();
        return new ServerInfo(
                nodeId,
                nodeStateManager.getServerState(),
                version,
                environment,
                coordinator,
                queryIdGenerator.map(QueryIdGenerator::getCoordinatorId),
                starting,
                nanosSince(startTime));
    }
}
```

The `InternalNode` object constructed from this info on the coordinator side contains:

```java
// io.trino.node.InternalNode
public class InternalNode implements Node {
    private final String nodeIdentifier;  // node.id
    private final URI internalUri;        // http(s)://host:port
    private final NodeVersion nodeVersion;
    private final boolean coordinator;
    private final long longHashCode;      // XxHash64 for consistent hashing
}
```

The `InternalNode` is constructed in `ServerMainModule` from Airlift's `NodeInfo`:

```java
// io.trino.server.ServerMainModule
@Provides @Singleton
public static InternalNode currentInternalNode(
        NodeInfo nodeInfo, HttpServerInfo httpServerInfo,
        NodeVersion nodeVersion, ServerConfig serverConfig,
        InternalCommunicationConfig internalCommunicationConfig) {
    return new InternalNode(
            nodeInfo.getNodeId(),
            internalCommunicationConfig.isHttpsRequired()
                ? httpServerInfo.getHttpsUri() : httpServerInfo.getHttpUri(),
            nodeVersion,
            serverConfig.isCoordinator());
}
```

There is also a `/v1/status` endpoint (`StatusResource`) that returns a richer `NodeStatus` with memory, CPU, and heap information. This is used by the failure detector for heartbeating (see Section 2.5) and by the UI, but is **not** used by the node state poller.

### 2.4 Coordinator Node State Polling: CoordinatorNodeManager and RemoteNodeState

The `CoordinatorNodeManager` is the coordinator's central node registry. It polls every 5 seconds.

```java
// io.trino.node.CoordinatorNodeManager
@PostConstruct
public void startPollingNodeStates() {
    nodeStateUpdateExecutor.scheduleWithFixedDelay(() -> {
        try { refreshNodes(false); }
        catch (Exception e) { log.error(e, "Error polling state of nodes"); }
    }, 5, 5, TimeUnit.SECONDS);
    refreshNodes(false);
}
```

The `refreshNodes` method performs three phases:

**Phase 1 -- Add new nodes from inventory:**

```java
for (URI uri : nodeInventory.getNodes()) {
    if (uri.equals(currentNode.getInternalUri())) continue;  // skip self
    RemoteNodeState remoteNodeState = nodeStates.computeIfAbsent(uri,
        _ -> new RemoteNodeState(uri, expectedNodeEnvironment,
            currentNode.getNodeVersion(), httpClient, ticker));
    remoteNodeState.setSeen();  // refresh the "last seen" timestamp
}
```

**Phase 2 -- Remove nodes missing from inventory for 30+ seconds:**

```java
for (var entry : nodeStates.entrySet()) {
    RemoteNodeState remoteNodeState = entry.getValue();
    if (remoteNodeState.isMissing()) {  // 30 seconds since last setSeen()
        if (remoteNodeState.hasBeenActive() &&
            remoteNodeState.getState() != NodeState.SHUTTING_DOWN) {
            log.info("Previously active node is missing: %s", entry.getKey());
        }
        nodeStates.remove(entry.getKey());
    }
}
```

**Phase 3 -- Async refresh of each node's state via HTTP:**

```java
ListenableFuture<List<Boolean>> future = Futures.allAsList(
    nodeStates.values().stream()
        .map(remoteNodeState -> remoteNodeState.asyncRefresh(forceAndWait))
        .toList());
```

The `RemoteNodeState.asyncRefresh()` makes an HTTP GET to `/v1/info` and processes the response:

```java
// io.trino.node.RemoteNodeState
public synchronized ListenableFuture<Boolean> asyncRefresh(boolean forceAndWait) {
    long millisSinceUpdate = millisSince(lastUpdateNanos.get());
    // Throttle: don't refresh more than once per second
    if (!forceAndWait && (millisSinceUpdate < 1_000 || future.get() != null)) {
        return requireNonNullElseGet(future.get(), () -> Futures.immediateFuture(true));
    }

    Request request = prepareGet()
            .setUri(infoUri)                          // {workerUri}/v1/info
            .setHeader(CONTENT_TYPE, JSON_UTF_8.toString())
            .build();
    HttpResponseFuture<JsonResponse<ServerInfo>> responseFuture =
            httpClient.executeAsync(request, createFullJsonResponseHandler(SERVER_INFO_CODEC));

    ListenableFuture<Boolean> transformSuccess = Futures.transform(responseFuture, result -> {
        NodeState nodeState;
        if (result == null || !result.hasValue()) {
            nodeState = INACTIVE;
        } else {
            ServerInfo serverInfo = result.getValue();
            // Validate environment
            if (!serverInfo.environment().equals(expectedNodeEnvironment)) {
                nodeState = INVALID;
            }
            // Validate version
            else if (!serverInfo.nodeVersion().equals(expectedNodeVersion)) {
                nodeState = INVALID;
            } else {
                nodeState = serverInfo.state();  // use the worker's self-reported state
            }
            internalNode.set(new InternalNode(
                    serverInfo.nodeId(), serverUri,
                    serverInfo.nodeVersion(), serverInfo.coordinator()));
        }
        RemoteNodeState.this.nodeState.set(nodeState);
        if (nodeState == ACTIVE) { hasBeenActive.set(true); }
        return nodeState == ACTIVE;
    }, directExecutor());

    // On connection errors, mark node as GONE (ConnectException) or INACTIVE (other)
    ListenableFuture<Boolean> catching = Futures.catching(transformSuccess, Throwable.class, t -> {
        nodeState.set(t instanceof ConnectException ? GONE : INACTIVE);
        return false;
    }, directExecutor());
    // ...
}
```

**Phase 4 -- Rebuild the AllNodes snapshot:**

```java
// io.trino.node.CoordinatorNodeManager.refreshNodesInternal()
private synchronized void refreshNodesInternal() {
    ImmutableSet.Builder<InternalNode> activeNodesBuilder = ImmutableSet.builder();
    ImmutableSet.Builder<InternalNode> inactiveNodesBuilder = ImmutableSet.builder();
    ImmutableSet.Builder<InternalNode> drainingNodesBuilder = ImmutableSet.builder();
    // ... also drainedNodes, shuttingDownNodes, invalidNodes, goneNodes

    // Classify current node first
    switch (currentNodeState.get()) {
        case ACTIVE -> activeNodesBuilder.add(currentNode);
        case INACTIVE, INVALID, GONE -> inactiveNodesBuilder.add(currentNode);
        case DRAINING -> drainingNodesBuilder.add(currentNode);
        // ...
    }

    // Classify each remote node
    for (RemoteNodeState remoteNodeState : nodeStates.values()) {
        InternalNode node = remoteNodeState.getInternalNode().orElse(null);
        if (node == null) continue;
        switch (remoteNodeState.getState()) {
            case ACTIVE -> activeNodesBuilder.add(node);
            case INACTIVE -> inactiveNodesBuilder.add(node);
            case DRAINING -> drainingNodesBuilder.add(node);
            // ...
        }
    }

    Set<InternalNode> coordinators = activeNodes.stream()
            .filter(InternalNode::isCoordinator).collect(toImmutableSet());

    AllNodes allNodes = new AllNodes(activeNodes, inactiveNodes, drainingNodes,
            drainedNodes, shuttingDownNodes, coordinators);

    if (!allNodes.equals(this.allNodes)) {
        this.allNodes = allNodes;
        // Notify listeners asynchronously
        List<Consumer<AllNodes>> listeners = ImmutableList.copyOf(this.listeners);
        nodeStateEventExecutor.submit(() ->
            listeners.forEach(listener -> listener.accept(allNodes)));
    }
}
```

The `AllNodes` record:

```java
// io.trino.node.AllNodes
public record AllNodes(
        Set<InternalNode> activeNodes,
        Set<InternalNode> inactiveNodes,
        Set<InternalNode> drainingNodes,
        Set<InternalNode> drainedNodes,
        Set<InternalNode> shuttingDownNodes,
        Set<InternalNode> activeCoordinators)
{
}
```

### 2.5 The HeartbeatFailureDetector (Airlift Discovery Mode Only)

The `HeartbeatFailureDetector` is a secondary health monitoring layer that only operates in `AIRLIFT_DISCOVERY` mode. It is installed by `FailureDetectorModule` which is only loaded from `AirliftNodeInventoryModule`. In `ANNOUNCE` mode and `DNS` mode, the `WorkerModule` binds `NoOpFailureDetector`.

```java
// io.trino.server.WorkerModule
binder.bind(FailureDetector.class).to(NoOpFailureDetector.class).in(Scopes.SINGLETON);
```

The failure detector uses high-frequency HTTP HEAD requests to `/v1/status`:

```java
// io.trino.failuredetector.HeartbeatFailureDetector
@PostConstruct
public void start() {
    if (isEnabled && started.compareAndSet(false, true)) {
        executor.scheduleWithFixedDelay(() -> updateMonitoredServices(),
            0, 5, TimeUnit.SECONDS);
    }
}
```

For each discovered service, it creates a `MonitoringTask` that pings at the configured heartbeat interval (default 500ms):

```java
private class MonitoringTask {
    public synchronized void enable() {
        if (future == null) {
            future = executor.scheduleAtFixedRate(() -> {
                ping();         // HTTP HEAD to /v1/status
                updateState();  // check failure ratio
            }, heartbeat.toMillis(), heartbeat.toMillis(), TimeUnit.MILLISECONDS);
        }
    }

    public synchronized boolean isFailed() {
        return future == null ||                    // disabled
               successTransitionTimestamp == null || // no successes yet
               Duration.nanosSince(successTransitionTimestamp).compareTo(warmupInterval) < 0;
    }

    private synchronized void updateState() {
        if (stats.getRecentFailureRatio() > failureRatioThreshold) {
            successTransitionTimestamp = null;  // mark as failed
        } else if (successTransitionTimestamp == null) {
            successTransitionTimestamp = System.nanoTime();  // start warmup
        }
    }
}
```

**Failure detector configuration:**

| Property | Default | Description |
|----------|---------|-------------|
| `failure-detector.enabled` | `true` | Enable/disable |
| `failure-detector.heartbeat-interval` | `500ms` | Ping frequency |
| `failure-detector.threshold` | `0.1` | Failure ratio threshold (10%) |
| `failure-detector.warmup-interval` | `5s` | Wait after transition to success |
| `failure-detector.expiration-grace-interval` | `10min` | How long to remember after disappearance |

### 2.6 Node State and Lifecycle: NodeStateManager

Each node (coordinator or worker) manages its own lifecycle state through `NodeStateManager`. State transitions are exposed via the `/v1/info/state` PUT endpoint:

```
Valid transitions:
  ACTIVE -> SHUTTING_DOWN (irreversible, triggers graceful shutdown)
  ACTIVE -> DRAINING (reversible, stops new task assignment)
  DRAINING -> ACTIVE (re-activation)
  DRAINING -> DRAINED (all tasks finished)
  DRAINING -> SHUTTING_DOWN
  DRAINED -> ACTIVE (re-activation)
  DRAINED -> SHUTTING_DOWN (immediate terminate)
```

The `CurrentNodeState` is stored as a versioned atomic reference:

```java
// io.trino.server.NodeStateManager.CurrentNodeState
public static class CurrentNodeState implements Supplier<NodeState> {
    private final AtomicReference<VersionedState> nodeState =
        new AtomicReference<>(new VersionedState(ACTIVE, 0));

    class VersionedState {
        private final NodeState state;
        private final long version;
    }
}
```

**Graceful shutdown sequence:**
1. State transitions to `SHUTTING_DOWN`
2. Wait `shutdown.grace-period` (default 2 min) for coordinator to observe the state
3. Wait for all active tasks to finish
4. Wait another grace period for task states to be observed
5. Call `lifeCycleManager.stop()` followed by `shutdownAction.onShutdown()`

### 2.7 NodeState Enum: The Full State Space

```java
// io.trino.node.NodeState
public enum NodeState {
    ACTIVE,        // Ready to handle tasks
    INACTIVE,      // Communication errors (internal state, not user-settable)
    DRAINING,      // Reversible graceful shutdown, no new tasks assigned
    DRAINED,       // All tasks finished, safe to stop
    SHUTTING_DOWN, // Non-reversible graceful shutdown
    INVALID,       // Wrong environment or version (internal state)
    GONE,          // Connection refused (internal state)
}
```

### 2.9 Worker Failure Detection: Timing Analysis

The coordinator detects worker failure through multiple mechanisms with different latencies:

**In ANNOUNCE mode (default):**

| Phase | Mechanism | Timeout/Interval |
|-------|-----------|-----------------|
| 1. Announcement stops | Worker stops POSTing to `/v1/announce` | Worker announces every 5s |
| 2. Inventory expiry | `AnnounceNodeInventory` entry expires | 30s after last announcement |
| 3. RemoteNodeState missing | `isMissing()` returns true (30s since last `setSeen()`) | 30s after inventory expiry |
| 4. Node eviction | `CoordinatorNodeManager.refreshNodes()` removes the node | Next 5s poll cycle |
| **Total worst case** | | **~65 seconds** |

However, the `/v1/info` poll in step 3 also runs in parallel. If the worker is down:

| Phase | Mechanism | Timeout/Interval |
|-------|-----------|-----------------|
| 1. Info poll fails | `RemoteNodeState.asyncRefresh()` gets ConnectException | 10s request timeout |
| 2. Node marked GONE | `nodeState.set(GONE)` | Immediate on failure |
| 3. Node enters GONE set | `refreshNodesInternal()` classifies it | Next 5s poll cycle |
| **Total typical case** | | **~15 seconds** |

The `GONE` vs `INACTIVE` distinction:
- `ConnectException` (connection refused) -> `GONE` (node is definitely down)
- Other exceptions (timeout, etc.) -> `INACTIVE` (node may be temporarily unreachable)

**In AIRLIFT_DISCOVERY mode:**

The `HeartbeatFailureDetector` provides faster detection:
- Heartbeat interval: 500ms
- Failure ratio threshold: 0.1 (10%)
- ~6 seconds of failures needed to cross threshold
- Failed nodes are excluded from `AirliftNodeInventory.getNodes()`

### 2.10 Server Bootstrap Sequence

The `Server.start()` method in `Server.java` orchestrates the startup:

```java
// io.trino.server.Server.doStart()
Injector injector = app.initialize();
// ... load plugins, catalogs, access control ...

// Start announcing AFTER everything is initialized
injector.getInstance(Announcer.class).start();

// Mark startup as complete (affects /v1/info "starting" field)
injector.getInstance(StartupStatus.class).startupComplete();
log.info("======== SERVER STARTED ========");
```

The `Announcer.start()` is called as one of the last steps, ensuring the worker does not appear in the cluster until it is fully initialized.

## 3. Key Design Insights

1. **Two-layer discovery is the fundamental pattern.** Layer 1 (inventory) answers "what URIs exist?" and Layer 2 (state verification) answers "is this URI healthy and compatible?" Separating these concerns allows the inventory mechanism to be swapped (Airlift discovery, announce, DNS) without changing the health-checking logic.

2. **The coordinator is the single source of truth for cluster membership.** Workers have zero knowledge of the cluster topology. The `WorkerInternalNodeManager` is a stub that throws on every operation. This is intentional -- workers only need to know where to announce and how to serve their `/v1/info` endpoint.

3. **Environment and version validation is a hard firewall.** If a worker's environment or version does not match the coordinator, it is marked `INVALID` and is completely invisible to the scheduler. This prevents mixed-version clusters from silently producing incorrect results.

4. **The announce-mode 30-second TTL cache is the main clock.** In the default `ANNOUNCE` mode, the `CacheBuilder.expireAfterWrite(30, SECONDS)` on `AnnounceNodeInventory` is the primary mechanism for evicting dead workers from the inventory. The worker's 5-second announcement interval gives it 6 chances to refresh before expiry.

5. **Node state is self-reported by the worker.** The coordinator trusts the worker's `/v1/info` response for its lifecycle state (ACTIVE, DRAINING, etc.). The coordinator only overrides this when it detects unreachability (marking the node INACTIVE or GONE). This means a misbehaving worker could theoretically lie about its state, but the failure detector provides an independent health signal.

6. **AllNodes is rebuilt on every poll cycle.** The `CoordinatorNodeManager.refreshNodesInternal()` creates a brand new immutable `AllNodes` record on every 5-second cycle. Change detection uses `equals()` to avoid spurious listener notifications. This is a classic pattern for safely sharing mutable state -- the shared reference is always an immutable snapshot.

7. **The HeartbeatFailureDetector is only active in legacy mode.** In the default `ANNOUNCE` mode, the failure detector is a `NoOpFailureDetector` on workers, and the coordinator-side failure detector is not installed at all. Only the `/v1/info` polling provides health detection. This simplifies the architecture but slightly increases detection latency.

8. **Graceful shutdown uses a "wait then drain" pattern.** When `SHUTTING_DOWN` is requested, the node first waits `shutdown.grace-period` (2 min) for the coordinator to notice, then waits for all tasks to complete, then waits another grace period for task states to propagate, then shuts down. The double grace period ensures no work is lost.

## 4. Porting Considerations (Java -> Rust)

### 4.1 The Announcer Pattern

In `ANNOUNCE` mode, the worker simply POSTs its own URI as plain text to the coordinator every 5 seconds. This is trivially implementable with any async HTTP client (e.g., `reqwest`). The key timing parameters are:
- **Announce interval:** 5 seconds (configurable via a simple `tokio::time::interval`)
- **Coordinator URIs:** From config, the list of coordinator URIs to POST to
- **Body:** The worker's own `http://host:port` URI as UTF-8 text

A Rust worker should send its announcement to all configured coordinator URIs in parallel (as the Java code does with `Futures.allAsList`).

### 4.2 The /v1/info Endpoint

The worker must serve a JSON endpoint at `/v1/info` returning a `ServerInfo`-equivalent struct. The fields are:
- `nodeId`: String (from config or auto-generated)
- `state`: Enum (`ACTIVE`, `DRAINING`, `DRAINED`, `SHUTTING_DOWN`)
- `nodeVersion`: Object with `version` string
- `environment`: String (from config)
- `coordinator`: Boolean
- `coordinatorId`: Optional string
- `starting`: Boolean
- `uptime`: Duration

This is a straightforward Axum/Actix handler with serde serialization.

### 4.3 Node State Machine

The `NodeStateManager` is a moderately complex state machine. For Rust:
- Use an `AtomicU64` for the versioned state (pack state enum + version into a single u64 for lock-free CAS)
- The drain/shutdown logic spawns background tasks that wait for active work to complete -- this maps to `tokio::spawn` with `tokio::sync::watch` or `tokio::sync::Notify` for task completion signaling
- The double grace period pattern (wait for coordinator to notice, wait for tasks, wait for propagation) should be preserved exactly

### 4.4 Coordinator-Side: Node Manager

If the Rust project also implements the coordinator, the `CoordinatorNodeManager` is the most complex piece:
- A `ConcurrentHashMap<URI, RemoteNodeState>` -> `DashMap<Uri, RemoteNodeState>` or `tokio::sync::RwLock<HashMap<...>>`
- Each `RemoteNodeState` makes async HTTP GET requests to `/v1/info` -- these should be concurrent `tokio::spawn` tasks
- The 30-second "missing" threshold on `RemoteNodeState.isMissing()` needs a monotonic clock (`tokio::time::Instant`)
- The `AllNodes` snapshot pattern maps to `Arc<AllNodes>` with an `AtomicArc` swap pattern or `tokio::sync::watch` for listener notification

### 4.5 Inventory Backend Selection

For initial Rust implementation, start with `ANNOUNCE` mode as it is the simplest and is now the default. The `AnnounceNodeInventory` is just a time-expiring set -- implementable with `moka::sync::Cache` or a custom `HashMap` with timestamps.

DNS mode is also simple -- just call `tokio::net::lookup_host()` on the configured hostnames.

`AIRLIFT_DISCOVERY` mode requires implementing a discovery protocol and is not recommended for a Rust port.

### 4.6 /v1/status Endpoint (Secondary)

The `/v1/status` endpoint returns detailed node metrics:

```
NodeStatus { nodeId, nodeVersion, environment, coordinator, uptime,
             externalAddress, internalAddress, memoryInfo,
             processors, processCpuLoad, systemCpuLoad,
             heapUsed, heapAvailable, nonHeapUsed }
```

For Rust, the JVM-specific metrics (heap, GC) are irrelevant. Replace with:
- RSS / VmSize from `/proc/self/status` (Linux) or `mach_task_basic_info` (macOS)
- Allocator stats from `jemalloc`/`mimalloc` if used
- CPU load from `sysinfo` crate

### 4.7 Critical Timing Parameters to Preserve

| Parameter | Java Value | Rust Equivalent |
|-----------|-----------|-----------------|
| Announce interval | 5s | `tokio::time::interval(Duration::from_secs(5))` |
| Inventory TTL | 30s | Cache TTL or manual expiry |
| Node state poll interval | 5s | Scheduler task interval |
| Info poll throttle | 1s minimum between polls | Debounce on per-node state |
| Missing threshold | 30s since last `setSeen()` | `Instant::elapsed() > 30s` |
| Info request timeout | 10s | `reqwest` timeout |
| Heartbeat interval (failure detector) | 500ms | Only if implementing Airlift mode |
| Shutdown grace period | 2min | Configurable duration |

### 4.8 Compatibility Note

A Rust worker joining a Java coordinator cluster only needs to:
1. POST its URI to the coordinator's `/v1/announce` every 5 seconds
2. Serve `/v1/info` returning a JSON `ServerInfo` with matching `environment` and `nodeVersion`
3. Serve `/v1/status` for the failure detector (if Airlift discovery mode)
4. Accept task assignments on `/v1/task` (covered in other research tasks)

The announce protocol is plain text over HTTP. The `/v1/info` response is simple JSON. There are no binary protocols, no custom framing, and no bidirectional connections. This makes the discovery layer one of the simplest components to port.
