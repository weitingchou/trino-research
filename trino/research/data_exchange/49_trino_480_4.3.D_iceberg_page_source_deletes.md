## 0. Research Focus

This teardown traces how Trino 480's Iceberg connector constructs page sources from splits and applies Merge-on-Read (MOR) delete files. The five focus areas are:

1. How `IcebergPageSourceProvider.createPageSource()` constructs the page source from an `IcebergSplit`, including format reader selection
2. How the page source wraps the underlying format-specific reader (Parquet/ORC/Avro)
3. Delete file handling: positional deletes, equality deletes, deletion vectors, and where filtering occurs in the pipeline
4. Schema evolution: column ID-based mapping and projection/coercion for older-schema data files
5. How `IcebergSplit.deleteFiles` flows into page source construction

**Key source files:**
- `IcebergPageSourceProvider.java` (~1925 lines) -- the central orchestrator
- `IcebergSplit.java` -- the split carrying delete file references
- `delete/DeleteManager.java` -- coordinates positional + equality delete loading
- `delete/PositionDeleteReader.java` -- reads position delete files into DeletionVectors
- `delete/EqualityDeleteFilter.java` -- builds hash-set-based equality delete predicates
- `delete/DeletionVector.java` -- RoaringBitmap-based deleted-row tracker
- `delete/RowPredicate.java` -- the unified row-level filter interface
- `delete/DeleteFile.java` -- delete file metadata record
- `TransformConnectorPageSource.java` (in trino-hive) -- column projection/transform layer
- `IcebergParquetColumnIOConverter.java` -- Parquet field ID-based column resolution
- `util/OrcIcebergIds.java` -- ORC field ID mapping

## 1. High-Level Overview

The Iceberg page source pipeline has a layered architecture with three stages:

```
IcebergSplit (with deleteFiles list)
        |
        v
IcebergPageSourceProvider.createPageSource()
        |
        |-- 1. Predicate evaluation (can skip entire split)
        |-- 2. Required columns expansion (adds columns needed by delete filters)
        |-- 3. createDataPageSource() -- format dispatch (Parquet/ORC/Avro)
        |      |
        |      |-- Format-specific reader (ParquetPageSource / OrcPageSource / IcebergAvroPageSource)
        |      |-- Wrapped by TransformConnectorPageSource for column projection
        |      |-- Returns ReaderPageSourceWithRowPositions (page source + row position range)
        |
        |-- 4. Delete filter wrapping (if deletes present)
        |      |
        |      |-- DeleteManager.getDeletePredicate()
        |      |      |-- Position deletes -> DeletionVector
        |      |      |-- Equality deletes -> EqualityDeleteFilter -> RowPredicate
        |      |      |-- Combined into single RowPredicate (AND composition)
        |      |
        |      |-- TransformConnectorPageSource wrapping that applies RowPredicate.applyFilter()
        |      |-- PrefixColumnsSourcePage to strip extra delete-only columns
        |
        v
  ConnectorPageSource (final, ready for the engine)
```

The key architectural insight: **deletes are applied AFTER the format reader produces pages, as a post-filter on the SourcePage**. The format reader produces full pages, then the delete predicate filters out deleted rows by calling `selectPositions()` on the SourcePage. This is a push-down-free approach -- the format reader is unaware of deletes.

## 2. Detailed Analysis

### 2.1 IcebergSplit and Delete File Flow

`IcebergSplit` carries all the information needed to read a single data file chunk, including its associated delete files:

```java
// IcebergSplit.java (key fields)
public class IcebergSplit implements ConnectorSplit {
    private final String path;
    private final long start;
    private final long length;
    private final long fileSize;
    private final long fileRecordCount;
    private final IcebergFileFormat fileFormat;
    private final List<DeleteFile> deletes;              // <-- delete files for this data file
    private final long dataSequenceNumber;               // <-- for equality delete ordering
    private final OptionalLong fileFirstRowId;            // <-- for row ID computation
    private final TupleDomain<IcebergColumnHandle> fileStatisticsDomain;
    ...
}
```

The `DeleteFile` record wraps Iceberg's native `org.apache.iceberg.DeleteFile` and extracts key metadata:

```java
// delete/DeleteFile.java
public record DeleteFile(
        FileContent content,        // POSITION_DELETES or EQUALITY_DELETES
        String path,
        FileFormat format,          // PARQUET, ORC, AVRO, or PUFFIN (for deletion vectors)
        long recordCount,
        long fileSizeInBytes,
        List<Integer> equalityFieldIds,       // field IDs for equality deletes
        OptionalLong rowPositionLowerBound,   // min row position in delete file
        OptionalLong rowPositionUpperBound,   // max row position in delete file
        long dataSequenceNumber,              // sequence number for ordering
        OptionalLong contentOffset,           // offset within Puffin file (deletion vectors)
        Optional<Integer> contentSizeInBytes) // size within Puffin file (deletion vectors)
{
    // A deletion vector is a special position delete in PUFFIN format with offset+size
    public boolean isDeletionVector() {
        return content == FileContent.POSITION_DELETES
                && format == FileFormat.PUFFIN
                && contentOffset.isPresent()
                && contentSizeInBytes.isPresent();
    }
}
```

The split-level `deletes` list is populated during split generation in `IcebergSplitSource` and flows through JSON serialization to the worker, where `IcebergPageSourceProvider.createPageSource()` receives it.

### 2.2 IcebergPageSourceProvider.createPageSource()

The main entry point receives the raw `ConnectorSplit` and performs the following sequence:

**Step 1: Schema and partition setup**
```java
IcebergSplit split = (IcebergSplit) connectorSplit;
Schema schema = SchemaParser.fromJson(tableHandle.getTableSchemaJson());
PartitionSpec partitionSpec = PartitionSpecParser.fromJson(schema, partitionSpecJson);
```

**Step 2: Effective predicate evaluation (early skip)**
```java
TupleDomain<IcebergColumnHandle> effectivePredicate = getUnenforcedPredicate(
        tableSchema, partitionKeys, dynamicFilter, unenforcedPredicate, fileStatisticsDomain);
if (effectivePredicate.isNone()) {
    return new EmptyPageSource();  // Skip entire split
}
```

The effective predicate intersects:
- The unenforced predicate (from the table handle)
- The file statistics domain (from the split, for min/max pruning)
- The current dynamic filter predicate

**Step 3: Partition-only fast path**

If the query only reads partition columns and there are no deletes, the provider skips opening the data file entirely and generates RLE (run-length-encoded) pages from partition values:

```java
if (effectivePredicate.isAll() && start == 0 && length == inputFile.length()
        && deletes.isEmpty()
        && icebergColumns.stream().allMatch(column -> partitionKeys.containsKey(column.getId()))) {
    return generatePages(fileRecordCount, icebergColumns, partitionKeys);
}
```

**Step 4: Required columns expansion for deletes**

Before creating the data page source, the provider determines what extra columns the delete filters need:

```java
Set<IcebergColumnHandle> deleteFilterRequiredColumns = requiredColumnsForDeletes(tableSchema, deletes);
deleteFilterRequiredColumns.stream()
        .filter(not(icebergColumns::contains))
        .forEach(requiredColumns::add);
```

For position deletes, `ROW_POSITION` is added. For equality deletes, the equality field ID columns are added. These extra columns will be present in the page produced by the format reader but stripped before returning to the engine.

**Step 5: Create the format-specific data page source**

```java
ReaderPageSourceWithRowPositions readerPageSourceWithRowPositions = createDataPageSource(
        session, inputFile, start, length, fileSize, ...);
```

This returns both the page source AND the row position range (start/end) for the split, which is critical for optimizing position delete loading.

**Step 6: Apply delete filtering**

```java
if (!deletes.isEmpty()) {
    Supplier<Optional<RowPredicate>> deletePredicate = memoize(() ->
        getDeleteManager(partitionSpec, partitionData).getDeletePredicate(
            path, dataSequenceNumber, deletes, requiredColumns, tableSchema,
            readerPageSourceWithRowPositions.startRowPosition(),
            readerPageSourceWithRowPositions.endRowPosition(),
            (deleteFile) -> readDeletionVector(fileSystem, deleteFile),
            (deleteFile, deleteColumns, tupleDomain) ->
                openDeleteFile(session, fileSystem, deleteFile, deleteColumns, tupleDomain)));

    pageSource = TransformConnectorPageSource.create(pageSource, page -> {
        Optional<RowPredicate> rowPredicate = deletePredicate.get();
        rowPredicate.ifPresent(predicate -> predicate.applyFilter(page));
        if (icebergColumns.size() == page.getChannelCount()) {
            return page;
        }
        return new PrefixColumnsSourcePage(page, icebergColumns.size());
    });
}
```

Critical details:
- The delete predicate is **lazily initialized** (via `memoize`). It is only computed on the first page.
- `applyFilter()` mutates the SourcePage in-place via `selectPositions()`.
- After filtering, if extra columns were added for deletes, `PrefixColumnsSourcePage` strips them by only exposing the first N channels (the original query columns).

### 2.3 Format Reader Selection (Parquet vs ORC vs Avro)

Format dispatch is a simple switch statement in `createDataPageSource()`:

```java
return switch (fileFormat) {
    case ORC -> createOrcPageSource(...);
    case PARQUET -> createParquetPageSource(...);
    case AVRO -> createAvroPageSource(...);
};
```

Each method returns `ReaderPageSourceWithRowPositions`, which bundles the page source with row position boundaries.

**Parquet path:** Creates a `ParquetReader`, wraps it in `ParquetPageSource`, then wraps that in a `TransformConnectorPageSource` for column projection. Row position range comes from the filtered row groups.

**ORC path:** Creates an `OrcRecordReader` via `reader.createRecordReader()`, wraps it in `OrcPageSource`, then wraps that in `TransformConnectorPageSource`. Row position range comes from `recordReader.getStartRowPosition()` / `getEndRowPosition()`.

**Avro path:** Creates an `IcebergAvroPageSource`, wraps it in `TransformConnectorPageSource`. Row positions are empty (not available for Avro).

### 2.4 TransformConnectorPageSource: The Column Projection Layer

The `TransformConnectorPageSource` is a critical abstraction that sits between the format reader and the rest of the system. It maps the "physical" columns (what the format reader produces) to the "logical" columns (what the query needs).

The builder pattern constructs a per-column transform list:

```java
TransformConnectorPageSource.Builder transforms = TransformConnectorPageSource.builder();
for (IcebergColumnHandle column : columns) {
    if (column.isIsDeletedColumn()) {
        transforms.constantValue(nativeValueToBlock(BOOLEAN, false));  // always false for data files
    } else if (partitionKeys.containsKey(column.getId())) {
        transforms.constantValue(...);  // partition value as constant
    } else if (column.isPathColumn()) {
        transforms.constantValue(...);  // file path as constant
    } else if (column.isRowPositionColumn()) {
        appendRowNumberColumn = true;
        transforms.transform(new GetRowPositionFromSource());  // last channel from reader
    } else if (!fileColumnExists) {
        // Column not in file -> initial default or null (schema evolution)
        transforms.constantValue(nativeValueToBlock(column.getType(), getInitialDefault(...)));
    } else if (column.isBaseColumn()) {
        transforms.column(ordinal);  // direct passthrough
    } else {
        // Nested field dereference (e.g., a.b.c)
        transforms.dereferenceField(ImmutableList.of(ordinal, ...dereference chain...));
    }
}
```

Transform types in `TransformConnectorPageSource`:
- **ConstantValue**: Returns an RLE block (e.g., partition values, file path)
- **InputColumn**: Direct passthrough of a source column by ordinal
- **DereferenceFieldTransform**: Extracts nested fields from a RowBlock via `getRowFieldsFromBlock()`
- **TransformBlock**: Applies an arbitrary `Block -> Block` function
- **Generic transform**: `SourcePage -> Block` for things like `GetRowPositionFromSource`

The `TransformSourcePage` lazily materializes blocks -- `getBlock(channel)` only invokes the transform on first access.

### 2.5 Delete File Handling (MOR)

#### 2.5.1 Delete Pipeline Architecture

The `DeleteManager` is the central coordinator. Trino maintains separate `DeleteManager` instances per partition (using `ConcurrentHashMap<PartitionKey, DeleteManager>`) and a single instance for unpartitioned tables. This is because equality delete filters are cached and shared across splits within the same partition.

```java
private DeleteManager getDeleteManager(PartitionSpec partitionSpec, PartitionData partitionData) {
    if (partitionSpec.isUnpartitioned()) {
        return unpartitionedTableDeleteManager;
    }
    PartitionKey partitionKey = partitionKeyFactories.computeIfAbsent(...)
            .apply(partitionData);
    return partitionedDeleteManagers.computeIfAbsent(partitionKey, _ -> new DeleteManager(typeManager));
}
```

`DeleteManager.getDeletePredicate()` classifies delete files and builds a combined predicate:

```java
public Optional<RowPredicate> getDeletePredicate(...) {
    // 1. Classify delete files
    Optional<DeleteFile> deletionVectorFile = Optional.empty();
    List<DeleteFile> positionDeleteFiles = new ArrayList<>();
    List<DeleteFile> equalityDeleteFiles = new ArrayList<>();
    for (DeleteFile deleteFile : deleteFiles) {
        switch (deleteFile.content()) {
            case POSITION_DELETES -> {
                if (deleteFile.isDeletionVector()) {
                    deletionVectorFile = Optional.of(deleteFile);
                } else {
                    positionDeleteFiles.add(deleteFile);
                }
            }
            case EQUALITY_DELETES -> equalityDeleteFiles.add(deleteFile);
        }
    }

    // 2. Build position delete predicate
    //    Deletion vectors take priority over position delete files (per Iceberg spec)
    Optional<DeletionVector> deletionVector = deletionVectorFile
            .map(deletionVectorReader::read)
            .or(() -> PositionDeleteReader.readPositionDeletes(...));

    Optional<RowPredicate> positionDeletes = deletionVector.map(vector -> {
        int filePositionChannel = /* find ROW_POSITION channel */;
        return (page, position) -> {
            long filePosition = BIGINT.getLong(page.getBlock(filePositionChannel), position);
            return !vector.isRowDeleted(filePosition);
        };
    });

    // 3. Build equality delete predicate
    Optional<RowPredicate> equalityDeletes = createEqualityDeleteFilter(...)
            .stream()
            .map(filter -> filter.createPredicate(readColumns, dataSequenceNumber))
            .reduce(RowPredicate::and);

    // 4. Combine both (AND)
    if (positionDeletes.isPresent() && equalityDeletes.isPresent()) {
        return Optional.of(positionDeletes.get().and(equalityDeletes.get()));
    }
    return positionDeletes.or(() -> equalityDeletes);
}
```

#### 2.5.2 Positional Delete Files

Position delete files contain `(file_path, pos)` pairs. `PositionDeleteReader` reads them into a `DeletionVector`:

```java
public static Optional<DeletionVector> readPositionDeletes(
        String dataFilePath,
        List<DeleteFile> positionDeleteFiles,
        OptionalLong startRowPosition,
        OptionalLong endRowPosition,
        DeletePageSourceProvider deletePageSourceProvider,
        TypeManager typeManager)
{
    IcebergColumnHandle deleteFilePath = getColumnHandle(DELETE_FILE_PATH, typeManager);
    IcebergColumnHandle deleteFilePos = getColumnHandle(DELETE_FILE_POS, typeManager);

    // Pushdown: filter to only rows for our data file path, within our row range
    TupleDomain<IcebergColumnHandle> deleteDomain =
            TupleDomain.fromFixedValues(ImmutableMap.of(deleteFilePath, NullableValue.of(VARCHAR, targetPath)));
    if (startRowPosition.isPresent()) {
        // Also filter by position range
        deleteDomain = deleteDomain.intersect(positionDomain);
    }

    DeletionVector.Builder deletionVector = DeletionVector.builder();
    for (DeleteFile deleteFile : positionDeleteFiles) {
        if (shouldLoadPositionDeleteFile(deleteFile, startRowPosition, endRowPosition)) {
            try (ConnectorPageSource pageSource = deletePageSourceProvider.openDeletes(deleteFile, deleteColumns, deleteDomain)) {
                readMultiFilePositionDeletes(pageSource, targetPath, deletionVector::add);
            }
        }
    }
    return deletionVector.build();
}
```

Key optimizations:
- **Predicate pushdown**: The delete domain pushes `file_path = targetPath` and `pos BETWEEN start AND end` into the delete file reader, allowing row group/stripe skipping.
- **Bound-based skipping**: `shouldLoadPositionDeleteFile()` checks if the delete file's `rowPositionLowerBound/UpperBound` overlaps with the split's row range. If not, the entire delete file is skipped.
- **Dictionary-aware path comparison**: `CachingVarcharComparator` caches the result of varchar comparison by object identity, exploiting the fact that dictionary-encoded blocks return the same Slice object for repeated values.
- **Early termination**: Because delete files are sorted by path, once the comparator finds a path greater than the target, reading stops immediately.

#### 2.5.3 Deletion Vectors (Puffin format)

Deletion vectors are a compact representation using `RoaringBitmap` arrays, stored in Puffin files. They are the preferred format (per Iceberg spec, "Readers can safely ignore position delete files if there is a DV for a data file").

```java
// DeletionVector.java - key structure
public final class DeletionVector {
    private final int bitmapCount;
    private final RoaringBitmap[] deletedRows;  // indexed by high 32 bits of file position

    public boolean isRowDeleted(long filePos) {
        int key = key(filePos);       // high 32 bits
        if (key >= deletedRows.length) return false;
        RoaringBitmap deletedRow = deletedRows[key];
        return deletedRow != null && deletedRow.contains(low(filePos));  // low 32 bits
    }

    private static int key(long pos) { return (int) (pos >>> 32); }
    private static int low(long pos) { return (int) pos; }
}
```

The deletion vector is read from the Puffin file using offset/size from the `DeleteFile` metadata:

```java
private static DeletionVector readDeletionVector(TrinoFileSystem fileSystem, DeleteFile delete) {
    TrinoInputFile trinoInputFile = fileSystem.newInputFile(Location.of(delete.path()), delete.fileSizeInBytes());
    try (TrinoInput trinoInput = trinoInputFile.newInput()) {
        Slice slice = trinoInput.readFully(delete.contentOffset().orElseThrow(),
                toIntExact(delete.contentSizeInBytes().orElseThrow()));
        return DeletionVector.builder().deserialize(slice).build().orElseThrow();
    }
}
```

The serialization format:
1. 4 bytes: body length (big-endian)
2. 4 bytes: magic number `0x64_39_d3_d1` (little-endian)
3. 8 bytes: bitmap count (little-endian)
4. For each bitmap: 4 bytes key (little-endian) + serialized RoaringBitmap
5. 4 bytes: CRC32 checksum (big-endian)

#### 2.5.4 Equality Delete Files

Equality delete files contain rows that, when matched against data rows on specific columns, indicate deletions. Trino implements this as an in-memory hash set of deleted row values.

The `EqualityDeleteFilter` has a two-phase lifecycle:

**Phase 1: Build (concurrent, deduplicated loading)**

```java
@ThreadSafe
public static class EqualityDeleteFilterBuilder {
    private final Schema deleteSchema;
    private final Map<StructLikeWrapper, DataSequenceNumber> deletedRows;  // ConcurrentHashMap
    private final Map<String, ListenableFutureTask<?>> loadingFiles = new ConcurrentHashMap<>();

    public ListenableFuture<?> readEqualityDeletes(DeleteFile deleteFile, ...) {
        // Ensure only one thread loads each file
        ListenableFutureTask<?> futureTask = loadingFiles.computeIfAbsent(
                deleteFile.path(),
                key -> ListenableFutureTask.create(() -> readEqualityDeletesInternal(...), null));
        futureTask.run();  // No-op if already running/completed
        return Futures.nonCancellationPropagating(futureTask);
    }
}
```

The `deletedRows` map uses `StructLikeWrapper` keys (which implement `equals`/`hashCode` based on field values) and `DataSequenceNumber` values. When the same row appears in multiple delete files, the highest sequence number wins.

**Phase 2: Predicate creation (per-split)**

```java
public RowPredicate createPredicate(List<IcebergColumnHandle> columns, long splitDataSequenceNumber) {
    StructLikeWrapper structLikeWrapper = StructLikeWrapper.forType(deleteStructType);
    StructProjection projection = StructProjection.create(fileStructType, deleteStructType);

    return (page, position) -> {
        StructProjection row = projection.wrap(new LazyTrinoRow(types, page, position));
        DataSequenceNumber maxDeleteVersion = deletedRows.get(structLikeWrapper.set(row));
        structLikeWrapper.set(null);  // clear reference to avoid memory leak
        return maxDeleteVersion == null || maxDeleteVersion.dataSequenceNumber() <= splitDataSequenceNumber;
    };
}
```

The `LazyTrinoRow` is used for the predicate evaluation path. Unlike `TrinoRow` (which eagerly materializes all values), `LazyTrinoRow` only materializes field values on demand -- important because the `StructProjection` may only access a subset of fields.

Key design: The equality delete check uses **data sequence numbers** for correctness. A delete entry only applies to a data file if the delete's sequence number is strictly greater than the data file's sequence number. This ensures that concurrent writes don't accidentally delete each other's data.

#### 2.5.5 RowPredicate: The Unified Filter Interface

```java
@ThreadSafe
public interface RowPredicate {
    boolean test(SourcePage page, int position);

    default RowPredicate and(RowPredicate other) {
        return (page, position) -> test(page, position) && other.test(page, position);
    }

    default void applyFilter(SourcePage page) {
        int positionCount = page.getPositionCount();
        int[] retained = new int[positionCount];
        int retainedCount = 0;
        for (int position = 0; position < positionCount; position++) {
            if (test(page, position)) {
                retained[retainedCount] = position;
                retainedCount++;
            }
        }
        if (retainedCount != positionCount) {
            page.selectPositions(retained, 0, retainedCount);
        }
    }
}
```

The `applyFilter()` method builds a position list of non-deleted rows and then calls `selectPositions()` on the `SourcePage`, which is an in-place mutation. For the `TransformSourcePage` implementation, `selectPositions()` filters both the underlying source page and any already-materialized blocks.

### 2.6 Schema Evolution and Column ID-Based Mapping

Iceberg uses column IDs (field IDs) rather than column names or positions to identify columns, enabling robust schema evolution. Trino leverages this in all three format paths.

#### 2.6.1 Parquet Column Mapping

For Parquet, the mapping process is:

1. **Build Iceberg-ID-to-Parquet-field map**: Walk the Parquet file schema, collecting `type.getId()` for each field:

```java
private static Map<Integer, org.apache.parquet.schema.Type> createParquetIdToFieldMapping(MessageType fileSchema) {
    ImmutableMap.Builder<Integer, org.apache.parquet.schema.Type> builder = ImmutableMap.builder();
    addParquetIdToFieldMapping(fileSchema, builder);  // recursive traversal
    return builder.buildOrThrow();
}
```

2. **Handle NameMapping fallback**: If the Parquet file lacks IDs (written by a non-Iceberg writer), apply the table's `NameMapping`:

```java
if (nameMapping.isPresent() && !ParquetSchemaUtil.hasIds(fileSchema)) {
    fileSchema = ParquetSchemaUtil.applyNameMapping(fileSchema, convertToLowercase(nameMapping.get()));
}
```

3. **Construct requested schema**: Build a Parquet `MessageType` from only the columns needed:

```java
MessageType requestedSchema = getMessageType(columnsForRead, fileSchema.getName(), parquetIdToFieldName);
```

4. **Build Field objects via IcebergParquetColumnIOConverter**: For each column, resolve the Parquet field by ID and construct the Trino `Field` hierarchy:

```java
Optional<Field> field = IcebergParquetColumnIOConverter.constructField(
        new FieldContext(baseColumn.getType(), baseColumn.getColumnIdentity()),
        messageColumnIO.getChild(parquetFieldName));
```

`IcebergParquetColumnIOConverter.constructField()` recursively maps RowType -> GroupField, ArrayType -> GroupField, MapType -> GroupField, and primitives -> PrimitiveField. For nested structs, it uses `lookupColumnById()` to match child columns by Iceberg field ID, which handles column reordering within structs.

#### 2.6.2 ORC Column Mapping

For ORC, the approach is similar but uses `OrcIcebergIds`:

```java
Map<Integer, OrcColumn> fileColumnsByIcebergId = fileColumnsByIcebergId(reader, nameMapping);
```

This builds a map from Iceberg field IDs to ORC column objects by reading the `iceberg.id` attribute from ORC column metadata. If IDs are missing and a NameMapping is available, IDs are injected via `setMissingFieldIds()`.

The `IdBasedFieldMapperFactory` and `IdBasedFieldMapper` classes handle nested column resolution:

```java
private static class IdBasedFieldMapper implements OrcReader.FieldMapper {
    private final Map<Integer, OrcColumn> idToColumnMappingForFile;
    private final Map<String, Integer> nameToIdMappingForTableColumns;

    @Override
    public OrcColumn get(String fieldName) {
        // Map field name -> Iceberg ID -> ORC column
        int fieldId = nameToIdMappingForTableColumns.get(fieldName);
        return idToColumnMappingForFile.get(fieldId);
    }
}
```

This two-step mapping (name -> ID -> file column) allows columns to be matched even when field names differ between the current schema and the file schema, as long as the Iceberg field IDs match.

#### 2.6.3 Initial Defaults for Missing Columns

When a column exists in the current schema but not in the data file (added after the file was written), Trino uses the column's initial default value:

```java
@Nullable
private static Object getInitialDefault(Schema schema, int columnId) {
    NestedField field = schema.findField(columnId);
    if (field == null) return null;
    var literal = field.initialDefaultLiteral();
    if (literal == null) return null;
    ByteBuffer buffer = literal.toByteBuffer();
    if (buffer == null) return null;
    Object icebergValue = Conversions.fromByteBuffer(field.type(), buffer);
    return convertIcebergValueToTrino(field.type(), icebergValue);
}
```

The default is emitted as a constant block via `TransformConnectorPageSource.Builder.constantValue()`.

### 2.7 Row Position and Row ID Handling

Row positions are fundamental to position delete processing. They are handled differently by format:

- **Parquet**: The `ParquetReader` is created with `appendRowNumberColumn = true`. Row positions come from the filtered row groups' `fileRowOffset`. The start/end range is:
  ```java
  startRowPosition = rowGroups.getFirst().fileRowOffset();
  endRowPosition = lastRowGroup.fileRowOffset() + lastRowGroup.prunedBlockMetadata().getRowCount();
  ```

- **ORC**: Row positions come from `OrcRecordReader`:
  ```java
  recordReader.getStartRowPosition();
  recordReader.getEndRowPosition();
  ```

- **Avro**: Row positions are empty (`OptionalLong.empty()`), meaning position delete files cannot use range-based optimizations for Avro data files.

When the reader has `appendRowNumberColumn = true`, the row number appears as the LAST channel in the source page. The `GetRowPositionFromSource` transform extracts it:

```java
private record GetRowPositionFromSource() implements Function<SourcePage, Block> {
    @Override
    public Block apply(SourcePage page) {
        return page.getBlock(page.getChannelCount() - 1);
    }
}
```

## 3. Key Design Insights

1. **Post-filter delete architecture**: Deletes are NOT pushed into the format reader. The format reader produces full pages, and deleted rows are filtered out afterward via `SourcePage.selectPositions()`. This keeps the format readers simple and format-independent, at the cost of reading and decoding rows that will be discarded.

2. **Lazy, memoized delete predicate construction**: The delete predicate is constructed lazily on first page access. This means if the format reader skips all row groups (due to predicate pushdown), the delete files are never loaded.

3. **Partition-scoped equality delete caching**: `DeleteManager` instances are per-partition, and `EqualityDeleteFilterBuilder` deduplicates file loads using `ConcurrentHashMap.computeIfAbsent()`. Multiple split threads for the same partition share the same equality delete hash set.

4. **Deletion vector priority over position deletes**: Per the Iceberg spec, if a deletion vector exists for a data file, traditional position delete files are ignored. This is a significant optimization since deletion vectors are much more compact.

5. **Sequence-number-based equality delete correctness**: The check `maxDeleteVersion.dataSequenceNumber() <= splitDataSequenceNumber` ensures that equality delete entries only suppress rows from data files written BEFORE the delete. This prevents the "phantom delete" problem in concurrent write scenarios.

6. **TransformConnectorPageSource as universal adapter**: This single layer handles partition value injection, constant columns, nested field dereferencing, row position extraction, and column reordering -- all in a lazy, on-demand fashion.

7. **Column ID-based mapping enables schema evolution**: Both the Parquet and ORC paths use Iceberg field IDs to match columns between the current table schema and the file's physical schema. This allows columns to be added, removed, renamed, and reordered between when the file was written and when it is read.

8. **PrefixColumnsSourcePage for column stripping**: When extra columns (like `ROW_POSITION` for position deletes, or equality field columns) are added to the reader's required columns, they are placed at the end. After delete filtering, `PrefixColumnsSourcePage` exposes only the original query columns by limiting `getChannelCount()`.

9. **DeletionVector uses 64-bit positions via high/low split**: The `DeletionVector` splits a 64-bit file position into a 32-bit "key" (high bits, indexing into the bitmap array) and 32-bit "low" bits (within a RoaringBitmap). In practice, most files have fewer than 2^32 rows, so only bitmap index 0 is used.

## 4. Porting Considerations (Java -> Rust)

1. **RowPredicate as a trait**: The `RowPredicate` interface maps naturally to a Rust trait with `fn test(&self, page: &SourcePage, position: usize) -> bool` and a blanket `and()` combinator. The `applyFilter()` method (building a retained positions array and calling `selectPositions`) should be a default method.

2. **DeletionVector with roaring crate**: Rust's `roaring` crate provides `RoaringBitmap` (32-bit) and `RoaringTreemap` (64-bit). The Iceberg serialization format with high/low split maps directly to `RoaringTreemap`'s internal structure. However, the serialization format is custom (Iceberg-specific), so the deserializer must be implemented manually, matching the magic number and CRC32 checks.

3. **EqualityDeleteFilter with concurrent hash maps**: The `ConcurrentHashMap<StructLikeWrapper, DataSequenceNumber>` pattern maps to `dashmap::DashMap` or `flurry::HashMap` in Rust. The `StructLikeWrapper` equivalent needs a custom `Hash + Eq` implementation over typed column values. Consider using `ahash` for faster hashing.

4. **TransformConnectorPageSource as an enum-based transform pipeline**: The Builder pattern with its `constantValue` / `column` / `dereferenceField` / `transform` variants maps well to a Rust enum:
   ```rust
   enum ColumnTransform {
       Constant(ArrayRef),
       PassThrough(usize),
       Dereference(Vec<usize>),
       Custom(Box<dyn Fn(&RecordBatch) -> ArrayRef>),
   }
   ```

5. **Lazy block materialization**: `TransformSourcePage.getBlock()` lazily computes blocks. In Rust, use `OnceCell<ArrayRef>` or `LazyCell` for per-column lazy evaluation within a page wrapper.

6. **Format reader abstraction**: The `ReaderPageSourceWithRowPositions` pattern should be a trait:
   ```rust
   trait FormatPageSource {
       fn next_page(&mut self) -> Option<RecordBatch>;
       fn start_row_position(&self) -> Option<u64>;
       fn end_row_position(&self) -> Option<u64>;
   }
   ```

7. **DeleteManager per-partition caching**: The `ConcurrentHashMap<PartitionKey, DeleteManager>` pattern maps to `DashMap<PartitionKey, Arc<DeleteManager>>`. The `PartitionKey` needs proper `Hash + Eq` based on the partition spec ID and partition data.

8. **Memory accounting**: The Java code carefully tracks `estimatedSizeInBytes` for equality delete filters. In Rust, use a custom allocator or manual size tracking, especially for the equality delete hash set which can grow large.

9. **The `openDeleteFile` callback pattern**: The delete pipeline uses functional interfaces (`DeletionVectorReader`, `DeletePageSourceProvider`) passed as lambdas from the page source provider. In Rust, these map to `Box<dyn Fn(...)>` or generic parameters with trait bounds.

10. **Thread safety model**: The Java code uses `@ThreadSafe` annotations and `ConcurrentHashMap` for shared state. In Rust, the equivalent is `Arc<Mutex<...>>` or lock-free structures. The equality delete filter builder's concurrent loading pattern (using `ListenableFutureTask.computeIfAbsent`) maps to `tokio::sync::OnceCell` or a similar async-once pattern.
