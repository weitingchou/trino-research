# Module Teardown: Exchange Protocol HTTP Headers & Framing (Task 4.2.G) [KG-12]

## 0. Research Focus
* **Task ID:** 4.2.G
* **Focus:** Document every HTTP header used in the exchange protocol, trace the full request/response construction on both sides (server via `TaskResource`, client via `HttpPageBufferClient`), understand the acknowledge/destroy endpoints and token semantics, the HTTP response body framing format, error handling status codes, and the complete page fetch cycle.

## 1. High-Level Overview

Trino's inter-worker data exchange is a pull-based HTTP protocol. A downstream worker (the "client" / exchange consumer) fetches serialized pages from an upstream worker (the "server" / task producer) via `GET /v1/task/{taskId}/results/{bufferId}/{token}`. The protocol is built on five custom `X-Trino-*` response headers that carry flow-control metadata, plus one custom request header to communicate buffer size limits. Authentication is handled by a separate JWT bearer header.

The protocol has three phases in each page fetch cycle:
1. **GET** -- Client sends a GET with the current sequence token in the URL path and a max-size header. Server returns 200 (with pages) or 204 (no data yet), both with full metadata headers.
2. **ACK** (optional, fire-and-forget) -- Client immediately sends an acknowledge GET to release server-side memory for consumed pages.
3. **DELETE** (end-of-stream) -- When the server signals buffer-complete, the client sends a DELETE to destroy the remote buffer and finalize cleanup.

All endpoints are annotated `@ResourceSecurity(INTERNAL_ONLY)`, meaning they require the `X-Trino-Internal-Bearer` JWT header. The body format uses a 16-byte envelope (magic + checksum + page count) followed by concatenated serialized pages, each self-framed by a 12-byte header that includes the compressed payload size.

## 2. Detailed Analysis

### 2.1 InternalHeaders -- The Canonical Header Definitions

All custom exchange headers are defined in a single file:

**Source:** `io.trino.server.InternalHeaders`

```java
public static final String TRINO_CURRENT_VERSION = "X-Trino-Current-Version";
public static final String TRINO_MAX_WAIT = "X-Trino-Max-Wait";
public static final String TRINO_MAX_SIZE = "X-Trino-Max-Size";
public static final String TRINO_TASK_INSTANCE_ID = "X-Trino-Task-Instance-Id";
public static final String TRINO_PAGE_TOKEN = "X-Trino-Page-Sequence-Id";
public static final String TRINO_PAGE_NEXT_TOKEN = "X-Trino-Page-End-Sequence-Id";
public static final String TRINO_BUFFER_COMPLETE = "X-Trino-Buffer-Complete";
public static final String TRINO_TASK_FAILED = "X-Trino-Task-Failed";
```

Note the naming mismatch between Java constant names and actual wire names: `TRINO_PAGE_TOKEN` maps to `X-Trino-Page-Sequence-Id` (not `X-Trino-Page-Token`), and `TRINO_PAGE_NEXT_TOKEN` maps to `X-Trino-Page-End-Sequence-Id`. The constants were evidently renamed after the wire format was established. The code references are consistent -- only the wire names matter for protocol compatibility.

Each `String` constant has a corresponding `HeaderName` constant (e.g., `TRINO_MAX_SIZE_HEADER`) used by the Airlift HTTP client when reading response headers.

### 2.2 Authentication Header

**Source:** `io.trino.server.InternalAuthenticationManager`

```java
private static final HeaderName TRINO_INTERNAL_BEARER = HeaderName.of("X-Trino-Internal-Bearer");

@Override
public Request filterRequest(Request request)
{
    return fromRequest(request)
            .addHeader(TRINO_INTERNAL_BEARER, getOrGenerateJwt())
            .build();
}
```

`InternalAuthenticationManager` implements `HttpRequestFilter`, which is registered as a Guice binding for the internal HTTP client. Every outgoing HTTP request from one Trino node to another is intercepted by `filterRequest()`, which attaches a JWT bearer token in the `X-Trino-Internal-Bearer` header.

The JWT is signed with HMAC-SHA256, derived from the shared secret via HKDF-SHA256:
```java
private static SecretKey expandKey(String sharedSecret)
{
    KDF hkdf = KDF.getInstance("HKDF-SHA256");
    AlgorithmParameterSpec params =
            HKDFParameterSpec.ofExtract()
                    .addIKM(sharedSecret.getBytes(UTF_8))
                    .thenExpand("internal-communication".getBytes(UTF_8), 32);
    return hkdf.deriveKey("HmacSHA256", params);
}
```

On the server side, `TaskResource` is annotated `@ResourceSecurity(INTERNAL_ONLY)`, which routes incoming requests through `InternalAuthenticationManager.handleInternalRequest()`. If the JWT is invalid, the server returns 401. If the node is still starting, it returns 503.

### 2.3 Request Headers (Client -> Server)

#### 2.3.1 GET /v1/task/{taskId}/results/{bufferId}/{token} (Page Fetch)

The client-side request is constructed in `HttpPageBufferClient.sendGetResults()`:

**Source:** `io.trino.operator.HttpPageBufferClient`

```java
private synchronized void sendGetResults()
{
    URI uri = HttpUriBuilder.uriBuilderFrom(location).appendPath(String.valueOf(token)).build();
    HttpResponseFuture<PagesResponse> resultFuture = httpClient.executeAsync(
            prepareGet()
                    .setHeader(TRINO_MAX_SIZE_HEADER, maxResponseSize.toString())
                    .setUri(uri).build(),
            new PageResponseHandler(dataIntegrityVerification != DataIntegrityVerification.NONE));
    // ...
}
```

**Request headers sent by the client:**

| Header | Wire Name | Value | Purpose |
|--------|-----------|-------|---------|
| `TRINO_MAX_SIZE_HEADER` | `X-Trino-Max-Size` | e.g., `"32MB"` | Maximum data size the client is willing to receive in one response. The server's `ClientBuffer` respects this when selecting pages to return. |
| `TRINO_INTERNAL_BEARER` | `X-Trino-Internal-Bearer` | JWT string | Authentication (added by `InternalAuthenticationManager` filter, not explicit in `sendGetResults`). |

The token (sequence ID) is embedded in the URL path, not as a header: `/v1/task/{taskId}/results/{bufferId}/{token}`.

**What the server reads from the request:**

```java
// TaskResource.getResults()
@GET
@Path("{taskId}/results/{bufferId}/{token}")
@Produces(TRINO_PAGES)
public void getResults(
        @PathParam("taskId") TaskId taskId,
        @PathParam("bufferId") PipelinedOutputBuffers.OutputBufferId bufferId,
        @PathParam("token") long token,
        @HeaderParam(TRINO_MAX_SIZE) DataSize maxSize,
        @Suspended AsyncResponse asyncResponse)
```

The server extracts `maxSize` from the `X-Trino-Max-Size` header. If absent, `maxSize` is null and the framework handles it (JAX-RS default handling -- in practice this header is always sent).

#### 2.3.2 GET /v1/task/{taskId}/results/{bufferId}/{token}/acknowledge (ACK)

```java
// HttpPageBufferClient, inside onSuccess callback:
URI uri = HttpUriBuilder.uriBuilderFrom(location)
        .appendPath(String.valueOf(result.getNextToken()))
        .appendPath("acknowledge").build();
httpClient.executeAsync(prepareGet().setUri(uri).build(), ...);
```

No custom headers -- just the JWT added by the filter. The token in the path is the *next* token (the end-sequence-id from the previous response), telling the server "I have consumed everything up to this sequence ID."

### 2.4 Response Headers (Server -> Client)

#### 2.4.1 GET Results Response Construction

The response is built in `TaskResource.createBufferResultResponse()`:

**Source:** `io.trino.server.TaskResource`

```java
private static Response createBufferResultResponse(
        PagesInputStreamFactory pagesInputStreamFactory,
        SqlTaskWithResults taskWithResults,
        BufferResult result)
{
    // Refresh heartbeat
    taskWithResults.recordHeartbeat();

    List<Slice> serializedPages = result.serializedPages();

    Response.ResponseBuilder response = Response.status(
            serializedPages.isEmpty() ? Status.NO_CONTENT : Status.OK)
            .header(TRINO_TASK_INSTANCE_ID, result.taskInstanceId())
            .header(TRINO_PAGE_TOKEN, result.token())
            .header(TRINO_PAGE_NEXT_TOKEN, result.nextToken())
            .header(TRINO_BUFFER_COMPLETE, result.bufferComplete())
            .header(TRINO_TASK_FAILED, taskWithResults.isTaskFailedOrFailing());

    if (serializedPages.isEmpty()) {
        return response.build();
    }

    return response
            .type(TRINO_PAGES)
            .entity((StreamingOutput) output ->
                    pagesInputStreamFactory.write(output, serializedPages))
            .build();
}
```

**Response headers:**

| Header | Wire Name | Type | Value | Purpose |
|--------|-----------|------|-------|---------|
| `TRINO_TASK_INSTANCE_ID` | `X-Trino-Task-Instance-Id` | `long` (as string) | Unique ID for this task incarnation | Detects task restarts after crashes. If a client sees a different instance ID than previously recorded, it means the remote task was restarted and data may be inconsistent. |
| `TRINO_PAGE_TOKEN` | `X-Trino-Page-Sequence-Id` | `long` | Starting sequence ID of this batch | Echoes back the token from the request. Allows the client to correlate response with request. |
| `TRINO_PAGE_NEXT_TOKEN` | `X-Trino-Page-End-Sequence-Id` | `long` | Next sequence ID to request | `nextToken = token + number_of_pages_returned`. The client uses this as the token in its next GET request. |
| `TRINO_BUFFER_COMPLETE` | `X-Trino-Buffer-Complete` | `boolean` | `true`/`false` | Signals that the upstream task has finished producing data for this buffer. When true, the client should finish consuming and then DELETE the buffer. |
| `TRINO_TASK_FAILED` | `X-Trino-Task-Failed` | `boolean` | `true`/`false` | Set by checking `taskWithResults.isTaskFailedOrFailing()`, which returns true if task state is ABORTED, ABORTING, FAILED, or FAILING. Allows the client to detect remote task failure even when data is still flowing. |
| `Content-Type` | `Content-Type` | string | `application/x-trino-pages` | Present only when HTTP status is 200 (pages exist). Not set for 204 (no content). |

**HTTP Status Codes for results:**

| Status | Meaning |
|--------|---------|
| **200 OK** | Body contains one or more serialized pages in `application/x-trino-pages` format. |
| **204 No Content** | No data available yet (timeout or empty buffer). Headers still carry metadata (token, complete flag, etc.). No body. |

### 2.5 Token Semantics and the Acknowledge/Destroy Endpoints

#### 2.5.1 Sequence ID Tokens

The "token" in the URL path is a **monotonically increasing sequence ID** that represents the position in the page stream. It starts at 0.

In `ClientBuffer.processRead()`:

```java
private synchronized BufferResult processRead(long sequenceId, DataSize maxSize)
{
    // if request is for pages before the current position, just return empty
    if (sequenceId < currentSequenceId.get()) {
        return emptyResults(taskInstanceId, sequenceId, false);
    }
    // ...
    // read the new pages
    List<Slice> result = new ArrayList<>();
    long bytes = 0;
    for (SerializedPageReference page : pages) {
        bytes += page.getRetainedSizeInBytes();
        if (!result.isEmpty() && bytes > maxBytes) {
            break;
        }
        result.add(page.getSerializedPage());
    }
    return new BufferResult(taskInstanceId, sequenceId, sequenceId + result.size(), false, result);
}
```

The key relationship: `nextToken = token + pages.size()`. Each page is implicitly assigned a sequence ID equal to `token + index_within_batch`.

#### 2.5.2 Implicit Acknowledge via GET

Every GET request implicitly acknowledges all pages with sequence IDs less than the requested token. In `ClientBuffer.getPages()`:

```java
public ListenableFuture<BufferResult> getPages(long sequenceId, DataSize maxSize, ...)
{
    // acknowledge pages first, outside of locks
    acknowledgePages(sequenceId);
    // ...
}
```

And `acknowledgePages()` drops all pages before the given sequence ID:

```java
public void acknowledgePages(long sequenceId)
{
    // ...
    int pagesToRemove = toIntExact(sequenceId - oldCurrentSequenceId);
    for (int i = 0; i < pagesToRemove; i++) {
        SerializedPageReference removedPage = pages.removeFirst();
        removedPages.add(removedPage);
        bytesRemoved += removedPage.getRetainedSizeInBytes();
    }
    verify(currentSequenceId.compareAndSet(oldCurrentSequenceId, oldCurrentSequenceId + pagesToRemove));
    verify(bufferedBytes.addAndGet(-bytesRemoved) >= 0);
}
```

#### 2.5.3 Explicit Acknowledge Endpoint

The explicit acknowledge endpoint (`GET .../acknowledge`) exists as an optimization for fast memory release. In `HttpPageBufferClient`, after receiving pages:

```java
if (shouldAcknowledge && acknowledgePages) {
    // Acknowledge token without handling the response.
    // The next request will also make sure the token is acknowledged.
    // This is to fast release the pages on the buffer side.
    URI uri = HttpUriBuilder.uriBuilderFrom(location)
            .appendPath(String.valueOf(result.getNextToken()))
            .appendPath("acknowledge").build();
    httpClient.executeAsync(prepareGet().setUri(uri).build(), ...);
}
```

This is fire-and-forget. The response is logged but not checked for correctness. Even if the ACK fails, the next GET will re-acknowledge via the implicit path. The ACK endpoint calls `taskManager.acknowledgeTaskResults()`, which calls `outputBuffer.acknowledge(bufferId, sequenceId)`.

#### 2.5.4 Destroy Buffer (DELETE)

The DELETE endpoint does *not* take a token -- it takes only `{taskId}/results/{bufferId}`. It destroys the entire client buffer:

```java
// SqlTask.destroyTaskResults():
public TaskInfo destroyTaskResults(PipelinedOutputBuffers.OutputBufferId bufferId)
{
    log.debug("Aborting task %s output %s", taskId, bufferId);
    outputBuffer.destroy(bufferId);
    return getTaskInfo();
}
```

In `ClientBuffer.destroy()`:

```java
public void destroy()
{
    List<SerializedPageReference> removedPages;
    synchronized (this) {
        removedPages = ImmutableList.copyOf(pages);
        pages.clear();
        bufferedBytes.set(0);
        noMorePages = true;
        destroyed.set(true);
        pendingRead = this.pendingRead;
        this.pendingRead = null;
    }
    dereferencePages(removedPages, onPagesReleased);
    // ...
}
```

The client sends DELETE after it sees `bufferComplete = true` in a response, confirming it has consumed all data. In `HttpPageBufferClient`, the lifecycle is:

```java
// After receiving a response with isClientComplete() == true:
synchronized (HttpPageBufferClient.this) {
    if (result.isClientComplete()) {
        completed = true;
    }
}
// On next initiateRequest(), since completed == true:
if (completed) {
    destroyTaskResults();
}
```

### 2.6 HTTP Response Body Framing

When the response has pages (status 200), the body is written by `PagesInputStreamFactory.write()`:

**Source:** `io.trino.server.PagesInputStreamFactory`

```java
public static final int SERIALIZED_PAGES_MAGIC = 0xfea4f001;

public void write(OutputStream stream, List<Slice> serializedPages) throws IOException
{
    try (stream) {
        SliceOutput header = new OutputStreamSliceOutput(stream);
        header.writeInt(SERIALIZED_PAGES_MAGIC);
        header.writeLong(dataIntegrityVerificationEnabled
                ? calculateChecksum(serializedPages) : NO_CHECKSUM);
        header.writeInt(serializedPages.size());
        header.flush();

        for (Slice page : serializedPages) {
            page.getInput().transferTo(stream);
            stream.flush();
        }
    }
    // EOF exception handling omitted -- client disconnect is silently ignored
}
```

**Response body layout:**

```
Offset | Size   | Field          | Notes
-------|--------|----------------|------
0      | 4      | magic          | int32 LE, always 0xfea4f001
4      | 8      | checksum       | int64 LE, XxHash64 of all pages, or NO_CHECKSUM (0x0123456789abcdef)
12     | 4      | pageCount      | int32 LE, number of serialized pages following
16     | varies | Page 0         | [12-byte header][compressed payload]
16+len0| varies | Page 1         | [12-byte header][compressed payload]
...    | ...    | ...            | ...
```

Each serialized page is self-framing via its 12-byte header:

```
Offset | Size | Field            | Notes
-------|------|------------------|------
0      | 4    | positionCount    | int32 LE, number of rows
4      | 4    | uncompressedSize | int32 LE, in bytes
8      | 4    | compressedSize   | int32 LE, in bytes (bit 31 = compressed flag)
```

The client reads this in `PageResponseHandler.handle()`:

```java
try (LittleEndianDataInputStream input = new LittleEndianDataInputStream(response.getInputStream())) {
    int magic = input.readInt();
    if (magic != SERIALIZED_PAGES_MAGIC) {
        throw new IllegalStateException(format(
                "Invalid stream header, expected 0x%08x, but was 0x%08x",
                SERIALIZED_PAGES_MAGIC, magic));
    }
    long checksum = input.readLong();
    int pagesCount = input.readInt();
    List<Slice> pages = ImmutableList.copyOf(readSerializedPages(input));
    verifyChecksum(checksum, pages);
    checkState(pages.size() == pagesCount,
            "Wrong number of pages, expected %s, but read %s", pagesCount, pages.size());
    return createPagesResponse(taskInstanceId, token, nextToken, pages, complete, remoteTaskFailed);
}
```

`readSerializedPages()` iterates by reading each page's 12-byte header, extracting `compressedSize` to know how many bytes to read for that page, then advancing to the next. This means pages are effectively length-prefixed (via the 12-byte header containing the compressed size), not delimiter-separated.

### 2.7 Error Handling

#### 2.7.1 HTTP Status Codes

| Status | Source | Meaning |
|--------|--------|---------|
| **200 OK** | `TaskResource.createBufferResultResponse` | Pages returned in body. |
| **204 No Content** | `TaskResource.createBufferResultResponse` | No pages available (timeout/empty). Headers still present. Also returned by successful DELETE. |
| **401 Unauthorized** | `InternalAuthenticationManager.handleInternalRequest` | Invalid JWT in `X-Trino-Internal-Bearer`. |
| **500 Internal Server Error** | `TaskResource.injectFailure` (failure injection) | Deliberate test failure, or unhandled exception. |
| **503 Service Unavailable** | `TaskResource.failRequestIfInvalid` | Worker not fully started. Also returned on hard timeout from `serviceUnavailable()`. |

The 503 case for startup status:

```java
private boolean failRequestIfInvalid(AsyncResponse asyncResponse)
{
    if (!startupStatus.isStartupComplete()) {
        asyncResponse.resume(new ServiceUnavailableException(
                "The server is not fully started yet"));
        return true;
    }
    return false;
}
```

The 503 for hard timeout:

```java
private static Response serviceUnavailable(Duration timeout)
{
    return status(Response.Status.SERVICE_UNAVAILABLE)
            .entity("Timed out after waiting for " + timeout.convertToMostSuccinctTimeUnit())

}
```

#### 2.7.2 Client-Side Error Handling

The client distinguishes several error categories in `HttpPageBufferClient`:

**1. Non-200/204 response codes** -- `PageTransportErrorException` (error code `PAGE_TRANSPORT_ERROR`, 65539):

```java
if (response.getStatusCode() != HttpStatus.OK.code()) {
    // Read up to 1000 lines / 100KB of error body for debugging
    throw new PageTransportErrorException(fromUri(uri),
            format("Expected response code to be 200, but was %s:%n%s",
                    response.getStatusCode(), body));
}
```

**2. Missing Content-Type header** -- `PageTransportErrorException`:

```java
Optional<String> contentType = response.getHeader(CONTENT_TYPE);
if (contentType.isEmpty()) {
    throw new PageTransportErrorException(fromUri(uri),
            format("%s header is not set: %s", CONTENT_TYPE, response));
}
```

**3. Wrong Content-Type** -- `PageTransportErrorException`:

```java
if (!mediaTypeMatches(contentType.orElseThrow(), TRINO_PAGES_TYPE)) {
    throw new PageTransportErrorException(fromUri(uri),
            format("Expected %s response from server but got %s",
                    TRINO_PAGES_TYPE, contentType.orElseThrow()));
}
```

**4. Missing required response headers** -- `PageTransportErrorException` for each:

```java
private static String getTaskInstanceId(Response response, URI uri) {
    return response.getHeader(TRINO_TASK_INSTANCE_ID_HEADER)
            .orElseThrow(() -> new PageTransportErrorException(fromUri(uri),
                    format("Expected %s header", TRINO_TASK_INSTANCE_ID_HEADER)));
}
```

The same pattern applies for `TRINO_PAGE_TOKEN_HEADER`, `TRINO_PAGE_NEXT_TOKEN_HEADER`, `TRINO_BUFFER_COMPLETE_HEADER`, and `TRINO_TASK_FAILED_HEADER`.

**5. Task instance ID mismatch** -- `TrinoException` (error code `REMOTE_TASK_MISMATCH`):

```java
if (!isNullOrEmpty(taskInstanceId) && !result.getTaskInstanceId().equals(taskInstanceId)) {
    throw new TrinoException(REMOTE_TASK_MISMATCH, format(
            "%s (%s). Expected taskInstanceId: %s, received taskInstanceId: %s",
            REMOTE_TASK_MISMATCH_ERROR, fromUri(uri), taskInstanceId, result.getTaskInstanceId()));
}
```

**6. Remote task failure** -- `TrinoException` (error code `REMOTE_TASK_FAILED`):

```java
if (result.isTaskFailed()) {
    throw new TrinoException(REMOTE_TASK_FAILED,
            format("Remote task failed: %s", remoteTaskId));
}
```

**7. Checksum mismatch** -- `ChecksumVerificationException`, handling depends on config:

```java
private void verifyChecksum(long readChecksum, List<Slice> pages)
{
    if (dataIntegrityVerificationEnabled) {
        long calculatedChecksum = calculateChecksum(pages);
        if (readChecksum != calculatedChecksum) {
            throw new ChecksumVerificationException(...);
        }
    }
    else {
        if (readChecksum != NO_CHECKSUM) {
            throw new ChecksumVerificationException(...);
        }
    }
}
```

With `DataIntegrityVerification.ABORT` -- becomes `TrinoException(GENERIC_INTERNAL_ERROR)` (non-retriable).
With `DataIntegrityVerification.RETRY` -- logged as warning, retried via backoff.

**8. Response too large** -- `PageTooLargeException` (error code `PAGE_TOO_LARGE`):

```java
private static Throwable rewriteException(Throwable t) {
    if (Throwables.getCausalChain(t).stream()
            .anyMatch(e -> nullToEmpty(e.getMessage()).contains("exceeded maximum length"))) {
        PageTooLargeException pageTooLargeException = new PageTooLargeException();
        pageTooLargeException.addSuppressed(t);
        return pageTooLargeException;
    }
    return t;
}
```

**9. Timeout / too many failures** -- `PageTransportTimeoutException` (error code `PAGE_TRANSPORT_TIMEOUT`):

```java
if (!(t instanceof TrinoException) && backoff.failure()) {
    String message = format("%s (%s - %s failures, failure duration %s, total failed request time %s)",
            WORKER_NODE_ERROR, uri, backoff.getFailureCount(),
            backoff.getFailureDuration().convertTo(SECONDS),
            backoff.getFailureRequestTimeTotal().convertTo(SECONDS));
    t = new PageTransportTimeoutException(fromUri(uri), message, t);
}
```

The `Backoff` class tracks cumulative failure duration. When `maxErrorDuration` is exceeded, the error escalates to `PageTransportTimeoutException`, which is a `TrinoException` and thus terminates the query rather than retrying.

**Error escalation rule:** Non-`TrinoException` errors are retried with exponential backoff. Any `TrinoException` immediately terminates the query by calling `clientCallback.clientFailed()`.

### 2.8 Complete Request/Response Sequence for a Page Fetch Cycle

Below is the full lifecycle traced through the code:

**Step 1: Client schedules request**

```
HttpPageBufferClient.scheduleRequest()
  -> sets scheduled = true
  -> computes backoff delay (0 on first attempt)
  -> scheduledExecutor.schedule(initiateRequest, delay)
```

**Step 2: Client builds and sends GET**

```
initiateRequest()
  -> if not completed: sendGetResults()
  -> URI = location + "/" + token  (e.g., /v1/task/0.1.0.0/results/0/0)
  -> httpClient.executeAsync(
       GET /v1/task/{taskId}/results/{bufferId}/{token}
       Headers:
         X-Trino-Max-Size: 32MB
         X-Trino-Internal-Bearer: <jwt>
     )
```

**Step 3: Server receives request**

```
TaskResource.getResults(taskId, bufferId, token, maxSize, asyncResponse)
  -> taskManager.getTaskResults(taskId, bufferId, token, maxSize)
    -> SqlTask.getTaskResults(bufferId, token, maxSize)
      -> outputBuffer.get(bufferId, token, maxSize)
        -> ClientBuffer.getPages(token, maxSize)
          -> acknowledgePages(token)    // implicit ACK
          -> if pages available: return immediate future with BufferResult
          -> if empty: return pending future (waits for enqueuePages)
```

**Step 4: Server waits (with timeout)**

```
bufferResultFuture = addTimeout(bufferResultFuture, emptyResults, waitTime)
  // waitTime = randomized in [DEFAULT_MAX_WAIT/2, DEFAULT_MAX_WAIT]
  // DEFAULT_MAX_WAIT = 2 seconds
  // Hard timeout = waitTime + 5 seconds (ADDITIONAL_WAIT_TIME)
```

**Step 5: Server builds response**

```
createBufferResultResponse(pagesInputStreamFactory, taskWithResults, result)
  -> taskWithResults.recordHeartbeat()
  -> if pages empty: 204 No Content + headers
  -> if pages present: 200 OK + headers + streaming body
     Headers:
       X-Trino-Task-Instance-Id: <taskInstanceId>
       X-Trino-Page-Sequence-Id: <token>
       X-Trino-Page-End-Sequence-Id: <nextToken>
       X-Trino-Buffer-Complete: false/true
       X-Trino-Task-Failed: false/true
       Content-Type: application/x-trino-pages  (only if 200)
     Body (if 200):
       [magic:4][checksum:8][pageCount:4]
       [page0_header:12][page0_payload:N]
       [page1_header:12][page1_payload:M]
       ...
```

**Step 6: Client processes response**

```
PageResponseHandler.handle()
  -> if 204: createEmptyPagesResponse(headers)
  -> if 200:
     -> verify Content-Type is application/x-trino-pages
     -> read 16-byte envelope (magic, checksum, pageCount)
     -> read each serialized page via readSerializedPages()
     -> verify checksum (if enabled)
     -> verify page count matches
     -> return PagesResponse
```

**Step 7: Client validates and updates state**

```
FutureCallback.onSuccess(PagesResponse result)
  -> check result.isTaskFailed() -> throw if true
  -> validate taskInstanceId matches
  -> if result.token == expected token: accept pages, advance token
  -> if acknowledgePages enabled: fire-and-forget ACK GET
  -> clientCallback.addPages(pages) -> buffers for operator consumption
  -> if result.isClientComplete(): set completed = true
  -> clientCallback.requestComplete()
```

**Step 8: ACK (if pages were received)**

```
GET /v1/task/{taskId}/results/{bufferId}/{nextToken}/acknowledge
  Headers: X-Trino-Internal-Bearer: <jwt>
  (fire and forget, response ignored)

Server: TaskResource.acknowledgeResults()
  -> taskManager.acknowledgeTaskResults(taskId, bufferId, nextToken)
  -> ClientBuffer.acknowledgePages(nextToken)
  -> drops pages with sequenceId < nextToken
  -> returns 200 OK
```

**Step 9: Repeat (Steps 1-8) until buffer complete**

**Step 10: Buffer complete -> Destroy**

```
// On next scheduleRequest(), since completed == true:
initiateRequest() -> destroyTaskResults()
  -> DELETE /v1/task/{taskId}/results/{bufferId}
     Headers: X-Trino-Internal-Bearer: <jwt>

Server: TaskResource.destroyTaskResults()
  -> taskManager.destroyTaskResults(taskId, bufferId)
  -> ClientBuffer.destroy() -> clears all pages, sets destroyed flag
  -> returns 204 No Content

Client onSuccess:
  -> verify status == 204
  -> set closed = true
  -> clientCallback.clientFinished()
```

## 3. Key Design Insights

1. **Implicit + Explicit ACK dual path.** Every GET request implicitly acknowledges all pages before the requested token, so the explicit ACK endpoint is purely an optimization for faster server-side memory release. The protocol is correct even if every ACK fails or is lost.

2. **Sequence IDs are monotonic page counters, not byte offsets.** Each page gets an integer sequence number starting from 0. The "token" is simply "how many pages have been consumed." This makes the protocol trivially idempotent: re-requesting the same token returns the same (or empty) result with no side effects.

3. **The server uses long-polling with bounded, randomized timeouts.** `DEFAULT_MAX_WAIT_TIME = 2s`, randomized to `[1s, 2s]`, with a hard 7-second ceiling. This avoids thundering-herd effects when many clients poll simultaneously.

4. **Task instance ID is the crash detection mechanism.** If a task restarts (e.g., after a node crash in fault-tolerant execution), it gets a new instance ID. The client detects this mismatch and fails the query, forcing the coordinator to reschedule.

5. **No `X-Trino-Buffer-Remaining-Bytes` header exists.** Despite being mentioned in the tracing guide task description, this header does not appear anywhere in the Trino 480 codebase. The server does not communicate remaining buffer size to the client.

6. **Headers are present on both 200 and 204 responses.** The five metadata headers are always set regardless of whether pages are returned. Only `Content-Type` is conditional on having a body.

7. **Pages are length-prefixed, not delimiter-separated.** Each page's 12-byte header contains `compressedSize`, allowing the reader to seek to the next page. The 16-byte response envelope provides `pageCount` as a redundant cross-check but is not strictly needed for parsing.

8. **Data integrity is opt-in and configurable per-mode.** The checksum field exists in every response but is set to the sentinel `NO_CHECKSUM` when verification is disabled. The special sentinel value was chosen so that zeroed-out corrupt data would not accidentally appear valid.

9. **The destroy endpoint has no token parameter.** Unlike the acknowledge endpoint which takes a specific sequence ID, destroy kills the entire buffer unconditionally. This is safe because destroy is only called after the client has seen buffer-complete and consumed everything.

10. **Fire-and-forget ACK with retry-via-next-GET.** The ACK is explicitly documented as "Acknowledge token without handling the response. The next request will also make sure the token is acknowledged." This is a pragmatic design that prioritizes latency over guaranteed immediate server memory release.

## 4. Porting Considerations (Java -> Rust)

### 4.1 HTTP Server Framework

The Java server uses JAX-RS (Jakarta) annotations (`@GET`, `@Path`, `@HeaderParam`, `@Suspended AsyncResponse`) with async response handling. In Rust, use `axum` or `actix-web` with their equivalents:
- `@PathParam` -> Path extractors
- `@HeaderParam` -> Typed header extractors
- `@Suspended AsyncResponse` -> Return `impl IntoResponse` from an async handler
- `StreamingOutput` -> `axum::body::Body::from_stream()` for streaming the page body

### 4.2 Header Handling

The header names are strings with specific wire format. In Rust, define them as `const &str`:

```rust
const TRINO_TASK_INSTANCE_ID: &str = "X-Trino-Task-Instance-Id";
const TRINO_PAGE_SEQUENCE_ID: &str = "X-Trino-Page-Sequence-Id";
const TRINO_PAGE_END_SEQUENCE_ID: &str = "X-Trino-Page-End-Sequence-Id";
const TRINO_BUFFER_COMPLETE: &str = "X-Trino-Buffer-Complete";
const TRINO_TASK_FAILED: &str = "X-Trino-Task-Failed";
const TRINO_MAX_SIZE: &str = "X-Trino-Max-Size";
const TRINO_INTERNAL_BEARER: &str = "X-Trino-Internal-Bearer";
```

All header values are stringified longs or booleans. Parse on the client side with strict error handling (Java throws `PageTransportErrorException` for any missing header).

### 4.3 DataSize Parsing

The `X-Trino-Max-Size` header carries Airlift `DataSize` format (e.g., `"32MB"`, `"16kB"`). The Rust port needs a parser that handles the same format including units: B, kB, MB, GB, TB, PB. Airlift's parsing is case-sensitive on the unit suffix.

### 4.4 JWT Authentication

The internal authentication uses HMAC-SHA256 JWTs with HKDF-SHA256 key derivation from a shared secret. In Rust, use `jsonwebtoken` crate for JWT and `hkdf` + `sha2` crates for key derivation. The HKDF info string is `"internal-communication"` and output length is 32 bytes.

### 4.5 Body Framing

The 16-byte response envelope and per-page 12-byte headers are all little-endian. Use `byteorder` crate or Rust's native `i32::from_le_bytes()`. Critical constants:
- `SERIALIZED_PAGES_MAGIC = 0xfea4f001`
- `NO_CHECKSUM = 0x0123456789abcdef`
- Page header: `[positionCount:i32][uncompressedSize:i32][compressedSize:i32]`
- Bit 31 of `compressedSize` is the "is compressed" flag (`SERIALIZED_PAGE_COMPRESSED_BLOCK_MASK`)

### 4.6 Async Long-Polling

The server uses async responses with Guava `ListenableFuture` and timeout-with-fallback. In Rust, use `tokio::time::timeout()` wrapping a `tokio::sync::oneshot` or `watch` channel. The fallback should return an empty `BufferResult` (204) rather than an error.

The randomized wait time (`[T/2, T]`) is important for avoiding thundering herd. Reimplement as:
```rust
let half = max_wait / 2;
let wait = half + rand::thread_rng().gen_range(0..half);
```

### 4.7 Backoff and Error Classification

The client-side error handling has a precise classification hierarchy that must be replicated exactly:
1. `TrinoException` -> query fails immediately (no retry)
2. `PageTransportErrorException` -> retried with backoff (eventually escalates to `PageTransportTimeoutException`)
3. `ChecksumVerificationException` -> behavior depends on `DataIntegrityVerification` config
4. `PageTooLargeException` -> detected via exception message string matching ("exceeded maximum length")

In Rust, model this as an error enum with explicit retry semantics.

### 4.8 Concurrency Safety

`HttpPageBufferClient` is heavily synchronized, with careful lock ordering (never hold lock when calling `clientCallback`). The `ClientBuffer` uses similar patterns with `@GuardedBy("this")` annotations. In Rust, use `Mutex<ClientState>` or `tokio::sync::Mutex`, and design the callback dispatch to always happen outside the lock scope.

### 4.9 Protocol Compatibility Testing

The Rust worker must interoperate with Java coordinators and workers. Create integration tests that:
1. Send GET with `X-Trino-Max-Size` header and validate response headers
2. Verify the 16-byte envelope is correctly formed
3. Test the implicit ACK behavior (GET with token N acknowledges everything < N)
4. Test the explicit ACK endpoint
5. Test the DELETE endpoint returns 204
6. Test task instance ID mismatch detection
7. Test behavior when server returns 503 (startup not complete)
