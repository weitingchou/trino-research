## Section 1: High-Level Overview

When the Trino coordinator sends a physical execution plan to a worker inside a `TaskUpdateRequest`, the plan is serialized as a JSON `PlanFragment`. For a Rust worker to receive and execute these plans, it must be able to perfectly deserialize this JSON format.

### Key Structural Layers

```
PlanFragment (top-level envelope)
  +-- id: PlanFragmentId (string)
  +-- root: PlanNode (polymorphic, recursive tree via @type discriminator)
  +-- symbols: Set<Symbol> (custom key serialization)
  +-- partitioning: PartitioningHandle (how fragment inputs are distributed)
  +-- partitionCount: OptionalInt
  +-- partitionedSources: List<PlanNodeId> (string list)
  +-- outputPartitioningScheme: PartitioningScheme (how outputs are distributed)
  +-- outputSkewedBucketCount: OptionalInt
  +-- statsAndCosts: StatsAndCosts
  +-- activeCatalogs: List<CatalogProperties>
  +-- languageFunctions: Map<FunctionId, LanguageFunctionData>
  +-- jsonRepresentation: Optional<String>
```

### Polymorphic Deserialization Strategy

Trino uses two distinct mechanisms for polymorphic JSON:

1. **PlanNode hierarchy:** Jackson `@JsonTypeInfo(use = Id.NAME, property = "@type")` with `@JsonSubTypes` -- 47 plan node types registered with string discriminators (e.g., `"tableScan"`, `"join"`, `"filter"`).

2. **Expression IR hierarchy:** Same pattern -- `@JsonTypeInfo(use = Id.NAME)` with 18 expression types (e.g., `"reference"`, `"comparison"`, `"call"`).

3. **Connector handles (ConnectorTableHandle, ColumnHandle, ConnectorPartitioningHandle, etc.):** Custom `AbstractTypedJacksonModule` that uses `@type` property with a class-loader-qualified ID format like `"system:io.trino.sql.planner.SystemPartitioningHandle"` or `"pluginId:com.example.MyTableHandle"`.

### 2.2 PlanFragment Top-Level Structure

**Source:** `core/trino-main/src/main/java/io/trino/sql/planner/PlanFragment.java`

The `@JsonCreator` constructor defines the exact fields deserialized from JSON:

```java
@JsonCreator
public PlanFragment(
        @JsonProperty("id") PlanFragmentId id,                                    // string (via @JsonValue)
        @JsonProperty("root") PlanNode root,                                       // polymorphic node tree
        @JsonProperty("symbols") Set<Symbol> symbols,                              // custom serialization
        @JsonProperty("partitioning") PartitioningHandle partitioning,             // fragment-level partitioning
        @JsonProperty("partitionCount") OptionalInt partitionCount,                // optional int
        @JsonProperty("partitionedSources") List<PlanNodeId> partitionedSources,   // list of node ID strings
        @JsonProperty("outputPartitioningScheme") PartitioningScheme outputPartitioningScheme,
        @JsonProperty("outputSkewedBucketCount") OptionalInt outputSkewedBucketCount,
        @JsonProperty("statsAndCosts") StatsAndCosts statsAndCosts,                // stats metadata
        @JsonProperty("activeCatalogs") List<CatalogProperties> activeCatalogs,    // connector info
        @JsonProperty("languageFunctions") Map<FunctionId, LanguageFunctionData> languageFunctions,
        @JsonProperty("jsonRepresentation") Optional<String> jsonRepresentation)   // human-readable plan text
```

**Field-by-field breakdown:**

| JSON Key | Type | Description |
|---|---|---|
| `id` | `string` | Fragment identifier (e.g., `"0"`, `"1"`). `PlanFragmentId` has `@JsonValue` returning string. |
| `root` | `object` | The top plan node of this fragment's tree. Polymorphic via `@type`. |
| `symbols` | `array` | Set of all `Symbol` objects used in the fragment. Each Symbol serializes with type info. |
| `partitioning` | `object` | `PartitioningHandle` -- describes how input data is partitioned to this fragment. |
| `partitionCount` | `int` or absent | Optional partition count for system partitioning. |
| `partitionedSources` | `array<string>` | PlanNodeIds of nodes that receive partitioned data (splits). |
| `outputPartitioningScheme` | `object` | How the fragment's output should be repartitioned for downstream. |
| `outputSkewedBucketCount` | `int` or absent | For skew handling in adaptive planning. |
| `statsAndCosts` | `object` | Contains `stats` and `costs` maps keyed by PlanNodeId. |
| `activeCatalogs` | `array` | Catalogs this fragment needs. Each has `name`, `version`, `connectorName`, `properties`. |
| `languageFunctions` | `object` | Map from FunctionId string to function data. |
| `jsonRepresentation` | `string` or absent | Human-readable plan text (coordinator may omit for workers). |

Note: Fields like `partitionedSourcesSet`, `partitionedSourceNodes`, `remoteSourceNodes`, and `containsTableScanNode` are NOT serialized -- they are computed at construction time from the deserialized fields.


### 2.3 Symbol Serialization (Critical Detail)

**Source:** `core/trino-main/src/main/java/io/trino/sql/planner/Symbol.java`, `SymbolKeySerializer.java`, `SymbolKeyDeserializer.java`

Symbol is a Java record:
```java
@JsonSerialize(keyUsing = SymbolKeySerializer.class)
public record Symbol(Type type, String name) { ... }
```

**When used as a MAP KEY** (e.g., in `assignments: Map<Symbol, ColumnHandle>`), Symbol uses a custom format:

```java
// SymbolKeySerializer
public void serialize(Symbol value, JsonGenerator generator, SerializerProvider serializers) {
    String type = value.type().getTypeId().getId();
    generator.writeFieldName(type.length() + "|" + type + "|" + value.name());
}
```

This produces keys like: `"7|varchar|col_name"` or `"6|bigint|total_amount"`

The format is: `<type_string_length>|<type_id>|<symbol_name>`

**SymbolKeyDeserializer** parses this back:
```java
public Object deserializeKey(String key, DeserializationContext context) {
    int lastDigitIndex = getLeadingDigitsLength(key, keyLength);
    int length = parseInt(key.substring(0, lastDigitIndex));
    String type = key.substring(lastDigitIndex + 1, lastDigitIndex + length + 1);
    String name = key.substring(lastDigitIndex + length + 2);
    return new Symbol(typeManager.getType(TypeId.of(type)), name);
}
```

**When used as a VALUE** (e.g., in a list or standalone), Symbol serializes as a normal JSON record with `type` and `name` fields.

### 2.6 Connector Handle Polymorphism (AbstractTypedJacksonModule)

**Source:** `core/trino-main/src/main/java/io/trino/metadata/AbstractTypedJacksonModule.java`, `HandleJsonModule.java`, `HandleResolver.java`

For SPI interfaces like `ConnectorTableHandle`, `ColumnHandle`, `ConnectorPartitioningHandle`, and `ConnectorTransactionHandle`, Trino uses a custom Jackson module rather than annotations.

The module uses `@type` as the property name (same as PlanNode), with a class-loader-qualified ID:

```java
// AbstractTypedJacksonModule.java
private static final String TYPE_PROPERTY = "@type";
```

The type ID format from `HandleResolver`:
```java
private static String classId(Class<?> handleClass) {
    return classLoaderId(handleClass) + ":" + handleClass.getName();
}
```

This produces IDs like:
- `"system:io.trino.sql.planner.SystemPartitioningHandle"` -- for system types
- `"tpch:io.trino.plugin.tpch.TpchTableHandle"` -- for plugin types (the classLoaderId is the plugin's ID)

The `HandleJsonModule` registers all SPI interfaces:
```java
@ProvidesIntoSet
public static Module tableHandleModule(HandleResolver resolver) {
    return new AbstractTypedJacksonModule<>(ConnectorTableHandle.class, resolver::getId, resolver::getHandleClass) {};
}
// ... similarly for ColumnHandle, ConnectorSplit, ConnectorPartitioningHandle, etc.
```

## Section 3: Key Design Insights

### 3.1 Three Layers of Polymorphism

The JSON uses the `@type` discriminator property at three distinct levels, each resolved differently:

1. **PlanNode**: Compile-time `@JsonSubTypes` -- fixed set of 47 known discriminator strings.
2. **Expression IR**: Compile-time `@JsonSubTypes` -- fixed set of 18 expression types.
3. **Connector handles**: Runtime `AbstractTypedJacksonModule` with `HandleResolver` -- the `@type` value is a fully-qualified `classLoaderId:className` string like `"system:io.trino.sql.planner.SystemPartitioningHandle"`.

This means the connector handle `@type` values are NOT known at compile time -- they depend on which plugins are loaded.

### 3.2 Symbol Key Encoding is Non-Standard

When `Symbol` is used as a JSON map key, it encodes as `"<type_len>|<type_id>|<name>"` (e.g., `"6|bigint|total"`). This is a performance optimization to avoid nested JSON objects as map keys. A Rust deserializer must parse this custom format.

### 3.3 Worker-Relevant vs Planner-Only Fields

Several fields are intentionally excluded from JSON sent to workers:
- `TableScanNode.enforcedConstraint` and `statistics` are `@JsonIgnore`
- `PlanFragment.partitionedSourcesSet`, `partitionedSourceNodes`, `remoteSourceNodes`, `containsTableScanNode` are computed, not serialized

### 3.4 Optional vs Null Ambiguity

Java `Optional` fields serialize as `null` when empty. `OptionalInt` also serializes as `null` when empty. The Rust deserializer must handle both `null` and absent fields for optional values.

### 3.5 Exchange vs RemoteSource Split

During planning, cross-fragment exchanges are `ExchangeNode` with `scope = REMOTE`. When the plan is fragmented, these become `RemoteSourceNode` in the downstream fragment (referencing the upstream fragment by ID) and disappear from the upstream fragment (which terminates at the output partitioning scheme). LOCAL exchanges remain as `ExchangeNode` within a single fragment.

### 3.6 Plan Fragment is Self-Contained

Each fragment carries everything a worker needs:
- The plan tree (`root`)
- All symbol type information (`symbols`)
- Input partitioning (`partitioning`, `partitionedSources`)
- Output partitioning (`outputPartitioningScheme`)
- Required catalogs (`activeCatalogs`)
- Custom functions (`languageFunctions`)
- Cost estimates (`statsAndCosts`)

### 3.7 Recursive Tree Structure

The plan tree is deeply nested JSON. A `JoinNode` contains two complete subtrees in its `left` and `right` fields. An `ExchangeNode` can have multiple children in its `sources` array. Single-child nodes like `FilterNode`, `ProjectNode`, `SortNode`, `LimitNode`, and `TopNNode` nest their child in a `source` field.

## Section 4: Porting Considerations (Java -> Rust)

### 4.1 Enum-Based Plan Node Representation

In Rust, the 47 plan node types map naturally to a `#[serde(tag = "@type")]` enum:

```rust
#[derive(Deserialize)]
#[serde(tag = "@type")]
enum PlanNode {
    #[serde(rename = "tableScan")]
    TableScan(TableScanNode),
    #[serde(rename = "filter")]
    Filter(FilterNode),
    #[serde(rename = "project")]
    Project(ProjectNode),
    #[serde(rename = "join")]
    Join(JoinNode),
    // ... 43 more variants
}
```

Similarly for expressions (18 variants).

### 4.2 Custom Symbol Key Deserialization

The Symbol map key format (`"6|bigint|orderkey"`) requires a custom `serde` `Deserialize` implementation for `HashMap<Symbol, T>`. This can use `deserialize_map` with a custom key parser.

```rust
struct SymbolKeyVisitor;
impl<'de> Visitor<'de> for SymbolKeyVisitor {
    fn visit_str(self, key: &str) -> Result<Symbol> {
        // Parse "6|bigint|orderkey" -> Symbol { type: "bigint", name: "orderkey" }
        let pipe1 = key.find('|').unwrap();
        let len: usize = key[..pipe1].parse().unwrap();
        let type_id = &key[pipe1+1..pipe1+1+len];
        let name = &key[pipe1+2+len..];
        Ok(Symbol { type_id: type_id.into(), name: name.into() })
    }
}
```

### 4.3 Connector Handle Extensibility Problem

The hardest porting challenge is `ConnectorTableHandle`, `ColumnHandle`, `ConnectorPartitioningHandle`, and `ConnectorTransactionHandle`. In Java, these use runtime class-loader resolution with `@type` values like `"system:io.trino.plugin.tpch.TpchTableHandle"`.

A Rust worker has two options:
1. **Opaque JSON passthrough**: Deserialize connector handles as `serde_json::Value` and only interpret the system-level ones (`SystemPartitioningHandle`). Connector-specific handles are passed through opaquely to connector code.
2. **Known connector registry**: Pre-register known connector handle types (e.g., Hive, Iceberg, TPCH) and deserialize them with a manual type dispatcher.

Option 1 is strongly recommended for initial implementation since the worker only needs to understand `SystemPartitioningHandle` for partitioning decisions. All other connector handles are just passed through to connector-side code.

### 4.4 Type System Bootstrap

`Symbol` values contain Trino type IDs (e.g., `"bigint"`, `"varchar"`, `"decimal(10,2)"`). The Rust worker needs a type registry that can resolve these strings into Rust type descriptors. This mirrors the `TypeManager` that Java's `SymbolKeyDeserializer` depends on.

### 4.5 Priority of Node Types for Phase 1

For an MVP Rust worker, the following node types should be implemented first (covers SELECT...FROM...WHERE...GROUP BY...ORDER BY...LIMIT queries):

1. **TableScanNode** -- reads splits
2. **FilterNode** -- predicate evaluation
3. **ProjectNode** -- column projection and expression evaluation
4. **RemoteSourceNode** -- receives data from upstream fragments
5. **AggregationNode** -- GROUP BY with aggregation functions
6. **JoinNode** -- hash join execution
7. **LimitNode** -- row count limiting
8. **TopNNode** -- sort+limit
9. **SortNode** -- full sort
10. **ExchangeNode** (LOCAL scope only) -- local repartitioning between pipelines
11. **OutputNode** -- terminal output to coordinator
12. **ValuesNode** -- literal data

### 4.6 Optional/Null Handling

Java's `Optional<T>` serializes as `null` when empty. Java's `OptionalInt` also serializes as `null` when empty (not absent). Rust's `serde` handles this with `Option<T>` and the `#[serde(default)]` attribute. Many fields require `#[serde(deserialize_with = "...")]` for custom null handling.

### 4.7 Nested Assignments Wrapper

The `ProjectNode.assignments` field uses the `Assignments` record which wraps the map in a nested `"assignments"` object. The JSON is:
```json
{ "assignments": { "assignments": { ... } } }
```

This is not a bug -- the record has a single field named `assignments` which itself contains the map. The Rust type needs to mirror this nesting.

### 4.8 StatsAndCosts Can Be Stub-Deserialized

The `statsAndCosts` field is informational (optimizer metadata). A Rust worker can safely deserialize it into a generic map or stub struct without interpreting the detailed statistics estimates. The worker never needs to read these values during execution.
