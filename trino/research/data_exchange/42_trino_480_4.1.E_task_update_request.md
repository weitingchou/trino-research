# Module Teardown: TaskUpdateRequest Full JSON Schema (Task 4.1.E)

## 0. Research Focus
* **Task ID:** 4.1.E
* **Knowledge Graph Tag:** KG-4
* **Focus:** Document every field in the `TaskUpdateRequest` JSON payload: top-level fields, the `session` object, `SplitAssignment` structure, `outputIds`, `dynamicFilterDomains`. Determine which fields are optional vs required, which are sent only on the first update vs every update, and how `HttpRemoteTask` constructs the request.

## 1. High-Level Overview

* **Core Responsibility:** `TaskUpdateRequest` is the single JSON payload that the coordinator sends to a worker to create or update a task. It is POSTed to the worker's `POST /v1/task/{taskId}` endpoint. This one message serves as both the "create task" and "update task" command -- Trino uses an idempotent update-based protocol where the coordinator repeatedly sends the full desired state of a task, and the worker converges toward it.

* **Key Design Decisions:**
  - **Single message for create and update:** There is no separate "create task" RPC. The first `TaskUpdateRequest` carries the `PlanFragment` and creates the task. Subsequent requests omit the fragment and carry only incremental split assignments, output buffer changes, and dynamic filter domains. This is a convergent, idempotent design.
  - **Session travels with every request:** The full `SessionRepresentation` (29 fields) is included in *every* update, not just the first. This is because the session contains identity, credentials, and properties needed to reconstruct the `Session` object on the worker for every request processing cycle.
  - **Fragment is optional after first success:** The `fragment` field is `Optional<PlanFragment>`. The coordinator sets `sendPlan = true` initially and flips it to `false` after the worker's response confirms `needsPlan() == false`. If the worker restarts, it will report `needsPlan() == true` again, and the coordinator will re-send the fragment.
  - **ExtraCredentials separated from session for security:** The `extraCredentials` map is deliberately stored outside of `SessionRepresentation` "to avoid being leaked" (per the Javadoc on the record). This is a conscious security design -- the session gets logged/serialized for debugging, but credentials are kept in a separate field.
  - **No Jackson annotations on the record:** `TaskUpdateRequest` is a Java `record` with no `@JsonCreator` or `@JsonProperty` annotations. Jackson's built-in record support serializes the record components directly by their names. This is the modern Jackson approach (Jackson 2.12+).
  - **Adaptive request sizing:** The coordinator dynamically adjusts the split batch size to keep the serialized JSON payload within a configurable maximum size (`maxRequestSizeInBytes`), preventing overly large HTTP bodies.

* **Source files analyzed:**
  - `io.trino.server.TaskUpdateRequest` -- the record definition (10 fields)

## 2. Detailed Analysis

### 2.1 TaskUpdateRequest: Top-Level Record Definition

`TaskUpdateRequest` is a Java record with 10 components. Because it is a `record`, Jackson serializes each component as a JSON property using the component name directly (no `@JsonProperty` annotations needed).

```java
// io.trino.server.TaskUpdateRequest
public record TaskUpdateRequest(
        SessionRepresentation session,           // REQUIRED - always sent
        Map<String, String> extraCredentials,    // REQUIRED - always sent (may be empty map)
        Span stageSpan,                          // REQUIRED - OpenTelemetry span (custom serializer)
        Optional<PlanFragment> fragment,         // OPTIONAL - only on first update (or re-send)
        Map<PlanNodeId, ConnectorTableCredentials> tableCredentials, // REQUIRED - always sent
        List<SplitAssignment> splitAssignments,  // REQUIRED - incremental (may be empty list)
        OutputBuffers outputIds,                 // REQUIRED - always sent (version-bumped)
        Map<DynamicFilterId, Domain> dynamicFilterDomains, // REQUIRED - incremental (may be empty)
        Optional<Slice> exchangeEncryptionKey,   // OPTIONAL - for encrypted exchange
        boolean speculative)                     // REQUIRED - whether task is speculative
```

The compact constructor enforces non-nullness on all fields and makes defensive copies:

```java
public TaskUpdateRequest
{
    requireNonNull(session, "session is null");
    requireNonNull(extraCredentials, "extraCredentials is null");
    requireNonNull(stageSpan, "stageSpan is null");
    requireNonNull(fragment, "fragment is null");       // The Optional itself must not be null
    tableCredentials = ImmutableMap.copyOf(tableCredentials);
    splitAssignments = ImmutableList.copyOf(splitAssignments);
    requireNonNull(outputIds, "outputIds is null");
    dynamicFilterDomains = ImmutableMap.copyOf(dynamicFilterDomains);
    requireNonNull(exchangeEncryptionKey, "exchangeEncryptionKey is null");
}
```

**JSON property names** (derived from record component names):
`session`, `extraCredentials`, `stageSpan`, `fragment`, `tableCredentials`, `splitAssignments`, `outputIds`, `dynamicFilterDomains`, `exchangeEncryptionKey`, `speculative`.

### 2.3 SplitAssignment Structure (splitAssignments)

The `splitAssignments` field is a `List<SplitAssignment>`. Each `SplitAssignment` maps a plan node to a set of splits:

```java
// io.trino.execution.SplitAssignment
@JsonCreator
public SplitAssignment(
    @JsonProperty("planNodeId") PlanNodeId planNodeId,   // String wrapper (@JsonValue)
    @JsonProperty("splits") Set<ScheduledSplit> splits,  // Sorted by sequenceId
    @JsonProperty("noMoreSplits") boolean noMoreSplits)  // Terminal signal
```

Each `ScheduledSplit` is a record with 3 components:

```java
// io.trino.execution.ScheduledSplit
public record ScheduledSplit(
    long sequenceId,           // Globally unique monotonic ID
    PlanNodeId planNodeId,     // Redundant with parent (used for internal bookkeeping)
    Split split)               // The actual split data
```

Each `Split` contains:

```java
// io.trino.metadata.Split
@JsonCreator
public Split(
    @JsonProperty("catalogHandle") CatalogHandle catalogHandle,
    @JsonProperty("connectorSplit") ConnectorSplit connectorSplit)
```

**CatalogHandle** serializes as a composite string via `@JsonValue`:
- Format: `"{catalogName}:{type}:{version}"` (e.g., `"tpch:normal:00000000-0000-0000-0000-000000000000"`)
- `@JsonCreator` is the static `fromId(String)` method that parses this format
- `CatalogHandleType` values: `NORMAL`, `INFORMATION_SCHEMA`, `SYSTEM`

**PlanNodeId** is a simple string wrapper:
- `@JsonCreator` takes a `String id`
- `@JsonValue` returns `id` (the `toString()` method)

**ConnectorSplit** is an SPI interface -- it is deserialized polymorphically by connector-specific implementations. The serialization format depends on the connector (e.g., `TpchSplit`, `HiveSplit`, etc.).

**Conceptual JSON for a split assignment:**

```json
{
  "planNodeId": "0",
  "splits": [
    {
      "sequenceId": 42,
      "planNodeId": "0",
      "split": {
        "catalogHandle": "hive:normal:v1",
        "connectorSplit": { /* connector-specific fields */ }
      }
    }
  ],
  "noMoreSplits": false
}
```

### 2.4 Fragment Field (Optional PlanFragment)

The `fragment` field is `Optional<PlanFragment>`. It is only sent when the task needs the plan (first update, or after worker restart).

`PlanFragment` is a large class with `@JsonCreator`/`@JsonProperty` annotations, covered separately in Task 4.1.F (KG-6). For this document, the key observation is:

- When present, the fragment contains the full execution plan tree (`PlanNode root`), output partitioning scheme, partition sources, remote sources, stats, catalog properties, language functions, and more.
- When the coordinator sends the fragment, it calls `planFragment.withoutEmbeddedJsonRepresentation()` to strip the redundant embedded JSON string (which is used only for UI display on the coordinator), reducing payload size.

### 2.5 OutputBuffers (outputIds)

`OutputBuffers` is a **polymorphic abstract class** using Jackson `@JsonTypeInfo`:

```java
@JsonTypeInfo(use = JsonTypeInfo.Id.NAME, property = "@type")
@JsonSubTypes({
    @JsonSubTypes.Type(value = PipelinedOutputBuffers.class, name = "pipelined"),
    @JsonSubTypes.Type(value = SpoolingOutputBuffers.class, name = "spool"),
})
public abstract class OutputBuffers {
    private final long version;     // Monotonically increasing version number
}
```

**PipelinedOutputBuffers** (for streaming, non-fault-tolerant execution):

```java
@JsonCreator
public PipelinedOutputBuffers(
    @JsonProperty("version") long version,
    @JsonProperty("type") BufferType type,          // PARTITIONED, BROADCAST, or ARBITRARY
    @JsonProperty("buffers") Map<OutputBufferId, Integer> buffers,  // buffer ID -> partition
    @JsonProperty("noMoreBufferIds") boolean noMoreBufferIds)
```

- `OutputBufferId` is a record wrapping an `int` with `@JsonValue`/`@JsonCreator`.
- `BufferType` is an enum: `PARTITIONED`, `BROADCAST`, `ARBITRARY`.

**SpoolingOutputBuffers** (for fault-tolerant execution with external exchange):

```java
@JsonCreator
public SpoolingOutputBuffers(
    @JsonProperty("version") long version,
    @JsonProperty("exchangeSinkInstanceHandle") ExchangeSinkInstanceHandle handle,
    @JsonProperty("outputPartitionCount") int outputPartitionCount)
```

**Conceptual JSON for pipelined output buffers:**

```json
{
  "@type": "pipelined",
  "version": 3,
  "type": "PARTITIONED",
  "buffers": { "0": 0, "1": 1, "2": 2 },
  "noMoreBufferIds": true
}
```

### 2.6 DynamicFilterDomains

The `dynamicFilterDomains` field is a `Map<DynamicFilterId, Domain>`.

**DynamicFilterId** is a simple string wrapper:
- `@JsonCreator` takes a `String id`
- `@JsonValue` returns `id` (the `toString()` method)

**Domain** (from `io.trino.spi.predicate`) represents a set of values for a column, used to filter data at scan time. It contains a `ValueSet` and a `boolean nullAllowed` field. Domain is a complex type with its own nested serialization hierarchy (covered in SPI documentation).

The coordinator collects dynamic filter domains from completed build-side tasks and pushes them to probe-side tasks via this field. The field is empty on most updates and only populated when new dynamic filter information is available.

### 2.7 TableCredentials

The `tableCredentials` field is a `Map<PlanNodeId, ConnectorTableCredentials>`.

`ConnectorTableCredentials` is an SPI **marker interface** with no methods:

```java
// io.trino.spi.connector.ConnectorTableCredentials
public interface ConnectorTableCredentials {}
```

Concrete implementations are provided by connectors. This field carries per-table access credentials (e.g., delegated tokens for cloud storage) keyed by the plan node that scans that table.

### 2.8 ExchangeEncryptionKey and Speculative Flag

- **`exchangeEncryptionKey`:** `Optional<Slice>` -- a binary encryption key for encrypted data exchange between workers. When present, the `Slice` is serialized as a base64-encoded byte array. This key is used to encrypt/decrypt pages in transit.

- **`speculative`:** `boolean` -- indicates whether this task is a speculative execution attempt (used in fault-tolerant execution mode). Speculative tasks may be canceled if the original task succeeds first.

### 2.9 First Update vs Subsequent Updates (HttpRemoteTask Construction Logic)

The coordinator's `HttpRemoteTask.sendUpdateInternal()` method constructs the `TaskUpdateRequest`. The key logic for first-vs-subsequent updates:

```java
// io.trino.server.remotetask.HttpRemoteTask
private final AtomicBoolean sendPlan = new AtomicBoolean(true);  // Start true

private void sendUpdateInternal() {
    // ...
    // Fragment is included only when sendPlan is true
    Optional<PlanFragment> fragment = sendPlan.get()
        ? Optional.of(planFragment.withoutEmbeddedJsonRepresentation())
        : Optional.empty();

    TaskUpdateRequest updateRequest = new TaskUpdateRequest(
            session.toSessionRepresentation(),        // ALWAYS sent
            session.getIdentity().getExtraCredentials(), // ALWAYS sent
            stageSpan,                                 // ALWAYS sent
            fragment,                                  // First update only (usually)
            tableCredentials,                          // ALWAYS sent
            splitAssignments,                          // Incremental batches
            outputBuffers.get(),                       // ALWAYS sent (latest version)
            dynamicFilterDomains.getDynamicFilterDomains(), // Incremental
            session.getExchangeEncryptionKey(),        // ALWAYS sent
            speculative.get());                        // ALWAYS sent
    // ...
}
```

After a successful response, the coordinator checks the worker's `TaskInfo.needsPlan()`:

```java
// In UpdateResponseHandler.success():
sendPlan.set(value.needsPlan());
```

On the worker side, `SqlTask.needsPlan` starts as `true` and is set to `false` after the `TaskExecution` is created from the fragment:

```java
// io.trino.execution.SqlTask
private final AtomicBoolean needsPlan = new AtomicBoolean(true);
// After creating TaskExecution with the fragment:
needsPlan.set(false);
```

**Summary of what varies between updates:**

| Field | First Update | Subsequent Updates |
|---|---|---|
| `session` | Full session | Full session (same) |
| `extraCredentials` | Credentials map | Credentials map (same) |
| `stageSpan` | Stage span | Stage span (same) |
| `fragment` | `Optional.of(planFragment)` | `Optional.empty()` |
| `tableCredentials` | Full credentials map | Full credentials map (same) |
| `splitAssignments` | Initial splits (may be batched) | New splits + noMoreSplits signals |
| `outputIds` | Initial output buffer config | Updated version (buffer additions, noMoreBufferIds) |
| `dynamicFilterDomains` | Usually empty | Populated as build-side filters complete |
| `exchangeEncryptionKey` | Encryption key (if enabled) | Same key |
| `speculative` | Initial speculative flag | May change to false via `setSpeculative(false)` |

### 2.10 Worker-Side Request Processing

The worker receives the request at `TaskResource.createOrUpdateTask()`:

```java
// io.trino.server.TaskResource
@POST
@Path("{taskId}")
@Consumes(MediaType.APPLICATION_JSON)
@Produces(MediaType.APPLICATION_JSON)
public void createOrUpdateTask(
        @PathParam("taskId") TaskId taskId,
        TaskUpdateRequest taskUpdateRequest,
        @Context UriInfo uriInfo,
        @Suspended AsyncResponse asyncResponse)
{
    // Reconstruct full Session from SessionRepresentation + extraCredentials + encryptionKey
    Session session = taskUpdateRequest.session().toSession(
            sessionPropertyManager,
            taskUpdateRequest.extraCredentials(),
            taskUpdateRequest.exchangeEncryptionKey());

    // Delegate to SqlTaskManager.updateTask
    TaskInfo taskInfo = taskManager.updateTask(
            session, taskId,
            taskUpdateRequest.stageSpan(),
            taskUpdateRequest.fragment(),
            taskUpdateRequest.tableCredentials(),
            taskUpdateRequest.splitAssignments(),
            taskUpdateRequest.outputIds(),
            taskUpdateRequest.dynamicFilterDomains(),
            taskUpdateRequest.speculative());

    asyncResponse.resume(Response.ok().entity(taskInfo).build());
}
```

The response is a `TaskInfo` object, which contains `needsPlan` to tell the coordinator whether to re-send the fragment on the next update.

## 3. Key Design Insights

1. **Convergent idempotent protocol:** The coordinator does not distinguish between "create" and "update" on the wire. It sends the same `TaskUpdateRequest` shape for both. The worker's `SqlTask.updateTask()` creates the task if it does not exist and updates it if it does. The `needsPlan` flag in the response closes the feedback loop so the coordinator knows when to stop sending the fragment. This makes error recovery trivial -- on any failure, the coordinator just re-sends the full desired state.

2. **Session-per-request architecture:** The full session (29+ fields) travels with every request, not just the first. This is deliberate -- the worker does not cache the session across HTTP requests. It reconstructs the `Session` object fresh from `SessionRepresentation.toSession()` on every update. This makes the worker stateless with respect to session state and simplifies recovery, at the cost of larger payloads.

3. **Security-conscious credential separation:** `extraCredentials` is kept outside `SessionRepresentation` explicitly to prevent accidental logging or serialization leakage. The `toString()` method of `TaskUpdateRequest` redacts `exchangeEncryptionKey` and only shows `extraCredentials` keys (not values). This is a conscious security pattern.

4. **Adaptive payload sizing:** The coordinator dynamically adjusts the number of splits per request (`splitBatchSize`) to keep the serialized JSON body below `maxRequestSizeInBytes`. When the serialized payload is too large, it reduces the batch size and re-schedules the update. This prevents OOM on workers from overly large request bodies and adapts to varying split sizes across connectors.

5. **Polymorphic OutputBuffers:** The `@JsonTypeInfo`/`@JsonSubTypes` pattern on `OutputBuffers` with the `"@type"` discriminator field is a standard Jackson polymorphic pattern. The two concrete types (`pipelined` and `spool`) correspond to the two execution modes: streaming (non-FTE) and fault-tolerant (FTE).

6. **Sequence IDs for split ordering:** `ScheduledSplit.sequenceId` is a monotonically increasing long assigned by `HttpRemoteTask.nextSplitId`. The `SplitAssignment` constructor sorts splits by sequence ID. Equality on `ScheduledSplit` is defined solely by `sequenceId`, not by split content. This provides deterministic ordering and deduplication.

7. **No explicit versioning on the request itself:** Unlike `OutputBuffers` (which has a `version` field) and dynamic filters (which use `VersionedDynamicFilterDomains` internally), the `TaskUpdateRequest` itself has no version number. The coordinator simply sends the latest state every time. The worker uses the split sequence IDs and output buffer version to determine what is new.

## 4. Porting Considerations (Java -> Rust)

### 4.1 Struct Definition

The `TaskUpdateRequest` maps to a Rust struct with `#[derive(Serialize, Deserialize)]`:

```rust
#[derive(Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct TaskUpdateRequest {
    pub session: SessionRepresentation,
    pub extra_credentials: HashMap<String, String>,
    pub stage_span: SpanContext,     // OpenTelemetry SpanContext
    pub fragment: Option<PlanFragment>,
    pub table_credentials: HashMap<PlanNodeId, ConnectorTableCredentials>,
    pub split_assignments: Vec<SplitAssignment>,
    pub output_ids: OutputBuffers,   // enum with tagged union
    pub dynamic_filter_domains: HashMap<DynamicFilterId, Domain>,
    pub exchange_encryption_key: Option<Vec<u8>>,  // Slice -> Vec<u8>
    pub speculative: bool,
}
```

### 4.2 SessionRepresentation Mapping

The session has 29+ fields. Key porting considerations:

- **`Optional<T>` -> `Option<T>`:** Many fields are Java `Optional`. In JSON these serialize as `null` when absent. Use `#[serde(default)]` or `Option<T>` with `#[serde(deserialize_with = ...)]` to handle null values.
- **`TimeZoneKey`:** Java uses a short numeric key with a lookup table. In Rust, use `chrono_tz::Tz` or a custom type. The wire format is a short integer.
- **`Locale`:** Java's `Locale` serializes as a string like `"en_US"`. Use a string in Rust.
- **`Instant`:** Serializes as an epoch timestamp. Use `chrono::DateTime<Utc>` or similar.
- **`SqlPath`:** Contains `path: Vec<CatalogSchemaName>` and `rawPath: String`.
- **`SelectedRole`:** Enum with `ROLE`/`ALL`/`NONE` + optional role name.
- **JSON property name mismatch:** The field `originalRoles` has `@JsonProperty("setOriginalRoles")` -- the JSON key differs from the Java field name. The Rust struct must use `#[serde(rename = "setOriginalRoles")]`.
- **Derived property `timeZone`:** The getter `getTimeZone()` is also annotated `@JsonProperty`, adding a derived string to the JSON output. This is not a constructor parameter and is ignored during deserialization. In Rust, use `#[serde(skip_deserializing)]` with a custom serialization function if needed.

### 4.3 Polymorphic OutputBuffers

Jackson uses `"@type": "pipelined"` or `"@type": "spool"` as a discriminator. In Rust, use `#[serde(tag = "@type")]`:

```rust
#[derive(Serialize, Deserialize)]
#[serde(tag = "@type")]
pub enum OutputBuffers {
    #[serde(rename = "pipelined")]
    Pipelined(PipelinedOutputBuffers),
    #[serde(rename = "spool")]
    Spool(SpoolingOutputBuffers),
}
```

### 4.4 ConnectorSplit Polymorphism

`ConnectorSplit` is an SPI interface with connector-specific implementations. This is the hardest part to port because Jackson uses class metadata for deserialization. Options:
- Use `serde_json::Value` as a pass-through for the initial port (defer full deserialization).
- Implement a connector registry that maps type names to deserialization functions.
- Since a Rust worker will likely only support specific connectors, hardcode the known split types.

### 4.5 Span Serialization

`io.opentelemetry.api.trace.Span` uses custom Trino serializers (`SpanSerializer`/`SpanDeserializer`). The wire format needs to be reverse-engineered from these classes. In Rust, use the `opentelemetry` crate's `SpanContext` or serialize as a trace ID + span ID pair.

### 4.6 Key Challenges

1. **Session reconstruction per request:** The Java worker creates a fresh `Session` from `SessionRepresentation` on every request. The Rust worker should do the same -- do not cache sessions across requests. This simplifies state management.

2. **`needsPlan` feedback loop:** The Rust worker must implement the `needsPlan` flag in its `TaskInfo` response so the coordinator knows when to stop sending the plan fragment. This is critical for payload size reduction.

3. **Adaptive batch sizing is coordinator-side only:** The split batching logic lives entirely in `HttpRemoteTask` on the coordinator. The Rust worker does not need to implement this -- it just receives whatever splits the coordinator sends.

4. **Encryption key as `Slice`:** Trino's `Slice` is an off-heap memory abstraction. On the wire it is just a byte array (base64 in JSON). Map to `Vec<u8>` in Rust.

5. **Map key serialization:** Several maps use custom types as keys (`PlanNodeId`, `DynamicFilterId`, `OutputBufferId`). These all serialize as simple strings or integers via `@JsonValue`. In Rust, implement `Serialize`/`Deserialize` for the key types so they work as `HashMap` keys.

6. **Empty vs absent:** All collection fields (`splitAssignments`, `dynamicFilterDomains`, `tableCredentials`) are always present in the JSON, even when empty. They are never `null`. Use default empty collections in the Rust struct.
