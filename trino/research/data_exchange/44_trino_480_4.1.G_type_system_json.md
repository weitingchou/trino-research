# Task 4.1.G: Type System JSON Serialization [KG-8]

**Source Version:** Trino 480
**Date:** 2026-04-06

---

## Section 1: High-Level Overview

Trino's SQL type system is serialized to JSON as **flat strings** -- not structured JSON objects. Every type, from simple `boolean` to complex `row("x" bigint,"y" varchar(256))`, is encoded as a single string that can be parsed back by the SQL parser. This is a critical design decision that greatly simplifies the JSON wire format but pushes parsing complexity to the deserialization side.

### Serialization Architecture Summary

```
Type (interface)
  |
  +-- @JsonValue getTypeId() --> TypeId
                                   |
                                   +-- @JsonValue getId() --> String
                                                                |
                                                                v
                                          "varchar(256)"  (flat string)
```

The chain is:
1. `Type.getTypeId()` (annotated `@JsonValue`) returns a `TypeId`
2. `TypeId.getId()` (annotated `@JsonValue`) returns a `String`
3. That string is `TypeSignature.toString()` -- a human-readable SQL-like representation

For deserialization, Jackson uses custom deserializers:
- `TypeDeserializer` (extends `FromStringDeserializer<Type>`) -- reads the string, creates `TypeId`, resolves via `TypeManager`
- `TypeSignatureDeserializer` (extends `FromStringDeserializer<TypeSignature>`) -- parses the string via `TypeSignatureTranslator.parseTypeSignature()`
- `TypeSignatureKeyDeserializer` (extends `KeyDeserializer`) -- same parsing, used when `TypeSignature` is a JSON map key

### Where Types Appear in Plan Fragment JSON

Types appear in three contexts within the plan fragment JSON:

1. **Symbol keys** -- `Symbol(Type type, String name)` is encoded as `"<typeLen>|<typeStr>|<name>"` when used as a map key (e.g., `"7|varchar|col_a"`)
2. **Symbol values** -- When serialized as a record (e.g., in `PlanFragment.symbols`), the `type` field is a string via `Type.@JsonValue`
3. **Expression IR** -- Every `Expression` has a `type()` field serialized as a string (e.g., in `Cast.type`, `Reference.type`, `Constant.type`)

## Section 2: Detailed Analysis

### 2.1 The TypeSignature Core

`TypeSignature` is the structural representation of a type. It has a `base` string and a list of `TypeParameter` values.

**Source:** `core/trino-spi/src/main/java/io/trino/spi/type/TypeSignature.java`

```java
public final class TypeSignature
{
    private final String base;
    private final List<TypeParameter> parameters;
    private final boolean calculated;

    @JsonValue
    public String jsonValue()
    {
        return formatValue(true);
    }

    @Override
    public String toString()
    {
        return formatValue(false);
    }
}
```

The `TypeParameter` sealed interface has three variants:

**Source:** `core/trino-spi/src/main/java/io/trino/spi/type/TypeParameter.java`

```java
public sealed interface TypeParameter
        permits TypeParameter.Type, TypeParameter.Numeric, TypeParameter.Variable
{
    record Type(Optional<String> name, TypeSignature type) implements TypeParameter { ... }
    record Numeric(long value) implements TypeParameter { ... }
    record Variable(String name) implements TypeParameter { ... }
}
```

- `Numeric` -- for precision/scale/length parameters (e.g., `DECIMAL(38, 18)`)
- `Type` -- for nested type parameters with optional field names (e.g., `ARRAY(bigint)`, `ROW("x" integer)`)
- `Variable` -- only used in function signature resolution, not in wire format

### 2.2 The formatValue() Method -- String Assembly

This is the core serialization logic:

**Source:** `core/trino-spi/src/main/java/io/trino/spi/type/TypeSignature.java`

```java
private String formatValue(boolean json)
{
    if (parameters.isEmpty()) {
        return base;
    }

    // Unbounded VARCHAR is serialized as just "varchar" (no parameters)
    if (base.equalsIgnoreCase(StandardTypes.VARCHAR) &&
            (parameters.size() == 1) &&
            parameters.get(0) instanceof TypeParameter.Numeric(long length) &&
            length == VarcharType.UNBOUNDED_LENGTH) {
        return base;
    }

    // Timestamp/time with time zone get special formatting
    if (base.equalsIgnoreCase(TIMESTAMP_WITH_TIME_ZONE)) {
        return format("timestamp(%s) with time zone", parameters.get(0));
    }
    if (base.equalsIgnoreCase(TIMESTAMP_WITHOUT_TIME_ZONE)) {
        return format("timestamp(%s) without time zone", parameters.get(0));
    }
    if (base.equalsIgnoreCase(TIME_WITH_TIME_ZONE)) {
        return format("time(%s) with time zone", parameters.get(0));
    }

    // Generic: "base(param0,param1,...)"
    StringBuilder typeName = new StringBuilder(base);
    typeName.append("(").append(json ? parameters.get(0).jsonValue() : parameters.get(0).toString());
    for (int i = 1; i < parameters.size(); i++) {
        typeName.append(",").append(json ? parameters.get(i).jsonValue() : parameters.get(i).toString());
    }
    typeName.append(")");
    return typeName.toString();
}
```

**Key observation:** For `TypeParameter.Numeric`, `jsonValue()` and `toString()` are identical. The `json` flag only matters for `TypeParameter.Variable` (which prefixes `@`), and these never appear in plan fragment wire format. So in practice, `toString()` and `jsonValue()` produce the same output for all types that appear on the wire.

**Special case -- unbounded VARCHAR:** When `VarcharType.UNBOUNDED_LENGTH` (`Integer.MAX_VALUE`) is the numeric parameter, `formatValue()` strips it and returns bare `"varchar"`. Only bounded VARCHAR shows the length: `"varchar(256)"`.

**Special case -- timestamp/time with time zone:** The base name includes spaces (`"timestamp with time zone"`), but `formatValue()` restructures it to SQL syntax: `"timestamp(3) with time zone"` (precision inserted between "timestamp" and "with time zone").

```java
{
```

- Mixed: `row("name" varchar,integer)`

### 2.5 DECIMAL Parameter Encoding

DECIMAL parameters are encoded **inline** in the signature string as comma-separated numeric parameters -- NOT as separate JSON fields.

**Source:** `core/trino-spi/src/main/java/io/trino/spi/type/DecimalType.java`

```java
DecimalType(int precision, int scale, Class<?> javaType, Class<? extends ValueBlock> valueBlockType)
{
    super(new TypeSignature(NAME, buildTypeParameters(precision, scale)), javaType, valueBlockType);
}

private static List<TypeParameter> buildTypeParameters(int precision, int scale)
{
    return List.of(numericParameter(precision), numericParameter(scale));
}
```

`formatValue()` produces: `"decimal(38,18)"` -- both precision and scale are `Numeric` parameters appended in order. There is no structured JSON like `{"precision": 38, "scale": 18}`.

### 2.6 Symbol Serialization -- Type Embedded in Key

When `Symbol` is used as a JSON map key (the most common case -- in `assignments`, `outputLayout`, etc.), it uses a custom format that embeds the type.

**Source:** `core/trino-main/src/main/java/io/trino/sql/planner/SymbolKeySerializer.java`

```java
public void serialize(Symbol value, JsonGenerator generator, SerializerProvider serializers)
        throws IOException
{
    String type = value.type().getTypeId().getId();
    generator.writeFieldName(type.length() + "|" + type + "|" + value.name());
}
```

Format: `<typeStringLength>|<typeString>|<symbolName>`

Examples:
- `"6|bigint|col_a"` -- a BIGINT column named `col_a`
- `"12|varchar(256)|col_b"` -- a VARCHAR(256) column named `col_b`
- `"25|timestamp(3) with time zone|ts"` -- a TIMESTAMP(3) WITH TIME ZONE column named `ts`
- `"28|row("x" bigint,"y" integer)|r"` -- a ROW column named `r`

**Source:** `core/trino-main/src/main/java/io/trino/sql/planner/SymbolKeyDeserializer.java`

```java
public Object deserializeKey(String key, DeserializationContext context)
{
    int lastDigitIndex = getLeadingDigitsLength(key, keyLength);
    int length = parseInt(key.substring(0, lastDigitIndex));
    String type = key.substring(lastDigitIndex + 1, lastDigitIndex + length + 1);
    String name = key.substring(lastDigitIndex + length + 2);
    return new Symbol(typeManager.getType(TypeId.of(type)), name);
}
```

The length prefix is necessary because type strings can contain `|` characters (they cannot, but more importantly they can contain any character not in `<>,`). The length-prefixed format avoids ambiguity with symbol names that might contain `|`.

**Source:** `core/trino-main/src/main/java/io/trino/metadata/TypeRegistry.java`

```java
{
    // TODO: ID should be encoded in a more canonical form than SQL
    return fromSqlType(id.getId());
}

{
    return sqlTypeCache.get(sqlType, () ->
        getType(toTypeSignature(SQL_PARSER.createType(sqlType))));

```

The deserialization path is: JSON string -> `TypeId.of(string)` -> `TypeRegistry.getType(TypeId)` -> `fromSqlType(string)` -> **SQL parser** parses the string as a `DataType` AST node -> `TypeSignatureTranslator.toTypeSignature()` converts to `TypeSignature` -> `TypeRegistry.getType(TypeSignature)` looks up or instantiates the type.

This means the wire format must be parseable by Trino's SQL type grammar.

### 2.9 Types in PlanNode Fields

Types do NOT appear as explicit fields on most PlanNodes. Instead, types are carried indirectly through:

1. **Symbols** -- `TableScanNode.outputSymbols` is `List<Symbol>` where each `Symbol` contains its type. When symbols appear as map keys (e.g., `TableScanNode.assignments: Map<Symbol, ColumnHandle>`), the type is embedded in the key string via the `SymbolKeySerializer` format.

2. **Expressions** -- Filter predicates, projection expressions, join conditions, etc. all carry types via the `Expression.type()` field.

3. **PlanFragment.symbols** -- The top-level `Set<Symbol>` in the fragment records all symbols used in the plan, each with their type.

There is no separate "type catalog" or type mapping table in the JSON. Each occurrence of a type is self-contained.

## Section 3: Key Design Insights

### 3.1 Flat String Serialization is Deliberate

The entire type system serializes to flat SQL-parseable strings rather than structured JSON objects. This was a deliberate choice:
- `TypeId` is described as "an opaque identifier for a Type that can be used for serialization or storage in external systems"
- The TODO comment in `TypeRegistry.getType(TypeId)` says `"ID should be encoded in a more canonical form than SQL"` -- they acknowledge it is a SQL string but have kept it this way
- The string format must round-trip through the SQL parser (`SqlParser.createType()`)

### 3.2 TypeSignature vs TypeId: Two Levels of String

- `TypeSignature` is the structural in-memory representation (base + parameters). Its `@JsonValue jsonValue()` method produces a string for direct serialization.
- `TypeId` is a wrapper around a string. `Type.getTypeId()` uses `TypeSignature.toString()` (NOT `jsonValue()`).
- For all types that appear on the wire (no `Variable` parameters), `toString()` and `jsonValue()` produce identical output.

### 3.3 Timestamp/Time With Time Zone: Non-Uniform Formatting

Most parameterized types use the generic `"base(param,param)"` format. But timestamp and time with time zone types get special SQL-like formatting: `"timestamp(3) with time zone"` rather than `"timestamp with time zone(3)"`. The base name contains spaces, and the precision is spliced into the middle. This requires special handling in any parser.

### 3.4 VARCHAR Unbounded is Special-Cased

Unbounded `VARCHAR` (length = `Integer.MAX_VALUE`) serializes as bare `"varchar"` -- the parameter is stripped. Bounded `VARCHAR(n)` serializes as `"varchar(256)"`. A deserializer must handle bare `"varchar"` by defaulting to unbounded length.

### 3.5 Symbol Key Format Uses Length Prefixing

The `SymbolKeySerializer` format `"<len>|<typeStr>|<name>"` uses a length prefix on the type string, not a simple split-on-pipe. This is necessary because the format must unambiguously separate the type string from the symbol name, even though in practice neither currently contains `|`.

### 3.6 No Schema-Level Type Registry in JSON

The plan fragment JSON contains no separate "types" section. Every type reference is independently encoded at the point of use. The same type string (e.g., `"varchar(256)"`) may appear dozens of times in a single fragment -- once per symbol key, once per expression, etc. There is no deduplication or indirection.

## Section 4: Porting Considerations (Java -> Rust)

### 4.1 Type String Parser is Required

A Rust worker must implement a parser for Trino type strings. This parser must handle:
- Simple base types: `"boolean"`, `"bigint"`, `"date"`, etc.
- Parameterized types with numeric params: `"varchar(256)"`, `"char(10)"`, `"decimal(38,18)"`
- Parameterized types with type params: `"array(bigint)"`, `"map(varchar,integer)"`
- ROW with named fields: `row("x" bigint,"y" varchar(256))` -- field names in double quotes, escaping via `""` for embedded quotes
- Timestamp/time special format: `"timestamp(3) with time zone"`, `"time(6) with time zone"`
- Interval types with spaces: `"interval year to month"`, `"interval day to second"`
- Nested composition: `"map(varchar,array(row("id" bigint,"name" varchar)))"` -- arbitrary nesting depth

A recursive descent parser or PEG grammar would be appropriate. The grammar is essentially:

```
type_string  := base_type
              | base_type "(" params ")"
              | timestamp_tz_type
              | time_tz_type
              | interval_type

base_type    := identifier (case-insensitive, may contain spaces for interval/timestamp with time zone)
params       := param ("," param)*
param        := number | type_string | '"' field_name '"' type_string
timestamp_tz := "timestamp(" number ") with time zone"
time_tz      := "time(" number ") with time zone"
interval     := "interval year to month" | "interval day to second"
```

### 4.2 Symbol Key Deserializer

Parsing the `"<len>|<typeStr>|<name>"` format:

```rust
fn parse_symbol_key(key: &str) -> (TypeString, SymbolName) {
    // Read leading digits for length
    let pipe1 = key.find('|').unwrap();
    let type_len: usize = key[..pipe1].parse().unwrap();
    let type_str = &key[pipe1 + 1..pipe1 + 1 + type_len];
    let name = &key[pipe1 + 1 + type_len + 1..];
    (type_str.into(), name.into())
}
```

### 4.5 Critical Format Details to Get Right

1. **No spaces after commas** in `formatValue()`: `"decimal(38,18)"` not `"decimal(38, 18)"`. The `StringBuilder` in Trino appends `,` with no space.

2. **Lowercase base names**: All standard type names are lowercase. Case-insensitive matching on input (Trino compares with `equalsIgnoreCase`), but canonical output is lowercase.

3. **ROW field name quoting**: Named fields use `"double-quoted"` names. Embedded double quotes are escaped as `""`. Anonymous fields have no quotes or name prefix.

4. **VARCHAR unbounded**: Must serialize as bare `"varchar"`, must deserialize bare `"varchar"` as unbounded. `UNBOUNDED_LENGTH = Integer.MAX_VALUE = 2147483647`.

5. **Timestamp precision placement**: `"timestamp(3) with time zone"` -- precision goes BETWEEN "timestamp" and "with time zone", not after. This is a SQL standard format, not the internal base name format.
