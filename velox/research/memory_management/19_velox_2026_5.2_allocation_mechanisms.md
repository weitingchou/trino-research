# Module Teardown: Velox Allocation Mechanisms

## 0. Research Focus
* **Task ID:** 5.2
* **Focus:** Trace the actual byte allocation mechanisms. How does Velox differentiate between contiguous allocations and non-contiguous `Allocation` chunks? How are these chunks provided to `Buffer`s?

## 1. High-Level Overview

Velox's allocation subsystem is a two-tier architecture: a `MemoryPool` tree provides accounting and quota enforcement, while an underlying `MemoryAllocator` performs the actual byte-level work. The allocator has two production implementations -- `MallocAllocator` (delegates to `std::malloc`) and `MmapAllocator` (manages its own address space via `mmap`/`madvise`). Both allocators expose the same three allocation modes: (1) non-contiguous multi-run `Allocation`s composed of page-aligned runs from predefined size classes, (2) contiguous `ContiguousAllocation`s backed by a single `mmap` region, and (3) raw byte allocations via `allocateBytes()`. This design allows all large memory consumers -- hash tables, IO buffers, caches, row containers -- to draw from a single capacity-bounded source, enabling dynamic balancing between cache and compute memory. The `Allocation` object itself uses a compact `PageRun` encoding that packs a pointer and page count into a single 64-bit word, achieving zero overhead per run.

## 2. Structural Architecture

### 2.1 Core Data Structures

**`AllocationTraits` -- Page Size Constants**
(`velox/common/memory/Allocation.h`)

```cpp
struct AllocationTraits {
  static constexpr uint64_t kPageSize = 4096;
  static constexpr uint64_t kHugePageSize = 2 << 20; // 2MB

  FOLLY_ALWAYS_INLINE static uint64_t pageBytes(MachinePageCount numPages) {
    return numPages * kPageSize;
  }
  FOLLY_ALWAYS_INLINE static MachinePageCount numPages(uint64_t bytes) {
    return bits::roundUp(bytes, kPageSize) / kPageSize;
  }
};
```

All page arithmetic in Velox is based on 4KB machine pages. This is the fundamental unit of allocation.

**`Allocation::PageRun` -- Compact Run Encoding**
(`velox/common/memory/Allocation.h`)

```cpp
class PageRun {
 public:
  static constexpr uint8_t kPointerSignificantBits = 48;
  static constexpr uint64_t kPointerMask = 0xffffffffffff;
  static constexpr uint32_t kMaxPagesInRun =
      (1UL << (64U - kPointerSignificantBits)) - 1;

  PageRun(void* address, MachinePageCount numPages) {
    auto word = reinterpret_cast<uint64_t>(address);
    data_ = word | (static_cast<uint64_t>(numPages) << kPointerSignificantBits);
  }

  template <typename T = uint8_t>
  T* data() const {
    return reinterpret_cast<T*>(data_ & kPointerMask);
  }

  MachinePageCount numPages() const {
    return data_ >> kPointerSignificantBits;
  }

 private:
  uint64_t data_;
};
```

This is a key design choice: on x86-64, virtual addresses use only the lower 48 bits, so the upper 16 bits are available for metadata. Velox packs the page count into those bits, encoding both pointer and size in a single 8-byte word. This means each `PageRun` costs exactly 8 bytes, and the maximum run size is `2^16 - 1 = 65535` pages (about 256MB).

**`Allocation` -- Non-Contiguous Multi-Run Collection**
(`velox/common/memory/Allocation.h`)

```cpp
class Allocation {
  MemoryPool* pool_{nullptr};
  std::vector<PageRun> runs_;
  int32_t numPages_ = 0;

  void append(uint8_t* address, MachinePageCount numPages);
  void findRun(uint64_t offset, int32_t* index, int32_t* offsetInRun) const;
};
```

A non-contiguous allocation is simply a vector of `PageRun`s. Each run is a contiguous chunk at a size-class boundary; the overall allocation is the union of all runs. To find a byte at offset N, `findRun()` performs a linear scan over runs:

```cpp
void Allocation::findRun(uint64_t offset, int32_t* index, int32_t* offsetInRun) const {
  uint64_t skipped = 0;
  for (int32_t i = 0; i < runs_.size(); ++i) {
    uint64_t size = AllocationTraits::pageBytes(runs_[i].numPages());
    if (offset - skipped < size) {
      *index = i;
      *offsetInRun = static_cast<int32_t>(offset - skipped);
      return;
    }
    skipped += size;
  }
}
```

**`ContiguousAllocation` -- Single mmap Region**
(`velox/common/memory/Allocation.h`)

```cpp
class ContiguousAllocation {
  MemoryPool* pool_{nullptr};
  void* data_{nullptr};
  uint64_t size_{0};     // committed/reserved portion
  uint64_t maxSize_{0};  // total mmap'd address space
};
```

A contiguous allocation wraps a single `mmap`'d region. The `maxSize_` vs `size_` distinction supports incremental growth: the system maps a large address range upfront (via `maxPages`) but only counts `numPages` against the capacity budget. The `growContiguous()` method increases `size_` toward `maxSize_` without remapping, which is important for hash table resizing with huge pages.

**Size Classes:**
(`velox/common/memory/MemoryAllocator.h`)

```cpp
const std::vector<MachinePageCount>
    sizeClassSizes_{1, 2, 4, 8, 16, 32, 64, 128, 256};
```

Powers of 2 from 4KB (1 page) to 1MB (256 pages). The `MmapAllocator` pre-maps the entire capacity for each size class at construction time, so any combination of sizes that adds up to capacity can be allocated without fragmentation.

**`SizeMix` -- Allocation Recipe**
(`velox/common/memory/MemoryAllocator.h`)

```cpp
struct SizeMix {
  std::vector<int32_t> sizeIndices;   // index into sizeClassSizes_
  std::vector<int32_t> sizeCounts;    // count per class
  int32_t numSizes{0};
  int32_t totalPages{0};
};
```

Computed by `allocationSize()`, the `SizeMix` determines how to decompose an N-page request into size-class runs. For example, 11 pages -> 1x8-page run + 1x2-page run + 1x1-page run.

### 2.3 MmapAllocator::SizeClass -- Per-Class Virtual Region

(`velox/common/memory/MmapAllocator.h`)

Each size class pre-maps the full capacity and tracks allocation/mapping state via bitmaps:

```cpp
class SizeClass {
  const uint64_t capacity_;           // number of size-class pages
  const MachinePageCount unitSize_;   // pages per unit (e.g., 8 for the 8-page class)
  const size_t byteSize_;             // total address range bytes
  uint8_t* address_;                  // start of mmap'd region

  std::vector<uint64_t> pageAllocated_;       // 1 = allocated
  std::vector<uint64_t> pageMapped_;          // 1 = backed by physical memory
  std::vector<uint64_t> mappedFreeLookup_;    // summary bits for SIMD scan
  ClassPageCount numMappedFreePages_ = 0;     // count of free+mapped pages
  int32_t clockHand_ = 0;                     // allocation sweep cursor
};
```

The two-bitmap design (`pageAllocated_` and `pageMapped_`) is fundamental. A page can be in four states:
- allocated+mapped: in use, backed by physical memory
- not-allocated+mapped ("mapped free"): available for immediate reuse without faulting
- not-allocated+not-mapped ("unmapped free"): address exists but physical pages advised away
- allocated+not-mapped: transiently possible during allocation before `ensureEnoughMappedPages`

### 2.4 The Buffer Layer

(`velox/buffer/Buffer.h`)

```cpp
class Buffer {
  velox::memory::MemoryPool* const pool_;
  uint8_t* const data_;
  uint64_t size_{0};
  uint64_t capacity_;
  std::atomic_int32_t referenceCount_{0};
  const Type type_;
};
static_assert(sizeof(Buffer) == 64);

class AlignedBuffer : public Buffer {
  static constexpr int16_t kAlignment = 64;
  static constexpr int32_t kPaddedSize = kSizeofAlignedBuffer + simd::kPadding;

  AlignedBuffer(velox::memory::MemoryPool* pool, size_t capacity)
      : Buffer{Type::kPOD,
               reinterpret_cast<uint8_t*>(this) + sizeof(*this),
               capacity, pool} {}
};
```

`AlignedBuffer` is placed inline at the start of its allocation using placement `new`. The data region immediately follows the 64-byte header, so `data_` points to `this + 64`. The allocation is obtained from `MemoryPool::allocate()` which calls `MemoryAllocator::allocateBytes()`. This means `Buffer`s use the raw-byte allocation path, not the page-level `Allocation` or `ContiguousAllocation` path.

## 3. Execution & Call Flow

### 3.1 Non-Contiguous Allocation (e.g., RowContainer)

The full call chain for a non-contiguous allocation:

```
RowContainer or RowPartitions
  -> MemoryPool::allocateNonContiguous(numPages, out)
    -> MemoryAllocator::allocateNonContiguous(numPages, out, reservationCB)
      1. Compute SizeMix via allocationSize(numPages, minSizeClass)
      2. Call reservationCB to reserve memory in the pool hierarchy
      3. If cache exists: cache->makeSpace(N, allocateFn)
         else: allocateNonContiguousWithoutRetry(sizeMix, out)
    -> [MmapAllocator] allocateNonContiguousWithoutRetry(sizeMix, out)
      1. Free any existing allocation in 'out'
      2. Atomic capacity check: numAllocated_ + totalPages <= capacity_
      3. For each size class in the mix:
         sizeClasses_[idx]->allocate(count, numUnmapped, out)
      4. If numUnmapped > 0: ensureEnoughMappedPages(numUnmapped)
      5. Mark newly allocated pages as mapped
```

**Size Mix Computation** (`MemoryAllocator::allocationSize`):
```cpp
SizeMix MemoryAllocator::allocationSize(
    MachinePageCount numPages, MachinePageCount minSizeClass) const {
  SizeMix mix;
  int32_t neededPages = numPages;
  for (int32_t sizeIndex = sizeClassSizes_.size() - 1; sizeIndex >= 0; --sizeIndex) {
    const MachinePageCount classPageSize = sizeClassSizes_[sizeIndex];
    const bool isSmallest =
        sizeIndex == 0 || sizeClassSizes_[sizeIndex - 1] < minSizeClass;
    // Skip if class is too large (>112.5% of remaining need)
    if (classPageSize > (neededPages + (neededPages / 8)) && !isSmallest) {
      continue;
    }
    MachinePageCount numClassPages = std::min<int32_t>(
        maxNumClassPages,
        std::max<MachinePageCount>(1, neededPages / classPageSize));
    neededPages -= numClassPages * classPageSize;
    // ...
    mix.sizeCounts.push_back(numClassPages);
    mix.sizeIndices.push_back(sizeIndex);
    if (neededPages <= 0) break;
  }
  return mix;
}
```

The algorithm sweeps from largest to smallest class, greedily filling the request. The "1/8 threshold" means a class is skipped if it would waste more than 12.5% of its allocation, unless it is the smallest usable class.

**SizeClass::allocateLocked -- Two-Phase Allocation:**
```cpp
bool SizeClass::allocateLocked(ClassPageCount numPages,
                                MachinePageCount* numUnmapped,
                                Allocation& out) {
  // Phase 1: Prefer mapped-free pages (no page faults needed)
  ClassPageCount considerMappedOnly = std::min(numMappedFreePages_, numPages);
  if (considerMappedOnly > 0) {
    allocateFromMappedFree(considerMappedOnly, out);
    numMappedFreePages_ -= numAllocated;
    numPagesToAllocate -= numAllocated;
  }
  if (numPagesToAllocate == 0) return true;

  // Phase 2: Clock-hand sweep for any free pages (may be unmapped)
  uint32_t cursor = clockHand_;
  for (;;) {
    if (++cursor >= numWords) cursor = 0;
    uint64_t bits = pageAllocated_[cursor];
    if (bits != kAllSet) {
      allocateAny(cursor, numPagesToAllocate, *numUnmapped, out);
      if (numPagesToAllocate == 0) {
        clockHand_ = previousCursor;
        return true;
      }
    }
  }
}
```

Phase 1 uses SIMD-accelerated bitmap scanning (`mappedFreeBits()`, `findMappedFreeGroup()`) to find pages that are both free and backed by physical memory. Phase 2 falls back to a clock-hand sweep over all free pages, counting unmapped ones separately.

**allocateAny -- Bit-Level Page Allocation:**
```cpp
void SizeClass::allocateAny(int32_t wordIndex, ClassPageCount& numPages,
                            MachinePageCount& numUnmapped, Allocation& allocation) {
  uint64_t freeBits = ~pageAllocated_[wordIndex];
  const auto toAlloc = std::min(numPages, __builtin_popcountll(freeBits));
  for (int32_t i = 0; i < toAlloc; ++i) {
    const int bit = __builtin_ctzll(freeBits);
    bits::setBit(&pageAllocated_[wordIndex], bit);
    if (!(pageMapped_[wordIndex] & (1UL << bit))) {
      numUnmapped += unitSize_;
    } else {
      --numMappedFreePages_;
    }
    allocation.append(
        address_ + AllocationTraits::kPageSize * unitSize_ * (bit + wordIndex * 64),
        unitSize_);
    freeBits &= freeBits - 1;  // clear lowest set bit
  }
  numPages -= toAlloc;
}
```

The address is computed directly from the bit index: `address_ + kPageSize * unitSize * globalBitIndex`. This is possible because each size class's entire virtual range is pre-mapped.

### 3.2 Contiguous Allocation (e.g., HashTable)

```
HashTable::allocateTable()
  -> pool->allocateContiguous(numPages, tableAllocation_)
    -> MemoryAllocator::allocateContiguous(numPages, collateral, allocation, reservationCB, maxPages)
      -> [MmapAllocator] allocateContiguousImpl(numPages, collateral, allocation, maxPages)
        1. Free collateral (non-contiguous pages) and previous allocation
        2. Atomic capacity check on numAllocated_
        3. ensureEnoughMappedPages(numToMap)  -- advise away from size classes
        4. mmap(nullptr, maxPages*4096, PROT_READ|PROT_WRITE, MAP_PRIVATE|MAP_ANONYMOUS, -1, 0)
           or managedArenas_->allocate(bytes) if useMmapArena_ is true
        5. allocation.set(data, numPages*4096, maxPages*4096)
        6. useHugePages(allocation, true)  -- madvise(MADV_HUGEPAGE)
```

**Key detail -- `ensureEnoughMappedPages`:**
```cpp
bool MmapAllocator::ensureEnoughMappedPages(int32_t newMappedNeeded) {
  std::lock_guard<std::mutex> l(sizeClassBalanceMutex_);
  const auto totalMaps = numMapped_.fetch_add(newMappedNeeded) + newMappedNeeded;
  if (totalMaps <= capacity_) {
    return true;  // Under capacity, no advise needed
  }
  const auto target = totalMaps - capacity_;
  const auto numAdvised = adviseAway(target);
  // ...
}
```

When a contiguous allocation needs new mapped pages, MmapAllocator steals physical backing from unused (free but mapped) pages in the size classes. This is the "dynamic balancing" mechanism: contiguous hash table memory can be carved from pages that were previously used for non-contiguous allocations.

**adviseAway -- Reclaiming Physical Pages:**
```cpp
MachinePageCount MmapAllocator::adviseAway(MachinePageCount target) {
  MachinePageCount numAway = 0;
  for (int32_t i = sizeClasses_.size() - 1; i >= 0; --i) {
    numAway += sizeClasses_[i]->adviseAway(target - numAway);
    if (numAway >= target) break;
  }
  numAdvisedPages_ += numAway;
  return numAway;
}
```

Iterates from largest to smallest class, calling `madvise(MADV_DONTNEED)` on free pages to return physical memory to the OS, making room for new mappings.

### 3.3 MallocAllocator -- Simpler Path

For `MallocAllocator`, the paths are simpler:

**Non-contiguous:** Each size class chunk is a separate `::malloc()` call:
```cpp
bool MallocAllocator::allocateNonContiguousWithoutRetry(const SizeMix& sizeMix, Allocation& out) {
  // ...
  for (int32_t i = 0; i < sizeMix.numSizes; ++i) {
    MachinePageCount numSizeClassPages =
        sizeMix.sizeCounts[i] * sizeClassSizes_[sizeMix.sizeIndices[i]];
    void* ptr = ::malloc(AllocationTraits::pageBytes(numSizeClassPages));
    out.append(reinterpret_cast<uint8_t*>(ptr), numSizeClassPages);
  }
  numAllocated_.fetch_add(sizeMix.totalPages);
  return true;
}
```

**Contiguous:** Uses `mmap` or `aligned_alloc` depending on configuration:
```cpp
void* MallocAllocator::dispatchAllocateContiguous(size_t maxBytes) {
  if (mallocContiguousEnabled_) {
    return ::aligned_alloc(AllocationTraits::kPageSize, maxBytes);
  }
  return ::mmap(nullptr, maxBytes, PROT_READ | PROT_WRITE,
                MAP_PRIVATE | MAP_ANONYMOUS, -1, 0);
}
```

### 3.4 Buffer Allocation Path

```
AlignedBuffer::allocate<T>(numElements, pool)
  -> pool->preferredSize(size + kPaddedSize)    // round up for efficiency
  -> pool->allocate(preferredSize)
    -> MemoryAllocator::allocateBytes(alignedSize, alignment)
      -> [MmapAllocator] allocateBytesWithoutRetry(bytes, alignment)
        if bytes <= maxMallocBytes_ (default 3072): ::malloc(bytes)
        elif bytes <= largestSizeClass (256 pages): allocateNonContiguous + return single run
        else: allocateContiguous
  -> new (memory) AlignedBuffer{pool, preferredSize - kPaddedSize}
  -> data_ = reinterpret_cast<uint8_t*>(this) + sizeof(*this)  // inline data
```

The `AlignedBuffer` is constructed via placement `new` into the allocated memory. Since `sizeof(AlignedBuffer) == 64`, the data payload starts at the 64-byte boundary immediately after the header, giving natural cache-line alignment.

**MmapAllocator::allocateBytesWithoutRetry -- Three Tiers:**
```cpp
void* MmapAllocator::allocateBytesWithoutRetry(uint64_t bytes, uint16_t alignment) {
  if (useMalloc(bytes)) {        // <= 3072 bytes: delegate to malloc
    auto* result = ::malloc(bytes);
    numMallocBytes_ += bytes;
    return result;
  }
  if (bytes <= pageBytes(sizeClassSizes_.back())) {  // <= 1MB: use size class
    Allocation allocation;
    auto numPages = roundUpToSizeClassSize(bytes, sizeClassSizes_);
    const SizeMix mix = allocationSize(numPages, numPages);
    allocateNonContiguousWithoutRetry(mix, allocation);
    auto run = allocation.runAt(0);
    allocation.clear();           // detach from Allocation, return raw pointer
    return run.data<char>();
  }
  // > 1MB: contiguous mmap
  ContiguousAllocation allocation;
  auto numPages = AllocationTraits::numPages(bytes);
  allocateContiguousWithoutRetry(numPages, nullptr, allocation);
  char* data = allocation.data<char>();
  allocation.clear();
  return data;
}
```

This three-tier approach in `MmapAllocator` avoids the overhead of bitmap management for small allocations (delegated to `malloc`), uses size classes for medium allocations (up to 1MB), and falls back to raw `mmap` for large allocations.

### 3.5 growContiguous -- Incremental Growth

```cpp
bool MmapAllocator::growContiguousWithoutRetry(
    MachinePageCount increment, ContiguousAllocation& allocation) {
  auto numAllocated = numAllocated_.fetch_add(increment) + increment;
  if (numAllocated > capacity_) { /* rollback */ }
  if (!ensureEnoughMappedPages(increment)) { /* rollback */ }
  numExternalMapped_ += increment;
  allocation.set(allocation.data(),
                 allocation.size() + AllocationTraits::pageBytes(increment),
                 allocation.maxSize());
  return true;
}
```

Growth simply advances `size_` within the already-mapped `maxSize_` range. No remapping, no copying. The physical pages are faulted in on demand by the OS. This is why `allocateContiguous` accepts `maxPages > numPages` -- it pre-reserves the address space.

## 4. Concurrency & State Management

### 4.1 MmapAllocator Locking Strategy

Three distinct lock scopes:

1. **`SizeClass::mutex_`** (per size class): Guards `pageAllocated_`, `pageMapped_`, `numMappedFreePages_`, and all bitmap operations. Held during `allocateLocked()`, `free()`, `setMappedBits()`.

2. **`sizeClassBalanceMutex_`** (global): Serializes capacity rebalancing. Held during `ensureEnoughMappedPages()` and `unmap()`, which need to advise away pages across multiple size classes atomically.

3. **`arenaMutex_`** (optional): Guards `managedArenas_` when `useMmapArena_` is enabled for contiguous allocations.

**Lock-free atomics for counters:**
- `numAllocated_`, `numMapped_`, `numExternalMapped_` are `std::atomic<MachinePageCount>`, updated with `fetch_add`/`fetch_sub` outside of locks.
- The contiguous allocation path carefully avoids releasing `numAllocated_` before the outcome is determined, preventing other threads from grabbing transiently free capacity.

### 4.2 MallocAllocator Sharded Counter

`MallocAllocator` uses `ConcurrentCounter<uint32_t>` to minimize contention on the global `allocatedBytes_` counter:

```cpp
inline bool incrementUsageWithReservation(uint32_t bytes) {
  return reservations_.update(bytes, reserveFunc_);
}
```

The `ConcurrentCounter` shards by thread ID. Each shard has its own local counter protected by a per-shard mutex. When a shard runs low, it reserves `reservationByteLimit_` (default 1MB) from the global atomic counter in bulk. Frees release back to the global counter when the local shard accumulates 2x the limit. This pattern is similar to how jemalloc uses thread-local caches.

```cpp
inline bool incrementUsageWithReservationFunc(
    uint32_t& counter, uint32_t increment, std::mutex& lock) {
  std::lock_guard<std::mutex> l(lock);
  if (counter > increment) {
    counter -= increment;
    return true;
  }
  if (!incrementUsageWithoutReservation(reservationByteLimit_)) {
    return false;
  }
  counter += reservationByteLimit_;
  counter -= increment;
  return true;
}
```

### 4.3 Buffer Reference Counting

`Buffer` uses `std::atomic_int32_t` with acquire-release semantics:

```cpp
void addRef() noexcept {
  referenceCount_.fetch_add(1, std::memory_order_acq_rel);
}

void release() {
  if (referenceCount_.fetch_sub(1, std::memory_order_acq_rel) == 1) {
    releaseResources();
    if (pool_) {
      freeToPool();
    } else {
      delete this;
    }
  }
}
```

The `BufferPtr` type alias is `boost::intrusive_ptr<Buffer>`, relying on free functions `intrusive_ptr_add_ref`/`intrusive_ptr_release`. A buffer is mutable if and only if `refCount() == 1 && !isView()`. This single-writer constraint eliminates the need for locks during buffer mutation.

## 5. Memory & Resource Profile

### 5.1 Allocation Patterns

| Allocation Type | Data Structure | Typical Consumer | Backing Mechanism (MmapAllocator) |
|---|---|---|---|
| Non-contiguous | `Allocation` (vector of `PageRun`) | RowContainer, RowPartitions | Size class bitmap allocation |
| Contiguous | `ContiguousAllocation` | HashTable, large sort buffers | Direct `mmap` or `MmapArena` |
| Raw bytes | `void*` via `allocateBytes()` | `AlignedBuffer` / `Buffer` | Tiered: malloc (<3KB), size class (<1MB), mmap (>1MB) |

### 5.2 MmapAllocator Memory Layout

At construction, MmapAllocator pre-maps **the entire capacity** for every size class:

```cpp
MmapAllocator::MmapAllocator(const Options& options) {
  for (const auto& size : sizeClassSizes_) {
    sizeClasses_.push_back(std::make_unique<SizeClass>(capacity_ / size, size));
  }
}

SizeClass::SizeClass(size_t capacity, MachinePageCount unitSize)
    : capacity_(capacity), unitSize_(unitSize),
      byteSize_(AllocationTraits::pageBytes(capacity_ * unitSize_)) {
  void* ptr = mmap(nullptr,
      AllocationTraits::pageBytes(capacity_ * unitSize_),
      PROT_READ | PROT_WRITE,
      MAP_PRIVATE | MAP_ANONYMOUS, -1, 0);
  address_ = reinterpret_cast<uint8_t*>(ptr);
}
```

For a 64GB capacity with 9 size classes (1,2,4,8,16,32,64,128,256 pages), the total virtual address space mapped is `9 * 64GB = 576GB`. However, physical memory is only committed on first touch (MAP_ANONYMOUS gives zero-fill-on-demand pages). The `madvise(MADV_DONTNEED)` mechanism returns physical pages to the OS without unmapping the virtual range.

**Overhead per SizeClass:**
- `pageAllocated_[]`: `capacity_ / 64` uint64_t words + 8 SIMD padding words
- `pageMapped_[]`: same
- `mappedFreeLookup_[]`: 1 bit per `kPagesPerLookupBit` (128 * batch_size) pages

### 5.3 Buffer Memory Layout

When `AlignedBuffer::allocate<T>(N, pool)` is called:

```
pool->preferredSize(N * sizeof(T) + kPaddedSize)  // kPaddedSize = 64 + simd::kPadding
```

The resulting allocation layout:
```
[AlignedBuffer header: 64 bytes][data: capacity bytes][end guard: 8 bytes][SIMD padding]
```

The SIMD padding (`simd::kPadding`, typically 32 bytes) past `capacity()` is guaranteed addressable. This allows SIMD operations to safely over-read without segfaulting. The `kEndGuard` (0xbadaddbadadddeadUL) sits at `data_ + capacity_` and is checked on destruction and resize to detect buffer overruns.

### 5.4 Cache Integration

Both allocators support an optional `Cache` registration. When allocation fails due to capacity pressure, the allocator invokes `cache->makeSpace()`:

```cpp
bool MemoryAllocator::allocateNonContiguous(...) {
  if (cache() == nullptr) {
    success = allocateNonContiguousWithoutRetry(mix, out);
  } else {
    success = cache()->makeSpace(
        pagesToAcquire(numPages, out.numPages()), [&](Allocation& acquired) {
          freeNonContiguous(acquired);
          return allocateNonContiguousWithoutRetry(mix, out);
        });
  }
}
```

The cache evicts data to produce an `Allocation`, which is freed to provide capacity for the new allocation. This allows dynamic balancing between cache and compute memory from a single capacity budget.

## 6. Key Design Insights

1. **Virtual address pre-mapping eliminates fragmentation.** MmapAllocator maps the full capacity for each size class at startup. Since any size-class page is a fixed offset into a pre-allocated virtual range, there is no address space fragmentation regardless of allocation/free patterns. This is fundamentally different from Trino, which relies on the JVM's garbage collector to defragment heap memory, and from DataFusion/Arrow, which uses jemalloc or the system allocator and can suffer from malloc fragmentation over long-running processes.

2. **Physical memory is a fungible resource managed via madvise.** The key insight is that virtual address space is cheap but physical backing is expensive. MmapAllocator uses `madvise(MADV_DONTNEED)` to return physical pages from free size-class slots, then faults in new pages where needed (contiguous allocations, different size classes). This creates a "physical page pool" that can float across all allocation types. Trino has no equivalent -- it cannot return JVM heap memory to the OS granularly. DataFusion inherits the Arrow allocator's approach where `free()` returns memory to the process heap, but cannot guarantee physical page reclamation.

3. **Two-bitmap design enables mapped-free preference.** The `pageAllocated_` and `pageMapped_` bitmaps allow MmapAllocator to prefer pages that are both free and backed by physical memory, avoiding TLB misses and page faults for reused pages. The SIMD-accelerated `mappedFreeBits()` and `findMappedFreeGroup()` make this preference fast even at scale. This optimization has no analog in Trino or DataFusion.

4. **PageRun packs pointer+size in 8 bytes using x86-64 address space layout.** By exploiting the 48-bit pointer invariant, Velox avoids a separate length field per run, halving the metadata overhead for `Allocation` objects. This is a C++-specific trick unavailable to JVM-based engines like Trino. Arrow's buffer metadata is heavier per-allocation (separate pointer + length + padding fields).

5. **Buffer uses inline placement-new for zero-indirection data access.** `AlignedBuffer` places the header and data in a single contiguous allocation, so `data_` is always `this + 64`. There is no heap pointer chase to reach the data. This differs from Trino's `Slice` which wraps a `byte[]` array (one indirection), and from Arrow's `Buffer` which stores a separate `data_` pointer that may or may not be in the same cache line as the metadata.

6. **Three-tier byte allocation adapts to allocation size.** MmapAllocator's `allocateBytesWithoutRetry` routes small allocations (<3KB) to malloc, medium allocations (3KB-1MB) through size classes, and large allocations (>1MB) to contiguous mmap. This prevents small allocations from consuming size-class bitmap capacity while ensuring large allocations get dedicated address ranges with huge page support. DataFusion's Arrow allocator does not differentiate -- all allocations go through the same jemalloc path regardless of size.

7. **Contiguous allocation supports over-reservation via maxPages.** The `maxPages` parameter allows mapping a large address range while only counting `numPages` against capacity. This is critical for hash tables that start small and grow: the address range is pre-reserved, and `growContiguous()` simply advances the committed size without remapping. Trino's equivalent (`long[] hashTable`) requires a full array copy on growth. DataFusion/Arrow has no equivalent; contiguous growth requires realloc.

8. **MallocAllocator's sharded counter minimizes contention for small allocations.** The `ConcurrentCounter` with `reservationByteLimit_`-sized bulk operations ensures that typical small allocations (<1MB) complete with only a per-shard mutex acquisition, never touching the global atomic. This is similar in spirit to jemalloc's thread cache but applied at the accounting level rather than the allocation level.

9. **The collateral pattern enables atomic-like capacity exchange.** `allocateContiguous` accepts both a `collateral` (non-contiguous) and a previous `allocation` (contiguous) that are freed atomically with the new allocation. The counters are carefully not decremented until the outcome is known, preventing other threads from grabbing the transiently free capacity. This "exchange" pattern avoids the need for a lock around the entire allocate-free cycle.

10. **Cache integration at the allocator level enables workload-aware memory management.** Unlike Trino where cache and compute memory are managed independently, and unlike DataFusion which has no built-in cache integration, Velox's allocator directly coordinates with the cache through the `Cache::makeSpace()` interface. When compute needs memory, the cache can evict entries to free pages. This single-source design prevents the OOM scenarios that occur when cache and compute independently try to use the same memory.
