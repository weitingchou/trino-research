# Module Teardown: The MemoryArbitrator (Spill Triggering)

## 0. Research Focus
* **Task ID:** 5.3
* **Focus:** Trace the exact path when a `MemoryPool` exceeds its limit. How does the `MemoryArbitrator` coordinate across the entire process to find victim queries/tasks? How does it signal operators to pause execution and invoke the `Spiller`? Compare this to Trino's `MemoryRevokingScheduler`.

## 1. High-Level Overview

The Velox `MemoryArbitrator` is a process-level coordinator that manages dynamic memory sharing among all running queries. When any query's `MemoryPool` tries to allocate beyond its current capacity, the allocation request propagates up the pool hierarchy to the root, which discovers the capacity shortfall and synchronously invokes the arbitrator's `growCapacity()`. The arbitrator then orchestrates a multi-phase strategy: first try to grow from the pool's own unused capacity, then reclaim free capacity from other pools, and finally trigger **global arbitration** -- a background thread that reclaims *used* memory from victim queries via disk spilling or, as a last resort, query abort.

The key architectural difference from Trino is that Velox performs arbitration **synchronously on the requesting thread** (blocking the operator that needs memory) and uses a **single global arbitration controller thread** to coordinate cross-query reclaim. Trino's `MemoryRevokingScheduler` runs asynchronously on a periodic schedule, scanning all pools for over-committed memory and setting "revoke requested" flags that operators check cooperatively. Velox's approach is more aggressive and deterministic -- the requesting thread is blocked until memory is available, with a configurable timeout (default 5 minutes).

### 2.2 The SharedArbitrator

The `SharedArbitrator` is the production implementation, registered with kind `"SHARED"`. It holds:

```cpp
// SharedArbitrator.h -- Key member variables

const uint64_t capacity_;                // Total system memory budget
const uint64_t reservedCapacity_;        // Reserved to guarantee min per-query capacity
const uint64_t maxArbitrationTimeNs_;    // Default 5 minutes timeout
const bool globalArbitrationEnabled_;    // Enables cross-query memory reclaim

tsan_atomic<uint64_t> freeReservedCapacity_{0};     // Available reserved pool
tsan_atomic<uint64_t> freeNonReservedCapacity_{0};  // Available general pool

// Background global arbitration
std::unique_ptr<std::thread> globalArbitrationController_;
std::condition_variable_any globalArbitrationThreadCv_;

// Participants: one per query root pool
std::unordered_map<std::string, std::shared_ptr<ArbitrationParticipant>>
    participants_;

// Waiting operations for global arbitration, ordered by participant ID
// (older queries served first)
std::map<uint64_t, ArbitrationWait*> globalArbitrationWaiters_;

// Thread pool for parallel memory reclaim across victims
std::unique_ptr<folly::CPUThreadPoolExecutor> memoryReclaimExecutor_;
```

The arbitrator maintains a **two-tier free capacity pool**: reserved capacity (guarantees minimum per-query allocation) and non-reserved capacity (available for elastic growth). This is analogous to Trino's memory pool structure with its general/reserved split.

### 2.3 ArbitrationParticipant

Each query root pool is wrapped in an `ArbitrationParticipant`:

```cpp
// ArbitrationParticipant.h (key fields)
class ArbitrationParticipant {
  const uint64_t id_;                          // Monotonically increasing, older = smaller
  const std::weak_ptr<MemoryPool> poolWeakPtr_; // Weak ref to query pool
  MemoryPool* const pool_;                      // Raw pointer for fast access
  const Config* const config_;                  // Growth/shrink parameters

  ArbitrationOperation* runningOp_{nullptr};     // Currently executing op
  std::deque<WaitOp> waitOps_;                   // Queued ops for serial execution

  mutable std::timed_mutex reclaimMutex_;        // Serializes reclaim on this pool
};
```

The participant enforces **serialized arbitration**: only one `ArbitrationOperation` per query pool runs at a time. Others queue in `waitOps_` and are resumed via `ContinuePromise` when the running operation finishes.

### 2.4 Memory Pool Hierarchy

```
MemoryPool (abstract)
  +-- MemoryPoolImpl
        Kind::kAggregate  -- root (query), task, node pools
        Kind::kLeaf       -- operator pools (actual allocation happens here)
```

Pools form a 4-level tree: query -> task -> plan node -> operator. Each level has a `MemoryReclaimer` registered. The hierarchy is critical because:
1. Leaf pools trigger allocation and detect capacity overflow
2. The capacity check propagates up to the root pool
3. The root pool calls the arbitrator
4. Reclaim traverses down from the root through the reclaimer tree

When the arbitrator tries to reclaim from an operator in a non-reclaimable section, it records a `numNonReclaimableAttempts` counter and skips that operator -- it does not block waiting. This differs from Trino where operators cooperatively check the revoke flag at safe points.

## 3. Execution & Call Flow

### 3.1 Trigger: MemoryPool Allocation Exceeds Capacity

The flow starts when a leaf operator allocates memory:

```
Operator allocates memory (e.g., HashBuild adding rows)
  -> MemoryPoolImpl::reserve() / allocate()
    -> MemoryPoolImpl::reserveThreadSafe()
      -> MemoryPoolImpl::incrementReservationThreadSafe(this, increment)
```

The `incrementReservationThreadSafe` propagates up the tree to the root:

```cpp
// MemoryPool.cpp:882
void MemoryPoolImpl::incrementReservationThreadSafe(
    MemoryPool* requestor, uint64_t size) {
  // Propagate to root first
  if (parent_ != nullptr) {
    toImpl(parent_)->incrementReservationThreadSafe(requestor, size);
  }

  if (maybeIncrementReservation(size)) {
    return;  // Fits within capacity
  }

  // At root pool: capacity exceeded, trigger arbitration
  VELOX_CHECK_NULL(parent_);
  growCapacity(requestor, size);
}
```

The root pool's `maybeIncrementReservation` check:

```cpp
// MemoryPool.cpp:934
bool MemoryPoolImpl::maybeIncrementReservation(uint64_t size) {
  std::lock_guard<std::mutex> l(mutex_);
  if (isRoot()) {
    checkIfAborted();
    // Allow overuse during arbitration itself
    if (FOLLY_UNLIKELY(
            (reservationBytes_ + size > capacity_) &&
            !underMemoryArbitration())) {
      return false;  // TRIGGERS ARBITRATION
    }
  }
  incrementReservationLocked(size);
  return true;
}
```

The root pool then calls the arbitrator:

```cpp
// MemoryPool.cpp:909
void MemoryPoolImpl::growCapacity(MemoryPool* requestor, uint64_t size) {
  MemoryPoolArbitrationSection arbitrationSection(requestor);
  arbitrator_->growCapacity(this, size);  // Calls SharedArbitrator
}
```

### 3.3 The Multi-Phase Growth Strategy

`growCapacity(ArbitrationOperation& op)` implements a cascading strategy:

```cpp
// SharedArbitrator.cpp:891
void SharedArbitrator::growCapacity(ArbitrationOperation& op) {
  checkIfAborted(op);
  checkIfTimeout(op);

  // Phase 1: Maybe the pool already has enough capacity (concurrent growth)
  RETURN_IF_TRUE(maybeGrowFromSelf(op));

  // Phase 2: Ensure capacity won't exceed pool's max or arbitrator's max
  //          May reclaim from the requesting pool itself (local reclaim)
  if (!ensureCapacity(op)) {
    MEM_POOL_CAP_EXCEEDED(...);
  }

  checkIfAborted(op);
  RETURN_IF_TRUE(maybeGrowFromSelf(op));

  // Phase 3: Set exponential growth targets, try free capacity
  op.setGrowTargets();
  RETURN_IF_TRUE(growWithFreeCapacity(op));

  // Phase 4: Reclaim unused (free) capacity from ALL other pools
  reclaimUnusedCapacity();
  RETURN_IF_TRUE(growWithFreeCapacity(op));

  // Phase 5: If global arbitration disabled, try local self-reclaim
  if (!globalArbitrationEnabled_) {
    reclaim(op.participant(), op.requestBytes(), op.timeoutNs(), true);
    ...
    return;
  }

  // Phase 6: Start global arbitration (cross-query spill/abort)
  startAndWaitGlobalArbitration(op);
}
```

### 3.4 ensureCapacity -- Local Self-Reclaim

Before trying other pools, the arbitrator tries to make room within the requesting pool itself:

```cpp
// SharedArbitrator.cpp:1190
bool SharedArbitrator::ensureCapacity(ArbitrationOperation& op) {
  if ((op.requestBytes() > capacity_) ||
      (op.requestBytes() > op.participant()->maxCapacity())) {
    return false;  // Impossible to satisfy
  }

  RETURN_TRUE_IF_TRUE(checkCapacityGrowth(op));

  shrink(op.participant(), /*reclaimAll=*/true);  // Release free capacity

  RETURN_TRUE_IF_TRUE(checkCapacityGrowth(op));

  // Local reclaim: spill from the requesting pool itself
  reclaim(op.participant(), op.requestBytes(), op.timeoutNs(),
          /*localArbitration=*/true);
  checkIfAborted(op);

  RETURN_TRUE_IF_TRUE(checkCapacityGrowth(op));

  shrink(op.participant(), /*reclaimAll=*/true);
  return checkCapacityGrowth(op);
}
```

### 3.5 startAndWaitGlobalArbitration -- Cross-Query Coordination

When local strategies fail, the operation enters the global arbitration queue:

```cpp
// SharedArbitrator.cpp:962
void SharedArbitrator::startAndWaitGlobalArbitration(ArbitrationOperation& op) {
  // Try allocating capacity one more time
  allocatedBytes = allocateCapacityLocked(
      op.participant()->id(), op.requestBytes(),
      op.maxGrowBytes(), op.minGrowBytes());

  if (allocatedBytes == 0) {
    // Add to wait queue (ordered by participant ID = age)
    arbitrationWait = std::make_unique<ArbitrationWait>(&op, ContinuePromise{});
    globalArbitrationWaiters_.emplace(op.participant()->id(), arbitrationWait);
    op.participant()->setPendingArbitrationGrowCapacity(op.requestBytes());
  }

  if (arbitrationWaitFuture.valid()) {
    wakeupGlobalArbitrationThread();  // Signal the controller thread

    // BLOCK: wait for the future with timeout
    const bool timeout =
        !std::move(arbitrationWaitFuture)
             .wait(std::chrono::microseconds(op.timeoutNs() / 1'000));
    // ...
  }

  CHECKED_GROW(op.participant(), allocatedBytes, op.requestBytes());
}
```

The requesting thread is now **blocked** on a `ContinueFuture`, waiting for the global arbitration controller to free enough memory and fulfill its promise.

### 3.6 Global Arbitration Controller Thread

The controller runs in a dedicated background thread:

```cpp
// SharedArbitrator.cpp:1045
void SharedArbitrator::globalArbitrationMain() {
  while (true) {
    {
      std::unique_lock<std::mutex> l(stateMutex_);
      globalArbitrationThreadCv_.wait(l, [&] {
        return hasShutdownLocked() || !globalArbitrationWaiters_.empty();
      });
      if (hasShutdownLocked()) break;
    }
    GlobalArbitrationSection section{this};
    runGlobalArbitration();
  }
}
```

`runGlobalArbitration()` executes in rounds:

```cpp
// SharedArbitrator.cpp:1076
void SharedArbitrator::runGlobalArbitration() {
  for (;; ++round) {
    const uint64_t targetBytes = getGlobalArbitrationTarget();
    if (targetBytes == 0) break;  // All waiters satisfied

    // Decide: spill or abort?
    shouldReclaimByAbort = globalArbitrationShouldReclaimByAbort(
        elapsedTime, shouldReclaimByAbort,
        allParticipantsReclaimed, reclaimedBytes);

    if (shouldReclaimByAbort) {
      reclaimedBytes = reclaimUsedMemoryByAbort(/*force=*/true);
    } else {
      reclaimedBytes = reclaimUsedMemoryBySpill(
          targetBytes, reclaimedParticipants,
          failedParticipants, allParticipantsReclaimed);
    }
    reclaimUnusedCapacity();  // Collect freed capacity after spill
  }
}
```

The target is calculated as the maximum of: (a) sum of all waiters' `maxGrowBytes`, and (b) `capacity * globalArbitrationMemoryReclaimPct / 100` (default 10%). This ensures each global run reclaims a meaningful chunk of memory.

### 3.7 Victim Selection for Spilling

```cpp
// SharedArbitrator.cpp:1260 -- reclaimUsedMemoryBySpill
auto candidates = getCandidates();
auto candidateGroups = sortAndGroupSpillCandidates(std::move(candidates));

// Select victims: pick from highest reclaimable capacity group first
for (auto& candidateGroup : candidateGroups) {
  for (auto& candidate : candidateGroup) {
    if (candidate.reclaimableUsedCapacity < participantConfig_.minReclaimBytes)
      continue;
    // Skip already-reclaimed or failed participants
    if (bytesToReclaim >= targetBytes) continue;
    bytesToReclaim += candidate.reclaimableUsedCapacity;
    victims.push_back(std::move(candidate));
  }
}
```

Candidates are **sorted and grouped** by capacity buckets (`globalArbitrationSpillCapacityLimits_`, defaulting from 4GB down by powers of 2). Within each bucket, they are sorted by **priority** (lower priority = reclaimed first) and **reclaimable used capacity** (largest first). This two-level sorting ensures:
1. Large memory consumers are targeted first (efficient reclaim)
2. Low-priority queries sacrifice memory before high-priority ones

### 3.8 Parallel Reclaim Execution

Victim reclaim is parallelized across the memory reclaim executor:


### 3.9 The Reclaim Path: Arbitrator -> Participant -> Pool -> Task -> Operator

```
SharedArbitrator::reclaim(participant, targetBytes, timeoutNs)
  -> ArbitrationParticipant::reclaim()      // Holds reclaimMutex_ (timed)
    -> MemoryPool::reclaim()                // Delegates to reclaimer
      -> Task::MemoryReclaimer::reclaim()   // THE CRITICAL PATH
        -> Task::MemoryReclaimer::reclaimTask()
```

**Task-level reclaim pauses execution:**

```cpp
// Task.cpp:3543
uint64_t Task::MemoryReclaimer::reclaimTask(
    const std::shared_ptr<Task>& task,
    uint64_t targetBytes, uint64_t maxWaitMs,
    memory::MemoryReclaimer::Stats& stats) {

  // CRITICAL: Resume task on exit (even on error)
  auto resumeGuard = folly::makeGuard([&]() {
    Task::resume(task);
  });

  // PAUSE THE TASK: wait for all drivers to go off-thread
  if (maxWaitMs == 0) {
    task->requestPause().wait();       // Unlimited wait
  } else {
    paused = task->requestPause().wait(
        std::chrono::milliseconds(maxWaitMs));  // Timed wait
  }

  if (!paused) {
    VELOX_FAIL("Memory reclaim failed to wait for task to pause...");
  }

  // Task is now paused -- safe to reclaim from operator pools
  reclaimedBytes = memory::MemoryReclaimer::reclaim(
      task->pool(), targetBytes, maxWaitMs, stats);
}
```

`requestPause()` sets `pauseRequested_ = true` and returns a future that completes when `numThreads_ == 0` (all drivers have left the execution loop or entered suspension).

### 3.10 Driver Suspension During Self-Arbitration

When an operator triggers arbitration for its own pool, the driver must enter a **suspended** state to avoid deadlock (the arbitrator might try to pause the same task):

```cpp
// MemoryReclaimer.cpp:32 (exec::MemoryReclaimer::enterArbitration)
void MemoryReclaimer::enterArbitration() {
  DriverThreadContext* driverThreadCtx = driverThreadContext();
  if (driverThreadCtx == nullptr) return;  // Not on driver thread

  Driver* const driver = driverThreadCtx->driverCtx()->driver;
  if (driver->task()->enterSuspended(driver->state()) != StopReason::kNone) {
    VELOX_FAIL("Terminate detected when entering suspension");
  }
}

void MemoryReclaimer::leaveArbitration() noexcept {
  // ...
  driver->task()->leaveSuspended(driver->state());
}
```

`enterSuspended` decrements `numThreads_` on the task. If this reaches 0, it fulfills the pause promise, allowing the reclaim thread to proceed. `leaveSuspended` reverses this, but if `pauseRequested_` is still set, it **blocks** until the task is resumed -- this prevents the driver from racing with the reclaim.

### 3.11 Operator-Level Spill (Terminal Action)

When reclaim reaches an operator:

### 3.12 Victim Selection for Abort (Last Resort)

When spilling is insufficient or takes too long:

```cpp
// SharedArbitrator.cpp:1064
bool SharedArbitrator::globalArbitrationShouldReclaimByAbort(
    uint64_t globalRunElapsedTimeNs,
    bool hasReclaimedByAbort,
    bool allParticipantsReclaimed,
    uint64_t lastReclaimedBytes) const {
  return globalArbitrationWithoutSpill_ ||
      (globalRunElapsedTimeNs >
           maxArbitrationTimeNs_ * globalArbitrationAbortTimeRatio_ &&
       (hasReclaimedByAbort ||
        (allParticipantsReclaimed && lastReclaimedBytes == 0)));
}
```

The abort threshold is: elapsed time > `maxArbitrationTime * abortTimeRatio` (default 0.5, so 2.5 minutes). Once in abort mode, `findAbortCandidate()` selects the youngest query (largest participant ID) with lowest priority whose capacity exceeds the abort limit threshold (starting at 1GB, halving down to 32MB). This strongly favors killing recently-started, low-priority, large-memory queries.

```cpp
// SharedArbitrator.cpp:1351
uint64_t SharedArbitrator::reclaimUsedMemoryByAbort(bool force) {
  const auto victim = findAbortCandidate(force);
  // Throw VELOX_MEM_POOL_ABORTED, then:
  abort(victim.participant, std::current_exception());
}
```

The abort propagates: `ArbitrationParticipant::abort()` -> `MemoryPool::abort()` -> `MemoryReclaimer::abort()` which visits all children recursively, setting the abort flag. The aborted query's future allocations will throw immediately.

### 3.13 Capacity Release and Waiter Resumption

When capacity is freed (from spill, shrink, or abort):

```cpp
// SharedArbitrator.cpp:1464
void SharedArbitrator::freeCapacity(uint64_t bytes) {
  std::vector<ContinuePromise> globalArbitrationWaitResumes;
  {
    std::lock_guard<std::mutex> l(stateMutex_);
    freeCapacityLocked(bytes, globalArbitrationWaitResumes);

    resume.setValue();  // Wake up blocked requesting threads
```

`resumeGlobalArbitrationWaitersLocked` iterates the ordered waiter map (oldest first) and allocates capacity:

```cpp
// SharedArbitrator.cpp:1494
void SharedArbitrator::resumeGlobalArbitrationWaitersLocked(
    std::vector<ContinuePromise>& resumes) {
  auto it = globalArbitrationWaiters_.begin();
  while (it != globalArbitrationWaiters_.end()) {
    auto* op = it->second->op;
    const uint64_t allocatedBytes = allocateCapacityLocked(
        op->participant()->id(), op->requestBytes(),
        op->maxGrowBytes(), op->minGrowBytes());
    if (allocatedBytes == 0) break;  // No more capacity available
    it->second->allocatedBytes = allocatedBytes;
    resumes.push_back(std::move(it->second->resumePromise));
    it = globalArbitrationWaiters_.erase(it);
  }
}
```

## 4. Concurrency & State Management

### 4.1 Lock Hierarchy

The SharedArbitrator uses a carefully layered locking scheme:

| Lock | Type | Scope | Protects |
|------|------|-------|----------|
| `participantLock_` | `folly::SharedMutex` | Arbitrator | Participant registration/lookup |
| `stateMutex_` | `std::mutex` | Arbitrator | Free capacity counters, waiter map, state |
| `ArbitrationParticipant::stateLock_` | `std::mutex` | Per-participant | Abort flag, operation queue |
| `ArbitrationParticipant::reclaimMutex_` | `std::timed_mutex` | Per-participant | Serializes reclaim operations |
| `MemoryPool::mutex_` | `std::mutex` | Per-pool | Capacity, reservation counters |
| `Task::mutex_` | `std::timed_mutex` | Per-task | Pause state, driver counts |

Key concurrency properties:

1. **Participant lookup** uses a `shared_lock` on `participantLock_` -- concurrent reads are lock-free.
2. **Capacity allocation** is serialized through `stateMutex_` but is an in-memory O(1) operation.
3. **Reclaim** is serialized per-participant via `reclaimMutex_` (timed to respect arbitration timeout).
4. **Global arbitration** runs on a single controller thread but dispatches actual reclaim to `memoryReclaimExecutor_` (thread pool sized at `hardware_concurrency * 0.5`).

### 4.2 Driver Suspension Protocol

The deadlock-prevention protocol:

1. Operator allocates memory -> capacity exceeded -> enters arbitration
2. `enterArbitration()` calls `Task::enterSuspended()`, decrementing `numThreads_`
3. If the arbitrator needs to reclaim from this same task, `requestPause()` succeeds immediately (the requesting driver is already suspended)
4. After arbitration completes, `leaveArbitration()` calls `leaveSuspended()`, which blocks if `pauseRequested_` is still set (another reclaim in progress)

This is a critical difference from Trino: Velox actively suspends the driver thread during arbitration, whereas Trino's drivers cooperatively yield at designated checkpoints.

### 4.3 Thread-Local Arbitration Context

```cpp
// MemoryArbitrator.h:466
struct MemoryArbitrationContext {
  enum class Type {
    kLocal,   // Pool requesting for itself
    kGlobal,  // System-wide reclaim
  };
  const Type type;

```

Set as a thread-local variable during arbitration via `ScopedMemoryArbitrationContext`. This enables:
- `underMemoryArbitration()` checks to allow pools to temporarily exceed capacity during arbitration
- `createAsyncMemoryReclaimTask()` to propagate the context to reclaim executor threads
- Sanity checks that memory allocations during reclaim come only from spill memory pools

### 4.4 Waiter Fairness

The `globalArbitrationWaiters_` is a `std::map<uint64_t, ArbitrationWait*>` keyed by participant ID. Since IDs are monotonically increasing, **older queries have smaller IDs and are served first**. This implements age-based fairness.

Additionally, `allocateCapacityLocked` enforces this:

```cpp
// SharedArbitrator.cpp:775
if (FOLLY_UNLIKELY(!globalArbitrationWaiters_.empty())) {
  if ((participantId > globalArbitrationWaiters_.begin()->first) &&
      (requestBytes > minAllocateBytes)) {
    return 0;  // Younger query must wait while older ones are pending
  }
  maxAllocateBytes = std::max(requestBytes, minAllocateBytes);
}
```

## 5. Memory & Resource Profile

### 5.1 Capacity Growth Strategy

The `ArbitrationParticipant` implements exponential growth with two phases:

- **Fast phase**: If `2 * currentCapacity <= fastExponentialGrowthCapacityLimit` (default 512MB), double the capacity.
- **Slow phase**: If above the limit, grow by `slowCapacityGrowPct * currentCapacity` (default 25%).

This minimizes the number of arbitration round-trips for small-to-medium queries while preventing over-allocation for large ones.

### 5.2 Shrink Strategy

When shrinking, the arbitrator preserves headroom:
- At least `minFreeCapacityRatio` (default 25%) of capacity remains free
- At least `minFreeCapacity` (default 128MB) remains free

This avoids a "capacity thrashing" pattern where a pool is shrunk aggressively and immediately needs to grow again.

### 5.3 Minimum Reclaim Threshold

The minimum reclaim bytes is `max(minReclaimBytes, minReclaimPct * capacity)` (defaults: 128MB, 25%). This prevents creating many small spill files from participants with minimal reclaimable memory.

### 5.4 Pool Initial Capacity

Each new pool receives up to `memoryPoolInitialCapacity` (default 256MB) from the arbitrator on creation, subject to availability. This pre-allocation avoids immediate arbitration for small queries.

## 6. Key Design Insights

1. **Synchronous blocking vs. asynchronous revocation.** Velox blocks the requesting operator thread on a `ContinueFuture` until memory is available. Trino's `MemoryRevokingScheduler` asynchronously sets "revoke requested" flags that operators check at yield points. Velox's approach guarantees deterministic memory resolution but can cause thread starvation under heavy memory pressure. Trino's approach is more latency-friendly but can delay memory reclamation.

2. **Centralized single-thread controller for global arbitration.** Velox uses one `GlobalArbitrationController` thread that coordinates all cross-query reclaim, dispatching actual spill work to a thread pool. This serialization prevents thundering-herd problems where multiple starving queries simultaneously try to reclaim from the same victims. Trino's `MemoryRevokingScheduler` runs on a single scheduled executor thread as well, but triggers revocation asynchronously.

3. **Priority-based victim selection with capacity bucketing.** Velox selects spill victims using a two-dimensional sort: capacity bucket (largest first) then reclaimer priority (lowest first). For abort, it additionally favors youngest queries (largest participant ID). This is more sophisticated than Trino's approach of revoking from the pool with the most revocable memory, as it explicitly considers query priority and age.

4. **Driver suspension to prevent deadlock.** Velox's `enterSuspended`/`leaveSuspended` protocol elegantly handles the case where an operator's memory request triggers reclaim of its own task. The driver thread suspends itself (decrementing `numThreads_`), so the task's `requestPause()` can complete without waiting for the requesting driver to exit. This is an inventive solution to a problem that Trino avoids entirely by decoupling revocation triggering (scheduler thread) from execution (operator threads).

5. **Two-tier free capacity (reserved + non-reserved).** The reserved capacity pool guarantees each query can get its minimum allocation even under system-wide pressure. Non-reserved capacity is available for elastic growth. This mirrors Trino's reserved pool concept but is implemented more efficiently as a simple counter split rather than a separate pool object.

6. **Abort as a timeout-based escalation.** Velox transitions from spill-based reclaim to query abort based on elapsed time (`globalArbitrationAbortTimeRatio * maxArbitrationTime`, default 50% of 5 min = 2.5 min). This prevents infinite spill cycles and guarantees forward progress. Trino handles this differently -- memory-intensive queries can be killed by an admin or by OOM protection, but there is no automatic escalation from spill to abort.

7. **Comparison with DataFusion.** DataFusion's memory management is significantly simpler. It uses a `MemoryPool` (via Apache Arrow's memory tracking) with hard limits and a `DiskManager` for spilling, but lacks a centralized arbitrator. Each operator independently decides when to spill based on its own memory usage against a configured threshold. There is no cross-query memory coordination or dynamic capacity adjustment. Velox's `SharedArbitrator` enables much more efficient memory utilization across concurrent queries at the cost of significant implementation complexity.

8. **The `nonReclaimableSection` guard pattern.** Rather than requiring operators to implement complex re-entrant reclaim logic, Velox allows operators to mark critical sections as non-reclaimable. The arbitrator simply skips these operators. This is pragmatic: it sacrifices some reclaim opportunity for implementation simplicity and correctness. The `numNonReclaimableAttempts` counter allows monitoring how often this fallback occurs.
