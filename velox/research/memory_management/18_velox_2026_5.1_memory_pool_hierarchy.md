# Module Teardown: The MemoryPool Hierarchy

## 0. Research Focus
* **Task ID:** 5.1
* **Focus:** Analyze how Velox structures memory pools (Root -> Query -> Task -> Node). Trace how bytes are reserved and tracked. How does it handle thread-safe aggregation of memory usage statistics without high lock contention?

## 1. High-Level Overview

Velox organizes memory into a four-level tree that mirrors the physical query execution plan: **Query (root) -> Task -> Node -> Operator (leaf)**. Every allocation ultimately flows through a leaf pool, but reservation accounting propagates up the tree to the root, where capacity limits are enforced. The design achieves low lock contention through three complementary strategies: (1) **quantized reservation** at the leaf level batches small allocations into MB-granularity increments so the parent chain is updated infrequently, (2) **atomic counters** (`tsan_atomic`) hold the hot-path fields (`reservationBytes_`, `usedReservationBytes_`) while a `std::mutex` guards consistency only when the quantized reservation needs to grow, and (3) the allocator layer itself uses **sharded concurrent counters** (`ConcurrentCounter`) to reduce global counter contention for small allocations.

A process-wide `MemoryManager` singleton owns the `MemoryAllocator` (physical allocation via malloc or mmap) and the `MemoryArbitrator` (dynamic capacity redistribution). Root pools register with the arbitrator, which assigns them an initial capacity and dynamically grows or shrinks capacity at runtime through spilling or query abort. This three-component separation (pool tree for accounting, allocator for physical memory, arbitrator for capacity governance) is central to Velox's memory architecture.

## 2. Structural Architecture

### 2.1 The Two Pool Kinds

The pool hierarchy has exactly two kinds of nodes. This enum is the foundation for all dispatch logic:

```cpp
// MemoryPool.h
enum class Kind {
    kLeaf = 0,       // Allocates memory. Cannot have children.
    kAggregate = 1,  // Manages hierarchy. Cannot allocate memory directly.
};
```

All actual `allocate()`/`free()` calls are enforced to go through `kLeaf` pools -- the `CHECK_AND_INC_MEM_OP_STATS` macro validates this on every allocation. The `kAggregate` pools exist solely to organize the tree and aggregate reservation accounting from their leaf descendants.

### 2.2 The Four-Level Tree

### 2.3 Class Hierarchy

The abstract `MemoryPool` base class defines the interface; `MemoryPoolImpl` is the sole concrete implementation:

```
MemoryPool (abstract, MemoryPool.h)
    |-- name_, kind_, parent_, maxCapacity_, trackUsage_, threadSafe_
    |-- children_ (weak_ptr map, protected by poolMutex_)
    |-- aborted_ (atomic bool)
    |
    +-- MemoryPoolImpl (concrete, MemoryPool.h:600)
            |-- manager_, allocator_, arbitrator_
            |-- capacity_ (tsan_atomic, only meaningful at root)
            |-- reservationBytes_ (tsan_atomic)

            |-- minReservationBytes_ (tsan_atomic, leaf only)

            |-- reclaimer_ (unique_ptr<MemoryReclaimer>)
            |-- destructionCb_ (root only, calls back to MemoryManager)
```

### 2.4 Ownership Model

Parent-child relationships use a **shared_ptr up, weak_ptr down** pattern:

- Each child holds a `std::shared_ptr<MemoryPool> parent_` -- this keeps the parent alive as long as any child exists.
- Each parent tracks children via `std::unordered_map<std::string, std::weak_ptr<MemoryPool>> children_` -- this avoids preventing child destruction.
- The `poolMutex_` (a `folly::SharedMutex`, i.e. a reader-writer lock) protects the `children_` map.

On destruction, a child first removes itself from the parent via `dropChild()`, then drops its shared reference. This is carefully ordered to avoid deadlock with `visitChildren()`:

```cpp
// MemoryPool.cpp (line 453-486)
MemoryPoolImpl::~MemoryPoolImpl() {
    if (parent_ != nullptr) {
        toImpl(parent_)->dropChild(this);
    }
    // ... leak checks, stats recording ...
    if (destructionCb_ != nullptr) {
        destructionCb_(this);  // Root pools: remove from MemoryManager::pools_
    }
}
```

## 3. Execution & Call Flow

### 3.1 Pool Creation Flow

**Root pool creation** (`MemoryManager::addRootPool`):

1. Create a `MemoryPoolImpl` with `Kind::kAggregate`, no parent, `capacity_` initialized to 0.
2. Register with the arbitrator via `arbitrator_->addPool(pool)` -- the arbitrator assigns initial capacity (default 256MB in `SharedArbitrator`).
3. Track in `MemoryManager::pools_` via `weak_ptr` under `folly::SharedMutex`.
4. Set the destruction callback.

```cpp
// Memory.cpp (line 265-280)
std::shared_ptr<MemoryPoolImpl> MemoryManager::createRootPool(...) {
    auto pool = std::make_shared<MemoryPoolImpl>(
        this, poolName, MemoryPool::Kind::kAggregate,
        nullptr, std::move(reclaimer), options);
    VELOX_CHECK_EQ(pool->capacity(), 0);
    arbitrator_->addPool(pool);    // Arbitrator assigns initial capacity
    return pool;
}
```

**Child pool creation** (`addLeafChild` / `addAggregateChild`):

1. Validate that `this` is `kAggregate` (CHECK_POOL_MANAGEMENT_OP).
2. Call `genChild()` which creates a new `MemoryPoolImpl` inheriting `trackUsage_`, `alignment_`, `debugOptions_` from parent.
3. Insert the child's `weak_ptr` into the parent's `children_` map under `poolMutex_` (unique lock).

```cpp
// MemoryPool.cpp (line 784-804)
std::shared_ptr<MemoryPool> MemoryPoolImpl::genChild(...) {
    return std::make_shared<MemoryPoolImpl>(
        manager_, name, kind, parent, std::move(reclaimer),
        Options{
            .alignment = alignment_,
            .trackUsage = trackUsage_,  // Inherited from parent/root
            .threadSafe = threadSafe,
            ...});
}
```

### 3.2 Allocation Flow (the Hot Path)

When a leaf pool's `allocate()` is called, the path is: **reserve -> allocate physical -> track debug info**:

```cpp
// MemoryPool.cpp (line 508-541)

    CHECK_AND_INC_MEM_OP_STATS(this, Allocs);  // Enforce kLeaf, ++numAllocs_
    const auto alignedSize = sizeAlign(size);
    reserve(alignedSize);                        // Step 1: Reserve in pool tree
    void* buffer = allocator_->allocateBytes(alignedSize, alignment_);  // Step 2: Physical alloc
    if (buffer == nullptr) {
        release(alignedSize);
        handleAllocationFailure(...);
    }
    DEBUG_RECORD_ALLOC(this, buffer, size);      // Step 3: Debug tracking
    return buffer;
}
```

### 3.3 Reservation Flow (Detailed)

The `reserve()` method dispatches to thread-safe or non-thread-safe paths:

```cpp
// MemoryPool.cpp (line 828-836)
void MemoryPoolImpl::reserve(uint64_t size, bool reserveOnly) {
    if (FOLLY_LIKELY(trackUsage_)) {
        if (FOLLY_LIKELY(threadSafe_)) {
            reserveThreadSafe(size, reserveOnly);
        } else {
            reserveNonThreadSafe(size, reserveOnly);
        }
    }
}
```

**`reserveThreadSafe` (the common path):**

```cpp
// MemoryPool.cpp (line 838-880)
void MemoryPoolImpl::reserveThreadSafe(uint64_t size, bool reserveOnly) {
    int32_t numAttempts = 0;
    int64_t increment = 0;
    for (;; ++numAttempts) {
        {
            std::lock_guard<std::mutex> l(mutex_);     // [A] Lock leaf's mutex
            increment = reservationSizeLocked(size);    // [B] Check if enough slack
            if (increment == 0) {                       // [C] Enough slack exists
                usedReservationBytes_ += size;
                cumulativeBytes_ += size;
                maybeUpdatePeakBytesLocked(usedReservationBytes_);
                break;                                  // FAST PATH: no parent update
            }
        }
        // SLOW PATH: Need more reservation from parent chain
        incrementReservationThreadSafe(this, increment);
    }
    if (numAttempts > 1) {
        numCollisions_ += numAttempts - 1;  // Track contention
    }
}
```

The key insight is `reservationSizeLocked()`:

```cpp
// MemoryPool.h (line 854-861)
FOLLY_ALWAYS_INLINE int64_t reservationSizeLocked(int64_t size) {
    const int64_t neededSize = size - (reservationBytes_ - usedReservationBytes_);
    if (neededSize <= 0) {
        return 0;   // Already have enough slack -- no parent update needed
    }
    return roundedDelta(reservationBytes_, neededSize);  // Quantized increment
}
```

If the leaf already has enough unused reservation (`reservationBytes_ - usedReservationBytes_ >= size`), the hot path completes with only the leaf's `mutex_` -- no parent locking at all. This is the **fast path**.

### 3.4 Parent Chain Propagation (Slow Path)

When the leaf needs more reservation, `incrementReservationThreadSafe` **recursively walks up** the tree:

```cpp
// MemoryPool.cpp (line 882-907)
void MemoryPoolImpl::incrementReservationThreadSafe(
    MemoryPool* requestor, uint64_t size) {
    // Step 1: Propagate UP to root first
    if (parent_ != nullptr) {
        toImpl(parent_)->incrementReservationThreadSafe(requestor, size);
    }
    // Step 2: Try to increment this node's reservation
    if (maybeIncrementReservation(size)) {
        return;
    }
    // Step 3: At root, capacity exceeded -- trigger arbitration
    VELOX_CHECK_NULL(parent_);
    growCapacity(requestor, size);
}
```

At each intermediate (aggregate) node, `maybeIncrementReservation` atomically increments `reservationBytes_` under `mutex_`:

```cpp
// MemoryPool.cpp (line 934-951)
bool MemoryPoolImpl::maybeIncrementReservation(uint64_t size) {
    std::lock_guard<std::mutex> l(mutex_);
    if (isRoot()) {
        checkIfAborted();
        if ((reservationBytes_ + size > capacity_) && !underMemoryArbitration()) {
            return false;  // Capacity exceeded at root
        }
    }
    incrementReservationLocked(size);  // reservationBytes_ += size
    return true;
}
```

### 3.5 Quantized Reservation (Batching Strategy)

The `quantizedSize()` function is the key to reducing parent chain updates:

```cpp
// MemoryPool.h (line 494-502)
FOLLY_ALWAYS_INLINE static uint64_t quantizedSize(uint64_t size) {
    if (size < 16 * kMB) {
        return bits::roundUp(size, kMB);       // 1MB granularity for small
    }
    if (size < 64 * kMB) {
        return bits::roundUp(size, 4 * kMB);   // 4MB granularity for medium
    }
    return bits::roundUp(size, 8 * kMB);       // 8MB granularity for large
}
```

When `reserveThreadSafe` needs more bytes, it computes `roundedDelta(reservationBytes_, neededSize)` which rounds up to the next quantum. For example, if the current reservation is 3MB and the operator needs 200KB more, the system will request 1MB from the parent chain, giving the leaf 800KB of slack for future allocations. This means subsequent small allocations will hit the fast path and not touch any parent locks.

### 3.6 Release Flow

Release is the mirror of reserve. At the leaf level, it computes how much excess reservation can be freed:

```cpp
// MemoryPool.cpp (line 976-1006)
void MemoryPoolImpl::releaseThreadSafe(uint64_t size, bool releaseOnly) {
    int64_t freeable = 0;
    {
        std::lock_guard<std::mutex> l(mutex_);
        usedReservationBytes_ -= size;
        const int64_t newCap = std::max(minReservationBytes_, usedReservationBytes_);
        int64_t newQuantized = quantizedSize(newCap);
        freeable = reservationBytes_ - newQuantized;
        if (freeable > 0) {
            reservationBytes_ = newQuantized;
        }
    }
    if (freeable > 0) {
        toImpl(parent_)->decrementReservation(freeable);  // Propagate down
    }
}
```

`decrementReservation` walks up the tree, decrementing `reservationBytes_` at each level under its own mutex:

```cpp
// MemoryPool.cpp (line 1008-1017)
void MemoryPoolImpl::decrementReservation(uint64_t size) noexcept {
    if (parent_ != nullptr) {
        toImpl(parent_)->decrementReservation(size);
    }
    std::lock_guard<std::mutex> l(mutex_);
    reservationBytes_ -= size;
}
```

### 3.7 Capacity Growth via Arbitration

When `maybeIncrementReservation` fails at the root (reservation would exceed capacity), the system triggers memory arbitration:

```cpp
// MemoryPool.cpp (line 909-932)
void MemoryPoolImpl::growCapacity(MemoryPool* requestor, uint64_t size) {
    ++numCapacityGrowths_;
    try {
        MemoryPoolArbitrationSection arbitrationSection(requestor);
        arbitrator_->growCapacity(this, size);
    } catch (const VeloxRuntimeError& veloxError) { ... }
}
```

The `MemoryPoolArbitrationSection` is an RAII guard that calls `enterArbitration()` / `leaveArbitration()` on the pool's reclaimer, which suspends the requesting driver thread to prevent deadlock during reclaim. The arbitrator (`SharedArbitrator`) then tries, in order: (1) shrink free capacity from other pools, (2) reclaim used memory via spilling, (3) abort younger queries.

`shrink()` and `grow()` are the root-level capacity adjustment primitives:

```cpp
// MemoryPool.cpp (line 1143-1181)
uint64_t MemoryPoolImpl::shrink(uint64_t targetBytes) {
    // Delegates to root
    std::lock_guard<std::mutex> l(mutex_);
    uint64_t freeBytes = std::max<uint64_t>(0, capacity_ - reservationBytes_);
    if (targetBytes != 0) freeBytes = std::min(targetBytes, freeBytes);
    capacity_ -= freeBytes;
    return freeBytes;
}

bool MemoryPoolImpl::grow(uint64_t growBytes, uint64_t reservationBytes) {
    // Delegates to root
    std::lock_guard<std::mutex> l(mutex_);
    if (capacity_ + growBytes > maxCapacity_) return false;
    if (reservationBytes_ + reservationBytes > capacity_ + growBytes) return false;
    capacity_ += growBytes;
    if (reservationBytes > 0) incrementReservationLocked(reservationBytes);
    return true;
}
```

## 4. Concurrency & State Management

### 4.1 The Three-Mutex Strategy

Velox uses a careful layering of locks:

| Lock | Scope | Protects | Type |
|------|-------|----------|------|
| `poolMutex_` | `MemoryPool` base | `children_` map | `folly::SharedMutex` (reader-writer) |
| `mutex_` | `MemoryPoolImpl` | `reservationBytes_`, `usedReservationBytes_`, `minReservationBytes_`, `capacity_` | `std::mutex` |
| `MemoryManager::mutex_` | Singleton | `pools_` map | `folly::SharedMutex` |

The `poolMutex_` is a reader-writer lock because `visitChildren()` (read) is far more frequent than `addLeafChild()`/`dropChild()` (write). The per-pool `mutex_` is a plain mutex because reservation updates need exclusive access for consistency.

### 4.2 Lock-Free Hot Path Design

The critical optimization: **most allocations never touch parent locks**. The quantized reservation system means the leaf pool typically has slack between `reservationBytes_` and `usedReservationBytes_`. Only the leaf's own `mutex_` is acquired on the fast path:

```
Allocation fast path (most common):
    leaf.mutex_ lock
    -> check reservationSizeLocked(size) == 0
    -> usedReservationBytes_ += size
    leaf.mutex_ unlock
    -> allocator_->allocateBytes(...)
```

When the leaf's reservation is exhausted (slow path), locks are acquired bottom-up:
```
Allocation slow path (infrequent):
    leaf.mutex_ lock -> determine increment needed -> leaf.mutex_ unlock
    -> parent.mutex_ lock -> incrementReservationLocked -> parent.mutex_ unlock
    -> root.mutex_ lock -> check capacity, incrementReservationLocked -> root.mutex_ unlock
    -> (if capacity exceeded: arbitrator takes over)
    -> leaf.mutex_ lock -> usedReservationBytes_ += size -> leaf.mutex_ unlock
```

### 4.3 tsan_atomic: Zero-Cost Atomics in Production

The `reservationBytes_`, `usedReservationBytes_`, `capacity_` fields are declared as `tsan_atomic<int64_t>`. In production builds, this is just a plain `int64_t` (no atomic overhead). Under ThreadSanitizer (TSAN) builds, it becomes `std::atomic<int64_t>` to suppress false-positive race reports:

```cpp
// Portability.h (line 55-81)
#ifdef TSAN_BUILD
template <typename T>
using tsan_atomic = std::atomic<T>;
#else
template <typename T>
using tsan_atomic = T;
#endif
```

This is safe because the fields are always accessed under `mutex_` in the thread-safe path. The `tsan_atomic` type simply tells TSAN that the access pattern is intentional.

### 4.4 Stats Counters as Atomic

The stats counters (`numAllocs_`, `numFrees_`, `numReserves_`, etc.) use `std::atomic_uint64_t`, allowing lock-free incrementing even from different threads:

```cpp
// MemoryPool.h (line 1077-1096)
std::atomic_uint64_t numAllocs_{0};
std::atomic_uint64_t numFrees_{0};
std::atomic_uint64_t numCollisions_{0};
std::atomic_uint64_t numCapacityGrowths_{0};
```

The retry-loop in `reserveThreadSafe` counts collisions -- situations where concurrent reserve/release on sibling pools temporarily invalidate the reservation calculation:

### 4.6 Allocator-Level Sharding (ConcurrentCounter)

The `MallocAllocator` adds a second layer of contention reduction for the global `allocatedBytes_` counter. Small allocations (< `reservationByteLimit_`, default 1MB) go through `ConcurrentCounter<uint32_t> reservations_`:

```cpp
// ConcurrentCounter.h
template <class T>
class ConcurrentCounter {
    struct alignas(folly::hardware_destructive_interference_size) Counter {
        mutable std::mutex lock;
        T value;
    };

    size_t shardIndex() const {
        const size_t hash = std::hash<std::thread::id>{}(std::this_thread::get_id());
        return hash & shardMask_;
    }
};
```

Each thread hashes to a shard (cache-line aligned to prevent false sharing). Shards locally accumulate reservation bytes. Only when a shard runs out does it "refill" by atomically incrementing the global `allocatedBytes_` counter by `reservationByteLimit_` (1MB):

```cpp
// MallocAllocator.h (line 139-159)
inline bool incrementUsageWithReservation(uint32_t bytes) {
    return reservations_.update(bytes, reserveFunc_);
}

// reserveFunc_ logic:
inline bool incrementUsageWithReservationFunc(
    uint32_t& counter, uint32_t increment, std::mutex& lock) {
    std::lock_guard<std::mutex> l(lock);
    if (counter > increment) {
        counter -= increment;
        return true;             // Fast path: no global update
    }
    if (!incrementUsageWithoutReservation(reservationByteLimit_)) {
        return false;            // Global capacity exceeded
    }
    counter += reservationByteLimit_;  // Refill from global
    counter -= increment;
    return true;
}
```

Frees symmetrically return excess local reservation back to the global counter when a shard accumulates more than `2 * reservationByteLimit_`.

### 4.7 visitChildren Lock-Safety

The `visitChildren()` method carefully avoids recursive locking by collecting child shared_ptrs under lock, then invoking the visitor outside the lock:

```cpp
// MemoryPool.cpp (line 287-315)
void MemoryPool::visitChildren(const std::function<bool(MemoryPool*)>& visitor) const {
    std::vector<std::shared_ptr<MemoryPool>> children;
    {
        std::shared_lock guard{poolMutex_};  // Read lock only
        for (auto& entry : children_) {
            auto child = entry.second.lock();
            if (child != nullptr) children.push_back(std::move(child));
        }
    }
    // Call visitor outside lock to prevent:
    // 1. User code re-acquiring poolMutex_
    // 2. Last-reference destruction triggering dropChild() deadlock
    for (auto& child : children) {
        if (!visitor(child.get())) return;
    }
}
```

## 5. Memory & Resource Profile

### 5.1 Reservation vs Used vs Capacity

Each leaf pool maintains three reservation levels:

- **`usedReservationBytes_`**: Actual bytes consumed by operator data structures. Only tracked at leaf level.
- **`reservationBytes_`**: Quantized reservation held at every level. Always >= `usedReservationBytes_` at leaf. Represents "committed" bytes that count against root capacity.
- **`minReservationBytes_`**: A floor set by `maybeReserve()` to pre-allocate headroom (e.g., 8MB growth quantum). Prevents premature release of slack.
- **`capacity_`**: Only meaningful at root. Set by the arbitrator. Hard upper bound for `reservationBytes_`.

The hierarchy invariant is:
```
root.capacity_ >= root.reservationBytes_ >= sum(child.reservationBytes_)
leaf.reservationBytes_ >= leaf.usedReservationBytes_
leaf.reservationBytes_ >= leaf.minReservationBytes_
```

### 5.2 Quantization Tiers

| Reservation Range | Granularity |
|-------------------|-------------|
| 0 - 16 MB | 1 MB |
| 16 MB - 64 MB | 4 MB |
| 64 MB+ | 8 MB |

This means a leaf pool always holds at most 8MB of unused reservation. The overhead is bounded and trades a small amount of over-reservation for dramatically fewer parent chain updates.

### 5.3 Pool Lifetime and Leak Detection

On `MemoryPoolImpl` destruction:
1. Debug mode: `leakCheckDbg()` verifies all allocations have been freed.
2. Warns if `usedReservationBytes_ > 0` (memory leak) or `minReservationBytes_ > 0` (reservation leak).
3. Debug-asserts that all three counters are zero.
4. Root pools record `numCapacityGrowths_` as a histogram metric.
5. Root pools invoke `destructionCb_` to deregister from `MemoryManager`.

### 5.4 maybeReserve: Explicit Pre-Allocation

Operators that know they will need a large chunk of memory can call `maybeReserve()` to pre-allocate:

```cpp
// MemoryPool.cpp (line 806-826)
bool MemoryPoolImpl::maybeReserve(uint64_t increment) {
    constexpr int32_t kGrowthQuantum = 8 << 20;  // 8 MB
    const auto reservationToAdd = bits::roundUp(increment, kGrowthQuantum);
    try {
        reserve(reservationToAdd, true);  // reserveOnly=true
    } catch (...) {
        if (aborted()) std::rethrow_exception(std::current_exception());
        return false;  // Graceful failure
    }
    return true;
}
```

This sets `minReservationBytes_` to prevent the reservation from being released until an explicit `release()` call. HashBuild and similar operators use this to ensure they can grow their hash tables without repeatedly hitting arbitration.

### 5.5 Shared Leaf Pools

For lightweight library usage that does not need per-query isolation, `MemoryManager` maintains 32 (configurable) shared leaf pools under the system root. Access is thread-hashed:

```cpp
// Memory.cpp (line 355-358)
```

## 6. Key Design Insights

1. **Quantized reservation is the core contention-reduction mechanism.** By rounding leaf reservations to MB boundaries, Velox ensures that the vast majority of allocations are "absorbed" locally at the leaf level. The parent chain is only updated when crossing a quantum boundary. This is analogous to Trino's approach where `MemoryPool.localMemoryContext` accumulates bytes locally and only propagates delta to the parent above a threshold, but Velox makes the quantization explicit with a tiered scheme (1MB/4MB/8MB) versus Trino's fixed-size approach.

2. **Two-phase capacity management separates accounting from physical allocation.** The `MemoryPool` tree handles reservation accounting (logical), while `MemoryAllocator` handles physical allocation (malloc/mmap). This is cleaner than Trino's model where the pool hierarchy mixes accounting with allocation, and more explicit than DataFusion's RAII model where `MemoryReservation` objects couple allocation tracking to `Drop` semantics. Velox's split allows the arbitrator to adjust capacity without touching actual memory.

3. **The arbitrator as a peer component, not part of the pool tree.** In Trino, memory limits are enforced by the pool tree itself (each pool checks against its parent's limit). In Velox, the root pool delegates capacity decisions to a pluggable `MemoryArbitrator`. This enables dynamic memory redistribution between queries (grow pool A by shrinking pool B) which Trino's static hierarchy cannot do without external intervention.

4. **Thread-safe vs non-thread-safe as a per-leaf option.** Velox recognizes that many operators execute single-threaded within a driver pipeline. By allowing `threadSafe=false` on individual leaf pools, it eliminates mutex overhead entirely for those cases. Neither Trino nor DataFusion offer this granularity -- Trino's `MemoryPool` always uses `AtomicLong` for thread safety, and DataFusion's `MemoryReservation` is single-owner by design.

5. **Shared_ptr up, weak_ptr down prevents zombie trees.** The ownership model where children hold strong references to parents (keeping them alive) while parents hold weak references to children (allowing them to die) ensures that the tree cannot have orphaned subtrees. Trino uses a similar parent-holds-child pattern but with explicit `close()` calls rather than reference counting. DataFusion relies on Rust's ownership model where pools are typically scoped to the `TaskContext` lifetime.

6. **ConcurrentCounter with cache-line alignment at the allocator level.** The `alignas(folly::hardware_destructive_interference_size)` on sharded counters prevents false sharing between CPU cores. Combined with the per-shard mutex and batch refill strategy (refill by 1MB at a time), this makes the global allocation counter effectively contention-free for small allocations. This is a level of optimization that neither Trino (which uses `AtomicLong` for similar counters) nor DataFusion (which uses `AtomicUsize`) attempt.

7. **Abort propagation through the pool tree.** When the arbitrator decides to abort a query, it calls `abort()` on the root pool, which sets `aborted_=true` and propagates via the reclaimer. Any subsequent allocation attempt on any pool in the subtree will check `aborted()` (which walks up to the root) and rethrow the abort error. This provides a uniform cancellation mechanism that integrates with the memory system rather than requiring a separate cancellation channel.

8. **The reclaimer hierarchy mirrors the pool hierarchy.** Each level of the pool tree can have its own `MemoryReclaimer` with priority-based ordering. The `SharedArbitrator` uses this to implement a policy where it first tries to reclaim from lower-priority reclaimers (e.g., hash join build sides that can spill) before escalating to higher-cost operations. This is a more sophisticated approach than Trino's flat memory revocation, which simply picks the largest memory consumer to revoke.
