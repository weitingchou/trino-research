## 1. High-Level Overview

Velox implements output buffering through a three-layer architecture that separates partition routing, serialization, and network-level buffering:

```
RowVector (columnar in-memory)
    |
    v
PartitionedOutput operator          -- per-driver: hashes rows, assigns to destinations
    |  (one detail::Destination per partition)
    v
OutputBufferManager (global singleton) -- taskId lookup
    |
    v
OutputBuffer (per-task)             -- holds N DestinationBuffer queues
    |
    v
DestinationBuffer                   -- per-downstream-consumer page queue
    |
    v
Network layer (Prestissimo HTTP)    -- getData() callback-driven pull
```

The design mirrors Trino's `OutputBuffer` / `PartitionedOutputBuffer`, but with two key Velox-specific differences: (1) serialization happens inside the operator on the driver thread, not in the buffer layer, and (2) the buffer uses a callback-based pull model (`DataAvailableCallback`) rather than Trino's `ListenableFuture`-based approach.

Three output distribution modes are supported, selected by `PartitionedOutputNode::Kind`:

| Kind | Behavior |
|------|----------|
| `kPartitioned` | Hash-based or round-robin: each row goes to exactly one destination |
| `kBroadcast` | Every page is cloned to all destinations |
| `kArbitrary` | Pages go to a shared pool; first active consumer takes them |

## 2. Structural Architecture

### 2.1 PartitionedOutput Operator

`PartitionedOutput` is an `Operator` subclass. It is the terminal (sink) operator of every stage that needs to shuffle data. Unlike most operators that produce output via `getOutput()`, this operator returns `nullptr` from `getOutput()` because its output is sent laterally to the `OutputBufferManager` rather than downstream in the pipeline.

Key members:

```cpp
class PartitionedOutput : public Operator {
  const std::vector<column_index_t> keyChannels_;       // partition key column indices
  const int numDestinations_;                            // number of downstream partitions
  const bool replicateNullsAndAny_;                      // Trino-compat null handling
  std::unique_ptr<core::PartitionFunction> partitionFunction_;  // hash or round-robin
  const std::vector<column_index_t> outputChannels_;     // column reordering/projection
  const std::weak_ptr<exec::OutputBufferManager> bufferManager_;
  const int64_t maxBufferedBytes_;                       // per-operator buffer limit (default 32MB)
  VectorSerde* const serde_;                             // Presto, CompactRow, or UnsafeRow
  std::vector<std::unique_ptr<detail::Destination>> destinations_;  // one per partition
  std::vector<uint32_t> partitions_;                     // reusable partition assignment vector
  // ...
};
```

### 2.2 detail::Destination

Each `detail::Destination` object manages the serialization buffer for one downstream partition. It accumulates rows (by index), serializes them in batches, and flushes serialized pages to the `OutputBufferManager`.

Key members:

```cpp
class Destination {
  const std::string taskId_;
  const int destination_;                          // partition index
  VectorSerde* const serde_;
  std::unique_ptr<VectorStreamGroup> current_;     // streaming serializer
  raw_vector<vector_size_t> rows_;                 // row indices assigned to this destination
  vector_size_t rowIdx_;                           // cursor within rows_
  uint64_t bytesInCurrent_;                        // bytes accumulated since last flush
  vector_size_t rowsInCurrent_;                    // rows accumulated since last flush
  int32_t targetSizePct_;                          // randomized flush threshold (70-120%)
  int32_t targetNumRows_;                          // row-count flush threshold
  // ...
};
```

### 2.3 OutputBuffer (Per-Task)

`OutputBuffer` is the central buffering structure, created once per task. It holds:

- A `vector<unique_ptr<DestinationBuffer>>` -- one queue per downstream consumer.
- An optional `ArbitraryBuffer` (for `kArbitrary` mode) that pools pages before dispatching.
- Backpressure state: `maxSize_`, `continueSize_`, `promises_` for blocking producers.

```cpp
class OutputBuffer {
  const std::shared_ptr<Task> task_;
  const core::PartitionedOutputNode::Kind kind_;
  const uint64_t maxSize_;              // default 32MB, from QueryConfig::maxOutputBufferSize
  const uint64_t continueSize_;         // 90% of maxSize_
  std::vector<std::unique_ptr<DestinationBuffer>> buffers_;
  int64_t bufferedBytes_;               // total live bytes across all destinations
  std::vector<ContinuePromise> promises_;  // blocked producer futures
  std::mutex mutex_;                    // single mutex for all operations
  // ...
};
```

### 2.4 DestinationBuffer

Each `DestinationBuffer` is a simple append-only queue of `shared_ptr<SerializedPageBase>`. The consumer calls `getData()` to pull pages; if no data is available, a `DataAvailableCallback` is registered for push notification.

```cpp
class DestinationBuffer {
  std::vector<std::shared_ptr<SerializedPageBase>> data_;
  int64_t sequence_;                    // sequence number of first element in data_
  DataAvailableCallback notify_;        // registered by consumer when no data available
  // ...
};
```

### 2.5 OutputBufferManager (Global Singleton)

`OutputBufferManager` is a process-wide singleton that maps `taskId -> shared_ptr<OutputBuffer>`. All interactions between the PartitionedOutput operator and the network layer go through this manager.

```cpp
class OutputBufferManager {
  folly::Synchronized<
      std::unordered_map<std::string, std::shared_ptr<OutputBuffer>>,
      std::mutex> buffers_;
};
```

## 3. Execution & Call Flow

### 3.1 addInput(): Row Routing

When a batch of rows arrives, `addInput()` performs four steps:

**Step 1 -- Input preparation (`initializeInput`):**

```cpp
void PartitionedOutput::initializeInput(RowVectorPtr input) {
  // If outputChannels_ is non-empty, project/reorder columns
  if (outputChannels_.empty()) {
    output_ = input_;
  } else {
    std::vector<VectorPtr> outputColumns;
    for (auto i : outputChannels_) {
      outputColumns.push_back(input_->childAt(i));
    }
    output_ = std::make_shared<RowVector>(..., outputColumns);
  }
  // Lazy-load all columns
  for (auto i = 0; i < output_->childrenSize(); ++i) {
    output_->childAt(i)->loadedVector();
  }
}
```

This is a zero-copy column projection -- it wraps existing column vectors in a new `RowVector` with a different column order.

**Step 2 -- Row size estimation (`estimateRowSizes`):**

The serde's `estimateSerializedSize()` is called to compute per-row byte sizes. This enables the serializer to decide when to flush based on byte counts rather than row counts alone.

**Step 3 -- Partition assignment:**

For single-destination output (`numDestinations_ == 1`), all rows go to `destinations_[0]`:

```cpp
if (numDestinations_ == 1) {
  destinations_[0]->addRows(IndexRange{0, numInput});
}
```

For multi-destination, the `PartitionFunction` is called:

```cpp
auto singlePartition = partitionFunction_->partition(*input_, partitions_);
```

The `partition()` call fills `partitions_[i]` with the target partition for row `i`. If all rows land in the same partition, `singlePartition` is set (optimization for skewed data).

Then rows are distributed:

```cpp
if (singlePartition.has_value()) {
  destinations_[singlePartition.value()]->addRows(IndexRange{0, numInput});
} else {
  for (vector_size_t i = 0; i < numInput; ++i) {
    destinations_[partitions_[i]]->addRow(i);
  }
}
```

**Step 3b -- replicateNullsAndAny handling:**

When `replicateNullsAndAny_` is true (used for outer joins in Trino-compatible mode):

1. The first row of the very first batch is replicated to ALL destinations.
2. Any row with a NULL in any partition key column is replicated to ALL destinations.
3. Other rows are hash-partitioned normally.

```cpp
if (!replicatedAny_) {
  for (auto& destination : destinations_) {
    destination->addRow(0);
  }
  replicatedAny_ = true;
  start = 1;
}
for (auto i = start; i < numInput; ++i) {
  if (nullRows_.isValid(i)) {
    for (auto& destination : destinations_) {
      destination->addRow(i);
    }
  } else {
    destinations_[partitions_[i]]->addRow(i);
  }
}
```

### 3.2 Hash Partitioning: HashPartitionFunction

The hash function uses Velox's `VectorHasher` framework:

```cpp
std::optional<uint32_t> HashPartitionFunction::partition(
    const RowVector& input, std::vector<uint32_t>& partitions) {
  if (hashers_.empty()) return 0u;  // no keys => all to partition 0

  // Compute combined hash for all key columns
  for (auto i = 0; i < hashers_.size(); ++i) {
    hashers_[i]->decode(*input.childAt(hasher->channel()), rows_);
    hashers_[i]->hash(rows_, i > 0, hashes_);  // i>0 = combine with previous
  }

  // Map hash to partition number
  partitions.resize(size);
  for (auto i = 0; i < size; ++i) {
    partitions[i] = hashes_[i] % numPartitions_;
  }
  return std::nullopt;  // different rows may go to different partitions
}
```

Two partition-to-number mapping strategies:

1. **Simple modulo:** `hashes_[i] % numPartitions_` -- used for remote shuffle.
2. **Bit-range extraction:** `HashBitRange::partition(hash)` extracts bits `[begin_, end_)` from the hash. This is equivalent to `(hash >> begin_) & fieldMask_` and produces `2^(end_-begin_)` partitions. Used when the number of partitions is a power of two.

For local exchange, the raw hash is further scrambled through `localExchangeHash()`:

```cpp
static inline uint32_t localExchangeHash(uint32_t rawHash) {
  bits::reverseBits(reinterpret_cast<uint8_t*>(&rawHash), sizeof(rawHash));
  return XXH32(&rawHash, sizeof(rawHash), 0);
}
```

This ensures that local and remote exchanges do not use the same hash bits, preventing correlated hot spots.

### 3.3 getOutput(): Serialization and Flush

`getOutput()` drives the serialization loop. Despite being called `getOutput()`, it always returns `nullptr` -- its purpose is to serialize and enqueue data into the OutputBufferManager.

```cpp
RowVectorPtr PartitionedOutput::getOutput() {
  const uint64_t maxPageSize = std::max(kMinDestinationSize,
      std::min(kMaxPageSize, maxBufferedBytes_ / numDestinations_));

  bool workLeft;
  do {
    workLeft = false;
    for (auto& destination : destinations_) {
      bool atEnd = false;
      blockingReason_ = destination->advance(
          maxPageSize, rowSize_, output_, ..., &atEnd, &future_, scratch_);
      if (blockingReason_ != BlockingReason::kNotBlocked) {
        blockedDestination = destination.get();
        break;
      }
      if (!atEnd) workLeft = true;
    }
  } while (workLeft);
  // ...
}
```

The `maxPageSize` calculation bounds individual serialized pages to `min(1MB, maxBufferedBytes / numDestinations)`, with a floor of `kMinDestinationSize = 60KB` (designed to fit a 64K network MTU).

### 3.4 Destination::advance(): The Serialization Core

```cpp
BlockingReason Destination::advance(uint64_t maxBytes, ...) {
  const uint32_t adjustedMaxBytes = (maxBytes * targetSizePct_) / 100;
  if (bytesInCurrent_ >= adjustedMaxBytes) {
    return flush(bufferManager, bufferReleaseFn, future);
  }

  // Accumulate rows until byte or row threshold
  while (rowIdx_ < rows_.size() && !shouldFlush) {
    bytesInCurrent_ += sizes[rows_[rowIdx_]];
    ++rowIdx_;
    ++rowsInCurrent_;
    shouldFlush = bytesInCurrent_ >= adjustedMaxBytes || rowsInCurrent_ >= targetNumRows_;
  }

  // Serialize the accumulated rows
  createVectorStreamGroup(output);
  if (serde_->kind() == "CompactRow") {
    current_->append(*outputCompactRow, rows, sizes);
  } else if (serde_->kind() == "UnsafeRow") {
    current_->append(*outputUnsafeRow, rows, sizes);
  } else {
    current_->append(output, rows, scratch);  // Presto columnar serde
  }

  if (shouldFlush || (eagerFlush_ && rowsInCurrent_ > 0)) {
    return flush(bufferManager, bufferReleaseFn, future);
  }
  return BlockingReason::kNotBlocked;
}
```

The `targetSizePct_` is randomized per-destination per-batch to between 70% and 120% of the nominal flush size. This deliberately staggers flush times across destinations to avoid bursty network traffic patterns where all destinations try to send simultaneously.

```cpp
void setTargetSizePct() {
  targetSizePct_ = 70 + (folly::Random::rand32(rng_) % 50);
  targetNumRows_ = (10'000 * targetSizePct_) / 100;
}
```

### 3.5 Destination::flush(): Serialization Finalization

```cpp
BlockingReason Destination::flush(OutputBufferManager& bufferManager, ...) {
  if (!current_ || rowsInCurrent_ == 0) return kNotBlocked;

  auto listener = bufferManager.newListener();
  IOBufOutputStream stream(*current_->pool(), listener.get(),
      std::max(kMinMessageSize, current_->size()));
  current_->flush(&stream);               // serialize to wire format

  clearVectorStreamGroup();                // reset serializer for next batch

  bool blocked = bufferManager.enqueue(
      taskId_, destination_,
      std::make_unique<PrestoSerializedPage>(
          stream.getIOBuf(bufferReleaseFn), nullptr, flushedRows),
      future);

  return blocked ? kWaitForConsumer : kNotBlocked;
}
```

The flush path:
1. Creates an `IOBufOutputStream` backed by a `StreamArena` allocation.
2. Calls `VectorStreamGroup::flush()` which invokes the serde's `IterativeVectorSerializer` to write the accumulated data in wire format.
3. Wraps the resulting `folly::IOBuf` in a `PrestoSerializedPage`.
4. Enqueues the page into the `OutputBufferManager`, which may return `blocked=true` if the total buffer is full.

### 3.6 OutputBuffer::enqueue(): Buffer-Side Ingestion

```cpp
bool OutputBuffer::enqueue(int destination, unique_ptr<SerializedPageBase> data,
                           ContinueFuture* future) {
  std::vector<DataAvailable> dataAvailableCallbacks;
  bool blocked = false;
  {
    std::lock_guard<std::mutex> l(mutex_);
    updateStatsWithEnqueuedPageLocked(data->size(), data->numRows().value());

    switch (kind_) {
      case kBroadcast:
        enqueueBroadcastOutputLocked(std::move(data), dataAvailableCallbacks);
        break;
      case kArbitrary:
        enqueueArbitraryOutputLocked(std::move(data), dataAvailableCallbacks);
        break;
      case kPartitioned:
        enqueuePartitionedOutputLocked(destination, std::move(data), dataAvailableCallbacks);
        break;
    }

    if (bufferedBytes_ >= maxSize_ && future) {
      promises_.emplace_back("OutputBuffer::enqueue");
      *future = promises_.back().getSemiFuture();
      blocked = true;
    }
  }
  // Notify consumers OUTSIDE the mutex
  for (auto& callback : dataAvailableCallbacks) {
    callback.notify();
  }
  return blocked;
}
```

The three enqueue strategies:

**Partitioned:** Direct 1:1 dispatch to the target `DestinationBuffer`.

```cpp
void OutputBuffer::enqueuePartitionedOutputLocked(int destination, ...) {
  auto* buffer = buffers_[destination].get();
  if (buffer != nullptr) {
    buffer->enqueue(std::move(data));
    dataAvailableCbs.emplace_back(buffer->getAndClearNotify());
  }
  // If buffer is null, downstream finished early -- data is dropped.
}
```

**Broadcast:** The page is wrapped in a `shared_ptr` and enqueued to every `DestinationBuffer`. All destinations share the same underlying serialized page (no copy).

```cpp
void OutputBuffer::enqueueBroadcastOutputLocked(...) {
  std::shared_ptr<SerializedPageBase> sharedData(data.release());
  for (auto& buffer : buffers_) {
    if (buffer != nullptr) {
      buffer->enqueue(sharedData);  // shared_ptr copy, not data copy
      dataAvailableCbs.emplace_back(buffer->getAndClearNotify());
    }
  }
  if (!noMoreBuffers_) {
    dataToBroadcast_.emplace_back(sharedData);  // save for late-arriving destinations
  }
}
```

**Arbitrary:** Pages go into a shared `ArbitraryBuffer` pool. The round-robin dispatch tries to load data into whichever destination has a pending `getData()` request.

```cpp
void OutputBuffer::enqueueArbitraryOutputLocked(...) {
  arbitraryBuffer_->enqueue(std::move(data));
  int32_t bufferId = nextArbitraryLoadBufferIndex_;
  for (int32_t i = 0; i < buffers_.size(); ...) {
    if (arbitraryBuffer_->empty()) break;
    buffer->maybeLoadData(arbitraryBuffer_.get());
    dataAvailableCbs.emplace_back(buffer->getAndClearNotify());
  }
}
```

### 3.7 Consumer-Side Pull: getData()

When a downstream consumer (typically Prestissimo's HTTP handler) wants data:

```cpp
void OutputBuffer::getData(int destination, uint64_t maxBytes, int64_t sequence,
                           DataAvailableCallback notify, ...) {
  std::lock_guard<std::mutex> l(mutex_);
  auto* buffer = buffers_[destination].get();
  // Implicit acknowledgement of previous data
  freed = buffer->acknowledge(sequence, true);
  // Get available data
  data = buffer->getData(maxBytes, sequence, notify, activeCheck, arbitraryBuffer_.get());
}
```

`DestinationBuffer::getData()` returns up to `maxBytes` of serialized pages starting at the given `sequence` number. If no data is available, the `notify` callback is registered and will be invoked (outside the mutex) when data arrives.

### 3.8 Serialization Format

Velox supports three serde formats, selected per-query:

| Format | Wire Representation | When Used |
|--------|-------------------|-----------|
| `Presto` (default) | Columnar PrestoPage format with type headers | Network shuffle (Prestissimo), Trino-compatible |
| `CompactRow` | Row-oriented compact binary format | Spilling, local exchange |
| `UnsafeRow` | Spark-compatible row format | Spark integration |

For the `Presto` format, `PrestoVectorSerde` uses an `IterativeVectorSerializer` that:
1. Flattens all encodings (dictionary, constant, etc.) during serialization.
2. Supports optional compression (`compressionKind` from `QueryConfig::shuffleCompressionKind`).
3. Supports a `PrestoOutputStreamListener` that computes CRC32 checksums on the fly.

The serializer is managed per-`Destination` through a `VectorStreamGroup`:

```
VectorStreamGroup
  +-- StreamArena (memory pool for serialization buffers)
  +-- IterativeVectorSerializer (format-specific serializer)
       +-- createStreamTree() -- initializes column-level streams
       +-- append() -- adds rows to the serialization buffer
       +-- flush() -- writes accumulated data to an OutputStream
```

## 4. Concurrency & State Management

### 4.1 Threading Model

The system has two concurrent access patterns:

1. **Producer threads** (driver threads running `PartitionedOutput`): call `OutputBufferManager::enqueue()`.
2. **Consumer threads** (HTTP handler threads in Prestissimo): call `OutputBufferManager::getData()`, `acknowledge()`, `deleteResults()`.

### 4.2 Locking Strategy

**OutputBufferManager level:** Uses `folly::Synchronized<map, mutex>` for the `taskId -> OutputBuffer` map. The lock is held only briefly to look up the `shared_ptr<OutputBuffer>`.

**OutputBuffer level:** A single `std::mutex mutex_` protects all state within a single task's output buffer. This is a coarse-grained lock, but contention is mitigated by:
- Keeping critical sections small (enqueue, acknowledge, getData are all brief).
- Performing expensive operations (notification callbacks, memory deallocation) outside the mutex.

```cpp
// Pattern used throughout OutputBuffer:
{
  std::lock_guard<std::mutex> l(mutex_);
  // ... modify state, collect callbacks/promises ...
}
// Outside mutex:
for (auto& callback : dataAvailableCallbacks) {
  callback.notify();  // may trigger HTTP response
}
releaseAfterAcknowledge(freed, promises);  // free memory, unblock producers
```

**DestinationBuffer level:** No internal locking -- always accessed under `OutputBuffer::mutex_`.

**detail::Destination level:** No locking needed -- each `Destination` is only accessed by its owning driver thread.

### 4.3 Backpressure

The system implements backpressure at two levels:

**Level 1 -- OutputBuffer backpressure (network-level):**

When `bufferedBytes_ >= maxSize_` (default 32MB), the producer driver is blocked:

```cpp
if (bufferedBytes_ >= maxSize_ && future) {
  promises_.emplace_back("OutputBuffer::enqueue");
  *future = promises_.back().getSemiFuture();
  blocked = true;
}
```

The producer is unblocked when `bufferedBytes_` drops below `continueSize_` (90% of `maxSize_`):

```cpp
void OutputBuffer::updateAfterAcknowledgeLocked(...) {
  // ... compute freedBytes ...
  if (bufferedBytes_ < continueSize_) {
    promises = std::move(promises_);  // wake all blocked producers
  }
}
```

**Level 2 -- PartitionedOutput operator backpressure:**

The `maxBufferedBytes_` config (default 32MB, from `QueryConfig::maxPartitionedOutputBufferSize`) controls the per-destination page size:

```cpp
const uint64_t maxPageSize = std::max(kMinDestinationSize,
    std::min(kMaxPageSize, maxBufferedBytes_ / numDestinations_));
```

When a destination flush results in a blocked future, the operator pauses and reports `BlockingReason::kWaitForConsumer` to the driver loop.

### 4.4 Sequence Numbers and Acknowledgement

Each `DestinationBuffer` maintains a monotonically increasing `sequence_` number. The protocol works as follows:

1. Producer enqueues pages, which get sequential positions in `data_`.
2. Consumer calls `getData(sequence=N)` to fetch pages starting from position N.
3. `getData()` implicitly acknowledges all pages with sequence < N (via `buffer->acknowledge(sequence, true)`).
4. Acknowledged pages are removed from `data_`, `sequence_` advances, and freed memory reduces `bufferedBytes_`.

This is similar to TCP's sliding window: the consumer advances the window by specifying the next expected sequence number.

### 4.5 End-of-Data Protocol

1. Each driver calls `OutputBuffer::noMoreData()` when it finishes producing.
2. `OutputBuffer` counts finished drivers via `numFinished_`.
3. When `numFinished_ == numDrivers_`, `atEnd_` is set.
4. For partitioned/broadcast: a `nullptr` sentinel is enqueued to each `DestinationBuffer`.
5. For arbitrary: `ArbitraryBuffer::noMoreData()` appends a null sentinel.
6. Consumers encountering the null sentinel know to stop requesting.

## 5. Memory & Resource Profile

### 5.1 Memory Ownership

| Component | Memory Source | Lifecycle |
|-----------|-------------|-----------|
| `VectorStreamGroup` (in `Destination`) | Operator's `MemoryPool` | Created per `Destination`, cleared on each flush |
| `IOBufOutputStream` | Pool from `VectorStreamGroup` | Transient during flush, becomes `PrestoSerializedPage` |
| `PrestoSerializedPage` (`folly::IOBuf`) | Pool-allocated, transferred to `DestinationBuffer` | Lives until acknowledged/deleted |
| `DestinationBuffer::data_` | `shared_ptr` references | Freed when `use_count == 1` after acknowledge |

### 5.2 Buffer Sizing

| Parameter | Default | Config Key |
|-----------|---------|------------|
| OutputBuffer max size (per task) | 32 MB | `max_output_buffer_size` |
| OutputBuffer continue threshold | 28.8 MB (90%) | Computed: `maxSize_ * kContinuePct / 100` |
| PartitionedOutput max buffered bytes | 32 MB | `max_partitioned_output_buffer_size` |
| Max page size | 1 MB | Hardcoded `kMaxPageSize = 1 << 20` |
| Min destination flush size | 60 KB | Hardcoded `kMinDestinationSize = 60 * 1024` |
| Target rows per page | 7,000-12,000 | `(10'000 * targetSizePct_) / 100` |

### 5.3 Broadcast Memory Optimization

For broadcast output, the serialized page is shared across all destinations via `shared_ptr`. The `updateAfterAcknowledgeLocked` logic only counts bytes as freed when `use_count() == 1`:

```cpp
for (const auto& free : freed) {
  if (free.use_count() == 1) {
    ++freedPages;
    freedBytes += free->size();
  }
}
```

This means the last destination to acknowledge a broadcast page actually frees the memory.

### 5.4 Utilization Monitoring

`OutputBuffer` exposes utilization metrics:

```cpp
double OutputBuffer::getUtilization() const {
  return bufferedBytes_ / static_cast<double>(maxSize_);
}

bool OutputBuffer::isOverutilized() const {
  return (bufferedBytes_ > (0.5 * maxSize_)) || atEnd_;
}
```

`isOverutilized()` is used by the Velox runtime to dynamically scale the number of consumers (e.g., spawning more TableWriter tasks when the output buffer is backing up).

### 5.5 Skew Detection

The `OutputBuffer::stats()` method computes `numTopBuffers` -- the number of destination buffers that handle 80% of the total data. This is a skew indicator:

```cpp
int32_t countTopBuffers(const vector<DestinationBuffer::Stats>& bufferStats,
                        int64_t totalBytes) {
  // Sort buffer sizes descending, count how many make up 80%
}
```

If `numTopBuffers` is much less than the total number of buffers, data is skewed.

## 6. Key Design Insights

### 6.1 Serialization on the Driver Thread

Unlike some systems that defer serialization to the network layer, Velox serializes data on the driver thread inside `PartitionedOutput::getOutput()`. This has important implications:

- **Pro:** The serialized data in `DestinationBuffer` is ready for zero-copy network transfer. The network thread only needs to call `getIOBuf()` (which is a `folly::IOBuf::clone()` -- a reference count bump, not a copy).
- **Pro:** Serialization memory is tracked under the operator's `MemoryPool`, making memory accounting per-task accurate.
- **Con:** Serialization work consumes driver thread CPU. In a CPU-bound query, this can reduce the throughput of the producing pipeline.

### 6.2 Staggered Flush Threshold

The randomized `targetSizePct_` (70-120%) is a clever anti-burst mechanism:

```cpp
void setTargetSizePct() {
  targetSizePct_ = 70 + (folly::Random::rand32(rng_) % 50);
  targetNumRows_ = (10'000 * targetSizePct_) / 100;
}
```

Without this, all destinations receiving equal-sized partitions would flush simultaneously, creating bursty network traffic. With randomization, flushes are spread in time, smoothing network utilization. The randomization is re-seeded per batch, so the average page size converges across destinations over time.

### 6.3 Comparison with Trino's OutputBuffer

| Aspect | Velox | Trino |
|--------|-------|-------|
| Serialization location | Driver thread (PartitionedOutput operator) | Separate serialization within OutputBuffer |
| Buffer types | kPartitioned, kBroadcast, kArbitrary | PARTITIONED, BROADCAST, ARBITRARY, PIPELINE |
| Backpressure mechanism | `ContinueFuture`/`ContinuePromise` | `ListenableFuture` |
| Consumer notification | `DataAvailableCallback` (function callback) | `OutputBufferListener` interface |
| Locking | Single mutex per OutputBuffer | `synchronized(this)` per SharedBuffer |
| Sequence management | Per-DestinationBuffer sequence numbers | Per-ClientBuffer token-based |
| Page format | `PrestoSerializedPage` wrapping `folly::IOBuf` | `Slice` / `SerializedPage` |
| Hash function | VectorHasher + simple modulo or bit-range | Murmur3-based `BucketFunction` |

Key differences:
- Trino's `OutputBuffer` has a `PIPELINE` mode for staged execution, absent in Velox.
- Velox's `ArbitraryBuffer` uses an active-check callback to skip inactive consumers, optimizing the HTTP streaming shuffle.
- Velox supports three serde formats (Presto, CompactRow, UnsafeRow) vs. Trino's single Presto format.

### 6.4 Comparison with DataFusion's Repartition

DataFusion's `RepartitionExec` takes a fundamentally different approach:

| Aspect | Velox | DataFusion |
|--------|-------|------------|
| Architecture | Operator -> global buffer manager -> network | ExecutionPlan with channels between partitions |
| Data format in transit | Serialized bytes (`folly::IOBuf`) | Native `RecordBatch` (no serialization for local) |
| Scope | Network exchange (across workers) | Local exchange (within a single process) |
| Partitioning | Inside PartitionedOutput operator | Inside RepartitionExec using `hash(columns) % N` |
| Backpressure | Future-based blocking | Tokio channel capacity limits |

DataFusion does not have an equivalent of the OutputBuffer for network exchange -- it relies on FlightSQL or external shuffle services for distributed queries. Velox's OutputBuffer subsystem is purpose-built for distributed execution as part of the Prestissimo ecosystem.

### 6.5 The "Opportunistic Flush on Block" Optimization

When a destination blocks (output buffer full), `getOutput()` flushes other destinations that have accumulated enough data:

```cpp
if (blockedDestination) {
  for (auto& destination : destinations_) {
    if (destination.get() == blockedDestination ||
        destination->serializedBytes() < kMinDestinationSize) {
      continue;
    }
    destination->flush(*bufferManager, bufferReleaseFn_, nullptr);
  }
}
```

Since the driver is going off-thread anyway (blocked waiting for consumer), flushing other destinations' accumulated data is "free" work that reduces memory pressure and makes data available to consumers sooner. The `kMinDestinationSize` (60KB) threshold avoids sending tiny pages that would be inefficient for network transfer.

### 6.6 ArbitraryBuffer: Load Balancing Without Partitioning

The `kArbitrary` mode is unique to Velox (and Prestissimo). It implements a consumer-driven load balancing strategy:

1. All produced pages go into a single shared `ArbitraryBuffer` (a `deque`).
2. When a consumer calls `getData()`, pages are pulled from the shared pool.
3. The `DataConsumerActiveCheckCallback` allows skipping inactive consumers.

This is used for table writer scaling, where the number of downstream consumers may change dynamically. Unlike partitioned output, there is no deterministic row-to-consumer mapping -- whichever consumer is ready takes the next batch.

### 6.7 Task Lifecycle Integration

The OutputBuffer holds a `shared_ptr<Task>`, and `PrestoSerializedPage` holds a reference via `bufferReleaseFn_`:

```cpp
bufferReleaseFn_([task = operatorCtx_->task()]() {})
```

This prevents the `Task` from being destroyed while serialized pages are still in flight (e.g., being sent over HTTP by Prestissimo). The lambda captures the task shared_ptr but does nothing -- its sole purpose is preventing premature destruction.
