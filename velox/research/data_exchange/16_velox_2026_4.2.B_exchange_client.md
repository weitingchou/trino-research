## 0. Research Focus
* **Task ID:** 4.2.B
* **Focus:** Trace how a downstream Velox task fetches data from upstream producers. Analyze the `Exchange` operator, the `ExchangeClient` that manages fetch requests, the `ExchangeSource` abstraction that hides the transport layer, and the `ExchangeQueue` that buffers serialized pages. Since the network RPC is host-dependent, identify the exact boundary where Velox expects the host to populate its internal exchange queues.

## 1. High-Level Overview

Velox's data exchange subsystem follows a **library-first** design: the engine provides the operator (`Exchange`), a fetch coordinator (`ExchangeClient`), and a shared buffer (`ExchangeQueue`), but delegates all actual network transport to the host application through an abstract `ExchangeSource` interface. The host registers a factory function that creates concrete `ExchangeSource` implementations -- for example, Prestissimo registers a `PrestoExchangeSource` that speaks HTTP to remote workers, while unit tests register a `LocalExchangeSource` that reads directly from the in-process `OutputBufferManager`.

The data flow for fetching proceeds as follows:

1. The coordinator (Prestissimo, Spark-Gluten, etc.) pushes `RemoteConnectorSplit` objects to the Velox `Task`, each identifying an upstream producer task by its task ID.
2. The `Exchange` operator (a `SourceOperator`) pulls these splits and passes them to the `ExchangeClient`.
3. The `ExchangeClient` creates an `ExchangeSource` per upstream task via the registered factory.
4. The `ExchangeClient` orchestrates fetch requests: it asks sources for available data sizes, then requests actual data within the queue's memory budget.
5. Data arrives as `SerializedPageBase` objects and is enqueued into the shared `ExchangeQueue`.
6. The `Exchange` operator dequeues pages, deserializes them using the configured `VectorSerde`, and produces `RowVector` output batches.

This design cleanly separates Velox's internal execution model (pull-based, future-driven) from the transport layer (HTTP, shared memory, RDMA, etc.).

class ExchangeSource : public std::enable_shared_from_this<ExchangeSource> {

## 3. Execution & Call Flow

### 3.1 Lifecycle: From Task Start to First Data

**Phase 1: ExchangeClient creation during Task startup**

When `Task::createDriversLocked()` runs, it inspects each `DriverFactory`. If the factory's leaf plan node is an `ExchangeNode` (which returns `true` from `requiresExchangeClient()`), the Task creates an `ExchangeClient`:

```cpp
// Task.cpp, line ~1161
if (auto exchangeNodeId = factory->needsExchangeClient()) {
    createExchangeClientLocked(
        pipeline, exchangeNodeId.value(), factory->numDrivers);
}
```

The creation wires together several query-config-driven parameters:

```cpp
// Task.cpp, line ~3453
exchangeClients_[pipelineId] = std::make_shared<ExchangeClient>(
    taskId_,
    destination_,
    queryCtx()->queryConfig().maxExchangeBufferSize(),   // default 32 MB
    numberOfConsumers,    // = factory->numDrivers
    queryCtx()->queryConfig().minExchangeOutputBatchBytes(),
    addExchangeClientPool(planNodeId, pipelineId),
    queryCtx()->executor(),
    queryCtx()->queryConfig().requestDataSizesMaxWaitSec(),
    queryCtx()->queryConfig().singleSourceExchangeOptimizationEnabled(),
    queryCtx()->queryConfig().exchangeLazyFetchingEnabled());
```

The same `ExchangeClient` pointer is shared across all drivers in the pipeline. Each driver's `Exchange` operator receives this shared client at construction.

**Phase 2: Split arrival and source creation**

The host pushes `RemoteConnectorSplit` objects via `Task::addSplitWithSequence()`. For exchange nodes, these are routed directly to the ExchangeClient:

```cpp
// Task.cpp, line ~1599
exchangeClientByPlanNode_[planNodeId]->addRemoteTaskId(remoteSplit->taskId);
```

Within `ExchangeClient::addRemoteTaskId()`, a new `ExchangeSource` is created via the factory chain:

```cpp
// ExchangeClient.cpp
source = ExchangeSource::create(remoteTaskId, destination_, queue_, pool_);
sources_.push_back(source);
queue_->addSourceLocked();
emptySources_.push(source);
if (!lazyFetching_) {
    requestSpecs = pickSourcesToRequestLocked();  // Start fetching immediately
}
```

`ExchangeSource::create()` iterates registered factories until one returns non-null:

```cpp
// ExchangeSource.cpp
for (auto& factory : factories()) {
    auto result = factory(remoteTaskId, destination, queue, pool);
    if (result) return result;
}
VELOX_FAIL("No ExchangeSource factory matches {}", remoteTaskId);
```

This is the **host-engine boundary**: the registered factory determines the transport. Prestissimo registers a factory that creates HTTP-based sources. Unit tests register `LocalExchangeSource` that talks to the in-process `OutputBufferManager`.

**Phase 3: Alternate split delivery -- the Exchange operator's getSplits()**

Splits can also arrive while drivers are running. Only driver 0 (`processSplits_ = true`) polls for splits from the Task's split store:

```cpp
// Exchange.cpp
Exchange::Exchange(...) : ..., processSplits_{operatorCtx_->driverCtx()->driverId == 0} {}
```

In `isBlocked()`, driver 0 calls `getSplits()` which loops fetching splits:

```cpp
void Exchange::getSplits(ContinueFuture* future) {
    if (!processSplits_) return;
    // ...
    for (;;) {
        exec::Split split;
        const auto reason = operatorCtx_->task()->getSplitOrFuture(..., split, *future);
        if (reason != BlockingReason::kNotBlocked) {
            addRemoteTaskIds(remoteTaskIds);
            return;   // Will resume when splits arrive
        }
        if (split.hasConnectorSplit()) {
            remoteTaskIds.push_back(remoteSplit->taskId);
            continue;
        }
        // No more splits
        addRemoteTaskIds(remoteTaskIds);
        exchangeClient_->noMoreRemoteTasks();
        noMoreSplits_ = true;
        return;
    }
}
```

Note the **task ID shuffling** before adding to the client:
```cpp
void Exchange::addRemoteTaskIds(std::vector<std::string>& remoteTaskIds) {
    std::shuffle(std::begin(remoteTaskIds), std::end(remoteTaskIds), rng_);
    for (const std::string& taskId : remoteTaskIds) {
        exchangeClient_->addRemoteTaskId(taskId);
    }
}
```
This randomization prevents multiple downstream tasks from hammering the same upstream task simultaneously.

### 3.2 The Fetch Protocol: Two-Phase Data Retrieval

The `ExchangeClient` uses a **two-phase fetch protocol** when multiple sources are present:

**Phase A: Size probing** -- Sources in `emptySources_` have unknown data availability. The client sends `requestDataSizes()` (maxBytes=0) to learn what pages are buffered at the source without actually transferring data:

```cpp
// ExchangeClient.cpp, request()
if (spec.maxBytes == 0) {
    future = spec.source->requestDataSizes(requestDataSizesMaxWaitSec_);
} else {
    future = spec.source->request(spec.maxBytes, kRequestDataMaxWait);
}
```

**Phase B: Data transfer** -- Sources that responded with non-empty `remainingBytes` move to `producingSources_`. The client requests actual data, bounded by available queue capacity:

```cpp
// pickSourcesToRequestLocked()
int64_t availableSpace = maxQueuedBytes_ - queue_->totalBytes() - totalPendingBytes_;
while (availableSpace > 0 && !producingSources_.empty()) {
    int64_t requestBytes = 0;
    for (auto bytes : producingSources_.front().remainingBytes) {
        availableSpace -= bytes;
        if (availableSpace < 0) break;
        requestBytes += bytes;
    }
    // ...
    requestSpecs.push_back({std::move(source), requestBytes});
    totalPendingBytes_ += requestBytes;
}
```

**Single-source optimization**: When `skipRequestDataSizeWithSingleSource` is enabled and there is exactly one source, the client skips the size-probing phase entirely. It requests data equal to the full remaining queue capacity:

```cpp
// pickupSingleSourceToRequestLocked()
const auto requestBytes = maxQueuedBytes_ - queue_->totalBytes();
if (requestBytes <= 0) return {};
requestSpecs.push_back({std::move(source), requestBytes});
```

**Deadlock prevention**: If no normal request can be issued (all remaining pages exceed capacity) but the queue lacks `minOutputBatchBytes_` worth of data, the client forces an out-of-band transfer of a single large page:

```cpp
if ((queue_->totalBytes() + totalPendingBytes_ < minOutputBatchBytes_) &&
    !producingSources_.empty()) {
    auto requestBytes = producingSources_.front().remainingBytes.at(0);
    LOG(INFO) << "Requesting large single page " << requestBytes
              << " bytes, exceeding capacity " << maxQueuedBytes_;
    requestSpecs.push_back({std::move(source), requestBytes});
}
```

### 3.3 Async Response Handling

Fetch requests return `folly::SemiFuture<Response>`. The response callback runs on the ExchangeClient's `executor_` (never inline, to avoid deadlocks):

```cpp
std::move(future)
    .via(executor_)
    .thenValue([self, spec, sendTimeMs](Response&& response) {
        // Lock queue mutex
        if (!response.atEnd) {
            if (!response.remainingBytes.empty()) {
                self->producingSources_.push({source, remainingBytes});
            } else {
                self->emptySources_.push(source);
            }
        }
        self->totalPendingBytes_ -= spec.maxBytes;
        requestSpecs = self->pickSourcesToRequestLocked();
        // Unlock

        // Pause source if not selected for next round
        if (pauseCurrentSource) currentSource->pause();
        // Issue next round of requests
        self->request(std::move(requestSpecs));
    })
    .thenError([self](const std::exception& e) {
        self->queue_->setError(e.what());
    });
```

This creates a **self-sustaining request chain**: each response triggers evaluation of whether more data should be fetched, maintaining the pipeline without driver thread involvement.

### 3.4 Consumer Side: Exchange::isBlocked() and getOutput()

The `Exchange` operator's `isBlocked()` is called by the Driver loop to determine if the operator can produce output:

```cpp
BlockingReason Exchange::isBlocked(ContinueFuture* future) {
    if (!currentPages_.empty() || atEnd_) return kNotBlocked;

    // Trigger split fetching
    if (!splitFuture_.valid()) getSplits(&splitFuture_);

    // Try to dequeue data
    ContinueFuture dataFuture;
    currentPages_ = exchangeClient_->next(
        driverId_, preferredOutputBatchBytes_, &atEnd_, &dataFuture);

    if (!currentPages_.empty() || atEnd_) return kNotBlocked;

    // Block on either data or splits, whichever arrives first
    if (splitFuture_.valid()) {
        std::vector<ContinueFuture> futures;
        futures.push_back(std::move(splitFuture_));
        futures.push_back(std::move(dataFuture));
        *future = folly::collectAny(futures).unit();
        return kWaitForSplit;
    }
    *future = std::move(dataFuture);
    return kWaitForProducer;
}
```

Key detail: `folly::collectAny` means the driver unblocks on whichever event fires first -- new splits or new data. This allows progressive fetching: the operator begins consuming data before all upstream tasks are known.

**Deserialization** in `getOutput()` supports two modes:

1. **Columnar pages** (`supportsAppendInDeserialize()`): Pages are processed one-by-one via `getOutputFromColumnarPages()`. The loop incrementally appends deserialized vectors until the batch target is hit, freeing each page as it is consumed.

2. **Row pages**: All current pages are merged into a single IOBuf chain, then deserialized in batches via `getOutputFromRowPages()`:
```cpp
std::unique_ptr<folly::IOBuf> mergedBufs = mergePages(currentPages_);
mergedRowPage_ = std::make_unique<PrestoSerializedPage>(std::move(mergedBufs));
inputStream_ = mergedRowPage_->prepareStreamForDeserialize();
```

Both paths use `estimatedRowSize_` (updated after each output) to calculate the target number of rows per batch:
```cpp
const auto numRows = estimatedRowSize_.has_value()
    ? std::max(preferredOutputBatchBytes_ / estimatedRowSize_.value(), kInitialOutputRows)
    : kInitialOutputRows;  // Conservative 64 rows initially
```

### 3.5 ExchangeQueue: Enqueue and Dequeue Mechanics

**Enqueue** (called by ExchangeSource implementation or its callback):
```cpp
void ExchangeQueue::enqueueLocked(unique_ptr<SerializedPageBase>&& page,
                                  vector<ContinuePromise>& promises) {
    if (page == nullptr) {
        ++numCompleted_;
        auto completedPromises = checkNoMoreInput();
        // Return promises to be fulfilled outside lock
        return;
    }
    totalBytes_ += page->size();
    queue_.push_back(std::move(page));
    // Wake consumers only if enough bytes for a batch
    const auto minBatchSize = minOutputBatchBytesLocked();
    while (!promises_.empty()) {
        const int32_t unblockedConsumers = numberOfConsumers_ - promises_.size();
        const int64_t unasignedBytes = totalBytes_ - unblockedConsumers * minBatchSize;
        if (unasignedBytes < minBatchSize) break;
        promises.push_back(std::move(it->second));  // Wake one consumer
        promises_.erase(it);
    }
}
```

The `minOutputBatchBytes` mechanism prevents consumer thrashing: consumers are not woken until enough bytes accumulate to fill a meaningful batch. This is dynamically capped at 1% of total received bytes to minimize latency for small exchanges:
```cpp
int64_t ExchangeQueue::minOutputBatchBytesLocked() const {
    if (noMoreInput_) return 0;  // Always unblock if done
    return std::min<int64_t>(minOutputBatchBytes_, receivedBytes_ / 100);
}
```

**Dequeue** (called by `ExchangeClient::next()` under lock):
```cpp
vector<unique_ptr<SerializedPageBase>> ExchangeQueue::dequeueLocked(
    int consumerId, uint32_t maxBytes, bool* atEnd,
    ContinueFuture* future, ContinuePromise* stalePromise) {
    if (totalBytes_ < minOutputBatchBytesLocked()) {
        addPromiseLocked(consumerId, future, stalePromise);
        return {};  // Not enough data yet
    }
    vector<unique_ptr<SerializedPageBase>> pages;
    uint32_t pageBytes = 0;
    for (;;) {
        if (queue_.empty()) {
            if (noMoreInput_) *atEnd = true;
            else if (pages.empty()) addPromiseLocked(consumerId, future, stalePromise);
            return pages;
        }
        if (pageBytes > 0 && pageBytes + queue_.front()->size() > maxBytes) return pages;
        pages.emplace_back(std::move(queue_.front()));
        queue_.pop_front();
        pageBytes += pages.back()->size();
        totalBytes_ -= pages.back()->size();
    }
}
```

Important behaviors:
- Returns at least one page if any data exists (the `maxBytes` check only fires after the first page).
- Completion detection: the queue tracks `numCompleted_` vs `numSources_` and signals `noMoreInput_` when all sources have sent their null terminator.

### 3.6 MergeExchange Variant

For ordered merge operations, `MergeExchange` creates one `ExchangeClient` per upstream source (via `MergeExchangeSource`), not a shared one. Each client feeds a `MergeSource` that produces deserialized `RowVector`s into a `TreeOfLosers` merge structure:

```cpp
// MergeSource.cpp
class MergeExchangeSource : public MergeSource {
    MergeExchangeSource(...) : client_(std::make_shared<ExchangeClient>(
            ..., 1,   // numberOfConsumers = 1
            0,        // minOutputBatchBytes = 0, deliver immediately
            ...)) {
        client_->addRemoteTaskId(taskId);
        client_->noMoreRemoteTasks();
    }
};
```

The `minOutputBatchBytes = 0` ensures the merge never blocks waiting for a batch -- it delivers each page immediately so the sorted merge can interleave rows from multiple sources.

## 4. Concurrency & State Management

### Locking Strategy

The system uses a **single-mutex design** centered on `ExchangeQueue::mutex_`:

1. **ExchangeClient** locks `queue_->mutex()` for all state mutations: adding sources, dequeuing pages, picking request targets.
2. **ExchangeQueue** enqueue/dequeue operations are called with the mutex already held (the `Locked` suffix convention).
3. **ExchangeSource** `shouldRequestLocked()` is called under the queue mutex, setting `requestPending_` atomically.
4. All promise fulfillment and `ExchangeSource::close()` happen **outside** the mutex to avoid deadlock.

This is critical: the comment in the constructor explicitly warns:
```cpp
// The provided executor must not be folly::InlineLikeExecutor, otherwise
// it might cause potential deadlock as the response callback in exchange
// client might call back into the exchange source under uncertain
// execution context.
```

### Thread Interactions

| Thread | Role | Synchronization |
|--------|------|-----------------|
| Driver thread (per-driver) | Calls `Exchange::isBlocked()` / `getOutput()` | Acquires queue mutex for `next()` |
| Executor thread | Runs async response callbacks from ExchangeSource | Acquires queue mutex to re-categorize sources and pick next requests |
| Host thread | Calls `Task::addSplit()` which may trigger `ExchangeClient::addRemoteTaskId()` | Acquires queue mutex |

### Multiple Consumer Support

The `ExchangeQueue` supports multiple consumers (one per driver in the pipeline). Each consumer has its own promise, identified by `consumerId`:
```cpp
folly::F14FastMap<int, ContinuePromise> promises_;
```

The enqueue path wakes consumers proportionally: it calculates "unassigned bytes" after accounting for already-unblocked consumers:
```cpp
const int32_t unblockedConsumers = numberOfConsumers_ - promises_.size();
const int64_t unasignedBytes = totalBytes_ - unblockedConsumers * minBatchSize;
if (unasignedBytes < minBatchSize) break;
```

This prevents all consumers from waking simultaneously when only enough data exists for one batch.

### Source Lifecycle State Machine

```
                     +--- shouldRequestLocked() returns true --->  PENDING
                     |       (requestPending_ = true)                 |
    IDLE / EMPTY ----+                                                |
         ^           |                                                v
         |           +--- shouldRequestLocked() returns false    Response arrives
         |                (already pending or atEnd)                   |
         |                                                            |
         +---- response.remainingBytes.empty() ---- EMPTY SOURCE -----+
         |                                                            |
         +---- response.remainingBytes non-empty -- PRODUCING SOURCE -+
         |                                                            |
         +---- response.atEnd = true -------------- COMPLETED --------+
```

### Lazy Fetching Optimization

When `lazyFetching_ = true`, `ExchangeClient::addRemoteTaskId()` skips the initial `pickSourcesToRequestLocked()` call. Data fetching only begins when `Exchange::isBlocked()` calls `next()` for the first time. This is designed for hash join build sides with cached tables: if the hash table is already cached, the `HashBuild` operator completes before `Exchange::isBlocked()` is ever called, avoiding unnecessary network I/O entirely.

## 5. Memory & Resource Profile

### Queue Memory Budget

The primary memory control is `maxQueuedBytes_` (default 32 MB from `ExchangeClient::kDefaultMaxQueuedBytes`). This governs the total bytes allowed in the queue plus in-flight requests:

```
Available space = maxQueuedBytes_ - queue_->totalBytes() - totalPendingBytes_
```

The `ExchangeClient` only issues new requests when `availableSpace > 0`. For single-source exchanges, this simplifies to `maxQueuedBytes_ - queue_->totalBytes()` since there are no competing sources.

### Memory Pool Hierarchy

The Task creates a dedicated memory pool for each ExchangeClient:
```cpp
// Task.cpp, addExchangeClientPool()
fmt::format("exchangeClient.{}.{}", planNodeId, pipelineId)
```

`ExchangeSource` holds a `shared_ptr<memory::MemoryPool>` (not a raw pointer) to survive beyond task lifetime -- important because async HTTP responses may arrive after the task has been destroyed.

### SerializedPage Lifecycle

Pages are `std::unique_ptr<SerializedPageBase>` (typically `PrestoSerializedPage` wrapping `folly::IOBuf`). The ownership flow:

1. **Creation**: The `ExchangeSource` implementation creates pages from received data and enqueues them into `ExchangeQueue`.
2. **Queue residence**: Pages sit in `ExchangeQueue::queue_` (a `std::deque`), tracked by `totalBytes_`.
3. **Dequeue**: `ExchangeClient::next()` transfers ownership to the caller (a vector of unique_ptrs).
4. **Consumption**: `Exchange::getOutput()` processes pages. Columnar deserialization frees each page immediately after processing (`page.reset()`). Row deserialization merges all pages into a single IOBuf chain.
5. **Cleanup**: On `Exchange::close()`, all held pages and state are cleared.

### Statistics Tracking

The queue tracks peak usage (`peakBytes_`), received pages (`receivedPages_`), and average page size (`receivedBytes_ / receivedPages_`). The `ExchangeClient` aggregates metrics from all sources and reports them as operator runtime stats:
- `peakBytes` -- Maximum queue fill level
- `numReceivedPages` -- Total pages received
- `averageReceivedPageBytes` -- Average page size
- `kBackgroundCpuTimeNanos` -- CPU time spent in async callbacks

## 6. Key Design Insights

### Insight 1: The Factory Pattern as the Transport Boundary

The `ExchangeSource::Factory` registration mechanism is the **single point where Velox's execution engine meets the network layer**. Velox never imports HTTP, gRPC, or any transport library. Instead:

```cpp
using Factory = std::function<std::shared_ptr<ExchangeSource>(
    const std::string& remoteTaskId, int destination,
    std::shared_ptr<ExchangeQueue> queue, memory::MemoryPool* pool)>;
```

The factory receives the queue and pool -- everything it needs to produce pages. The `ExchangeSource` contract is minimal: `shouldRequestLocked()`, `request()`, `requestDataSizes()`, `pause()`, `close()`. The host can implement any transport behind this interface.

This contrasts with **Trino**, where the `ExchangeClient` directly uses `HttpPageBufferClient` to make HTTP calls. Trino's exchange is tightly coupled to its HTTP-based protocol. In Velox, the transport is fully pluggable -- the same engine code works with HTTP (Prestissimo), shared memory (testing), or any custom protocol.

Compared to **DataFusion**, which relies on Apache Arrow Flight or custom `ExecutionPlan` partitioning schemes and lets the host handle all data movement before calling `execute()`, Velox takes a middle path: it provides the fetch coordination logic (request scheduling, memory budgeting, backpressure) but delegates the actual data transport.

### Insight 2: Two-Phase Fetch With Capacity-Aware Scheduling

The `pickSourcesToRequestLocked()` algorithm is sophisticated:

1. First, probe all empty sources for data sizes (zero-cost metadata fetch).
2. Then, greedily fill available capacity from producing sources.
3. If the queue is starving (below `minOutputBatchBytes_`), force a large transfer even if it exceeds capacity.

This is more nuanced than Trino's approach, where `ExchangeClient` uses a simpler `scheduleRequestIfNecessary()` that requests from a single source at a time with a fixed `maxResponseSize`. Velox's per-page size awareness allows it to make precise decisions about which pages to fetch, avoiding queue overflows while maximizing throughput.

### Insight 3: Shared ExchangeClient, Independent Consumers

All drivers in a pipeline share a single `ExchangeClient` (and thus a single `ExchangeQueue`), but each driver dequeues independently using its `driverId` as the consumer ID. Only driver 0 manages splits (`processSplits_ = true`).

This design has implications:
- **Good**: Fewer ExchangeSource connections than drivers. A single ExchangeClient coordinates all fetching, avoiding thundering herd on upstream tasks.
- **Good**: The queue's batch-aware wake-up prevents multiple consumers from waking when only one batch is available.
- **Trade-off**: All consumers contend on the same queue mutex. For high-parallelism pipelines, this could become a bottleneck.

In Trino, each `ExchangeOperator` has its own buffer from the `ExchangeClient`, and the `ExchangeClient` is also shared per exchange. The key difference is Trino splits response data across destination buffers at the source side, while Velox uses a single shared queue at the destination.

### Insight 4: The Pause Mechanism for Backpressure

The `ExchangeSource::pause()` method is a backpressure signal: when the `ExchangeClient` determines that a source will not be requested in the next round, it calls `pause()` on that source. The `LocalExchangeSource` implements this by acknowledging received data to the `OutputBufferManager`, freeing upstream buffer space:

```cpp
void pause() override {
    buffers->acknowledge(remoteTaskId_, destination_, ackSequence);
}
```

This is an elegant way to propagate backpressure upstream without a separate control channel. When the downstream queue is full, sources are paused, which (in a production HTTP implementation) can release server-side buffers or slow down the streaming connection.

### Insight 5: Null-Page Termination Protocol

End-of-stream is signaled by enqueuing a `nullptr` page:
```cpp
void ExchangeQueue::enqueueLocked(unique_ptr<SerializedPageBase>&& page, ...) {
    if (page == nullptr) {
        ++numCompleted_;
        auto completedPromises = checkNoMoreInput();
        // ...
    }
}
```

When `numCompleted_ == numSources_` and `noMoreSources_` is true, the queue enters `noMoreInput_` state and wakes all waiting consumers. This is the same termination protocol as Trino's `SerializedPage.isMarkerPage()` / `end-of-data` marker, though Velox uses a null pointer rather than a sentinel object.

### Insight 6: Progressive Fetching Before All Splits Arrive

The `Exchange` operator calls `getSplits()` and `next()` concurrently. If splits are still arriving but some data is already available, the operator processes that data immediately. The `folly::collectAny` on split and data futures means:

```cpp
// Block until data becomes available or more splits arrive.
std::vector<ContinueFuture> futures;
futures.push_back(std::move(splitFuture_));
futures.push_back(std::move(dataFuture));
*future = folly::collectAny(futures).unit();
```

This maximizes pipeline utilization in large clusters where split discovery is gradual -- the exchange does not wait for all upstream tasks to be registered before consuming their output.
