## 1. High-Level Overview

Velox's Connector SPI is a pluggable abstraction that decouples the execution engine from storage formats. The design is deliberately narrow: the engine provides exactly three things to a connector -- the output schema, a table handle carrying filters, and per-split metadata -- and the connector returns row batches (`RowVectorPtr`). Filter pushdown is the core optimization in this interface and operates at three distinct levels:

1. **SubfieldFilters** -- `unordered_map<Subfield, FilterPtr>` -- simple, single-column comparisons against constants (e.g., `a > 10`, `b IN ('x','y')`) that can be evaluated inside the columnar file reader at the decode level, skipping entire row groups and applying SIMD-accelerated per-value tests.
2. **MetadataFilter** -- a boolean expression tree over the same leaf filters, used for row-group/stripe-level pruning via column statistics.
3. **Remaining Filter** -- an arbitrary `ExprSet` expression evaluated row-by-row after the file reader produces output, for predicates too complex to push into the reader (e.g., cross-column comparisons like `a < b`).

Additionally, **subfield projections** allow the engine to prune nested structures: only requested array indices, map keys, or struct fields are materialized from the file reader.

Unlike Trino, where pushdown is negotiated through a multi-round planner protocol (`ConnectorMetadata.applyFilter()` / `applyProjection()`), Velox performs a single-shot construction: the plan node carries all filters in the `HiveTableHandle`, and the `HiveDataSource` constructor builds the final `ScanSpec` tree in one pass. Dynamic filters (from hash join build sides) are injected later via `addDynamicFilter()`.

## 2. Structural Architecture

### 2.1 The Connector Interface Hierarchy

```
ConnectorFactory
  +-- newConnector() -> Connector
        |
Connector (abstract)
  +-- createDataSource() -> DataSource     [reading]
  +-- createDataSink() -> DataSink         [writing]
  +-- createIndexSource() -> IndexSource   [index lookup]
  +-- canAddDynamicFilter() -> bool
  +-- supportsSplitPreload() -> bool
        |
DataSource (abstract)
  +-- addSplit(ConnectorSplit)
  +-- next(size, future) -> optional<RowVectorPtr>
  +-- addDynamicFilter(channel, Filter)
  +-- getFilters() -> SubfieldFilters*
  +-- getCompletedBytes/Rows()
  +-- estimatedRowSize()
```

The `Connector::createDataSource()` signature encodes the pushdown contract:

```cpp
virtual std::unique_ptr<DataSource> createDataSource(
    const RowTypePtr& outputType,              // columns to produce
    const ConnectorTableHandlePtr& tableHandle, // filters + table metadata
    const connector::ColumnHandleMap& columnHandles, // column mapping
    ConnectorQueryCtx* connectorQueryCtx) = 0;
```

- `outputType`: The projection list. Only these columns appear in the output `RowVector`.
- `tableHandle`: A `HiveTableHandle` carrying `SubfieldFilters`, `remainingFilter`, `dataColumns`, and `sampleRate`.
- `columnHandles`: Maps output column names to `HiveColumnHandle` objects, each carrying `requiredSubfields` (subfield projections) and `columnType` (regular, partition, synthesized, rowIndex, rowId).

### 2.2 The Filter Type System

The `SubfieldFilters` type is the central pushdown currency:

```cpp
using SubfieldFilters = std::unordered_map<Subfield, FilterPtr>;
```

**Subfield** (`velox/type/Subfield.h`) is a path into a nested column:
- `Subfield::NestedField("a")` -- top-level column `a`
- `Subfield::NestedField("a") / LongSubscript(1)` -- `a[1]`
- `Subfield::NestedField("m") / StringSubscript("key")` -- `m['key']`
- `Subfield::AllSubscripts()` -- wildcard `[*]`

**Filter** (`velox/type/Filter.h`) is a hierarchy of 20+ concrete filter types:

| FilterKind | Description | SIMD-optimized? |
|---|---|---|
| `kBigintRange` | `lower <= x <= upper` on int64 | Yes (`testValues(batch)`) |
| `kBigintValuesUsingHashTable` | IN-list via hash table | No |
| `kBigintValuesUsingBitmask` | IN-list via bitmask | Yes |
| `kBigintValuesUsingBloomFilter` | Bloom filter for large sets | Yes |
| `kBytesRange` | String range comparison | No |
| `kBytesValues` | String IN-list | No |
| `kDoubleRange` / `kFloatRange` | Floating-point ranges | Yes |
| `kIsNull` / `kIsNotNull` | Null checks | N/A |
| `kBoolValue` | Boolean equality | N/A |
| `kMultiRange` | Union of ranges | No |
| `kTimestampRange` | Timestamp range | No |
| `kHugeintRange` | 128-bit integer range (DECIMAL) | No |

Every concrete filter provides:
- `testInt64(value)` / `testBytes(ptr, len)` / etc. -- single-value tests
- `testValues(xsimd::batch<T>)` -- SIMD batch test returning a batch bool mask
- `testInt64Range(min, max, hasNull)` -- range test for metadata/statistics pruning
- `mergeWith(other)` -- AND-merge two filters on the same subfield
- `clone()` -- deep copy with optional `nullAllowed` override

### 2.3 The HiveTableHandle: Packaging Filters

```cpp
class HiveTableHandle : public ConnectorTableHandle {
  const SubfieldFilters subfieldFilters_;     // Subfield -> Filter map
  const TypedExprPtr remainingFilter_;        // Arbitrary boolean expression
  const double sampleRate_;                   // Random sampling rate
  const RowTypePtr dataColumns_;              // Full file schema
  const vector<HiveColumnHandlePtr> filterColumnHandles_; // Filter-only columns
};
```

This is created by the planner/client and passed into `createDataSource()`. The `subfieldFilters_` carry the "fast-path" filters; `remainingFilter_` carries everything else.

### 2.4 The HiveColumnHandle: Subfield Projections

```cpp
class HiveColumnHandle : public ColumnHandle {
  const string name_;
  const ColumnType columnType_;  // kRegular, kPartitionKey, kSynthesized, kRowIndex, kRowId
  const TypePtr dataType_;
  const TypePtr hiveType_;       // May differ from dataType for schema evolution
  const vector<Subfield> requiredSubfields_;  // Subfield projection pruning
  const vector<NamedExtraction> extractions_; // Advanced extraction chains
};
```

The `requiredSubfields` field is the projection pushdown mechanism for complex types. For example, `SELECT a[1], a[3] FROM t` produces a column handle for `a` with `requiredSubfields = [a[1], a[3]]`. The connector uses this to prune array elements beyond index 3, and to skip materializing indices 0 and 2.

## 3. Execution & Call Flow

### 3.1 Phase 1: DataSource Construction (one-time setup)

When the first split arrives at `TableScan::getSplit()`, the data source is created:

```
TableScan::getSplit()
  -> createDataSource()  [local function in TableScan.cpp]
       -> connector.createDataSource(outputType, tableHandle, columnHandles, ctx)
            -> new HiveDataSource(...)
```

Inside `HiveDataSource::HiveDataSource()`, the entire filter and projection tree is built:

**Step 1: Process column handles.** Each `HiveColumnHandle` is classified:
- Regular columns -> read from file
- Partition keys -> constant values from split metadata
- Synthesized columns -> `$path`, `$file_size`, etc.
- RowIndex/RowId -> special auto-generated columns

```cpp
for (const auto& [_, columnHandle] : assignments) {
    auto handle = checkedPointerCast<const HiveColumnHandle>(columnHandle);
    processColumnHandle(handle);
}
```

**Step 2: Collect subfield projections.** For each output column, the `requiredSubfields` are gathered into a `subfields_` map:

```cpp
for (auto& subfield : handle->requiredSubfields()) {
    subfields_[handle->name()].push_back(&subfield);
}
```

**Step 3: Clone SubfieldFilters from the table handle.** The table handle's filters are copied into the data source's mutable `filters_` map:

```cpp
for (const auto& [k, v] : hiveTableHandle_->subfieldFilters()) {
    filters_.emplace(k.clone(), v);
}
```

**Step 4: Extract additional filters from the remaining filter.** The `extractFiltersFromRemainingFilter()` function breaks the `remainingFilter` expression tree into three buckets:
- Simple column-vs-constant comparisons -> merged into `filters_` (SubfieldFilters)
- `rand() < K` conjuncts -> combined into `sampleRate`
- Everything else -> kept as `remainingFilter` (compiled into `ExprSet`)

This extraction is recursive, handling AND/OR/NOT structures. OR expressions are converted into `MultiRange` or disjunctive filters when all branches reference the same column. NOT(x OR y) is converted via De Morgan's law to (NOT x) AND (NOT y).

```cpp
auto remainingFilter = extractFiltersFromRemainingFilter(
    hiveTableHandle_->remainingFilter(),
    expressionEvaluator_,
    filters_,        // <-- filters are added here
    sampleRate);
```

**Step 5: Build the ScanSpec tree.** The `makeScanSpec()` function constructs a tree of `ScanSpec` nodes that mirrors the column nesting structure:

```cpp
scanSpec_ = makeScanSpec(
    readerOutputType_,
    subfields_,     // subfield projections
    filters_,       // SubfieldFilters
    hiveTableHandle_->dataColumns(),
    partitionKeys_,
    infoColumns_,
    specialColumns_,
    disableStatsBasedFilterReorder,
    pool_);
```

`makeScanSpec()` does the following:
1. Creates a root `ScanSpec` node.
2. For each output column, adds a child `ScanSpec`. If the column has subfield projections, only the specified subfields are added recursively (pruning the rest).
3. For each filter-only column (referenced in filters but not in the output), adds a child `ScanSpec` without a projection channel.
4. Applies each filter from `SubfieldFilters` to the corresponding `ScanSpec` node via `fieldSpec->setFilter(pair.second)`.

**Step 6: Build the MetadataFilter.** If a remaining filter exists, it is compiled into a `MetadataFilter` that can evaluate against row-group statistics:

```cpp
if (remainingFilter) {
    metadataFilter_ = make_shared<MetadataFilter>(
        *scanSpec_, *remainingFilter, expressionEvaluator_);
}
```

### 3.2 Phase 2: Split Processing

When a split is added:

```
TableScan::getSplit()
  -> dataSource_->addSplit(connectorSplit)
       -> HiveDataSource::addSplit()
            -> createSplitReader()
            -> splitReader_->configureReaderOptions(randomSkip)
            -> splitReader_->prepareSplit(metadataFilter, runtimeStats)
```

`SplitReader::prepareSplit()` does:
1. **Validates synthesized column filters** against the split's info column values.
2. **Creates the file reader** (`baseReader_`) -- opens the file, reads metadata.
3. **Adapts the ScanSpec** to the file schema (`getAdaptedRowType()`), handling schema evolution (column type mismatches, missing columns, extra columns).
4. **Checks if the split is empty** (`checkIfSplitIsEmpty()`) by testing filters against file-level and stripe-level statistics. If all statistics fail the filters, the entire split is skipped without reading any data.
5. **Creates the RowReader** (`baseRowReader_`) with the `ScanSpec` and `MetadataFilter` passed via `RowReaderOptions`:

```cpp
rowReaderOptions.setScanSpec(scanSpec);
rowReaderOptions.setMetadataFilter(std::move(metadataFilter));
rowReaderOptions.setRequestedType(rowType);
rowReaderOptions.range(hiveSplit->start, hiveSplit->length);
```

The RowReader constructs `SelectiveColumnReader` instances for each column in the ScanSpec. Each reader receives a reference to its `ScanSpec` node and uses the filter during decode. This is where the physical pushdown happens -- filters are tested at the DWIO decode layer, inline with data extraction.

### 3.3 Phase 3: Reading Data

```
TableScan::getOutput()
  -> dataSource_->next(readBatchSize, blockingFuture_)
       -> HiveDataSource::next()
            -> splitReader_->next(size, output_)
                 -> baseRowReader_->next(size, output, mutation)
                      [SelectiveColumnReader applies ScanSpec filters during decode]
            -> evaluateRemainingFilter(rowVector)  [if remaining filter exists]
            -> wrap output columns with post-processors
```

The `HiveDataSource::next()` method:
1. Calls `splitReader_->next()` which delegates to `baseRowReader_->next()`. The DWIO row reader applies `ScanSpec` filters at the decode level -- using `testInt64Range()` for row-group pruning, then `testInt64()` / `testValues()` for per-value filtering within selected row groups.
2. If a `remainingFilterExprSet_` exists, evaluates it against the output batch. This produces a `selectedIndices` buffer that masks out non-passing rows.
3. Wraps the output columns, applying any post-processors from the column handles.
4. Returns the filtered `RowVector` to the `TableScan` operator.

### 3.4 Phase 4: Dynamic Filter Injection

Dynamic filters arrive from hash join build sides during execution. The flow is:

```
HashBuild completes
  -> Task distributes dynamic filters to TableScan operators
       -> TableScan::addDynamicFilterLocked(producer, filters)
            -> dataSource_->addDynamicFilter(outputChannel, filter)
                 -> HiveDataSource::addDynamicFilter()
                      -> scanSpec_->getChildByChannel(channel).setFilter(filter)
                      -> scanSpec_->resetCachedValues(true)  [triggers reorder]
                      -> splitReader_->resetFilterCaches()
```

The `PushdownFilters` struct in `Driver.h` holds a shared filter state across all drivers for the same pipeline:

```cpp
struct PushdownFilters {
    F14FastMap<column_index_t, FilterPtr> filters;
    F14FastSet<column_index_t> dynamicFilteredColumns;
    bool staticFiltersInitialized = false;
};
```

When a new `DataSource` is created (for a new driver), the `createDataSource()` function in `TableScan.cpp` merges static SubfieldFilters with any already-received dynamic filters:

```cpp
auto* staticFilters = dataSource->getFilters();
// ...
for (column_index_t i = 0; i < outputType->size(); ++i) {
    auto field = Subfield::create(handle->second->name());
    if (auto it = staticFilters->find(*field); it != staticFilters->end()) {
        Filter::merge(it->second, lk->filters[i]);
    }
}
lk->staticFiltersInitialized = true;
// Apply any pending dynamic filters
for (auto outIndex : lk->dynamicFilteredColumns) {
    dataSource->addDynamicFilter(outIndex, lk->filters.at(outIndex));
}
```

Dynamic filters are merged with static filters via `Filter::mergeWith()`, producing a combined filter (e.g., a `BigintRange` ANDed with a `BloomFilter` produces a `BigintMultiRange` or the intersection of the two).

## 4. Concurrency & State Management

### 4.1 Thread Safety of PushdownFilters

The `PushdownFilters` struct is wrapped in `folly::Synchronized`. Multiple drivers for the same pipeline share the same `PushdownFilters` instance. The locking protocol is:
- Read lock (`rlock()`) during `dataSource_->next()` -- the scan runs while holding a read lock, preventing filter modifications from corrupting the reader state.
- Write lock (`wlock()`) during `addDynamicFilterLocked()` and initial static filter merging.
- Lock ordering: downstream-to-upstream (prevents deadlock in multi-stage pipelines).

### 4.2 DataSource is Single-Threaded

The `DataSource` interface is NOT thread-safe. Each driver has its own `DataSource` instance. The `ConnectorQueryCtx` documentation states: "One instance per DataSource. This may be passed between threads but methods must be invoked sequentially."

### 4.3 ScanSpec Mutability

The `ScanSpec` tree is mutable at runtime:
- `setFilter()` can be called to add dynamic filters.
- `resetCachedValues()` invalidates cached `hasFilter()` results and triggers filter reordering.
- `moveAdaptationFrom()` transfers filter order and dynamic filters between splits, preserving runtime adaptations.
- `selectivity()` tracks per-filter selectivity statistics. The `newRead()` method periodically reorders filters by selectivity (most selective first), improving performance over time.

### 4.4 Split Preloading

The `HiveConnector` supports split preloading (`supportsSplitPreload() == true`). When enabled, the `TableScan` operator schedules background `DataSource` creation on the connector's IO executor:

```cpp
void TableScan::preload(const shared_ptr<ConnectorSplit>& split) {
    split->dataSource = make_unique<AsyncSource<DataSource>>(...);
}
```

The preloaded `DataSource` is later consumed via `setFromDataSource()`, transferring file handles and metadata without re-opening. Dynamic filters and adaptive filter ordering are preserved through this transfer.

## 5. Memory & Resource Profile

### 5.1 Memory Pool Hierarchy

Each `ConnectorQueryCtx` carries two memory pools:
- `operatorPool_` -- leaf pool for direct allocations (buffers, vectors)
- `connectorPool_` -- aggregate pool for hierarchical management (used by DataSink for write buffering)

The `TableScan` operator creates the connector pool via `task->addConnectorPoolLocked()`, ensuring it is tracked under the task's memory accounting.

### 5.2 Filter Memory

Filters are `shared_ptr<Filter>` and are reference-counted. The `SubfieldFilters` map in `HiveDataSource` owns one copy; the `ScanSpec` tree holds another reference via `setFilter()`. Dynamic filters from `PushdownFilters` add a third reference path.

`BigintRange` filters are lightweight (a few int64 fields). `BigintValuesUsingHashTable` and `BigintValuesUsingBloomFilter` can be larger (hash tables / bloom filter bitmaps), but they are shared across the pipeline via `PushdownFilters`.

### 5.3 Lazy Loading

Velox uses lazy column loading extensively. The `ScanSpec` tree's `projectOut()` flag determines whether a column's values are materialized. Filter-only columns (`projectOut_ == false`) are read and filtered but their values are not materialized into the output. Projected columns may be lazily loaded (deferred until actually accessed), unless they are referenced in the remaining filter -- in which case `shouldEagerlyMaterialize()` determines whether early materialization is needed.

## 6. Key Design Insights

### 6.1 Single-Shot vs. Multi-Round Pushdown (Velox vs. Trino)

**Trino** uses a multi-round protocol:
- `ConnectorMetadata.applyFilter()` is called during planning. The connector returns a new `TableHandle` with pushed-down predicates and an `unenforced` predicate remainder. The planner may call this multiple times as it optimizes.
- `ConnectorMetadata.applyProjection()` similarly negotiates projections.
- The connector has agency -- it can accept or reject pushdowns, and the planner adapts.

**Velox** uses a single-shot construction:
- The `HiveTableHandle` arrives fully formed with all `SubfieldFilters` and the `remainingFilter` already decided by the upstream planner (e.g., Prestissimo's Presto coordinator).
- The `HiveDataSource` constructor processes everything in one pass: clones filters, extracts more from the remaining filter, builds the `ScanSpec` tree.
- There is no negotiation -- the connector always accepts all filters.

This is a deliberate design choice. Velox is an execution-only library, not a full query engine. The planning-time pushdown negotiation belongs to the coordinator (Presto/Spark), and Velox receives the final, already-negotiated pushdown plan.

### 6.2 SubfieldFilters as a Flat Map vs. Trino's TupleDomain

Trino represents pushdown predicates as a `TupleDomain<ColumnHandle>`, which maps columns to `Domain` objects (ranges and discrete value sets). Velox's `SubfieldFilters` (`unordered_map<Subfield, FilterPtr>`) is structurally similar but more expressive:

- **Nested pushdown:** Trino's `TupleDomain` is flat -- it maps top-level columns to domains. Velox's `Subfield` paths can address nested elements: `a.b[1].c` can have its own filter. This allows pushdown into deeply nested Parquet/ORC columns.
- **SIMD-native filters:** Velox's `Filter` classes are designed for SIMD evaluation (`testValues(batch)`). Trino's `Domain` predicates are evaluated in the Java domain without direct SIMD support.
- **Bloom filter pushdown:** Velox has a first-class `BigintValuesUsingBloomFilter` filter kind used for dynamic filter pushdown from hash joins. Trino recently added similar support but through a separate mechanism.

### 6.3 The ScanSpec Tree: Unified Filter + Projection Specification

The `ScanSpec` is Velox's most distinctive abstraction. It merges filter and projection specifications into a single tree that mirrors the column nesting structure:

```
ScanSpec("root")
  +-- ScanSpec("a")  [channel=0, projectOut=true, filter=BigintRange(10,100)]
  +-- ScanSpec("b")  [channel=1, projectOut=true]
  |     +-- ScanSpec("keys")    [filter=IsNotNull]
  |     +-- ScanSpec("values")  [projectOut=true]
  +-- ScanSpec("c")  [no channel, projectOut=false, filter=BytesValues("x","y")]
```

Each `ScanSpec` node can carry:
- A `filter_` -- applied at the decode level
- A `channel_` -- output position in the result `RowVector`
- A `projectOut_` flag -- whether to materialize values
- A `constantValue_` -- for partition keys or missing columns
- Children -- for nested columns (struct fields, map keys/values, array elements)

This design allows the `SelectiveColumnReader` to make integrated filter+projection decisions at each level of nesting. A filter on `a.b[1]` can skip materializing irrelevant array elements at the DWIO layer.

**Comparison to DataFusion:** DataFusion's `TableProvider::scan()` receives separate `projection` (column indices) and `filters` (Expr list) parameters. There is no unified tree structure -- projections are flat column indices, and filters are expression trees that the provider may or may not push down. The provider returns a `TableProviderFilterPushDown` enum to indicate whether each filter was handled. This is more flexible (the provider has agency) but less integrated with the physical reader.

### 6.4 Dynamic Filter Merging is Lazy and Lock-Aware

The dynamic filter system demonstrates careful concurrency design:
- Static filters are initialized once (from the first driver to create a `DataSource`), not per-driver.
- Dynamic filters are merged into `ScanSpec` via `setFilter()`, which simply replaces the filter pointer. The old filter is not explicitly freed -- it is dropped when the last `shared_ptr` reference goes away.
- `resetCachedValues(true)` forces filter reordering, so newly added dynamic filters are positioned based on their estimated selectivity relative to existing filters.

### 6.5 The Remaining Filter as a Safety Net

The three-tier filter architecture (SubfieldFilters -> MetadataFilter -> Remaining Filter) ensures correctness while maximizing pushdown:

1. `extractFiltersFromRemainingFilter()` aggressively tries to convert expression conjuncts into `SubfieldFilters`. It handles negation (De Morgan), OR-to-MultiRange conversion, and random sampling extraction.
2. Whatever cannot be converted stays as the `remainingFilter`, compiled into an `ExprSet` and evaluated row-by-row after the reader produces output.
3. The `MetadataFilter` bridges the two: it takes the remaining filter's leaf predicates and maps them to row-group statistics tests, enabling stripe-level pruning even for expressions that cannot be pushed into per-value filters.

This is more aggressive than Trino's approach, where the coordinator decides what to push down during planning. Velox's `HiveDataSource` constructor does a second pass of filter extraction at execution time, potentially pushing more predicates than the coordinator requested.

### 6.6 Filter Adaptation and Reordering

The `ScanSpec::newRead()` method implements runtime filter reordering:
- Each filter's `selectivity()` is tracked across batches.
- Periodically, children of a struct `ScanSpec` are reordered so that the most selective (highest rejection rate) filters come first.
- This is critical for performance: if column `c` rejects 99% of rows, reading it before column `a` (which rejects 10%) avoids materializing 99% of `a`'s values.

This adaptation persists across splits via `moveAdaptationFrom()`, so the optimal filter order discovered from the first split carries over to subsequent splits.
